### AI Testing / Software Testing（10 项）

1. **Measuring and Exploiting Confirmation Bias in LLM-Assisted Security Code Review** (arXiv:2603.18740)
   - **来源：** arXiv (cs.SE/cs.AI/cs.CR)
   - **领域：** AI Testing / Security / Code Review
   - **技术介绍：** 研究 LLM 辅助安全代码审查中的确认偏误问题及其可利用性。研究 1 通过 250 个 CVE 漏洞/补丁对在 4 个 SOTA 模型上进行控制实验，发现将变更框定为"无 bug"可使漏洞检测率降低 16-93%，注入漏洞比内存破坏漏洞更易受影响。研究 2 模拟对抗性 PR，在 GitHub Copilot 上一击成功率 35%，在 Claude Code 自主代理上迭代攻击成功率 88%。通过元数据脱敏和明确指令可恢复所有交互式场景和 94% 自主场景的检测能力。
   - **应用场景：** AI 辅助代码审查、CI/CD 安全集成、供应链攻击防御
   - **为什么值得关注：** 首次系统量化 LLM 代码审查的确认偏误漏洞，揭示供应链攻击新向量

2. **Who Tests the Testers? Systematic Enumeration and Coverage Audit of LLM Agent Tool Call Safety** (arXiv:2603.18245)
   - **来源：** arXiv (cs.SE/cs.CR)
   - **领域：** AI Testing / Agent Safety / Meta-Audit
   - **技术介绍：** 提出 SafeAudit 元审计框架，解决现有代理安全基准测试完整性问题。核心贡献：1) 基于 LLM 的枚举器系统生成测试用例，枚举有效工具调用工作流和多样用户场景；2) 提出 rule-resistance 指标，从现有基准提炼紧凑安全规则并识别未覆盖的不安全交互模式。在 3 个基准 12 个环境上，SafeAudit 发现 20%+ 现有基准未暴露的残留不安全行为，覆盖率随测试预算增加单调增长。
   - **应用场景：** 代理安全测试、基准评估、安全合规审计
   - **为什么值得关注：** 首次提出代理安全测试的元审计方法，揭示现有基准重大覆盖缺口

3. **Can LLMs Reason Like Automated Theorem Provers for Rust Verification? VCoT-Bench: Evaluating via Verification Chain of Thought** (arXiv:2603.18334)
   - **来源：** arXiv (cs.SE/cs.AI/cs.LG)
   - **领域：** Software Testing / Formal Verification / Rust
   - **技术介绍：** 评估 LLM 在 Rust 程序验证中的推理能力。提出 VCoT-Lift 框架，将底层求解器推理提升为高层人类可读的验证步骤，通过显式验证思维链提供细粒度评估的真实依据。基于 VCoT-Lift 构建 VCoT-Bench 基准，包含 1,988 个 VCoT 完成任务，从三个正交维度评估：对不同缺失证明程度的鲁棒性、不同证明类型的能力、对证明位置的敏感性。10 个 SOTA 模型评估显示严重脆弱性，当前 LLM 远未达到自动定理证明器的推理能力。
   - **应用场景：** Rust 安全开发、形式化验证、LLM 推理评估
   - **为什么值得关注：** 首个 Rust 验证推理链基准，揭示 LLM 形式化推理能力严重不足

### GitHub 项目（3 项）

1. **langchain-ai/open-swe** - 开源异步编码代理框架
   - **链接：** https://github.com/langchain-ai/open-swe
   - **深度分析：** Open SWE 代表了精英工程组织（Stripe、Ramp、Coinbase）内部编码代理架构的开源化。核心设计决策：1) **沙箱优先**：每个任务独立云沙箱，隔离后赋予完全权限，限制错误爆炸半径；2) **触发面集成**：Slack/Linear原生集成，工程师在现有工作流中调用；3) **子代理编排**：基于 Deep Agents 的 task 工具原生支持子代理派生；4) **中间件安全网**：确定性中间件确保关键步骤（如 PR 创建）无论 LLM 行为如何都会发生；5) **AGENTS.md 注入**：仓库级规则文件编码约定、测试要求、架构决策。这反映了内部代理开发的核心洞察：工具精选比工具数量重要、隔离是安全前提、确定性钩子补充 LLM 不可靠性。7,352 星今日 965 星增长显示社区强烈需求。

2. **jarrodwatts/claude-hud** - Claude Code 可观测性插件
   - **链接：** https://github.com/jarrodwatts/claude-hud
   - **深度分析：** Claude HUD 解决了 AI 辅助开发中的关键痛点：上下文盲盒。传统终端中开发者无法直观看到：1) 上下文窗口使用率（直到截断才发现问题）；2) 工具活动（Claude 在读/编辑哪些文件）；3) 子代理状态（后台在做什么）；4) TODO 进度。HUD 通过 Claude Code 原生 statusline API 实现：解析 transcript JSONL 获取工具/代理活动，直接读取 token 数据（非估算），约 300ms 更新频率。支持 1M 上下文会话自动缩放，提供 Full/Essential/Minimal 预设和细粒度自定义。9,033 星今日 1,851 星增长登顶 Trending #1，反映 AI 编码可观测性是普遍需求。这是 AI 工具链成熟度的标志：从"能用"到"好用"需要可观测性。

3. **opendataloader-project/opendataloader-pdf** - AI 就绪 PDF 解析器
   - **链接：** https://github.com/opendataloader-project/opendataloader-pdf
   - **深度分析：** PDF 解析是 RAG 和企业知识库的关键瓶颈。传统 PDF 解析问题：1) 布局信息丢失（表格、多栏、图文混排）；2) 公式/图表处理差；3) 扫描版 PDF OCR 质量不稳定；4) 元数据提取不完整。opendataloader-pdf 定位"AI 就绪"：输出格式直接适配向量数据库和 LLM 上下文需求。Java 实现适合企业集成。6,508 星今日 1,416 星增长显示市场需求强烈。与 LlamaParse、Unstructured.io 等商业方案相比，开源替代对企业成本敏感场景有吸引力。

1. **Confirmation Bias in LLM-Assisted Security Code Review** (arXiv:2603.18740)
   - **链接：** https://arxiv.org/abs/2603.18740
   - **深度分析：** 这篇论文揭示了 AI 辅助开发的一个隐蔽但严重的安全漏洞。确认偏误是人类认知偏差，但论文证明 LLM 同样受影响且可被利用。研究设计精巧：研究 1 控制实验量化偏误程度（16-93% 检测率下降），研究 2 实战模拟供应链攻击（Copilot 35% 一击成功，Claude Code 88% 迭代成功）。关键洞察：1) ** framing 效应**：PR 描述 framing 为"安全改进"或"紧急修复"可显著降低警惕性；2) **模型差异**：自主代理比交互式助手更易受攻击（88% vs 35%），因为迭代优化 framing；3) **缓解有效**：元数据脱敏 + 明确指令可恢复 94-100% 检测能力。实际影响：1) CI/CD 集成 AI 代码审查需默认启用 debiasing；2) 代码托管平台应限制 PR 描述对审查模型的影响；3) 安全团队需审计 AI 审查日志寻找 framing 攻击痕迹。这是 AI 安全从"模型安全"扩展到"系统安全"的典型案例。

2. **SafeAudit: Meta-Audit for Agent Tool Call Safety** (arXiv:2603.18245)
   - **链接：** https://arxiv.org/abs/2603.18245
   - **深度分析：** 论文提出了一个元问题：如何测试安全测试本身？当前代理安全基准（AgentBench、ToolBench 等）存在覆盖缺口，但无法量化。SafeAudit 的两个核心贡献：1) **LLM 枚举器**：系统生成测试用例，枚举有效工具调用工作流和多样用户场景，解决人工标注成本高问题；2) **rule-resistance 指标**：从现有基准提炼紧凑安全规则，定量识别未覆盖的不安全交互模式。关键结果：3 基准 12 环境发现 20%+ 残留不安全行为，覆盖率随测试预算单调增长。这意味着：1) 现有基准通过≠真正安全；2) 元审计应成为代理部署前必需步骤；3) rule-resistance 可作为安全成熟度指标。这是代理安全评估方法论的重要进步，从"跑基准"到"审计基准"。

3. **VCoT-Bench: Rust Verification Reasoning Evaluation** (arXiv:2603.18334)
   - **链接：** https://arxiv.org/abs/2603.18334
   - **深度分析：** 论文回答了关键问题：LLM 能否像自动定理证明器那样推理？答案是否定的，且差距显著。VCoT-Lift 框架的创新在于将求解器级推理"提升"为人类可读的验证步骤，提供细粒度评估的真实依据。VCoT-Bench 包含 1,988 个任务，从三个维度评估：1) **鲁棒性**：对缺失证明的容忍度；2) **能力**：不同证明类型的表现；3) **敏感性**：证明位置变化的响应。10 个 SOTA 模型显示"严重脆弱性"，意味着：1) LLM 形式化推理仍是早期阶段；2) Rust 验证等安全关键场景不能依赖 LLM 独立证明；3) 需要混合方法（LLM 生成提示 + 求解器验证）。这对 AI 辅助安全开发有实际指导意义：LLM 可作为辅助工具，但不能替代形式化验证。
