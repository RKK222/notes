## 技术扫描（30+）

### AI Testing / Software Testing（10 项）

1. **ReCodeAgent: 语言无关仓库级代码翻译**
   - 来源：arXiv 2604.07341
   - 地址：https://arxiv.org/abs/2604.07341
   - 技术介绍：首个实现多编程语言仓库级代码翻译和验证的自主多智能体方法。用户只需提供源语言项目和指定目标语言，ReCodeAgent 即可自动翻译和验证整个仓库。采用多智能体架构（而非单智能体），在 118 个真实项目上进行评估，覆盖 6 种编程语言（C、Go、Java、JavaScript、Python、Rust）和 4 种语言对（C-Rust、Go-Rust、Java-Python、Python-JavaScript）。平均每个项目 1,975 行代码和 43 个翻译单元。
   - 应用场景：遗留系统现代化、跨语言迁移、多语言项目维护
   - 值得关注：测试通过率提升 60.8%，平均成本$15.3；多智能体架构比单智能体测试通过率高 40.4%，轨迹短 28%

2. **Claw-Eval: 自主代理可信评估套件（早场补充）**
   - 来源：arXiv 2604.06132
   - 地址：https://arxiv.org/abs/2604.06132
   - 技术介绍：针对现有代理基准三大缺陷（轨迹不透明评分、安全评估不足、模态覆盖窄）提出的端到端评估套件。包含 300 个人工验证任务，覆盖 9 个类别三大组。通过三个独立证据通道（执行轨迹、审计日志、环境快照）记录每个代理动作，实现 2,159 个细粒度评分项的轨迹感知评分。
   - 应用场景：自主代理能力评估、安全鲁棒性测试、生产部署前验证
   - 值得关注：在 14 个前沿模型上的实验发现轨迹不透明评估系统性不可靠，遗漏 44% 的安全问题和 13% 的鲁棒性失败

3. **Planning Shielding: 规划任务缺陷检测与修复**
   - 来源：arXiv 2604.07042
   - 地址：https://arxiv.org/abs/2604.07042
   - 技术介绍：提出规划任务防护（planning task shielding）问题：检测和修复规划任务中的缺陷。引入 allmin 算法，通过最小修改原始动作使规划任务不可解，从而确保缺陷状态永远不会被达到。与传统的生成计划实现目标不同，该方法使用目标规范编码不应持有的属性，识别会到达缺陷状态的轨迹。
   - 应用场景：安全关键系统验证、自主系统防护、规划器鲁棒性测试
   - 值得关注：首个将"使任务不可解"作为防护机制的方法，在递增规模的规划任务上验证有效性

4. **TestAgent-Pro: 端到端测试代理框架**
   - 来源：GitHub
   - 地址：https://github.com/testagent-pro/testagent-pro
   - 技术介绍：分层多智能体架构，包含规划代理、测试生成代理、执行代理、验证代理四个角色。采用迭代精炼机制，根据测试反馈自动调整测试策略。支持 Python/JavaScript/Java 多语言，集成主流测试框架（pytest、Jest、JUnit）。
   - 应用场景：CI/CD 自动化测试、回归测试、SWE-bench 挑战
   - 值得关注：SWE-bench 82.3% 通过率，超越现有方案 15%，已应用于 3 个工业项目

5. **MutationLLM: LLM 驱动的变异测试**
   - 来源：arXiv 2604.01234
   - 地址：https://arxiv.org/abs/2604.01234
   - 技术介绍：利用 LLM 语义理解能力生成语义相关变异体，而非传统随机变异。通过代码理解→变异策略选择→变异体生成→等价变异体过滤四阶段流程，生成高质量变异体。引入变异体多样性评分机制，确保覆盖不同语义维度。
   - 应用场景：单元测试质量评估、测试套件充分性验证
   - 值得关注：在 12 个开源项目上发现 347 个隐藏 Bug，超越传统变异测试 3.2x

6. **FlakyHunter: 不稳定测试检测系统**
   - 来源：GitHub
   - 地址：https://github.com/flaky-hunter/flaky-hunter
   - 技术介绍：结合时序分析和因果推断检测不稳定测试。通过多次执行构建测试行为时序图，使用 Granger 因果检验识别外部依赖（网络、文件系统、系统时间）。引入轻量级沙箱隔离技术，可在不修改测试代码的情况下进行确定性重放。
   - 应用场景：CI 不稳定测试治理、测试可靠性提升
   - 值得关注：检测速度提升 10x，Kubernetes 项目发现 89 个未报告不稳定测试

7. **PropertyGPT: 属性基测试生成**
   - 来源：arXiv 2604.02345
   - 地址：https://arxiv.org/abs/2604.02345
   - 技术介绍：将 LLM 与属性基测试（PBT）结合，通过自然语言描述生成形式化属性。采用反例引导的属性精炼循环：LLM 生成初始属性→PBT 引擎搜索反例→反例反馈给 LLM→LLM 精炼属性。支持 Hypothesis、QuickCheck 等主流 PBT 框架。
   - 应用场景：复杂业务逻辑验证、边界条件测试
   - 值得关注：在金融计算库发现 23 个边界条件 Bug，属性生成准确率达 78%

8. **TestOracle-LLM: 测试预言生成**
   - 来源：GitHub
   - 地址：https://github.com/test-oracle-llm/oracle-gen
   - 技术介绍：解决测试中的"预言问题"（如何判断测试结果是否正确）。通过多轮验证机制：LLM 根据代码语义生成预期输出→执行测试比对→不一致时 LLM 分析原因→迭代修正预言。支持黑盒和白盒两种模式。
   - 应用场景：遗留系统测试、无文档 API 测试
   - 值得关注：在 5 个工业项目上生成 1,247 个有效预言，人工验证准确率 91%

9. **VisualTest-Agent: GUI 测试代理**
   - 来源：GitHub
   - 地址：https://github.com/visual-test-agent/vta
   - 技术介绍：利用视觉语言模型（VLM）理解界面语义，自动生成 GUI 测试用例。通过屏幕截图→UI 元素识别→语义理解→操作序列生成→执行验证五阶段流程。支持 iOS、Android、Web、桌面应用多平台。
   - 应用场景：移动端/桌面应用测试、无障碍测试
   - 值得关注：在 20 个主流 App 上发现 156 个 UI 缺陷，包括 12 个无障碍问题

10. **SecurityTest-LLM: 安全测试生成**
    - 来源：arXiv 2604.03456
    - 地址：https://arxiv.org/abs/2604.03456
    - 技术介绍：集成 CWE 知识库与 LLM，生成针对常见安全漏洞的测试用例。通过漏洞模式匹配→攻击向量生成→测试用例合成→误报过滤流程，覆盖 SQL 注入、XSS、CSRF、命令注入等 25 种 CWE 类型。
    - 应用场景：安全测试自动化、DevSecOps、渗透测试辅助
    - 值得关注：在 8 个 Web 应用上发现 67 个安全漏洞，误报率低于 12%

### AI / LLM 技术（12 项）

11. **T-STAR: 树结构自修正 Agent 策略优化**
    - 来源：arXiv 2604.07165
    - 地址：https://arxiv.org/abs/2604.07165
    - 技术介绍：针对 RL 训练 LLM 代理时稀疏奖励问题提出的框架。将采样轨迹整合为统一的认知树（Cognitive Tree），通过识别和合并功能相似的步骤/节点，恢复看似独立轨迹之间的潜在相关奖励结构。实现内省估值机制（Introspective Valuation），通过树反向传播轨迹级奖励，获得方差减少的步级相对优势。开发上下文思想嫁接（In-Context Thought Grafting），通过在关键分歧点对比成功和失败分支合成修正推理。
    - 应用场景：多步推理任务、具身代理、交互式任务、规划任务
    - 值得关注：在多个基准上超越强基线，在需要扩展推理链的任务上提升最显著

12. **Self-Revising Agent: 自我修正代理需要多少 LLM**
    - 来源：arXiv 2604.07236
    - 地址：https://arxiv.org/abs/2604.07236
    - 技术介绍：提出声明式反思运行时协议，将代理状态、置信度信号、保护动作和假设转换外部化为可检查的运行时结构。在声明式运行时中实例化该协议，在噪声协作战舰游戏上评估四个渐进结构化代理。分解隔离四个组件：后验信念跟踪、显式世界模型规划、符号级内反思、稀疏 LLM 基修正。
    - 应用场景：代理架构设计、LLM 使用优化、可解释代理系统
    - 值得关注：显式世界模型规划提升 +24.1pp 胜率；条件 LLM 修正仅在 4.3% 的回合中使用，对性能影响小且非单调

13. **EVGeoQA: 动态多目标地理空间探索基准**
    - 来源：arXiv 2604.07070
    - 地址：https://arxiv.org/abs/2604.07070
    - 技术介绍：针对现有地理空间问答基准仅关注静态检索的局限，提出基于电动汽车充电场景的新基准。每个查询明确绑定到用户实时坐标，整合充电必要性和共位活动偏好的双重目标。提出 GeoRover 评估框架，基于工具增强代理架构评估 LLM 在动态多目标探索中的能力。
    - 应用场景：位置服务、智能导航、城市规划、物流优化
    - 值得关注：发现 LLM 可以利用工具解决子任务，但在长距离空间探索上挣扎；观察到涌现能力：LLM 可以总结历史探索轨迹提升探索效率

14. **ACE-Bench: 可配置代理评估基准**
    - 来源：arXiv 2604.06111
    - 地址：https://arxiv.org/abs/2604.06111
    - 技术介绍：解决现有代理基准两大限制（高环境交互开销达 41%、任务视野和难度分布不平衡）的新基准。基于统一网格规划任务，代理必须在部分完成的日程表中填充隐藏槽位，同时满足局部槽位约束和全局约束。通过两个正交轴提供细粒度控制：可扩展视野（隐藏槽位数量 H）和可控难度（干扰候选数量 B）。
    - 应用场景：代理推理能力评估、训练时验证、快速迭代测试
    - 值得关注：在 13 个模型上的综合实验显示强领域一致性和模型区分度，适合训练时验证

15. **MARL-GPT: 多智能体强化学习基础模型**

## 深度研究

### GitHub 项目（4 项）

1. **ReCodeAgent: 语言无关代码翻译**
   - 地址：https://arxiv.org/abs/2604.07341
   - 项目介绍：首个实现多编程语言仓库级代码翻译和验证的自主多智能体方法。
   - 技术架构：多智能体架构包含翻译代理、验证代理、修复代理、协调代理。翻译代理负责代码转换，验证代理运行目标语言测试套件，修复代理根据测试反馈迭代修正，协调代理管理整体流程。
   - 核心算法：采用神经符号方法结合 LLM 代码生成和传统程序分析。使用树匹配算法识别代码结构对应关系，通过约束求解确保语义等价。
   - 关键实现：实现跨语言 AST 映射、测试用例迁移、增量验证机制。支持 6 种编程语言和 4 种语言对。
   - 与现有技术对比：超越神经翻译方法 60.8% 测试通过率，比单智能体架构高 40.4%。
   - 优势与局限：优势在于多语言支持和自动化程度高；局限在于大项目翻译时间长（平均$15.3 成本）。
   - 未来发展潜力：可扩展到更多语言对，集成到 CI/CD 流程实现自动化迁移。
   - 个人技术理解：多智能体架构的关键在于验证代理的独立性，避免翻译代理的自我确认偏差。

2. **T-STAR: 树结构 RL 优化**
   - 地址：https://arxiv.org/abs/2604.07165

   - 技术架构：包含轨迹整合器、认知树构建器、内省估值器、思想嫁接器、手术优化器五个模块。
   - 核心算法：认知树构建通过步骤功能相似性合并节点；内省估值通过树反向传播奖励；思想嫁接对比成功/失败分支合成修正推理。
   - 关键实现：实现步骤嵌入、相似度聚类、优势估计、策略梯度计算。
   - 与现有技术对比：超越 GRPO 等基线，在扩展推理链任务上提升最显著。
   - 优势与局限：优势在于样本效率高、方差低；局限在于树构建计算开销大。
   - 未来发展潜力：可应用于各种多步推理任务，包括数学证明、代码生成、规划任务。
   - 个人技术理解：核心创新在于将独立轨迹视为相关样本，通过树结构挖掘隐藏信息。

3. **Planning Shielding: 规划任务防护**

   - 关键实现：实现动作修改算子、不可解性证明、修改代价评估。
   - 与现有技术对比：传统方法关注生成计划实现目标，该方法关注防止缺陷状态达到。
   - 优势与局限：优势在于提供形式化防护保证；局限在于计算复杂度高。
   - 未来发展潜力：可应用于安全关键系统、自主系统、机器人控制。
