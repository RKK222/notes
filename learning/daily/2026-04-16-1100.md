## 技术扫描（35 项）

### AI Testing / Software Testing（12 项）

1. **TestAgent-Eval: 测试代理标准化评估框架**
   - URL: https://arxiv.org/abs/2604.14001
   - 来源：arXiv
   - 介绍：首个测试代理标准化评估框架，定义 5 维度评估指标：测试覆盖率、缺陷发现率、误报率、执行效率、可解释性。包含 500+ 测试任务基准，覆盖单元测试、集成测试、系统测试场景。引入 TestAgent-Score 综合评分，防止单一指标刷分。评估 15 个主流测试代理，发现覆盖率与缺陷发现率相关性仅 0.43，揭示高覆盖率≠高质量测试。
   - 应用：测试代理评估、测试质量度量
   - 价值：测试代理评估标准化，避免指标操纵

2. **Memorization Advantage in Code LLMs (ICSE 2026)**
   - URL: https://arxiv.org/abs/2604.13997
   - 来源：arXiv (ICSE 2026 Accepted)
   - 介绍：基于扰动的代码 LLM 记忆化优势量化方法，定义"记忆化优势"为可能见过与未见输入的性能差距。评估 8 个开源代码 LLM、19 个基准、4 类任务（代码生成、理解、漏洞检测、Bug 修复）。发现敏感性模式因模型和任务差异巨大：StarCoder 某些基准敏感性达 0.8，QwenCoder 大多低于 0.4。CVEFixes 和 Defects4J 记忆化优势低于 0.1，表明模型更多依赖学习泛化而非直接记忆。
   - 应用：代码 LLM 评估、数据泄漏检测
   - 价值：首个记忆化优势量化框架，揭示评估协议改进需求

3. **Commit Signing on GitHub 实证研究**
   - URL: https://arxiv.org/abs/2604.14014
   - 来源：arXiv
   - 介绍：平台级 Commit 签名实证研究，追踪 71,694 活跃 GitHub 用户、1600 万 + commits、874,198 仓库。发现生态表观高签名率是幻觉：排除平台生成签名后，仅<6% 开发者曾自行签名 commit，大多数"签名者"仅在网页浏览器中签名。本地签名者中，签名很少随时间或跨仓库持续，约 1/8 开发者管理签名因密钥未上传而验证失败。过期密钥几乎不被撤销，1/4+ 用户携带至少一个失效密钥。
   - 应用：软件供应链安全、代码来源验证
   - 价值：揭示 Commit 签名生态级 gap，提出具体改进建议

4. **AISafetyBenchExplorer v2: 基准治理工具**
   - URL: https://arxiv.org/abs/2604.14002

   - 介绍：基准治理工具 v2 版本，新增基准质量评分（BQS）和基准选择推荐系统。BQS 考虑仓库活跃度、文档完整性、社区采用度、复现性 4 维度。推荐系统基于任务类型、评估目标、资源约束智能推荐基准组合。新增 23 个基准，总数达 218 个。
   - 应用：基准选择、元评估
   - 价值：基准治理实用化，降低基准选择成本

5. **MutationLLM v3: 跨语言变异测试**

2. **Archon v2: AI 编程 Harness**

### 学术论文（4 项）

1. **Memorization Advantage in Code LLMs (ICSE 2026)**
   - URL: https://arxiv.org/abs/2604.13997
   - 介绍：代码 LLM 记忆化问题首个系统量化研究。提出"记忆化优势"概念：模型在可能见过输入 vs 未见输入上的性能差距。方法是对基准进行语义保持扰动（变量重命名、代码重构、注释增删），比较扰动前后性能变化。评估 8 模型×19 基准×4 任务，发现敏感性模式高度依赖模型和任务：StarCoder 某些基准敏感性达 0.8（高度记忆），QwenCoder 大多低于 0.4（更好泛化）。任务维度：代码总结敏感性低（模式固定），测试生成敏感性高（创意性强）。关键发现：CVEFixes 和 Defects4J 记忆化优势<0.1，反驳了常见泄漏担忧，表明模型更多依赖学习泛化。
   - 技术贡献：扰动基记忆化量化方法
   - 核心价值：代码 LLM 评估协议改进依据，安全场景需更强协议
   - 未来方向：开发记忆化鲁棒基准，改进训练数据去重策略

2. **Commit Signing on GitHub 实证研究**
   - URL: https://arxiv.org/abs/2604.14014
   - 介绍：软件供应链安全首个平台级实证研究。追踪 71,694 活跃用户×1600 万 commits×874K 仓库，揭示 Commit 签名生态级 gap。关键发现：(1) 排除 GitHub 自动签名后，仅<6% 开发者曾自行签名；(2) 大多数"签名者"仅在网页 UI 签名，本地 Git 未配置；(3) 本地签名者中，签名很少跨时间/仓库持续；(4) 1/8 开发者签名因密钥未上传而验证失败；(5) 过期密钥几乎不被撤销；(6) 1/4+ 用户携带失效密钥。结论：Commit 签名作为来源信号在生态规模上不可靠。建议：强制密钥上传、自动过期检测、签名持续性激励。
   - 技术贡献：平台级供应链安全实证方法
   - 核心价值：揭示供应链安全实践 gap，推动改进
   - 未来方向：改进签名 UX，自动化密钥管理

3. **CollabCoder: Plan-Code Co-Evolution (ACL 2026)**
   - URL: https://arxiv.org/abs/2604.13946
   - 介绍：代码生成效率新突破。传统多智能体框架问题：静态规划、孤立执行、高开销、低适应性。CollabCoder 核心创新：计划模块与代码模块协作决策，动态决定调试过程执行哪个模块。关键机制：(1) 计划 - 代码双向反馈，计划指导代码，代码执行结果反哺计划优化；(2) 协作决策器基于任务复杂度、历史成功率、资源约束选择执行模块；(3) 早期终止机制，检测到收敛即停止。LiveCodeBench/xCodeEval 性能提升 11-20%，API 调用减少 4-10 次/执行，效率增益随难度增加更显著。
   - 技术贡献：计划 - 代码协同进化框架
   - 核心价值：代码生成效率与质量双提升
   - 未来方向：扩展到多语言、多框架场景

4. **Log-based vs Graph-based Fault Diagnosis**
   - URL: https://arxiv.org/abs/2604.14019
   - 介绍：AIOps 故障诊断方法对比研究。传统日志分析将日志视为线性序列，丢失结构上下文（父子依赖、分支、时序）。图方法用 GNN 捕获这些关系。研究对比：(1) 日志编码器（BERT）；(2) 图模型（GNN）；(3) 混合模型（日志编码器表示 + 图结构）。在 TraceBench（trace 导向）和 BGL（传统系统日志）上评估异常检测和故障分类。发现：纯图模型未超越编码器基线，但混合模型 achieves 最强性能。启示：图结构有价值，但需与语义表示结合。
   - 技术贡献：日志故障诊断对比研究
   - 核心价值：图增强架构条件优势，指导 AIOps 系统设计
   - 未来方向：探索更高效的图 - 序列融合架构
