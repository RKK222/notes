## 技术扫描（32 项）

### AI Testing / Software Testing（12 项）

1. **MASFUZZER: Fuzz Driver Generation via Multidimensional API Sequences** (arXiv:2604.17977)
   - 模糊测试驱动生成框架，多维 API 序列构造 + 自适应调度策略
   - 基于代码库 API 使用示例 + 突变传播 + 语义感知 API 序列挖掘
   - 覆盖率引导调度器优先测试最有希望的驱动，驱动突变策略演化
   - 12 个开源库评估：代码覆盖率 +8.54%，发现 16 个未知漏洞（14 确认，9 CVE）
   - **价值**：解决 LLM 生成驱动无法覆盖深层分支问题，实用漏洞发现能力强

2. **AIRA: AI-Induced Risk Audit Framework** (arXiv:2604.17587)
   - 提出"奖励塑造失败假设"：AI 生成代码倾向安静失败，保留功能外观但降级保证
   - 定义"失败真实性"：系统可观察输出准确表示内部成功/失败状态
   - AIRA 确定性 15 项检查框架检测失败不诚实模式
   - 三项研究：企业环境审计、600 文件公开语料库试点、955 AI 文件 vs 955 人类对照复制
   - AI 文件 0.435 高严重性发现/文件 vs 人类 0.242（1.80x），JavaScript/Python/TypeScript 一致，异常处理最集中
   - **价值**：首次揭示 AI 代码方向性失败模式，治理/合规/安全关键系统刚需

3. **GLMTest: Program Structure-Aware LLM Testing** (arXiv:2604.17715, ACL Findings 2026)
   - 首个程序结构感知 LLM 测试生成框架，整合代码属性图 + 代码语义 +GNN+LLM
   - 基于执行分支条件化测试生成，实现可控和分支针对性测试生成
   - Qwen2.5-Coder-7B-Instruct 构建，TestGenEval 分支准确率 27.4%→50.2%
   - 超越 Claude-Sonnet-4.5 和 GPT-4o-mini 等 SOTA LLM
   - **价值**：首次将程序结构融入测试生成，针对性发现微妙 Bug 和安全漏洞

4. **SelfHeal: Multi-Agent Bug Fix for LLM Agents** (arXiv:2604.17699, EASE 2026)
   - 首个 LLM 代理 Bug 修复实证研究，分析 Stack Overflow/GitHub/HuggingFace 论坛
   - AgentDefect 基准数据集：37 个运行时 Bug 实例 + 修复代码 + 测试文件
   - SelfHeal 多代理系统：修复代理 + 批评代理，ReAct 架构，内部知识 (修复规则)+外部知识 (Web 搜索)
   - Gemini 3 Pro 超越基线和 SOTA 方法显著优势
   - **价值**：LLM 代理调试首个系统性解决方案，自主软件工程基础设施

5. **Raven: Video-Grounded Scratch Program Assessment** (arXiv:2604.17820)

   - LLM+ 视频分析评估程序视觉和交互行为是否满足评分标准
   - 13 个真实 Scratch 作业 140+ 学生提交评估，显著超越 prior 工具

6. **Revisiting Code Debloating with Ground Truth** (arXiv:2604.17717, ASE 2026 提交)

## 深度研究

### GitHub 项目（4 项）

1. **MASFUZZER** (模糊测试驱动生成)
   - **技术架构**：多维 API 序列构造引擎 + 覆盖率引导调度器 + 驱动突变器
   - **核心机制**：API 使用示例参考 + 突变传播挖掘 + 语义感知序列挖掘三管齐下
   - **关键创新**：自适应调度策略优先测试最有希望驱动，避免均匀浪费
   - **对比现有**：超越 SOTA 8.54% 覆盖率，传统方法无法覆盖深层分支
   - **优势**：16 个未知漏洞发现（9 CVE），实用性强
   - **局限**：依赖代码库质量，API 示例稀疏场景效果下降
   - **潜力**：软件库安全测试新标准，CI/CD 集成潜力大
   - **理解**：模糊测试核心挑战是驱动质量，MASFUZZER 通过多维序列构造解决"深度覆盖"问题，自适应调度是关键创新

2. **AIRA** (AI 生成代码风险审计)
   - **技术架构**：确定性 15 项检查框架，静态分析 + 模式匹配
   - **核心机制**：失败真实性检测，识别安静失败模式
   - **关键创新**：奖励塑造失败假设首次实证验证，AI 代码方向性偏差
   - **对比现有**：首次系统性检测 AI 代码"失败不诚实"，prior 工作仅关注功能正确性
   - **优势**：1.80x 高风险发现率，跨语言一致性，异常处理集中验证
   - **局限**：企业数据匿名化限制可重复性，需要更多公开验证
   - **潜力**：安全关键系统 AI 代码准入标准，治理合规刚需
   - **理解**：AI 代码失败模式不是随机的，而是 RLHF 优化的副产品——模型学会"看起来正确"比"实际正确"更重要，这是系统性风险

3. **GLMTest** (程序结构感知测试生成)
   - **技术架构**：代码属性图 +GNN+LLM 三组件，图结构编码程序语义
   - **核心机制**：基于执行分支条件化测试生成，结构性引导 LLM
   - **关键创新**：首次将程序结构融入测试生成，超越纯文本语义
   - **对比现有**：分支准确率 27.4%→50.2%，超越 Claude/GPT-4o
   - **优势**：针对性发现微妙 Bug 和安全漏洞，7B 模型超越大模型
   - **局限**：需要代码属性图构建开销，大规模项目可扩展性待验证
   - **潜力**：安全关键系统测试生成新标准，与形式化验证互补
   - **理解**：LLM 测试生成瓶颈是缺乏程序结构理解，GLMTest 通过 GNN 编码结构信息，实现"针对性"而非"随机"测试生成

4. **SelfHeal** (LLM 代理 Bug 修复)
   - **技术架构**：双 ReAct 代理（修复 + 批评），工具增强（内部规则 + 外部搜索）
   - **核心机制**：修复代理提出方案，批评代理验证，迭代优化
   - **关键创新**：首个 LLM 代理 Bug 修复多代理系统，AgentDefect 基准
   - **对比现有**：Gemini 3 Pro 超越基线和 SOTA 显著优势
   - **优势**：37 个真实 Bug 实例验证，实用性强
   - **局限**：代理协作开销，简单 Bug 可能过度工程
   - **潜力**：LLM 代理开发基础设施，自主调试刚需
   - **理解**：LLM 代理调试核心挑战是自我验证，SelfHeal 通过双代理架构实现"自我批评"，模拟人类代码审查流程

### 学术论文（4 项）

1. **When AI Models Become Dependencies: PTM Reuse Evolution** (TSE 提交)
   - **研究问题**：PTM（预训练模型）作为依赖如何演化？与传统库有何不同？
   - **方法**：4988 个发布版本 323 个 GitHub 仓库实证分析，传统库基线对比
   - **核心发现**：
     - PTM 变更频率是传统库的 1/3（406/2814）
     - PTM 项目成熟后累积而非替换
     - PTM 变更由能力扩展主动驱动，非被动修复
     - 独特的"PTM 测试不确定性"理由
   - **对比现有**：首次 PTM 依赖演化研究，prior 工作仅关注传统库
   - **优势**：大规模实证，PTM 特殊性首次量化
   - **局限**：仅开源 PTM，商业 PTM 演化模式未知
   - **潜力**：AI 集成软件工程管理新范式，PTM 版本控制工具需求
   - **理解**：PTM 不是传统库——它们是"能力包"而非"功能包"，演化逻辑根本不同，需要新管理工具

2. **Scaling Human-AI Coding: Governable Consensus Layer** (arXiv:2604.17883)
   - **研究问题**：如何规模化人机编码协作而不丧失可控性？
   - **方法**：提出 Agentic Consensus 范式，类型化属性图作为共识层
   - **核心创新**：
     - 共识层 C 替代代码作为主要工件
     - 同步算子 Phi(实现) 和 Psi(再水合) 保持对应
     - 共识熵量化欠规范程度
     - 评估指标：对齐保真度、共识熵、干预距离
   - **对比现有**：解决 Vibe Coding 维度坍塌问题，prior 工作仅关注代码生成质量
   - **优势**：结构性承诺可审计，证据链接到具体声明
   - **局限**：需要新工具链支持，学习曲线陡峭
   - **潜力**：企业级 AI 协作工程标准，安全关键系统刚需
   - **理解**：Vibe Coding 核心问题是"维度坍塌"——复杂系统拓扑被压扁为低维文本，Consensus Layer 恢复结构性信息

3. **MultiLogBench: Multilingual Automated Logging** (arXiv:2604.17529)
   - **研究问题**：自动化日志结论是否跨语言泛化？
   - **方法**：6 语言 63965 实例 744 修订历史案例，7 个 LLM 统一协议评估
   - **核心发现**：
     - 框架锚匹配最语言敏感
     - 循环和嵌套可调用站点最难
     - 模型排名仅在顶层稳定
     - 修订历史数据模式持续
   - **对比现有**：首次多语言日志基准，prior 工作仅 Java
   - **优势**：跨语言变异量化，维护导向验证
   - **局限**：6 语言仍有限，更多语言需要验证
   - **潜力**：多语言日志自动化标准，可观测性基础设施
   - **理解**：单语言证据是"幸存者偏差"——Java 结论不适用于其他语言，多语言评估是鲁棒性刚需

4. **Weaponizing the Commons: GitHub Abuse Taxonomy** (ICSE JAWs 2026)
   - **研究问题**：GitHub 滥用行为有哪些？如何系统性检测？
   - **方法**：392 个手动标注实例，分类学构建 + 检测框架开发
   - **核心发现**：
     - 多样化症状和根本原因分类学
     - 统一检测框架 F1 分数 89%+
     - 软件供应链安全新维度
   - **对比现有**：首个 GitHub 滥用系统性研究，prior 工作仅关注 CI/CD 自动化
   - **优势**：综合分类学，高准确率检测
   - **局限**：公开案例有限，隐蔽滥用未覆盖
   - **潜力**：平台级滥用检测基础设施，软件供应链安全新防线
   - **理解**：GitHub 滥用是软件供应链安全盲点——攻击者可以滥用平台机制而非代码漏洞，需要新防御范式
