## 技术扫描（35 项）

### AI Testing / Software Testing（12 项）

1. **TestGeneralizer: Generalizing Test Cases for Comprehensive Scenario Coverage** (FSE 2026)
   - 地址：https://arxiv.org/abs/2604.21771
   - 领域：AI Testing
   - 介绍：提出从开发者编写的初始测试用例出发，泛化覆盖更多测试场景的框架。现有自动化测试生成主要关注代码覆盖率，但真实项目中的测试场景源于需求而非控制流分支。TestGeneralizer 通过三阶段流程：（1）增强对目标方法和初始测试背后需求的理解；（2）生成测试场景模板并实例化；（3）从实例生成和精炼可执行测试用例。在 12 个开源 Java 项目上评估，相比 ChatTester 在变异场景覆盖上提升 31.66%，LLM 评估场景覆盖提升 23.08%。
   - 价值：FSE 2026 接受，从"覆盖率驱动"进化到"需求驱动"的测试生成范式

2. **Measuring LLM Involvement in Static Analysis**
   - 地址：https://arxiv.org/abs/2604.21746
   - 领域：AI Testing
   - 介绍：比较三种 LLM 参与静态分析翻译的架构：直接查询生成、结构化 JSON 中间表示、工具增强代理生成。在 20 个代码分析任务上评估 4 个开源模型。发现结构化中间表示（方法 2）结果匹配率最高，比直接生成高 15-25 个百分点，甚至超越消耗 8 倍 token 的代理方法。大模型受益于结构化中间表示，小模型则受限于 schema 合规性。
   - 价值：在形式化结构化领域，约束 LLM 输出到良类型中间表示比无约束生成或迭代工具使用效果更好

3. **Agentic AI-assisted Coding: Epistemic Grounding**
   - 地址：https://arxiv.org/abs/2604.21744
   - 领域：AI Testing
   - 介绍：提出 GROUNDING.md 概念——社区治理的领域范围认知接地文档，编码硬性约束（不可协商的有效性不变量）和约定参数（社区同意的默认值），覆盖所有其他上下文强制执行有效性。以质谱蛋白质组学为例，使非领域专家能生成具有最佳实践内嵌的代码和工具。
   - 价值：AI 辅助编程质量保证新方向，领域知识编码到代理上下文

4. **AeroReq2LTL: Automated LTL Specification from Aerospace Requirements**
   - 地址：https://arxiv.org/abs/2604.21715
   - 领域：形式化验证 / AI Testing
   - 介绍：将自然语言航空航天需求自动翻译为线性时序逻辑（LTL）属性的框架。两大工业创新：（1）数据字典将技术术语规范化为精确原子命题；（2）模板化需求语言使时间线索和逻辑关系在翻译前显式化。在真实航空航天数据集上实现 85% 精确率和 88% 召回率，输出可直接被现有验证工具消费。
   - 价值：安全关键系统形式化验证的工业级突破

5. **LLMs for ROS2 Architecture Comprehension**
   - 地址：https://arxiv.org/abs/2604.21699
   - 领域：AI Testing
   - 介绍：评估 9 个 LLM 在理解 ROS2 机器人系统架构方面的能力。1230 个提示覆盖 3 个增量规模的 ROS2 系统。几乎所有问题都被正确回答（平均 98.22%），gemini-2.5-pro 表现最佳（100%），gpt-4.1 最差（95%）。仅 300/1230 提示回答错误，其中 249 个来自最复杂系统。
   - 价值：LLM 辅助复杂系统架构理解的首次系统评估

6. **Verifying ML Interpretability via Provenance**
   - 地址：https://arxiv.org/abs/2604.21599
   - 领域：AI Testing
   - 介绍：将 ML 可解释性这一非功能性需求（NFR）从不可度量转化为可验证。通过保存模型和数据溯源信息使模型行为透明可解释，形成可量化的功能性需求（FR），验证 FR 即验证可解释性 NFR。
   - 价值：ML 工程需求验证方法论突破

7. **DryRUN: Eliminating Ground-Truth Test Dependency in Code Generation**
   - 地址：https://arxiv.org/abs/2604.21598
   - 领域：AI Testing
   - 介绍：发现多代理代码生成框架依赖人工提供的公共测试用例，导致"过度自信差距"——框架过拟合简单示例而在隐藏评估中失败。DryRUN 允许 LLM 自主生成有效输入并模拟执行轨迹来自我修正，在 LiveCodeBench v6 上匹配 CodeSIM 性能，同时完全不需要公共测试用例或外部执行反馈。
   - 价值：摆脱对公共测试用例的依赖，真实世界代码生成关键突破

8. **Metamorphic Testing for Diagnosing Memorization in LLM-based APR**
   - 地址：https://arxiv.org/abs/2604.21579
   - 领域：AI Testing
   - 介绍：结合变异测试（MT）与负对数似然（NLL）揭示 LLM 基于自动程序修复（APR）中的数据泄露。对 Defects4J 和 GitBug-Java 应用语义保持变换构建变异基准。所有评估的 SOTA LLM 在变换基准上修复成功率显著下降（GPT-4o -4.1%，Llama-3.1 -15.98%），且退化与原始基准 NLL 强相关。
   - 价值：MT+NLL 联合提供更强的数据泄露证据，APR 评估协议改进依据

9. **SpecSyn: LLM-based Formal Specification Synthesis for Real-world Verification**
   - 地址：https://arxiv.org/abs/2604.21570
   - 领域：形式化验证 / AI Testing
   - 介绍：LLM 驱动的程序验证规范生成方法。将输入程序分解为独立段，通过迭代规范生成处理，创新性地引入基于语义非等价程序变异和变体区分的规范精炼机制。精确率 >90%，召回率 >75%，在 1365 个目标属性中成功处理 1071 个。
   - 价值：LLM 辅助形式化验证处理真实世界程序的关键突破

10. **Orchid: Requirement Ambiguity Impact on Code Generation**
    - 地址：https://arxiv.org/abs/2604.21505
    - 领域：AI Testing
    - 介绍：首个针对模糊需求设计的代码生成基准，包含 1304 个函数级任务，覆盖四种模糊类型：词汇、句法、语义和模糊性。发现模糊性一致降低所有 LLM 性能，高级模型受影响最严重。LLM 对同一模糊需求产生功能分歧的实现，且无法自主识别或消歧。
    - 价值：揭示 LLM 代码生成在真实需求模糊性面前的系统性弱点

11. **Hidden Dependencies in SBOM-based Software Composition Analysis**
    - 地址：https://arxiv.org/abs/2604.21278
    - 领域：供应链安全 / AI Testing
    - 介绍：研究 SBOM 中两种不匹配模式：未表示为组件级依赖的隐藏代码级依赖，以及扫描仪无法一致识别的组件变体（克隆）。这些不匹配导致不一致的漏洞报告和 VEX 声明处理。
    - 价值：供应链安全工具链的底层数据质量问题揭示

12. **CrossCommitVuln: Multi-Commit Python Vulnerabilities**
    - 地址：https://arxiv.org/abs/2604.21917 (AIware 2026)
    - 领域：安全测试
    - 介绍：15 个真实 Python 漏洞基准，其中可利用条件跨多个提交引入——每个单独对 per-commit 静态分析良性，但 collectively 关键。per-commit 检测率（CCDR）仅 13%，87% 的链对 per-commit SAST 不可见。即使在累积模式下检测率仅 27%。
    - 价值：揭示 per-commit SAST 的根本局限性，跨提交安全分析新方向

### AI / LLM 技术（13 项）

13. **Tool Attention: Dynamic Tool Gating for MCP/Tools Tax Elimination**
    - 地址：https://arxiv.org/abs/2604.21816
    - 领域：AI Infrastructure
    - 介绍：MCP 依赖无状态 eager schema injection 产生隐藏 per-turn 开销（10k-60k tokens）。Tool Attention 将"Attention Is All You Need"范式从 token 自注意力泛化到工具门控注意力。结合意图 Schema 重叠分数、状态感知门控函数和两阶段延迟 schema 加载，在 120 工具 6 服务器基准上减少 95.0% per-turn 工具 token（47.3k→2.4k），有效上下文利用率从 24% 提升到 91%。
    - 价值：协议级效率而非原始上下文长度才是可扩展代理系统的约束

14. **DiffMAS: End-to-End Optimization of Multi-Agent Language Systems**
    - 地址：https://arxiv.org/abs/2604.21794 (COLM 2026 Under Review)
    - 领域：多智能体系统
    - 介绍：将隐式通信（通过 KV cache 等内部表示）作为多智能体系统的可学习组件。DiffMAS 对多智能体隐式轨迹执行参数高效监督训练，使代理联合学习信息如何在交互间编码和解释。在 AIME24 达 26.7%，GPQA-Diamond 达 20.2%。
    - 价值：从固定文本通信进化到可学习隐式通信协议

15. **Thinking with Reasoning Skills: Fewer Tokens, More Accuracy**
    - 地址：https://arxiv.org/abs/2604.21764 (ACL 2026 Industry Track)
    - 领域：LLM 推理优化
    - 介绍：从大量深思熟虑和试错探索中提炼可复用的推理技能并存储，在推理时检索这些技能指导未来推理。不同于"从头推理"范式，先回忆相关技能帮助模型避免冗余迂回。在编码和数学推理任务上显著减少推理 token 同时提升性能。
    - 价值：推理技能蒸馏，降低 per-request 成本

16. **Bounding the Black Box: Statistical Certification Framework for AI Risk Regulation**
    - 地址：https://arxiv.org/abs/2604.21854
    - 领域：AI 安全 / 合规
    - 介绍：借鉴航空认证范式，提出两阶段框架将 AI 风险监管转化为工程实践。阶段一：权威机构正式确定可接受失败概率 δ 和操作输入域 ε；阶段二：RoMA 和 gRoMA 统计验证工具计算系统真实失败率的可审计上限，无需访问模型内部。
    - 价值：EU AI Act 合规的缺失验证工具

17. **Alignment has a Fantasia Problem**
    - 地址：https://arxiv.org/abs/2604.21827 (ICLR 2026 Workshop)
    - 领域：AI 对齐
    - 介绍：现代 AI 助手训练遵循指令，隐含假设用户能清晰表达目标。但行为研究表明人们常在目标未完全形成时就与 AI 交互。将此类失败称为"Fantasia 交互"，主张 AI 应通过主动帮助用户随时间形成和精炼意图来提供认知支持。
    - 价值：对齐研究从"理性oracle"到"认知伙伴"范式转变

18. **From Research Question to Scientific Workflow: Agentic AI for Science Automation**
    - 地址：https://arxiv.org/abs/2604.21910
    - 领域：AI 自动化
    - 介绍：三层代理架构：LLM 将自然语言解释为结构化意图（语义层）；验证生成器产生可复现工作流 DAG（确定性层）；领域专家编写"Skills"（知识层）。在 1000 Genomes 工作流上评估，Skills 将完全匹配意图准确率从 44% 提升到 83%，技能驱动的延迟工作流生成减少 92% 数据传输。
    - 价值：科学研究自动化新范式，LLM 非确定性局限在意图提取

19. **MathDuels: Evaluating LLMs as Problem Posers and Solvers**
    - 地址：https://arxiv.org/abs/2604.21916
    - 领域：LLM 评估
    - 介绍：自博弈基准，模型担任双重角色：在对抗提示下创作数学问题并解决其他参与者创作的问题。三阶段生成管道（元提示、问题生成、难度放大），独立验证器排除病态问题。Rasch 模型联合估计求解能力和问题难度。19 个前沿模型实验揭示创作和求解能力部分解耦。
    - 价值：超越静态基准的自进化评估方法

20. **Who Defines "Best"? Interactive User-Defined LLM Leaderboard Evaluation**
    - 地址：https://arxiv.org/abs/2604.21769 (FAccT 2026)
    - 领域：LLM 评估
    - 介绍：分析 LMArena 数据集，发现数据集严重偏向某些主题，模型排名在提示切片间变化，偏好判断使用方式模糊了预期范围。设计交互式可视化界面允许用户定义自己的评估优先级。
    - 价值：LLM 基准排名透明度和上下文特异性评估

21. **GS-Quant: Granular Semantic Quantization for Knowledge Graph Completion**
    - 地址：https://arxiv.org/abs/2604.21649 (ACL 2026)
    - 领域：知识图谱 / LLM
    - 介绍：生成语义连贯且结构分层的离散代码用于 KG 实体。Granular Semantic Enhancement 模块将层次知识注入码本，Generative Structural Reconstruction 模块对代码序列施加因果依赖。扩展 LLM 词汇表实现图结构推理。
    - 价值：弥合连续图嵌入和离散 LLM token 之间的模态鸿沟

22. **Transformers in Symbolic Reasoning: Generalization and (Un)embedding Collapse**
    - 地址：https://arxiv.org/abs/2604.21632
    - 领域：LLM 理论基础
    - 介绍：研究 decoder-only transformer 在命题逻辑推理中泛化到未见变量名的能力。发现表示崩溃（unembedding 权重在训练中坍缩到几乎相同向量）是关键原因。结合架构改变、数据多样性和冻结/重置嵌入实现未见 token 泛化。在 Gemma 3 家族中观察到 (un)embedding 坍缩证据。
    - 价值：Transformer 符号推理泛化能力的机制解释

23. **CoFEE: Reasoning Control for LLM-based Feature Discovery**
    - 地址：https://arxiv.org/abs/2604.21584
    - 领域：AI 工程
    - 介绍：推理控制框架在 LLM 功能发现中强制执行认知行为：从结果反向链接、子目标分解、可观察性和泄漏验证、显式回溯拒绝的推理路径。成功率比 vanilla 高 15.2%，生成特征少 29%，成本降低 53.3%。
    - 价值：结构化归纳偏置提升 LLM 特征工程效率

24. **Separable Expert Architecture: Privacy-Preserving LLM Personalization**
    - 地址：https://arxiv.org/abs/2604.21571
    - 领域：隐私计算 / LLM
    - 介绍：三层架构解耦个人数据和共享权重：静态基础模型 + 可组合领域专家 LoRA 适配器 + per-user 代理工件。删除代理工件即确定性遗忘。Phi-3.5-mini 和 Llama-3.1-8B 验证，KL 散度约 0.21 nats，82-89% 验证通过率。
    - 价值：将机器遗忘从棘手的权重编辑问题转化为确定性删除操作

#### 1. Tool Attention: Dynamic Tool Gating for MCP/Tools Tax Elimination

**地址：** https://arxiv.org/abs/2604.21816  
**GitHub：** https://github.com/asadani/tool-attention

**项目介绍（200-400 字）：**  
Tool Attention 解决了一个被广泛忽视但影响深远的问题：MCP（Model Context Protocol）的无状态 eager schema injection 导致代理每轮产生 10k-60k token 的"MCP Tax"或"Tools Tax"。这个开销膨胀 KV cache，在上下文利用率接近 70% 的"断裂点"时与推理退化相关联，并将 token 预算转化为持续的运营成本。Tool Attention 将 Transformer 的"Attention Is All You Need"范式从 token 自注意力泛化到工具门控注意力，提出三层机制：（1）意图 Schema 重叠（ISO）分数通过句子嵌入计算工具相关性；（2）状态感知门控函数执行前置条件和访问范围检查；（3）两阶段延迟 schema 加载——紧凑摘要池保持在上下文中，仅对 top-k 门控工具提升完整 JSON schema。

**技术架构：**  
中间件层架构，独立于 LLM 和 MCP 服务器。ISO 模块使用 sentence-transformers 计算用户意图与每个工具 schema 描述的余弦相似度。门控模块维护工具状态（前置条件满足性、访问范围、调用历史）。延迟加载器维护紧凑摘要池（每个工具 ~50 token）和完整 schema 池（每个工具 ~2k-5k token），根据门控结果动态提升。

**核心算法：**  
- ISO 分数：cosine(embedding(intent), embedding(schema_description))
- 门控函数：g(t) = ISO_score(t) × precondition_met(t) × access_allowed(t)
- 两阶段加载：阶段一加载摘要池（固定开销），阶段二仅对 top-k 工具加载完整 schema

**与现有技术对比：**  
传统 MCP 部署每轮加载所有工具 schema（47.3k tokens），Tool Attention 降至 2.4k tokens（-95%）。有效上下文利用率从 24% 提升到 91%。

**优势与局限：**  
优势：协议级优化，无需修改 LLM 或 MCP 服务器；通用适用于任何 MCP 部署。局限：评估基于模拟而非真实 LLM 代理；投影值（任务成功率、延迟、成本）未在实际部署中验证。

**未来发展潜力：**  
如果投影值得到验证，Tool Attention 可能成为 MCP 部署的标准中间件。扩展到动态工具发现、工具组合优化、跨会话工具缓存等方向。

**个人技术理解：**  
这是代理基础设施领域被忽视的关键优化。大多数研究关注模型能力和上下文长度，但协议级效率才是可扩展代理系统的实际约束。95% 的 token 减少意味着 20 倍的成本节约，这对生产部署至关重要。

#### 2. DiffMAS: End-to-End Optimization of Multi-Agent Language Systems

**地址：** https://arxiv.org/abs/2604.21794

**项目介绍（200-400 字）：**  
DiffMAS 提出将隐式通信作为多智能体系统的可学习组件。现有工作聚焦代理角色和编排，将代理间通信视为固定接口（通常是文本）。DiffMAS 探索通过内部表示（如 KV cache）的隐式通信替代文本协议，执行参数高效的监督训练来学习多智能体隐式轨迹。代理联合学习信息如何在交互间编码和解释。在数学推理、科学 QA、代码生成和常识基准上一致超越单代理推理、文本多代理系统和先前隐式通信方法。AIME24 达 26.7%，GPQA-Diamond 达 20.2%。

**技术架构：**  
训练框架而非推理系统。核心组件：（1）多代理轨迹收集器，记录代理间隐式交互；（2）参数高效微调模块（LoRA/Adapter），在隐式轨迹上训练；（3）通信解码器，学习从其他代理的隐式表示中提取信息。

**核心算法：**  
参数高效监督训练：给定多代理交互轨迹 {(h_1, h_2, ..., h_T)}，训练解码器 D 使得 D(h_i) 预测代理 j 需要的信息。损失函数结合任务准确性和通信效率。

**与现有技术对比：**  
传统文本通信：高开销但可解释。DiffMAS：低开销但需要训练。在推理基准上超越文本多代理系统。

**优势与局限：**  
优势：端到端优化通信和推理；参数高效；跨任务泛化。局限：需要训练数据；隐式通信不可解释；当前仅评估推理任务。

**未来发展潜力：**  
扩展到多模态代理、实时协作、跨模型家族通信。可能成为多代理系统的标准通信协议。

**个人技术理解：**  
隐式通信是多代理系统从"聊天"进化到"协作"的关键一步。类似人类团队从语言交流进化到默契配合。COLM 2026 下审，值得关注后续发表。

### 学术论文

#### 1. DryRUN: Eliminating Ground-Truth Test Dependency in Code Generation

**地址：** https://arxiv.org/abs/2604.21598

**论文介绍（200-400 字）：**  
发现多代理代码生成框架（如 CodeSIM）依赖人工提供的公共测试用例来锚定调试和模拟循环，这在实际软件工程中存在根本性限制——真实世界中 ground-truth 输入输出示例在实现前几乎不可用。更关键的是，依赖公共测试导致"过度自信差距"：框架过拟合简单示例而在隐藏评估中失败。DryRUN 证明 LLM 可以自主生成有效输入并模拟执行轨迹来自我修正，不需要 ground-truth 样本。在 LiveCodeBench v6（2025 年 3 月后数据）上评估，DryRUN 匹配 CodeSIM 性能同时完全不需要公共测试用例或外部执行反馈，且减少输出 token 消耗。

**技术方法：**  
三阶段自主循环：（1）计划——LLM 分析问题描述生成解决策略；（2）自主输入生成——LLM 生成多样化的有效输入（不依赖 ground-truth）；（3）模拟执行——LLM 模拟代码执行并检测逻辑错误，迭代修正。

**核心创新：**  
- 消除对公共测试用例的依赖，使代码生成框架适用于真实世界场景
- 识别并解决"过度自信差距"问题
- 模拟执行替代真实执行，降低成本

**与现有技术对比：**  
CodeSIM 依赖公共测试，DryRUN 完全自主。性能匹配但适用性更广。

**优势与局限：**  
优势：真实世界适用、降低 token 消耗、无外部依赖。局限：模拟执行可能不如真实执行准确；仅评估编程竞赛基准。

**未来发展潜力：**  
结合真实执行反馈的混合方法可能最优。扩展到 web 开发、API 集成等更复杂场景。

**个人技术理解：**  
这是代码生成从"基准游戏"到"真实工程"的关键一步。摆脱公共测试依赖意味着代理可以在没有 ground-truth 的真实世界中工作。

#### 2. Metamorphic Testing for Diagnosing Memorization in LLM-based APR

**地址：** https://arxiv.org/abs/2604.21579

**论文介绍（200-400 字）：**  
LLM 基于自动程序修复（APR）技术显示出降低调试成本的 promising 结果，但先验结果可能受数据泄露影响——LLM 可能在预训练数据中记忆了评估基准中的 bug 修复，导致性能估计膨胀。本文结合变异测试（MT）与负对数似然（NLL）提供更强的数据泄露证据。对 Defects4J 和 GitBug-Java 应用语义保持变换构建变异基准，评估 7 个 LLM 在原始和变换版本上的修复成功率。所有 SOTA LLM 在变换基准上修复成功率显著下降（GPT-4o -4.1%，Llama-3.1 -15.98%），且退化与原始基准 NLL 强相关。

**技术方法：**  
- 语义保持变换：重命名变量、重构控制流、提取方法等
- NLL 计算：测量 LLM 对原始 bug 修复的似然
- 相关性分析：变换基准退化 vs 原始基准 NLL

**核心发现：**  
- 所有 LLM 存在数据泄露，但程度差异大
- NLL 可作为记忆化的代理指标
- MT 单独使用可缓解数据泄露对 APR 评估的影响

**优势与局限：**  
优势：方法论严谨，多个基准验证。局限：语义保持变换可能引入微妙语义变化；仅评估 APR 领域。

**未来发展潜力：**  
扩展到更多代码生成任务（代码翻译、代码补全）。结合更多变换技术。

**个人技术理解：**  
这是 AI 代码生成评估领域的重要方法论贡献。MT+NLL 联合为数据泄露检测提供了更强证据，对 APR 评估协议改进有直接指导意义。

#### 3. Orchid: Requirement Ambiguity Impact on Code Generation

**地址：** https://arxiv.org/abs/2604.21505

**论文介绍（200-400 字）：**  
现有代码生成基准通常假设清晰无歧义的需求，留下实证空白。Orchid 是首个针对模糊需求设计的代码生成基准，包含 1304 个函数级任务，覆盖四种模糊类型：词汇歧义（多义词）、句法歧义（结构不清晰）、语义歧义（含义不确定）和模糊性（边界不明确）。系统评估 LLM 在模糊需求下的行为。发现模糊性一致降低所有 LLM 性能，高级模型受影响最严重。LLM 对同一模糊需求产生功能分歧的实现，且无法自主识别或消歧。

**技术方法：**  
- 四种模糊类型系统分类
- 1304 个函数级任务，每个有明确和模糊两个版本
- 多模型对比评估

**核心发现：**  
- 模糊性对代码生成的负面影响被严重低估
- 高级模型在模糊需求下表现更差（可能因为更自信地错误解释）
- LLM 缺乏自主消歧能力

**优势与局限：**  
优势：首个模糊需求基准，系统分类。局限：仅函数级，未覆盖仓库级；模糊类型可能不完整。

**未来发展潜力：**  
扩展到仓库级代码生成。结合消歧技术（主动询问、上下文推断）。

**个人技术理解：**  
这是代码生成从"理想世界"到"真实世界"的关键一步。真实需求几乎总是模糊的，LLM 需要学会处理模糊性而非假设清晰需求。
