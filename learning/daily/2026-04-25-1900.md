### AI Testing / Software Testing（8 项）

#### 1. TestGeneralizer: Generalizing Test Cases for Comprehensive Test Scenario Coverage
- **地址**：https://arxiv.org/abs/2604.21771
- **来源**：arXiv cs.SE（FSE 2026 Accepted）
- **领域**：Software Testing / Test Generation
- **技术介绍**：开发者通常从隐式模式推导多个测试用例，但全面测试耗时且易出错。现有自动化测试生成技术主要关注代码覆盖率，但实际测试场景不一定与控制流分支对齐。TestGeneralizer 从单个初始测试出发，推断底层需求并生成多样化测试场景。三阶段框架：(1) 增强对目标方法和初始测试背后需求的理解；(2) 生成测试场景模板并实例化为多种场景变体；(3) 从场景实例生成和精炼可执行测试用例。在 12 个开源 Java 项目上评估，相比 ChatTester 实现 +31.66%（变异覆盖）和 +23.08%（LLM 评估场景覆盖）的显著提升。
- **应用场景**：自动化测试生成、回归测试扩展、代码变更影响分析
- **为什么值得关注**：从"覆盖率驱动"进化到"场景驱动"，更贴近开发者实际测试思维

#### 2. Assessing the Impact of Requirement Ambiguity on LLM-based Code Generation (Orchid)
- **地址**：https://arxiv.org/abs/2604.21505
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Code Generation Evaluation
- **技术介绍**：软件需求模糊性在真实开发中无处不在，但现有基准测试通常假设清晰明确的需求。Orchid 是首个专门设计模糊需求的代码生成基准，包含 1,304 个函数级任务，覆盖四种模糊类型：词汇歧义、句法歧义、语义歧义和模糊性。系统评估发现模糊性一致降低所有 LLM 的性能，高级模型受影响最严重。LLM 频繁为同一模糊需求生成功能迥异的实现，且无法自主识别或消歧。
- **应用场景**：代码生成模型评估、需求工程、AI 辅助开发质量保障
- **为什么值得关注**：揭示了 LLM 代码生成在真实场景中的关键瓶颈——无法处理模糊需求

#### 3. A Metamorphic Testing Approach to Diagnosing Memorization in LLM-Based Program Repair
- **地址**：https://arxiv.org/abs/2604.21579
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Data Leakage Detection
- **技术介绍**：LLM 自动化程序修复（APR）可能因数据泄露而高估性能。本文结合变异测试（MT）与负对数似然（NLL）来更好地揭示数据泄露。对 Defects4J 和 GitBug-Java 应用语义保持变换构建变体基准。结果显示所有 SOTA LLM 在变换基准上修复成功率显著下降：GPT-4o -4.1%，Llama-3.1 -15.98%。退化与原始基准的 NLL 强相关，表明模型在更可能记忆化的实例上表现更好。
- **应用场景**：APR 基准测试公正性验证、数据泄露检测、模型评估可信度
- **为什么值得关注**：MT+NLL 联合方法为 APR 评估提供了更强有力的数据泄露证据

#### 4. Measuring How LLM Involvement Affects Chatbot Accuracy in Static Analysis
- **地址**：https://arxiv.org/abs/2604.21746
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Static Analysis
- **技术介绍**：比较三种 LLM 参与程度架构用于将自然语言翻译为 Joern 查询语言 CPGQL：(1) 直接查询生成；(2) 生成 schema 约束的 JSON 中间表示；(3) 工具增强代理生成。在 20 个代码分析任务、三种复杂度层级、四种开源模型上评估。结构化中间表示（方法 2）实现最高结果匹配率，在大模型上比直接生成高 15-25 个百分点，尽管代理方法消耗 8 倍 token。
- **应用场景**：自然语言代码查询、静态分析工具易用性、LLM 在形式化领域的最佳实践
- **为什么值得关注**：在形式化结构化领域，约束 LLM 输出到类型化中间表示比无约束生成或迭代工具使用效果更好

#### 5. LLM-based Synthesis and Refinement of Formal Specifications for Real-world Program Verification (SpecSyn)
- **地址**：https://arxiv.org/abs/2604.21570
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Formal Verification
- **技术介绍**：程序验证需要全面的跨过程规格说明，但手动构建耗时费力。SpecSyn 将输入程序分解为独立片段，分别进行迭代规格生成。创新性地引入基于语义非等价程序变异和变体区分的规格精炼机制，评估和增强生成规格的语义强度。保持 90%+ 精确率和 75%+ 召回率，成功处理 1365 个目标属性中的 1071 个。
- **应用场景**：形式化验证自动化、安全关键软件验证、程序正确性保证
- **为什么值得关注**：LLM 在形式化验证中的规模化应用突破

#### 6. Automated LTL Specification Generation from Industrial Aerospace Requirements (AeroReq2LTL)
- **地址**：https://arxiv.org/abs/2604.21715
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Formal Methods / Aerospace
- **技术介绍**：航空航天安全关键软件中，将自然语言需求翻译为线性时序逻辑（LTL）属性是劳动密集且易错的过程。AeroReq2LTL 使用 LLM 自动化 LTL 属性生成，两个工业创新：(1) 数据字典将技术术语标准化为精确原子命题；(2) 模板化需求语言使时序线索和逻辑关系显式化。在真实航空航天数据集上实现 85% 精确率和 88% 召回率，输出可直接被现有验证工具消费。
- **应用场景**：航空航天安全验证、形式化规格自动生成、安全关键系统测试
- **为什么值得关注**：工业级形式化方法自动化的重要里程碑

#### 7. Quantifying Software Performance Requirements via Interactive Retrieval-Augmented Preference Elicitation (IRAP)
- **地址**：https://arxiv.org/abs/2604.21380
- **来源**：arXiv cs.SE（ACL 2026 Accepted）
- **领域**：Software Testing / Performance Engineering
- **技术介绍**：软件性能需求以自然语言记录，量化为数学形式对软件工程至关重要。IRAP 通过交互式检索增强偏好 elicitation 将性能需求量化为数学函数。与现有方法不同，IRAP 显式地从问题特定知识中推导检索和推理偏好，指导与利益相关者的渐进交互，同时减少认知开销。在四个真实数据集上对比 10 种 SOTA 方法，所有情况下均表现优越，最少 5 轮交互即可实现高达 40 倍改进。
- **应用场景**：性能需求量化、SLA 制定、性能测试目标设定
- **为什么值得关注**：将模糊性能需求转化为可测试数学指标

#### 8. Hidden Dependencies and Component Variants in SBOM-Based Software Composition Analysis
- **地址**：https://arxiv.org/abs/2604.21278
- **来源**：arXiv cs.SE
- **领域**：Software Testing / Supply Chain Security
- **技术介绍**：软件物料清单（SBOM）已成为漏洞管理的重要技术，但其有效性取决于 SBOM 对组件身份和实际依赖关系的准确表示。本文研究两种不匹配模式：(1) 未表示为组件级依赖的隐藏代码级依赖；(2) 扫描器无法一致识别的组件变体（克隆）。这些不匹配会导致漏洞报告不一致和 VEX 声明处理不一致。
- **应用场景**：供应链安全分析、SBOM 质量评估、漏洞管理
- **为什么值得关注**：揭示 SBOM 基础设施的隐藏脆弱性

### AI / LLM 技术（10 项）

#### 9. MMTR-Bench: Can MLLMs "Read" What is Missing?
- **地址**：https://arxiv.org/abs/2604.21277
- **来源**：arXiv cs.AI
- **领域**：MLLM Evaluation / Visual Reasoning
- **技术介绍**：MMTR-Bench 评估多模态大语言模型从视觉上下文直接重建被遮蔽文本的内在能力。与常规 QA 任务不同，MMTR-Bench 消除显式提示，要求模型从单页或多页输入中恢复遮蔽文本，涵盖文档和网页等真实领域。包含 2,771 个测试样本，多种语言和不同目标长度。实验显示基准构成重大挑战，尤其是句子和段落级重建。
- **应用场景**：MLLM 能力评估、文档理解、视觉-语言对齐测试
- **为什么值得关注**：首次系统评估 MLLM 的"补全"能力，而非问答能力

#### 10. Enhancing Online Recruitment with Category-Aware MoE and LLM-based Data Augmentation
- **地址**：https://arxiv.org/abs/2604.21264
- **来源**：arXiv cs.AI（ACL Industry Track 2026 Accepted）
- **领域**：LLM Application / HR Tech
- **技术介绍**：人岗匹配（PJF）在线招聘关键组件。LLM 数据增强使用 CoT 提示润色低质量职位描述；类别感知 MoE 通过类别嵌入动态分配专家权重，学习更具区分度的模式。离线评估和在线 A/B 测试显示 AUC 相对提升 2.40%，GAUC 提升 7.46%，在线点击转化率提升 19.4%。
- **应用场景**：智能招聘、人岗匹配、HR 技术
- **为什么值得关注**：LLM + MoE 在工业级招聘平台的成功部署

#### 11. Robustness Analysis of POMDP Policies to Observation Perturbations
- **地址**：https://arxiv.org/abs/2604.21256
- **来源**：arXiv cs.AI
- **领域**：Reinforcement Learning / Robustness
- **技术介绍**：POMDP 策略通常使用标称系统模型设计，但部署时可能因校准漂移或传感器退化而偏离真实系统。本文研究策略对 POMDP 观测模型偏差的鲁棒性，引入策略观测鲁棒性问题：确定保证策略价值高于指定阈值的最大可容忍观测模型偏差。提出 Robust Interval Search 算法，在非粘性变体中具有多项式时间复杂度。
- **应用场景**：机器人控制鲁棒性测试、自动驾驶安全验证、关键系统 RL 部署
- **为什么值得关注**：为 RL 策略部署提供可量化的鲁棒性保证

#### 12. Hierarchical Predictive Correction to Mitigate Cascading Failures in VLA Systems (ReCAPA)
- **地址**：https://arxiv.org/abs/2604.21232
- **来源**：arXiv cs.AI
- **领域**：Vision-Language-Action / Embodied AI
- **技术介绍**：VLA 系统执行多步任务时，中间步骤的错误会传播并积累为级联失败。ReCAPA 使用预测和对比在三层面调整偏差：动作、子目标和轨迹。使用 Sinkhorn 模块和 Score-field 模块在所有层面强制执行语义对齐。在 VisualAgentBench、MineDojo 和 AI2-THOR 等具身代理基准上实现竞争性结果。
- **应用场景**：具身 AI 可靠性、机器人任务执行、VLA 系统测试
- **为什么值得关注**：引入两个新指标量化错误传播和恢复过程

#### 13. DAVinCI: Dual Attribution and Verification in Claim Inference
- **地址**：https://arxiv.org/abs/2604.21193
- **来源**：arXiv cs.AI
- **领域**：LLM Factuality / Hallucination Detection
- **技术介绍**：DAVinCI 两阶段框架：(1) 将生成声明归因于内部模型组件和外部来源；(2) 使用蕴含推理和置信度校准验证每个声明。在 FEVER 和 CLIMATE-FEVER 数据集上评估，分类准确率、归因精确率、召回率和 F1 分数提升 5-20%。模块化实现可集成到现有 LLM 管道。
- **应用场景**：事实核查、幻觉检测、可信 AI 输出
- **为什么值得关注**：将归因和验证桥接，提供可扩展的可审计 AI 路径

#### 14. Multi-Agent Empowerment and Emergence of Complex Behavior in Groups

#### 15. Agentic AI for Personalized Physiotherapy: Multi-Agent Framework

#### 16. AI Governance under Political Turnover: The Alignment Surface of Compliance Design

#### 17. Propensity Inference: Environmental Contributors to LLM Behaviour
- **地址**：https://arxiv.org/abs/2604.21098
- **来源**：arXiv cs.AI
- **领域**：AI Safety / LLM Behaviour Analysis
- **技术介绍**：开发测量语言模型非授权行为倾向的方法。三个方法论改进：分析环境因素变化对行为的影响、通过贝叶斯广义线性模型量化效应大小、明确防止循环分析。在 23 个语言模型和 11 个评估环境上，发现战略和非战略因素对行为解释的贡献大致相等，未找到战略因素随能力提高而变得更重要或更不重要的证据。
- **应用场景**：AI 对齐风险评估、LLM 行为分析、安全测试
- **为什么值得关注**：为 AI 倾向研究提供可实证检验的理论框架

#### 19. Can Large Language Models Assist the Comprehension of ROS2 Software Architectures?
- **地址**：https://arxiv.org/abs/2604.21699

- **领域**：Robotics / Software Architecture / LLM
- **技术介绍**：ROS2 架构高度复杂，数千个组件以去中心化方式通信。控制实验对 9 个 LLM 提交 1,230 个提示，关于 3 个递增规模 ROS2 系统的架构相关问题。几乎所有问题在所有 LLM 上回答正确（均值 98.22%）。gemini-2.5-pro 表现最佳（100% 准确率），gpt-4.1 表现最差（95%）。

- **为什么值得关注**：LLM 在机器人软件架构理解上的系统性评估

#### 20. Agentic AI-assisted Coding and Epistemic Grounding (GROUNDING.md)

- **来源**：arXiv cs.SE

- **应用场景**：领域特定代码生成、科学软件质量保证、代理编码治理

## 深度研究

### GitHub 项目

#### 1. TestGeneralizer (FSE 2026)
**地址**：https://arxiv.org/abs/2604.21771

**项目介绍（300 字）**：
TestGeneralizer 代表了软件测试生成的范式转变——从"覆盖率驱动"到"场景驱动"。传统自动化测试生成工具（如 EvoSuite、ChatTester）主要关注代码覆盖率指标，但实际开发中的测试场景源于需求理解而非控制流分支。TestGeneralizer 的核心洞察是：开发者编写的测试本质上是可执行规格说明，即使单个初始测试也能揭示底层需求和应验证的多样化场景。

框架采用三阶段架构：
1. **需求理解增强**：使用 LLM 分析目标方法签名、初始测试代码和项目上下文，推断隐含需求和测试场景模式
2. **场景模板生成与实例化**：从推断的需求中生成抽象测试场景模板，然后实例化为具体的测试场景变体（边界条件、异常路径、组合场景等）
3. **可执行测试生成与精炼**：从场景实例生成实际可执行测试代码，并通过编译反馈和运行时断言进行精炼

在 12 个开源 Java 项目上的评估显示，相比 ChatTester 实现 +31.66% 变异覆盖率和 +23.08% LLM 评估场景覆盖率。

**技术架构**：
- LLM 驱动的需求推断引擎（基于代码上下文和测试意图分析）
- 场景模板生成器（抽象→具体的层次化场景建模）
- 测试代码生成与精炼循环（编译验证 + 运行时断言）

**核心创新**：
- 从单测试推导多场景的"需求逆向工程"方法
- 场景覆盖率作为独立于代码覆盖率的评估指标
- 更贴近开发者实际测试思维的生成范式

**优势与局限**：
- 优势：生成的测试更贴近真实需求，场景覆盖更全面
- 局限：依赖 LLM 的需求推断准确性，对复杂业务逻辑的理解可能受限

**未来发展潜力**：
与 Orchid（模糊需求基准）结合，可以研究 TestGeneralizer 在模糊需求场景下的表现，形成"需求模糊性→场景推断→测试生成"的完整链条。

#### 2. SpecSyn: LLM-based Formal Specification Synthesis
**地址**：https://arxiv.org/abs/2604.21570

**项目介绍（300 字）**：
SpecSyn 解决了程序验证中的核心瓶颈——形式化规格说明的手动构建极其耗时且需要专业形式化方法知识。现有 LLM 赋能方法在处理大规模输入程序和保证生成规格强度方面存在显著局限。

SpecSyn 的核心创新在于两个机制的结合：
1. **程序分解**：将大型输入程序分解为独立片段，分别处理，解决规模问题
2. **语义非等价变异精炼**：通过生成语义非等价的程序变异体来测试生成规格的区分能力，如果规格无法区分正确和错误实现，则自动精炼

实验结果显示 SpecSyn 保持 90%+ 精确率和 75%+ 召回率，在 1365 个目标属性中成功处理 1071 个，显著优于现有 LLM 基线方法。

**技术架构**：
- 程序片段分解器（基于函数边界和控制流分析）
- LLM 迭代规格生成（逐片段生成前置/后置条件）
- 变异驱动的规格精炼（语义非等价变异体生成 + 区分验证）

**与现有技术对比**：
相比 NL2SPEC 等工具，SpecSyn 在规模处理能力和规格强度保证上有显著提升。相比 AeroReq2LTL（专注于航空航天需求），SpecSyn 面向通用程序验证。

**优势与局限**：
- 优势：可扩展到真实世界程序，规格强度可验证
- 局限：变异生成质量影响精炼效果，对复杂数据结构的规格可能不够精确

### 学术论文

#### 1. Orchid: Ambiguous Requirements Benchmark
**地址**：https://arxiv.org/abs/2604.21505

**论文介绍（300 字）**：
Orchid 是首个专门设计模糊需求的代码生成基准，揭示了 LLM 代码生成在真实场景中的关键瓶颈。现有基准测试（如 HumanEval、MBPP）假设清晰明确的需求，但真实软件开发中需求模糊性无处不在——来自自然语言的固有不精确性和利益相关者的不同解释。

Orchid 包含 1,304 个函数级任务，覆盖四种模糊类型：
- **词汇歧义**：术语多义性（如 "sort" 可以指升序或降序）
- **句法歧义**：句子结构多义性（如 "process each file and directory" 的 scope 不明确）
- **语义歧义**：含义多义性（如 "fast" 可以指时间效率或资源效率）
- **模糊性**：缺乏精确边界（如 "handle large inputs" 的阈值未定义）

关键发现：
1. 模糊性一致降低所有 LLM 性能，高级模型受影响最严重（这可能是因为高级模型过度自信于模糊需求的解释）
2. LLM 频繁为同一模糊需求生成功能迥异的实现
3. LLM 缺乏自主识别或消歧模糊需求的能力

**技术价值**：
Orchid 为代码生成评估提供了一个更真实的基准，揭示了从"清晰需求代码生成"到"模糊需求代码生成"的巨大性能差距。这暗示下一代 AI 辅助开发工具需要模糊需求处理能力——要么主动消歧，要么明确标注不确定性。

**与 TestGeneralizer 的关系**：
TestGeneralizer 从单测试推断多场景，Orchid 评估模糊需求下的代码生成。两者结合可以研究：当需求模糊时，TestGeneralizer 能否生成覆盖多种解释的测试场景？

#### 2. Metamorphic Testing + NLL for APR Data Leakage
**地址**：https://arxiv.org/abs/2604.21579

**论文介绍（300 字）**：
这篇论文解决了 LLM 自动化程序修复（APR）评估中的一个关键可信度问题：数据泄露。当评估基准与 LLM 预训练数据重叠时，模型可能记住了 bug 修复方案，导致性能估计虚高。

方法论创新：
1. **变异测试（MT）**：对 Defects4J 和 GitBug-Java 应用语义保持变换（如变量重命名、表达式等价替换、控制流重构），构建语义相同但表面形式不同的变体基准
2. **负对数似然（NLL）**：测量模型对原始基准实例的似然，作为记忆化程度的代理指标
3. **联合分析**：比较模型在原始基准和变体基准上的修复成功率差异，并与 NLL 关联

关键发现：
- 所有 SOTA LLM 在变换基准上修复成功率显著下降
- GPT-4o 下降 -4.1%（相对稳健），Llama-3.1 下降 -15.98%（高度依赖记忆）
- 退化程度与原始基准的 NLL 强相关
- MT+NLL 联合方法比单独使用 MT 提供更强的数据泄露证据

**技术价值**：
为 APR 基准测试提供了可重复的数据泄露检测框架。建议 APR 研究社区采用 MT 变体基准作为标准评估实践，以确保性能报告的真实性。

**与 Orchid 的关系**：
Orchid 关注需求模糊性对代码生成的影响，本文关注数据泄露对代码修复评估的影响。两者共同指向一个趋势：AI 代码工具评估需要更严格的真实场景模拟。
