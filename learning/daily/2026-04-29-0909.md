### AI Testing / Software Testing（10 项）

#### 1. When Prompt Under-Specification Improves Code Correctness
- **地址**：https://arxiv.org/abs/2604.24712
- **来源**：arXiv 2604.24712 (cs.SE)
- **领域**：AI Testing / 提示工程 / 代码生成
- **技术介绍**：传统研究认为提示欠规范会显著降低代码正确性，但这项探索性研究在 LiveCodeBench 上发现反直觉结果：欠规范提示反而提升正确性。原因是结构化丰富的任务描述（含描述、约束、示例、I/O 约定）提供了冗余，欠规范破坏了误导性的词汇或结构线索，阻止了不正确的检索式解决策略。10 个模型评估显示，提示鲁棒性不是 LLM 的固定属性，而是高度依赖提示结构。
- **应用场景**：提示工程优化、代码生成质量保障、基准测试设计
- **为什么值得关注**：颠覆"越详细越好"的提示工程常识，揭示结构化冗余的防御价值

#### 2. Defective Task Descriptions in LLM-Based Code Generation

- **为什么值得关注**：从任务端而非模型端提升代码生成质量

#### 3. Leveraging LLMs for Multi-File DSL Code Generation: Industrial Case Study

### 学术论文

#### 1. When Prompt Under-Specification Improves Code Correctness

**地址**：https://arxiv.org/abs/2604.24712

**论文介绍（200-400 字）**：
这篇探索性研究颠覆了提示工程的一个核心假设——"越详细的提示越好"。传统研究（基于 HumanEval 等最小规范基准）表明提示欠规范会显著降低代码正确性。但本研究在 LiveCodeBench（结构化更丰富的基准）上发现反直觉结果：欠规范提示反而提升了正确性。研究评估了 10 个 LLM，发现提示鲁棒性不是 LLM 的固定属性，而是高度依赖提示结构。在结构化丰富的任务描述中（含描述、约束、示例、I/O 约定），欠规范破坏了误导性的词汇或结构线索，阻止了不正确的检索式解决策略。手动分析揭示了三种改进机制：(1) 打破过度拟合的术语触发；(2) 移除误导性约束；(3) 消除虚假标识符触发。

**技术架构**：
- 实验设计：HumanEval（最小规范）vs LiveCodeBench（丰富规范）
- 提示变异：系统性地移除/修改提示中的不同元素
- 模型评估：10 个 LLM 的交叉对比

**核心算法或系统设计**：
- 提示变异策略：系统性移除约束、示例、I/O 约定等
- 正确性评估：自动化测试通过率
- 机制分析：手动分类改进/退化案例

**关键实现机制**：
- 结构化冗余作为"防御层"抵消欠规范负面影响
- 欠规范破坏"捷径"策略（基于表面线索的检索式解决）
- 模型被迫从第一性原理推理而非模式匹配

**与现有技术对比**：
| 方面 | 传统观点 | 本研究发现 |
|------|----------|------------|
| 提示详细度 | 越详细越好 | 结构化丰富时欠规范有益 |
| 鲁棒性 | LLM 固定属性 | 高度依赖提示结构 |
| 机制 | 更多信息=更好 | 信息过载=误导 |

**优势与局限**：
- 优势：颠覆性发现、实证严谨、实用价值高
- 局限：探索性研究（非系统性）、仅覆盖代码生成场景

**未来发展潜力**：
这一发现将影响提示工程的最佳实践，特别是在结构化丰富的任务中，"少即是多"可能成为新原则。

**个人技术理解**：
这项研究的核心洞察是：LLM 的代码生成不是"信息处理"而是"模式匹配"。当提示过于详细时，LLM 倾向于基于表面线索进行检索式解决（retrieval-based solution），而非从第一性原理推理。欠规范提示打破了这种捷径，迫使模型进行更深层的推理。这与人类认知科学中的"干扰效应"类似——有时移除信息反而提升决策质量。

#### 2. Measuring the Unmeasurable: Markov Chain Reliability for LLM Agents

**地址**：https://arxiv.org/abs/2604.24579

**论文介绍**：
这篇论文提出了首个 LLM 代理可靠性的量化框架，使用马尔可夫链建模代理行为的状态转移。传统软件可靠性理论（如 MTBF、故障率）难以直接应用于 LLM 代理，因为代理行为具有高度随机性和上下文依赖性。本研究将代理行为抽象为马尔可夫链，通过状态转移概率计算长期可靠性指标，解决代理行为随机性导致的可靠性评估难题。

**技术架构**：
- 状态空间：代理可能的行为状态集合
- 转移概率：基于历史交互数据估计
- 可靠性指标：长期稳态下的成功概率

**核心算法或系统设计**：
- 马尔可夫链建模：将代理行为序列抽象为状态转移
- 稳态分析：计算长期可靠性指标
- 置信区间估计：基于采样不确定性量化

**关键实现机制**：
- 行为状态定义：成功/失败/重试/放弃等
- 转移概率估计：从交互日志中学习
- 可靠性计算：稳态分布下的成功概率

**优势与局限**：
- 优势：首个代理可靠性量化框架、理论基础扎实
- 局限：假设马尔可夫性（可能不成立）、需要大量交互数据

**未来发展潜力**：
这一框架有望成为生产部署 LLM 代理的 SLA 保障基础，类似于传统软件的可靠性工程。

**个人技术理解**：
代理可靠性量化是 AI 工程化的关键瓶颈。没有可靠性指标，就无法进行容量规划、SLA 承诺和故障预算分配。这篇论文填补了这一空白，但马尔可夫假设可能过于简化——代理行为往往具有长程依赖性，未来可能需要更高阶的马尔可夫模型或 RNN 建模。

#### 3. Defective Task Descriptions in LLM-Based Code Generation

**地址**：https://arxiv.org/abs/2604.24703

**论文介绍**：
系统分析 LLM 代码生成中缺陷任务描述的类型、频率和影响。研究识别出缺失约束、矛盾需求、模糊目标等缺陷模式，并提出自动检测方法。与上篇"欠规范提升正确性"形成有趣对比——本研究关注的是"坏的不详细"而非"好的不详细"。

**技术架构**：
- 缺陷分类：缺失约束、矛盾需求、模糊目标、过时信息
- 检测模型：基于 LLM 的缺陷检测器
- 影响评估：缺陷类型与代码质量的相关性

**核心算法或系统设计**：
- 缺陷模式库：预定义的缺陷类型和检测规则
- 自动检测：LLM 驱动的缺陷识别
- 影响量化：缺陷存在与否的代码正确性对比

**优势与局限**：
- 优势：系统化缺陷分类、自动检测工具
- 局限：仅覆盖代码生成场景、缺陷定义依赖人工标注

**未来发展潜力**：
缺陷任务描述检测有望成为代码生成流水线的前置检查环节，类似于静态分析在编译前的角色。

**个人技术理解**：
这篇研究与 2604.24712 形成有趣对比：24712 说"欠规范有时有益"，24703 说"缺陷描述有害"。关键区别在于——欠规范是"信息不足"，缺陷是"信息错误"。LLM 可以处理信息不足（从第一性原理推理），但难以处理信息错误（被误导）。这提示我们：提示工程的核心不是"信息量"而是"信息质量"。
