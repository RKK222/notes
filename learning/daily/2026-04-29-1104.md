### AI Testing / Software Testing（10 项）

#### 1. BenchGuard: Automated Auditing of LLM Agent Benchmarks
- **地址**：https://arxiv.org/abs/2604.24955
- **来源**：arXiv 2604.24955 (cs.CL/AI/SE)
- **领域**：AI Testing / 基准审计 / Agent 评估
- **技术介绍**：随着基准复杂度增加，许多看似 Agent 失败实际上是基准自身的失败——损坏的规格说明、隐式假设、惩罚有效替代方案的刚性评估脚本。BenchGuard 是首个面向任务导向、执行式 Agent 基准的自动化审计框架。它通过结构化 LLM 协议交叉验证所有基准工件，可选择性地纳入 Agent 解决方案或执行痕迹作为额外诊断证据。在 ScienceAgentBench 上识别出 12 个作者确认问题（包括使任务不可解的致命错误），在 BIXBench Verified-50 子集上精确匹配 83.3% 的专家识别问题。50 个复杂生物信息学任务的完整审计成本低于 15 美元。
- **应用场景**：基准质量保证、Agent 评估基础设施验证、自动化基准开发
- **为什么值得关注**：标志着"AI 辅助基准开发"新范式——前沿模型不仅作为评估对象，还作为验证评估基础设施的积极参与者

#### 2. CoRE: Code Reasoning Benchmark Beyond Output Prediction
- **地址**：https://arxiv.org/abs/2604.25399
- **来源**：arXiv 2604.25399 (cs.SE), ACL'26 Findings
- **领域**：AI Testing / 代码推理 / 基准测试
- **技术介绍**：尽管 LLM 在代码生成任务上表现强劲，但是否真正推理代码执行仍不清楚。现有代码推理基准主要评估单一规范实现下的最终输出正确性，忽略了两个关键方面：(1) LLM 是否能对功能等价实现保持一致性；(2) LLM 是否能准确推理中间执行状态。CoRE 通过实现不变性和过程透明度评估代码推理。对 8 个前沿 LLM 的广泛评估揭示两个根本局限：模型在等价实现间存在显著的鲁棒性差距；观察到"表面执行"现象——模型到达正确最终输出但未正确推理中间执行状态。
- **应用场景**：代码推理质量评估、LLM 代码能力基准、推理过程透明度检测
- **为什么值得关注**：证明仅输出评估不足以评估代码推理，CoRE 是评估鲁棒和忠实代码推理的必要基准

#### 3. SAFEdit: Multi-Agent Framework for Instructed Code Editing
- **地址**：https://arxiv.org/abs/2604.25737
- **来源**：arXiv 2604.25737 (cs.SE/AI), EQUISA@EASE 2026
- **领域**：AI Testing / 多Agent / 代码编辑
- **技术介绍**：在 EditBench 基准上，40 个评估模型中 39 个任务成功率低于 60%。SAFEdit 将编辑过程分解为专业化角色：Planner Agent 生成显式、可见性感知的编辑计划，Editor Agent 应用最小字面代码修改，Verifier Agent 执行真实测试运行。测试失败时，SAFEdit 使用失败抽象层（FAL）将原始测试日志转换为结构化诊断反馈，反馈给 Editor 以支持迭代优化。SAFEdit 达到 68.6% TSR，超过单模型基线 3.8 个百分点，超过 ReAct 单 Agent 基线 8.6 个百分点。迭代优化循环贡献了 17.4 个百分点。
- **应用场景**：代码编辑质量保障、多Agent协作编码、指令驱动代码修改
- **为什么值得关注**：首次系统性地证明多Agent分解能显著提升代码编辑可靠性，减少指令级幻觉

#### 4. RESTestBench: REST API Test Cases from NL Requirements
- **地址**：https://arxiv.org/abs/2604.25862
- **来源**：arXiv 2604.25862 (cs.SE/AI), EASE 2026
- **领域**：AI Testing / API 测试 / 需求驱动测试
- **技术介绍**：现有 REST API 测试工具通常用代码覆盖率和崩溃故障指标评估。但 LLM 方法越来越多地从 NL 需求生成测试以验证功能行为，使传统指标成为弱代理。RESTestBench 包含三个 REST 服务配对手动验证的 NL 需求（精确和模糊变体），引入基于需求的变异测试指标衡量生成测试用例的故障检测有效性。结果显示当生成器与故障或变异代码交互时，测试有效性显著下降，尤其是模糊需求场景。
- **应用场景**：API 测试质量评估、需求驱动测试生成、LLM 测试工具有效性验证
- **为什么值得关注**：首个面向需求驱动 API 测试的基准，揭示 SUT 行为暴露对测试有效性的负面影响

#### 5. MARD: Multi-Agent Framework for Android Malware Detection
- **地址**：https://arxiv.org/abs/2604.25264
- **来源**：arXiv 2604.25264 (cs.CR/SE)
- **领域**：AI Testing / 安全测试 / 恶意软件检测
- **技术介绍**：传统 ML 检测模型受概念漂移困扰，且受限于浅层特征。MARD 将确定性分析引擎作为按需执行工具，LLM 编排整个决策过程。基于 ReAct 范式设计自主多Agent交互机制，构建高度可解释的证据链。单次复杂 APK 深度分析成本低于 $0.10。无需任何领域特定微调，F1 达到 93.46%，在跨度五年的评估中展现对概念漂移的鲁棒性和强跨域泛化能力。
- **应用场景**：Android 恶意软件检测、安全测试自动化、概念漂移场景下的持续检测
- **为什么值得关注**：首次将多Agent+ReAct范式应用于恶意软件检测，成本极低且无需微调

#### 6. Commit-Aware Test Case Prioritization for CI
- **地址**：https://arxiv.org/abs/2604.25363
- **来源**：arXiv 2604.25363 (cs.SE)
- **领域**：AI Testing / CI/CD / 测试优先级
- **技术介绍**：CI 管道中的回归测试成本日益增长。现有 TCP 技术主要依赖历史执行数据和覆盖率指标，忽视代码变更中的丰富结构信息。本文提出结合版本控制 diff 的结构属性、测试覆盖关系和历史执行行为的统一预测模型。在五个 Defects4J 项目上评估，使用留一项目交叉项目验证设置，结果显示 commit 感知 TCP 在分类和优先级排序有效性上显著优于非 commit 感知基线。
- **应用场景**：CI/CD 测试优化、回归测试优先级排序、变更感知测试调度
- **为什么值得关注**：证明 commit 结构语义能显著提升回归故障检测，实现鲁棒可泛化的 CI 环境 TCP

#### 7. Code Evaluation Metrics for Plagiarism Detection
- **地址**：https://arxiv.org/abs/2604.25778
- **来源**：arXiv 2604.25778 (cs.SE/AI/IR), LEARNER@EASE 2026
- **领域**：AI Testing / 代码抄袭检测 / 评估指标
- **技术介绍**：代码评估指标（CEMs）用于评估代码生成任务，但是否能可靠检测不同修改级别（L1-L6）的抄袭仍不清楚。使用 ConPlag 和 IRPlag 数据集评估五个 CEMs（CodeBLEU、CrystalBLEU、RUBY、TSED、CodeBERTScore），与 JPlag 和 Dolos 对比。结果显示无预处理时 Dolos 整体排名性能最高，CrystalBLEU/CodeBLEU/RUBY 优于 JPlag。有预处理时 CrystalBLEU 超越 Dolos。
- **应用场景**：学术诚信保障、代码抄袭检测、CEM 工具选型
- **为什么值得关注**：证明 CEMs 在排名指标上可与专用工具媲美，为代码生成评估工具复用提供依据

#### 8. Scenario-Based System Testing for Distributed Robotics
- **地址**：https://arxiv.org/abs/2604.25772
- **来源**：arXiv 2604.25772 (cs.SE)
- **领域**：AI Testing / 分布式系统测试 / 机器人
- **技术介绍**：SCSL（Scenario Specification Language）用于自动生成和执行系统级测试。针对复杂分布式系统（如协作自主机器人），其中经典模型测试因系统复杂性过高、测试行为无法完全预计算、SUT 配置动态变化而变得不切实际。SCSL 通过场景分解解决复杂性，支持在线测试执行，并提供协作构造支持动态重新配置。
- **应用场景**：分布式机器人系统测试、自主系统验证、动态配置场景测试
- **为什么值得关注**：首个面向分布式机器人场景的自动化系统测试语言，解决传统模型测试的三大挑战

#### 9. MultiVul: Multimodal Vulnerability Detection
- **地址**：https://arxiv.org/abs/2604.25711
- **来源**：arXiv 2604.25711 (cs.SE/AI)
- **领域**：AI Testing / 漏洞检测 / 多模态
- **技术介绍**：源代码和 accompanying 注释是互补且自然对齐的模态——代码编码结构逻辑，注释捕获开发者意图。MultiVul 通过双重相似性学习和一致性正则化对齐代码和注释表示，增强多样化代码-文本对以提高鲁棒性。在 DiverseVul 和 Devign 数据集上，四个 LLMs（DeepSeek-Coder-6.7B、Qwen2.5-Coder-7B、StarCoder2-7B、CodeLlama-7B）上达到比提示方法高 27.07% F1，比仅代码微调高 13.37% F1。
- **应用场景**：代码漏洞检测、多模态代码分析、安全测试
- **为什么值得关注**：首次利用代码-注释多模态互补性提升漏洞检测，推理效率可比

#### 10. PLM-GNN Hybrids for Code Classification
- **地址**：https://arxiv.org/abs/2604.25599
- **来源**：arXiv 2604.25599 (cs.SE/LG)
- **领域**：AI Testing / 代码分类 / 漏洞检测
- **技术介绍**：系统配对三种代码专用 PLM 与三种基础 GNN 架构，在 Java250 和 Devign 上对比 PLM-only、GNN-only 基线。混合模型始终优于 GNN-only 基线，在 Devign 上性能和鲁棒性对 PLM 特征源比 GNN 骨干更敏感。发现更大的 PLM 不一定是更好的特征提取器，PLM 选择比 GNN 选择影响更大。
- **应用场景**：代码分类、漏洞检测、PLM-GNN 架构设计
- **为什么值得关注**：为 PLM-GNN 混合架构提供实用设计指南，挑战"越大越好"的假设

#### 11. Carbon-Taxed Transformers: Green Compression Pipeline
- **地址**：https://arxiv.org/abs/2604.25903
- **来源**：arXiv 2604.25903 (cs.SE/LG)
- **领域**：AI / 模型压缩 / 绿色 AI
- **技术介绍**：LLM 在 SE 中的加速采用带来不可持续的计算成本危机。CTT 从经济碳定价概念出发，操作化计算碳税惩罚架构低效并奖励部署就绪压缩。在代码克隆检测、代码摘要和代码生成三个核心 SE 任务上评估，跨越编码器-only、编码器-解码器和解码器-only 架构。CTT 实现：(1) 最高 49x 内存减少，(2) 克隆检测时间减少 8-10x、摘要 3x、生成 4-7x，(3) CO2 排放减少 81%，(4) 克隆检测保留约 98% 准确率、摘要 89%、生成 91%（文本指标）和 68%（pass@1）。
- **应用场景**：绿色 AI、模型压缩、SE 任务高效部署
- **为什么值得关注**：首个将经济碳税原则应用于模型压缩的系统方法，在激进压缩下保持高性能

#### 12. SWE-MIMIC-Bench: Multi-LLM Pipeline for GitHub Issue Extraction
- **地址**：https://arxiv.org/abs/2604.25880
- **来源**：arXiv 2604.25880 (cs.SE)
- **领域**：AI / 知识提取 / 多LLM协作
- **技术介绍**：大规模 OSS 项目中复杂生产后问题的解决需要大量认知努力。SWE-MIMIC-Bench 使用自动化多 LLM 管道从原始 GitHub 讨论生成问题轨迹数据集。管道使用五个闭源 LLM 配置执行不同任务：标签分类、内联代码块和外部链接摘要、评论分析、标签特定字段分类和轨迹合成。在 800 个真实 GitHub 问题上达到 91.7% 成功率，提取 734 个高保真推理轨迹。
- **应用场景**：OSS 知识提取、Agent 训练数据生成、问题诊断自动化
- **为什么值得关注**：生成结构化推理轨迹用于训练现代 LLM Agent 像专家开发者一样思考和行动

#### 13. Intent Compilation: Closure Gaps for Open-World AI Agents
- **地址**：https://arxiv.org/abs/2604.25000
- **来源**：arXiv 2604.25000 (cs.AI/SE)
- **领域**：AI / Agent 理论 / 开放世界部署
- **技术介绍**：近期工作将智能框架为通过 learned 结构和测试时搜索减少解决时间，但无法解释为什么能力模型仍难以在开放机构中部署。提出意图编译：将部分指定的人类目的转换为可检查的绑定执行工件。相关部署区别是闭世界求解器 vs 开放世界 Agent。形式化剩余开放性为闭包差距向量，定义委托信封为动作空间的预授权区域，区分错误闭包与欠搜索。

- **为什么值得关注**：为开放世界 Agent 部署提供首个形式化理论框架，超越闭世界求解范式

#### 14. Enterprise Deep Research: Controlled Information Flow
- **地址**：https://arxiv.org/abs/2604.24978
- **来源**：arXiv 2604.24978 (cs.CL/SE), ACL Industry 2026
- **领域**：AI / 深度研究 / 企业应用
- **技术介绍**：企业深度研究常因信息覆盖不均、上下文爆炸和过早停止而失败。EDR 架构：(1) 通过大纲生成与反思将请求分解为覆盖驱动的目标，(2) 通过依赖引导执行和显式信息共享定位上下文，(3) 强制执行基于证据的完成标准使 Agent 迭代收集信息直到满足充分性条件。在内部销售赋能任务和公共 DeepResearch Bench 上达到最强整体性能。
- **应用场景**：企业研究自动化、深度信息检索、销售赋能
- **为什么值得关注**：依赖控制上下文和显式证据充分性标准减少过早停止，提升企业研究输出一致性

#### 15. Safety Drift After Fine-Tuning
- **地址**：https://arxiv.org/abs/2604.24902
- **来源**：arXiv 2604.24902 (cs.CY/SE)
- **领域**：AI / 安全对齐 / 微调
- **技术介绍**：基础模型通常为特定领域微调，但安全评估通常仅在基础模型上进行，隐含假设安全属性在下游适配中持续存在。分析 100 个模型（包括广泛部署的医疗和法律领域微调以及开放基础模型的受控适配），发现在通用和领域特定安全基准上，良性微调引发大的、异构的、经常矛盾的安全变化：模型在某些工具上改进同时在其他工具上退化，评估间存在实质性分歧。
- **应用场景**：AI 安全治理、微调后安全评估、高风险领域部署
- **为什么值得关注**：挑战当前以基础模型评估为中心的责任范式，证明安全行为在普通下游适配下不稳定

#### 16. Topology-faithful Dimensionality Reduction at Scale
- **地址**：https://arxiv.org/abs/2604.25209
- **来源**：arXiv 2604.25209 (cs.LG/AI/SE/SI)
- **领域**：AI / 降维 / 拓扑数据分析
- **技术介绍**：UMAP 和 t-SNE 等降维方法的局部邻域目标在保留采样噪声的同时扭曲全局拓扑。标准局部指标奖励这种噪声记忆：顶级嵌入发明数据中不存在的周期和断开岛屿。引入基于已知同调的噪声流形的拓扑忠实度基准，调整 DiRe 并找到 Pareto 最优配置在分类上匹配或击败 GPU 加速 UMAP，同时在压力测试上恢复精确的第一 Betti 数。在 723K arXiv 论文嵌入上，DiRe 在可比运行时间下保留比 UMAP 多 3-4 倍的拓扑结构。
- **应用场景**：高维数据可视化、拓扑数据分析、论文嵌入降维
- **为什么值得关注**：解决降维中全局拓扑保留的根本问题，在大规模论文嵌入上展现显著优势

#### 17. R³-SQL: Ranking Reward and Resampling for Text-to-SQL

#### 2. CoRE: Code Reasoning Benchmark Beyond Output Prediction

**地址**：https://arxiv.org/abs/2604.25399
**类型**：论文, ACL'26 Findings

**论文介绍**（200-400 字）：
CoRE 是一个代码推理基准，通过实现不变性和过程透明度评估 LLM 的代码推理能力。现有代码推理基准主要评估单一规范实现下的最终输出正确性，忽略了两个关键方面：(1) LLM 是否能对功能等价实现保持一致性；(2) LLM 是否能准确推理中间执行状态。

对 8 个前沿 LLM 的广泛评估揭示两个根本局限：模型在等价实现间存在显著的鲁棒性差距（性能变化显著）；观察到"表面执行"现象——模型到达正确最终输出但未正确推理中间执行状态。这些发现证明仅输出评估不足以评估代码推理，CoRE 是评估鲁棒和忠实代码推理的必要基准。

**技术架构**：
- 实现不变性测试（功能等价实现的性能一致性）
- 过程透明度评估（中间执行状态的正确性）
- 多语言支持

**核心算法**：
- 功能等价代码变换生成
- 中间执行状态追踪和验证
- 鲁棒性差距量化

**与现有技术对比**：
- 传统代码推理基准：仅评估最终输出
- CoRE：同时评估输出正确性和推理过程忠实性

**优势**：首次系统评估代码推理过程透明度、揭示"表面执行"现象
**局限**：功能等价变换的生成质量、中间状态验证的覆盖度

**发展潜力**：可能成为代码推理评估的新标准，推动 LLM 从"表面执行"向"深度推理"演进。

#### 3. Safety Drift After Fine-Tuning

**地址**：https://arxiv.org/abs/2604.24902
**类型**：论文

**论文介绍**（200-400 字）：
本文挑战了 AI 安全评估中的一个基本假设：安全属性在下游适配中持续存在。分析 100 个模型（包括广泛部署的医疗和法律领域微调以及开放基础模型的受控适配），发现在通用和领域特定安全基准上，良性微调引发大的、异构的、经常矛盾的安全变化。

模型在某些工具上改进同时在其他工具上退化，评估间存在实质性分歧。这些结果表明安全行为在普通下游适配下不稳定。在没有对部署相关上下文中的微调模型进行显式重新评估的情况下，当前以基础模型评估为中心的责任范式不足以管理下游风险，忽视了高风险设置中特别严重的实际危害来源。

**技术架构**：
- 100 模型安全行为横向对比
- 通用安全基准 + 领域特定安全基准
- 基础模型 vs 微调模型安全差异量化

**核心算法**：
- 安全行为变化矩阵构建
