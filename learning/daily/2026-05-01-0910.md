#### 1. Agentic AI in the SDLC: Architecture, Empirical Evidence, and Reshaping of SE
- **地址**：https://arxiv.org/abs/2604.26275
- **来源**：arXiv 2604.26275 (cs.SE)
- **领域**：AI Testing / Agentic SDLC / 综述
- **技术介绍**：合成 Anthropic、OpenAI、Google DeepMind、Microsoft Research、Princeton、Stanford 等工作，提出 Agentic 软件工程六层参考架构。对比传统 SDLC 与新兴 Agentic SDLC，汇总性能证据（SWE-bench Verified 从 2023 年 10 月 1.96% 到 2026 年 4 月 78.4%）、生产力证据（13.6%-55.8% 时间节省）和劳动力市场影响（Anthropic 2026 年抽样 49% 工作至少 25% 任务使用 AI）。核心论点：研究焦点已从代码生成转向人类监督下的委托执行。识别五个开放问题：评估、治理、技术债务、技能再分配、注意力经济学。
- **应用场景**：Agentic 开发架构设计、SDLC 转型参考、研究趋势把握
- **为什么值得关注**：最全面的 Agentic SDLC 综述，定义了从代码生成到委托执行的研究范式转变

#### 2. CoRE: Code Reasoning Benchmark Beyond Output Prediction
- **地址**：https://arxiv.org/abs/2604.25399
- **来源**：arXiv 2604.25399 (cs.SE), ACL'26 Findings
- **领域**：AI Testing / 代码推理评估 / 基准
- **技术介绍**：现有代码推理基准主要评估最终输出正确性，在单一规范实现下评估。CoRE 通过实现不变性（implementation invariance）和过程透明度（process transparency）评估代码推理。对 8 个前沿 LLM 的评估揭示两个根本局限：(1) 鲁棒性差距——模型在等价实现间性能差异显著；(2) 表面执行——模型得出正确最终输出但未正确推理中间执行状态。证明仅输出评估不足以评估代码推理。
- **应用场景**：代码推理能力基准测试、LLM 代码理解深度评估
- **为什么值得关注**：首次系统揭示 LLM 代码推理的"表面执行"问题，挑战现有评估范式

#### 3. SAFEdit: Multi-Agent Framework for Instructed Code Editing
- **地址**：https://arxiv.org/abs/2604.25737
- **来源**：arXiv 2604.25737 (cs.SE), EQUISA@EASE 2026
- **领域**：AI Testing / 多代理代码编辑 / EditBench
- **技术介绍**：EditBench 基准上 39/40 模型 TSR 低于 60%。SAFEdit 将编辑分解为专业化子代理：Planner Agent 生成可见性感知的编辑计划，Editor Agent 应用最小字面代码修改，Verifier Agent 执行真实测试运行。测试失败时，SAFEdit 使用 Failure Abstraction Layer (FAL) 将原始测试日志转换为结构化诊断反馈，反馈给 Editor 支持迭代优化。在 EditBench 5 种语言 445 个编辑实例上，SAFEdit 达到 68.6% TSR，超越单模型基线 3.8 个百分点，超越 ReAct 单代理基线 8.6 个百分点。迭代优化循环贡献 17.4 个百分点。
- **应用场景**：多代理代码编辑可靠性提升、指令驱动代码修改
- **为什么值得关注**：首次系统性地用多代理分解解决代码编辑可靠性问题，迭代优化贡献显著

#### 4. PICKLES: MBT + BDD Testing Framework
- **地址**：https://arxiv.org/abs/2604.26572
- **来源**：arXiv 2604.26572 (cs.SE)
- **领域**：Software Testing / MBT / BDD / 形式化测试
- **技术介绍**：结合模型基测试（MBT）和行为驱动开发（BDD）方法，定义人类可读规格和测试用例的测试方法。引入 PICKLES（Precise Input and Control-flow Keyword-based Language for tEst Scenarios），Gherkin 风格 BDD 场景的扩展，以结构化自然语言指定。提供从 Pickles 场景到形式化模型的双向翻译，以及获取组合所有翻译场景的主模型的方法。标准 MBT 算法可自动从中派生测试用例。在交通管理软件组件工业案例研究上，PICKLES 比 BDD 实现显著更高的覆盖率。
- **应用场景**：工业软件测试、形式化测试生成、BDD 增强
- **为什么值得关注**：首次将 MBT 的形式化能力与 BDD 的可读性结合，工业案例验证

#### 5. RepoDoc: Knowledge Graph-Driven Auto Documentation
- **地址**：https://arxiv.org/abs/2604.26523
- **来源**：arXiv 2604.26523 (cs.SE)
- **领域**：AI Testing / 文档生成 / 知识图谱
- **技术介绍**：使用仓库知识图谱（RepoKG）作为整个文档生命周期的语义基础。三阶段框架：(1) RepoKG 构建，提取代码实体及其关系；(2) 模块聚类，按功能内聚性分组代码；(3) 技能代理生成，查询图谱创建模块化、交叉引用的文档（含自动 Mermaid 图）。增量维护时，语义影响传播机制双向导航 RepoKG 精确定位所有受影响部分，实现选择性、针对性重新生成。在 8 种编程语言 24 个仓库上评估，API 覆盖率提升 32.5%，完整性提升 10.4%，文档生成速度提升 3 倍，token 减少 85%。增量更新节省 73% 时间和 77% token。
- **应用场景**：大规模代码库文档自动化、增量文档维护
- **为什么值得关注**：知识图谱驱动的文档生成，增量更新效率显著提升

#### 6. RESTestBench: REST API Test Benchmark from NL Requirements
- **地址**：https://arxiv.org/abs/2604.25862
- **来源**：arXiv 2604.25862 (cs.SE), EASE 2026
- **领域**：AI Testing / REST API 测试 / 自然语言需求
- **技术介绍**：现有 REST API 测试工具通常用代码覆盖率和崩溃基故障指标评估，但 LLM 方法越来越多地从自然语言需求生成测试。RESTestBench 包含三个 REST 服务，配对手动验证的自然语言需求（精确和模糊两种变体），支持受控和可重复的需求基测试生成评估。引入基于需求的变异测试指标，衡量生成测试用例对特定需求的故障检测有效性。评估非精炼生成和精炼生成（与运行 SUT 交互）两种方法。发现当生成器与有缺陷或变异代码交互时，测试有效性显著下降，尤其在模糊需求下。
- **应用场景**：REST API 测试基准、需求驱动测试生成评估
- **为什么值得关注**：首次系统评估自然语言需求驱动的 API 测试生成质量

#### 7. MultiVul: Multimodal Contrastive Vulnerability Detection
- **地址**：https://arxiv.org/abs/2604.25711
- **来源**：arXiv 2604.25711 (cs.SE)
- **领域**：Software Testing / 漏洞检测 / 多模态
- **技术介绍**：源代码和 accompanying 注释是互补但天然对齐的模态——代码编码结构逻辑，注释捕获开发者意图。MultiVul 通过双重相似性学习和一致性正则化对齐代码和注释表示，用多样化代码-文本对增强鲁棒性。在 DiverseVul 和 Devign 数据集上评估 4 个 LLM（DeepSeek-Coder-6.7B、Qwen2.5-Coder-7B、StarCoder2-7B、CodeLlama-7B），F1 比基于提示的方法提升 27.07%，比仅代码微调提升 13.37%，同时保持可比推理效率。
- **应用场景**：代码漏洞检测、多模态代码分析
- **为什么值得关注**：首次将代码+注释多模态对比学习用于漏洞检测

#### 8. SWE-MIMIC-Bench: Issue Trajectory Dataset from GitHub Discussions
- **地址**：https://arxiv.org/abs/2604.25880
- **来源**：arXiv 2604.25880 (cs.SE)
- **领域**：AI Testing / 问题轨迹 / 知识提取
- **技术介绍**：从原始 GitHub 讨论中用自动化多 LLM 流水线生成问题轨迹数据集。不同于简单摘要，流水线使用一组闭源 LLM 执行细粒度任务：分析单个评论（感知外部链接资源）、将评论分析分类为标签特定字段（根因、解决方案计划、实现进度）、合成标签感知轨迹。用 5 种闭源 LLM 配置用于不同目的：标签分类、内联代码块和外部链接摘要、评论分析、标签特定字段分类、轨迹合成。在 800 个真实 GitHub Issue 上评估，91.7% 成功率提取 734 个高保真推理轨迹。
- **应用场景**：Agent 训练数据生成、问题诊断知识提取
- **为什么值得关注**：为训练 LLM Agent 像专家开发者一样思考提供高质量轨迹数据

#### 9. Commit-Aware Test Case Prioritization for CI
- **地址**：https://arxiv.org/abs/2604.25363
- **来源**：arXiv 2604.25363 (cs.SE)
- **领域**：Software Testing / CI / 测试优先级
- **技术介绍**：CI 管道中回归测试因测试套件规模和执行频率增长而日益昂贵。测试用例优先级排序（TCP）通过重新排序测试以更早暴露故障来缓解此问题。本文提出结合版本控制 diff 结构属性、测试覆盖关系和历史执行行为的统一预测模型。在 5 个 Defects4J 项目上评估，使用留一项目交叉项目验证。发现包含 commit 结构语义显著提升回归故障检测。
- **应用场景**：CI 回归测试优化、测试优先级排序
- **为什么值得关注**：首次将 commit 结构语义系统性地融入学习基 TCP

#### 10. Code Evaluation Metrics for Plagiarism Detection
- **地址**：https://arxiv.org/abs/2604.25778
- **来源**：arXiv 2604.25778 (cs.SE), LEARNER 2026
- **领域**：Software Testing / 代码抄袭检测 / CEM
- **技术介绍**：评估 5 种代码评估指标（CodeBLEU、CrystalBLEU、RUBY、TSED、CodeBERTScore）在不同修改级别（L1-L6）的抄袭检测能力。与 SOTA 工具 JPlag 和 Dolos 对比。发现 CrystalBLEU、CodeBLEU 和 RUBY 超越 JPlag。CrystalBLEU 在 L6（最困难级别）保持竞争力。预处理后 CrystalBLEU 超越 Dolos 总体排名。
- **应用场景**：学术诚信、代码抄袭检测
- **为什么值得关注**：证明通用代码评估指标可媲美专用抄袭检测工具

#### 11. Bug-Report Fault Localization at ABB Robotics
- **地址**：https://arxiv.org/abs/2604.25700
- **来源**：arXiv 2604.25700 (cs.SE)
- **领域**：Software Testing / 故障定位 / 工业实证
- **技术介绍**：仅用自然语言 bug 报告内容进行故障定位。将问题建模为监督文本分类，评估 3 种传统 ML 模型（Logistic Regression、SVM、Random Forest）和 2 种微调 Transformer（RoBERTa-Base、Distil-RoBERTa）。使用 ABB Robotics 瑞典专有数据：5 年已解决工业 bug 报告。发现使用 TF-IDF 特征的传统模型一致超越微调语言模型，数据增强提升 Random Forest 性能。挑战 Transformer 模型在工业语境下普遍优于经典方法的假设。
- **应用场景**：工业故障定位、bug 报告分析
- **为什么值得关注**：工业实证挑战"Transformer 万能"假设

#### 12. LLM Black-Box Testing for FMU-Based Simulations
- **地址**：https://arxiv.org/abs/2604.25650
- **来源**：arXiv 2604.25650 (cs.SE)
- **领域**：Software Testing / 黑盒测试 / FMU 仿真
- **技术介绍**：提出人在回路的黑盒测试方法，用 LLM 生成功能模拟单元（FMU）的结构化场景目标（Given-When-Then 格式）。方法以 FMU 功能和接口规格为输入，提示 LLM 生成结构化场景目标，定义仿真初始输入条件、可能的条件变化和系统对变化的期望输出行为。在润滑油冷却系统上评估。
- **应用场景**：动态仿真模型测试、工业仿真验证
- **为什么值得关注**：将 LLM 能力扩展到工业仿真测试领域

#### 1. Warp: Agentic Development Environment

1. 终端即 Agentic 平台——不是插件而是原生设计
2. OpenAI 作为创始赞助商，GPT 模型驱动代理工作流
3. 实时可视化代理工作流（build.warp.dev）
4. 混合代理模式（内置 + 自带）

**与现有技术对比**：

#### 2. Superpowers: Agentic Skills Framework

**地址**：https://github.com/obra/superpowers

**项目介绍（300 字）**：
Superpowers 是一个完整的软件开发方法论，构建在可组合技能集之上，专为编码代理设计。它从启动编码代理的那一刻开始介入——不是让代理直接写代码，而是先问用户真正想做什么。在提炼出规格后，分块展示设计供用户验证。批准后生成实施计划（每个任务 2-5 分钟），然后启动子代理驱动开发流程。

Superpowers 的核心创新在于它将软件开发方法论（TDD、YAGNI、DRY）直接编码为代理技能，而非依赖提示工程。每个技能都是自动触发的强制性工作流，不是建议。系统支持 Claude Code、Codex、Cursor、Copilot、Gemini 等多平台。

技能集包括 7 个核心技能：brainstorming（启动前需求提炼）、git worktrees（隔离工作空间）、writing-plans（任务分解）、subagent-driven-development（子代理执行）、test-driven-development（RED-GREEN-REFACTOR）、requesting-code-review（任务间审查）、finishing-a-development-branch（分支完成）。

**技术架构**：
- 技能触发：自动检测任务类型，触发对应技能
- 子代理调度：每个任务分配独立子代理，两阶段审查
- 平台适配：Claude Code / Codex / Cursor / Copilot / Gemini
- 许可：开源（具体许可待确认）

**核心创新**：
1. 方法论即技能——TDD/YAGNI/DRY 编码为代理工作流
2. 强制性技能触发——不是建议而是必须执行
3. 子代理驱动开发——每个任务独立代理 + 两阶段审查
4. 跨平台兼容——支持 5 个主流编码代理平台

**优势与局限**：
- 优势：方法论系统化、跨平台、子代理隔离、TDD 强制执行
- 局限：技能质量依赖底层代理能力、复杂项目可能 token 消耗大

**个人技术理解**：
Superpowers 代表 Agentic 开发的"方法论工程化"趋势——不再依赖提示技巧，而是将软件开发最佳实践编码为可组合、可触发、可验证的技能模块。这是对 mattpocock/skills 生态的重要补充和深化。

CUA（Computer-Use Agent）是开源 Computer-Use Agent 基础设施，提供沙箱、SDK 和基准，用于训练和评估能控制完整桌面（macOS、Linux、Windows、Android）的 AI 代理。核心卖点：统一 API 覆盖 5 个平台 + 6 种运行时。

cuabot 为任何编码代理提供无缝沙箱——独立窗口原生显示在桌面，H.265 视频流、共享剪贴板、音频。支持 Claude Code、OpenClaw 等代理在沙箱中运行 GUI 工作流。cua-bench 支持 OSWorld、ScreenSpot、Windows Arena 等基准评估，导出轨迹用于训练。lume 提供 Apple Silicon 上近原生性能的 macOS/Linux VM 管理。

技术栈包括 6 个包：cuabot（多代理沙箱 CLI）、cua-agent（AI 代理框架）、cua-sandbox（SDK）、cua-computer-server（UI 交互驱动）、cua-bench（基准/RL 环境）、lume（VM 管理）。

**技术架构**：
- 跨平台支持：Linux 容器/VM、macOS、Windows、Android、BYOI
- 视频传输：H.265 低延迟流式传输
- 输入模拟：鼠标/键盘/多触手势
- 基准评估：OSWorld/ScreenSpot/Windows Arena
- VM 管理：Apple Virtualization.Framework

**核心创新**：
1. 统一 API 覆盖 5 平台——同一 Python API 控制任何 OS
2. 非 AX 表面支持——Chromium/Blender/Figma/DAW 等
3. 背景运行——不抢占光标/焦点/Space
4. 可回放轨迹——每个会话记录为训练数据

**优势与局限**：
- 优势：跨平台统一、非 AX 表面支持、基准评估、轨迹回放
- 局限：本地 VM 需要 Apple Silicon、Windows 支持仍在完善

**个人技术理解**：
CUA 是 Computer-Use Agent 领域的"Kubernetes"——提供标准化的基础设施层，让上层代理开发聚焦于智能而非平台适配。统一 API 设计是核心差异化。

### 学术论文

#### 1. CoRE: Code Reasoning Benchmark Beyond Output Prediction

**地址**：https://arxiv.org/abs/2604.25399

**论文介绍（300 字）**：
CoRE（Code Reasoning）基准挑战现有代码推理评估的根基。现有基准（如 HumanEval、MBPP）只评估最终输出正确性，在单一规范实现下评估 LLM 的代码推理能力。CoRE 揭示这是不够的——模型可能在得出正确输出的同时完全误解了执行过程。

CoRE 通过两个维度评估代码推理：(1) 实现不变性——模型是否对功能等价的不同实现保持一致推理？(2) 过程透明度——模型是否能准确推理中间执行状态？对 8 个前沿 LLM 的评估揭示两个根本局限：

第一，鲁棒性差距——模型在等价实现间性能差异显著。同一逻辑用不同变量名或控制结构实现，模型推理结果可能完全不同。

第二，表面执行——模型得出正确最终输出但未正确推理中间执行状态。这类似于学生猜对答案但不会解题过程。

CoRE 的数据和代码已开源，定位为评估鲁棒和忠实代码推理的必要基准。ACL'26 Findings 接收。

**技术架构**：
- 实现不变性测试：同一逻辑的多种等价实现
- 过程透明度测试：中间执行状态推理
- 8 个前沿 LLM 评估：GPT-4/Claude/Gemini 等

**核心算法**：
1. 等价实现生成——自动生成功能等价但结构不同的代码变体
2. 中间状态标注——人工标注关键中间执行状态
3. 鲁棒性差距量化——测量模型在不同实现间的一致性

**与现有技术对比**：
| 基准 | 评估维度 | 实现不变性 | 过程透明度 |
|------|---------|-----------|-----------|
| HumanEval | 输出正确性 | ❌ | ❌ |
| MBPP | 输出正确性 | ❌ | ❌ |
| CoRE | 输出+过程 | ✅ | ✅ |

**优势与局限**：
- 优势：首次系统评估过程透明度、揭示表面执行问题、开源数据
- 局限：中间状态标注成本高、仅覆盖 Python

**未来发展潜力**：
CoRE 可能成为代码推理评估的新标准，推动从"输出正确"到"推理忠实"的评估范式转变。

**个人技术理解**：
CoRE 的发现对 Agentic 代码生成有深远影响——如果模型"表面执行"，那么依赖模型代码推理的代理可能在复杂场景下产生系统性错误。这解释了为什么 SWE-Agent 在复杂 Issue 上表现不稳定。

#### 2. SAFEdit: Multi-Agent Framework for Instructed Code Editing

**地址**：https://arxiv.org/abs/2604.25737

**论文介绍（300 字）**：
SAFEdit 解决 instructed code editing（指令驱动代码编辑）的可靠性挑战。EditBench 基准上 39/40 模型 TSR 低于 60%，揭示通用代码生成与指令驱动编辑之间的巨大差距。

SAFEdit 的核心创新是将编辑过程分解为三个专业化子代理：
1. Planner Agent——生成显式的、可见性感知的编辑计划
2. Editor Agent——应用最小、字面的代码修改
3. Verifier Agent——执行真实测试运行

关键机制是 Failure Abstraction Layer (FAL)——当测试失败时，FAL 将原始测试日志转换为结构化诊断反馈，反馈给 Editor 支持迭代优化。这不是简单的"重试"，而是有结构的错误理解和修复。

在 EditBench 5 种语言（英语、波兰语、西班牙语、中文、俄语）445 个编辑实例上评估。SAFEdit 达到 68.6% TSR，超越单模型基线 3.8 个百分点，超越 ReAct 单代理基线 8.6 个百分点。迭代优化循环贡献 17.4 个百分点——这是最大的单一贡献因素。

EQUISA@EASE 2026 接收。

**技术架构**：
- 三代理分解：Planner → Editor → Verifier
- FAL：Failure Abstraction Layer（结构化错误诊断）
- 迭代优化：测试失败 → FAL 分析 → Editor 修复 → 重新验证

**核心算法**：
1. 可见性感知规划——Planner 理解代码结构和编辑影响范围
2. 最小修改原则——Editor 只做必要的字面修改
3. FAL 错误抽象——将原始测试日志转换为结构化反馈
4. 迭代优化循环——最多 N 轮测试-修复循环

**优势与局限**：
- 优势：多代理分解有效、FAL 结构化工、迭代优化贡献显著、多语言支持
- 局限：token 消耗增加、编辑计划质量依赖 Planner 能力

**个人技术理解**：
SAFEdit 的 17.4 个百分点迭代优化贡献是关键发现——它证明错误理解和修复比初始规划更重要。这对 Agentic 代码编辑的设计有直接指导意义：投资错误处理比投资规划能力回报更高。

#### 3. PICKLES: MBT + BDD Testing Framework

**地址**：https://arxiv.org/abs/2604.26572

**论文介绍（250 字）**：
PICKLES 结合模型基测试（MBT）和行为驱动开发（BDD），定义人类可读规格和测试用例的测试方法。引入 PICKLES 语言——Gherkin 风格 BDD 场景的扩展，以结构化自然语言指定。提供从 Pickles 场景到形式化模型的双向翻译，以及组合所有翻译场景的主模型方法。标准 MBT 算法可自动从中派生测试用例。在交通管理软件组件工业案例研究上，PICKLES 比 BDD 实现显著更高的覆盖率。

**核心创新**：
1. Gherkin → 形式化模型双向翻译
2. 主模型组合——多场景合并为统一模型
3. 工业验证——交通管理系统案例

**个人技术理解**：
PICKLES 填补了 BDD 可读性和 MBT 自动化之间的空白。对于需要形式化保证但团队习惯 BDD 工作流的工业场景，这是实用方案。
