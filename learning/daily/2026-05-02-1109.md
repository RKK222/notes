#### 1. PuzzleMark: Implicit Jigsaw Learning for Robust Code Dataset Watermarking
- **地址**：https://arxiv.org/abs/2604.27677
- **来源**：arXiv cs.SE, FSE 2026
- **领域**：Software Testing / 数据集安全
- **技术介绍**：提出 PuzzleMark，一种鲁棒的代码数据集水印方法。传统共现模式水印易受检测和移除攻击。PuzzleMark 引入载体选择策略，利用代码复杂度评估代码片段作为水印载体的适用性，选择高适用性片段进行水印。提出新颖的拼接模式替代传统共现模式，通过变量名拼接实现两种水印策略。自适应嵌入基于代码固有特征，更隐蔽且设计简单。使用 Fisher 精确测试在黑盒设置下验证可疑模型。100% 验证成功率，0% 误报率，对模型性能影响可忽略。人类研究和 4 种 SOTA 水印检测方法评估显示强不可感知性，平均可疑率 ≤0.24，平均召回率 ≤30.41%。
- **应用场景**：代码数据集知识产权保护、模型训练溯源
- **为什么值得关注**：FSE 2026 录用，代码数据集 IP 保护从理论走向实践

#### 2. ScaleBox: Enabling High-Fidelity and Scalable Code Verification for LLMs
- **地址**：https://arxiv.org/abs/2604.27467
- **来源**：arXiv cs.SE, ACL 2026 Demo
- **GitHub**：https://github.com/icip-cas/ScaleBox
- **领域**：AI Testing / 代码验证基础设施
- **技术介绍**：代码沙箱是推进 LLM 编码能力的关键基础设施，为 RL 训练和评估提供可验证反馈。现有系统在高质量验证和高并发工作负载下效率不足。ScaleBox 引入自动化特殊裁判生成和管理、测试用例级别的细粒度并行执行（无缝多节点协调）、配置驱动的评估套件用于可复现基准测试。实验表明 ScaleBox 显著提升代码验证准确性和效率。RLVR 实验显示在 LiveCodeBench 上性能大幅提升，训练稳定性显著优于启发式匹配基线。
- **应用场景**：大规模代码训练验证、RL 训练基础设施
- **为什么值得关注**：ACL 2026 Demo 录用，填补高并发代码验证基础设施空白

#### 3. Tail-aware N-version ML Models for Reliable API Recommendation (NvRec)
- **地址**：https://arxiv.org/abs/2604.27647
- **来源**：arXiv cs.SE, EASE 2026
- **领域**：AI Testing / API 推荐可靠性
- **技术介绍**：ML 基 API 推荐中代码数据集的长尾分布导致不可靠推荐，尤其对尾部低频 API 方法。NvRec 利用 N 个不同版本的 ML 模型增强 API 序列推荐的可靠性，抑制涉及尾部 API 的不可靠输出。使用 5 个 API 推荐模型（CodeBERT, CodeT5, MulaRec, UniXcoder, CodeT5+）实现，对每个 API 方法的性能和尾部属性进行画像。三版本组合（CodeT5 + MulaRec + UniXcoder）在限制高度可靠候选的多数投票下实现 83.8% 真实接受率和 80.7% 拒绝率。五版本配置在简单多数投票下实现 83.1% 真实接受率和 69.0% 拒绝率，提供更好的平衡。
- **应用场景**：API 推荐系统可靠性保障、长尾问题缓解
- **为什么值得关注**：N 版本编程思想在 ML 时代的复兴，解决长尾分布问题

#### 4. Understanding Bugs in Template Engine-Based Applications
- **地址**：https://arxiv.org/abs/2604.27692
- **来源**：arXiv cs.SE (submitted to TOSEM)
- **领域**：Software Testing / 模板引擎 Bug 分析
- **技术介绍**：首个全面的模板引擎应用 Bug 研究。分析 15 个模板引擎（5 种编程语言）的 1,004 个应用 Bug，识别症状和根因及常见修复模式。异常渲染结果（意外或空白输出）是最普遍症状（48.61%），通常表现为难以诊断的静默失败。识别 17 种根因，语法误用、数据上下文不匹配、不兼容集成为主导类别。67.92% 的 Bug 在模板内修复，但超过 20% 需要修改宿主侧逻辑以解决数据上下文问题。为 Jinja 引擎开发两个原型工具辅助开发和调试。
- **应用场景**：模板引擎应用质量保证、IaC 和数据工程
- **为什么值得关注**：模板引擎在 IaC 和数据工程中广泛应用但 Bug 研究几乎空白

#### 12. LLMs as ASP Programmers: Self-Correction Enables Nonmonotonic Reasoning
- **地址**：https://arxiv.org/abs/2604.27960
- **来源**：arXiv cs.AI
- **领域**：AI / 非单调推理
- **技术介绍**：现有神经符号方法依赖单调逻辑（如 SMT），无法表示可废止推理——人类认知的关键组成部分。LLM+ASP 将自然语言翻译为答案集编程（ASP），一种基于稳定模型语义的非单调形式化。无需每任务工程，统一应用于多样推理任务。自动化自我纠错循环中 ASP 求解器的结构化反馈实现迭代精炼。6 个多样基准评估：(1) 稳定模型语义允许 LLM 自然表达默认规则和例外，在非单调任务上显著优于 SMT；(2) 迭代自我纠错是性能的主要驱动力；(3) 紧凑的上下文参考指南显著优于冗长文档，揭示"context rot"现象。
- **应用场景**：复杂推理任务、可废止推理
- **为什么值得关注**：填补 SMT 单调逻辑无法表达可废止推理的空白

#### 15. Synthetic Computers at Scale for Long-Horizon Productivity Simulation
- **地址**：https://arxiv.org/abs/2604.28181
- **来源**：arXiv cs.AI
- **领域**：AI / 合成数据
- **技术介绍**：现实长期生产力工作强烈依赖于用户特定计算机环境。提出可扩展方法论创建具有真实文件夹层次结构和内容丰富的工件（文档、电子表格、演示文稿）的合成环境。在每个合成计算机上运行长期仿真：一个 Agent 创建特定于计算机用户的生产力目标（需要多个专业交付物和约一个月人类工作），另一个 Agent 作为用户在计算机上持续工作（导航文件系统、协调模拟协作者、生成专业工件）。1,000 个合成计算机，每次运行超过 8 小时 Agent 运行时和 2,000+ 轮次。仿真产生丰富的经验学习信号，在域内和域外生产力评估上显著提升 Agent 性能。
- **应用场景**：Agent 自我改进、Agentic 强化学习
- **为什么值得关注**：可扩展合成计算机创建作为 Agent 自我改进的基础底物

现有神经符号方法依赖单调逻辑（如 SMT），无法表示可废止推理——人类认知的关键组成部分。LLM+ASP 将自然语言翻译为答案集编程（ASP），一种基于稳定模型语义的非单调形式化。无需每任务工程，统一应用于多样推理任务。自动化自我纠错循环中 ASP 求解器的结构化反馈实现迭代精炼。6 个多样基准评估显示：稳定模型语义允许 LLM 自然表达默认规则和例外，在非单调任务上显著优于 SMT；迭代自我纠错是性能的主要驱动力；紧凑的上下文参考指南显著优于冗长文档，揭示"context rot"现象。

**技术架构**：
- NL→ASP 翻译器：自然语言到 ASP 程序的自动转换
- 自我纠错循环：ASP 求解器反馈驱动迭代精炼
- 稳定模型语义：非单调推理的形式化基础

**核心算法/系统设计**：
- NL→ASP 翻译：基于 LLM 的结构化翻译
- 自我纠错：ASP 求解器反馈 → LLM 修正 → 再求解
- Context rot 发现：冗长文档降低约束遵循

**与现有技术对比**：
- LLM+SMT：单调逻辑，无法表达可废止推理
- LLM+ASP：非单调逻辑，自我纠错驱动

**优势与局限**：
- 优势：无需每任务工程，统一应用于多样推理任务
- 局限：ASP 求解器性能可能成为瓶颈

**未来发展潜力**：
- 扩展到更多非单调推理场景
- 与 LLM 工具调用深度集成
- 支持实时自我纠错

**个人技术理解**：
LLM+ASP 填补了神经符号推理中非单调推理的关键空白。自我纠错作为主要性能驱动力的发现，为神经符号方法提供了新的设计思路——与其依赖大量领域知识，不如依赖迭代纠错。
