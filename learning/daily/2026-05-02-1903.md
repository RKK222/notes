#### 1. Test Before You Deploy: Governing Updates in the LLM Supply Chain
- **地址**：https://arxiv.org/abs/2604.27789
- **来源**：arXiv cs.SE, LLMSC 2026 (FSE 2026 工作坊)
- **领域**：AI Testing / LLM 供应链治理
- **技术介绍**：LLM 作为软件系统的核心依赖，其提供商侧的静默更新会导致行为漂移——功能回归、格式变化、安全约束失效等。现有方法聚焦回归测试或版本控制，但缺乏部署侧的兼容性治理机制。本文提出基于三个组件的部署侧治理框架：明确定义模型行为规则（生产契约）、按部署风险类别组织的聚焦测试（风险类别测试套件）、以及除非满足定义的安全和性能标准否则阻止更新的发布检查点（兼容性门控）。通过多 LLM 版本的探索性验证，证明在特定风险区域的靶向测试可以发现整体指标遗漏的性能回归。
- **应用场景**：LLM 驱动应用的部署治理、模型版本管理
- **为什么值得关注**：首次将 LLM 更新管理框架化为软件供应链治理问题，提出 deployer-side 兼容性控制研究议程

#### 2. The Grand Software Supply Chain of AI Systems
- **地址**：https://arxiv.org/abs/2604.27781
- **来源**：arXiv cs.SE
- **领域**：AI Testing / AI 供应链安全
- **技术介绍**：AI 系统建立在低完整性机制的软件之上，从数据获取到最终推理的每个阶段都暴露风险。本文将 AI 供应链作为一等分析对象，分解为四个架构层：数据获取、模型训练、模型推理和交叉基底。在这些层中识别出传统供应链机制无法解决的四个结构性差距：可验证性、版本控制、可观测性和可追溯性。AI 系统在所有四个方面都存在不足：携带未声明的行为耦合、无法回退到已知工作组件、静默降级而非暴露破坏性变化、谱系几乎无法近似。通过测量 48 个生产级开源项目的参考栈进行说明——声明 4,664 个直接依赖，解析为 11,508 个传递包，总计约 3.92 亿行代码。
- **应用场景**：AI 系统供应链安全分析、依赖治理
- **为什么值得关注**：量化揭示了 AI 系统供应链的惊人规模——3.92 亿行代码的传递依赖，为 AI 供应链安全提供了第一手数据

#### 3. CI-Repair-Bench: Repository-Aware Benchmark for Automated Patch Validation via CI Workflows
- **地址**：https://arxiv.org/abs/2604.27148
- **来源**：arXiv cs.SE
- **领域**：Software Testing / CI 修复基准
- **技术介绍**：CI 通过多阶段工作流强制执行仓库级正确性，但诊断和修复 CI 失败仍然具有挑战性。与传统的程序修复不同，CI 失败经常涉及非代码工件、环境和依赖问题、嘈杂的执行日志和工作流级约束。现有程序修复基准在此场景下存在不足：主要面向测试、将修复限制在源代码、假设固定执行环境、在简化的 CI 工作流下评估。CI-Repair-Bench 从真实 GitHub Actions 执行构建，包含 103 个仓库的 567 个 CI 失败实例，仅通过原始工作流下的完整 CI 重新执行来评估修复正确性。失败分为 12 种 CI 错误类型。实证结果表明，自动修复对格式化和 linting 等局部工具强制执行失败最有效，而环境、依赖和配置相关失败仍然具有挑战性——最佳 LLM 仅实现 18.9% 修复成功率。
- **应用场景**：CI/CD 自动化修复、仓库级程序修复评估
- **为什么值得关注**：首个 CI 原生的自动化程序修复基准，揭示了 CI 修复的真实难度

#### 4. emote: Effective Modular Testing with EvoSuite
- **地址**：https://arxiv.org/abs/2604.27112
- **来源**：arXiv cs.SE
- **领域**：Software Testing / 模块化测试
- **技术介绍**：模块化测试涉及隔离测试程序的各个组件。通常，为了有效生成测试，必须包含一系列非目标设置调用以获得目标组件的高覆盖率。本文评估和改进 EvoSuite 测试生成器的模块化测试。发现由于禁止调用非目标设置方法的严格限制，EvoSuite 的模块化测试模式效率低下，通常导致低分支覆盖率。提出 emote 增强 EvoSuite，放宽此限制，允许在测试前缀中包含非目标方法。此修改从开发者编写的模糊驱动中获得灵感——这些驱动通常在测试目标方法之前调用设置方法来正确初始化状态。修改 EvoSuite 的适应度函数以将分支覆盖率贡献集中在源自目标方法的调用链上。在 SF100 基准的子集上评估，显示目标方法覆盖率提高 15.15%。
- **应用场景**：Java 面向对象程序的模块化单元测试生成
- **为什么值得关注**：经典测试生成工具的实用增强，15% 覆盖率提升在工业级基准上显著

#### 5. PhantomRun: Reproducing Build Failures in Embedded Open Source Software
- **地址**：https://arxiv.org/abs/2604.27075
- **来源**：arXiv cs.SE, EASE 2026
- **领域**：Software Testing / CI 构建复现
- **技术介绍**：由于嵌入式系统中的硬件-软件协同开发，CI 构建经常因复杂的交叉编译、板配置和工具链约束而失败。虽然 CI 构建日志包含有价值的诊断信息，但由于异构运行器、工具链和日志格式，它们寿命短且难以重用。PhantomRun 提供统一的抽象层和公开可重用的数据集，标准化 CI 构建日志和元数据的检索、存储和复现。在 4,628 个失败的 CI 运行中，重建了 91.8% 的构建并在 98% 的评估案例中保留了执行结果。PhantomRun 提供两个核心能力：检索任何提交的构建日志和在受控环境中忠实重新执行相应的构建。通过将所有构建工件和元数据以统一的机器可读格式暴露，实现 CI 失败的可复现和纵向研究。
- **应用场景**：嵌入式系统 CI 失败诊断、构建历史重建
- **为什么值得关注**：EASE 2026 录用，91.8% 构建重建率解决了嵌入式 CI 诊断的核心痛点

#### 6. Adaptive and AI-Augmented Security Testing: A Systematic Survey
- **地址**：https://arxiv.org/abs/2604.27000
- **来源**：arXiv cs.SE/cs.CR
- **领域**：Software Testing / 安全测试
- **技术介绍**：现代软件系统在快速 CI/CD 管道中开发，在发布前确保安全面临技术和组织挑战。传统静态和动态分析工具提供有价值的结构和行为洞察，但通常在非自适应工作流中运行并产生大量需要手动分类的警告。反馈驱动模糊测试和基于搜索的测试方法展示了由执行信号引导的迭代输入精炼的力量，而 LLM 在自动测试生成中显示潜力但经常缺乏程序结构的语义基础。本文对五个领域的自适应和 AI 增强安全测试研究进行系统综述：(1) 漏洞检测的结构程序分析、(2) DevSecOps 和持续安全测试、(3) 反馈驱动模糊测试和基于搜索的测试、(4) LLM 自动测试生成、(5) 整合程序分析与自适应学习的新兴混合系统。分析 55 项同行评审研究，揭示结构性程序表示（AST、CFG、CPG）和自适应测试机制之间的持续脱节——表征为"结构-自适应碎片化"。
- **应用场景**：安全测试框架设计、DevSecOps 实践
- **为什么值得关注**：系统性综述识别出"结构-自适应碎片化"这一关键研究空白

#### 7. Sliceformer: Static Program Slicing with Dataflow-Aware Pretraining
- **地址**：https://arxiv.org/abs/2604.26961
- **来源**：arXiv cs.SE, ACL 2026 录用
- **领域**：Software Testing / 程序分析
- **技术介绍**：静态程序切片是隔离与特定变量相关代码的基础软件工程 technique。最近使用 LM 的学习方法在自动化切片预测中显示希望，但遭受不准确的依赖建模和无约束生成的困扰——LM 无法捕获精确的数据流关系并产生包含幻觉 token 和语句的切片。Sliceformer 将静态程序切片重新定义为使用小型语言模型（如 CodeT5+）的序列到序列任务。引入两个关键创新：首先，设计数据流感知预训练目标，利用数据流图（DFG）通过数据流保持语句排列和数据流感知跨度损坏来教授模型数据依赖；其次，开发约束解码机制强制执行词法和语法约束。在 Java 和 Python 程序切片基准上评估，比 SOTA 基线最多提升 22% 的 ExactMatch。
- **应用场景**：程序切片、代码理解、调试辅助
- **为什么值得关注**：ACL 2026 录用，首次将 DFG 引入 LM 预训练目标解决程序切片问题

#### 8. LLM-as-a-Judge for Human-AI Co-Creation: Reliability-Aware Evaluation Framework
- **地址**：https://arxiv.org/abs/2604.27727
- **来源**：arXiv cs.SE
- **领域**：AI Testing / LLM 评估
- **技术介绍**：LLM 越来越多地被用作评估开放式输出的评判者和 AI 辅助编程中的共创伙伴；但人类-AI 共创环境中的严格评估仍然不足——评判必须可靠、跨模型可比、并在多轮交互中可解释。提出基于 rubric 的 LLM-as-a-Judge 框架用于竞赛风格的人类-AI 共创编码和 SE。框架围绕 schema 约束的评判输出、验证和修复机制、按用户和问题分组以防止轨迹泄漏、以及参与者级 NONBLIND 上下文构建。通过多指标协议评估多个 LLM 评判者——涵盖区分度（ROC-AUC、PR-AUC）、阈值决策质量（MCC）、概率可靠性（LogLoss、Brier score、ECE）和评判者间一致性（Cohen's 和 Fleiss' κ）。人类-AI 共创通过轨迹级信号进一步检查——包括轮次置信度、Success-at-Turn、time-to-success、修订 churn 和 CodeBLEU。
- **应用场景**：AI 辅助编程评估、人类-AI 协作质量度量
- **为什么值得关注**：首次将可靠性感知 LLM 评判与轨迹分析结合，提供可审计和可复现的评估方法

#### 9. How Code Representation Shapes False-Positive Dynamics in Cross-Language LLM Vulnerability Detection
- **地址**：https://arxiv.org/abs/2604.27714
- **来源**：arXiv cs.CR/cs.SE
- **领域**：Software Testing / 漏洞检测
- **技术介绍**：代码表示格式如何影响跨语言 LLM 漏洞检测中的误报行为仍然知之甚少。系统变化训练强度和代码表示格式，比较原始源文本与修剪的 AST，在训练和推理时跨两个 8B 参数 LLM（Qwen3-8B 和 Llama 3.1-8B-Instruct），在 NIST Juliet 测试套件（v1.3）的 C/C++ 数据上微调，并在 Java（OWASP Benchmark v1.2）和 Python（BenchmarkPython v0.1）上评估。跨语言 FPR 反映训练时和推理时表示的联合效应，而非单独任何一个。文本微调驱动 FPR 单调上升（Qwen3-8B：零样本 0.763、试点 0.866、全规模 1.000），而 F1 保持稳定（0.637-0.688），掩盖了崩溃。提出表面线索记忆是主要机制：文本微调将 C/C++ 特定 API 名称和语法习惯编码为漏洞触发器，在目标语言代码上无差别触发。跨表示探针——将文本训练的权重应用于 AST 编码输入而不重新训练——隔离了这一点：Qwen3-8B FPR 从 0.866 降至 0.583。
- **应用场景**：跨语言漏洞检测、LLM 安全测试
- **为什么值得关注**：揭示了文本微调导致 FPR 崩溃的关键机制，提出无需重新训练的 AST 探针过滤方案

#### 10. An Empirical Security Evaluation of LLM-Generated Cryptographic Rust Code
- **地址**：https://arxiv.org/abs/2604.27001
- **来源**：arXiv cs.CR/cs.SE, EASE 2026
- **领域**：Software Testing / 密码学代码安全
- **技术介绍**：开发者和组织越来越多地使用 LLM 生成安全关键代码，包括密码学解决方案。本研究对 240 个 Rust 代码样本（两种加密算法：AES-256-GCM 和 ChaCha20-Poly1305）由三个 LLM（Gemini 2.5 Pro、GPT-4o、DeepSeek Coder）使用四种不同提示策略生成进行实证评估。对于每个成功编译的代码样本，使用 CodeQL 静态分析和基于规则的密码学特定分析器检测漏洞。评估结果显示仅 23.3% 的生成代码样本成功编译。在编译的代码中，CodeQL 仅产生两个误报，而基于规则的密码学特定分析器在 57% 的编译样本中识别出漏洞且零误报。两种算法的编译成功率差异显著（AES-256-GCM 34.2% vs ChaCha20-Poly1305 12.5%）。提示策略显著影响结果（P=0.002），思维链提示比零样本差 5 倍。所有三个模型都表现出系统性失败，包括 nonce 重用和 API 幻觉。
- **应用场景**：LLM 生成密码学代码安全评估
- **为什么值得关注**：57% 的编译代码存在密码学漏洞，思维链提示反而更差——反直觉发现

#### 11. When Your LLM Reaches End-of-Life: Framework for Confident Model Migration
- **地址**：https://arxiv.org/abs/2604.27082
- **来源**：arXiv cs.AI/cs.SE
- **领域**：AI Testing / 模型迁移
- **技术介绍**：提出当底层 LLM 达到 EOL 或需要替换时迁移生产 LLM 系统的框架。关键贡献是贝叶斯统计方法，将自动评估指标针对人类判断进行校准，即使在有限的人工评估数据下也能实现自信的模型比较。在服务于 530 万月交互的商业问答系统上演示该框架，跨六个全球区域评估正确性、拒绝行为和风格遵守以成功识别合适的替换模型。框架广泛适用于任何部署 LLM 产品的企业，提供平衡质量保证和评估效率的原则性、可复现的模型迁移方法。
- **应用场景**：生产环境 LLM 替换、模型版本迁移
- **为什么值得关注**：首个针对 LLM EOL 迁移的系统化框架，530 万月交互规模的实证

#### 12. UCSC-NLP at SemEval-2026 Task 13: Machine-Generated Code Detection
- **地址**：https://arxiv.org/abs/2604.26990
- **来源**：arXiv cs.SE, SemEval-2026
- **领域**：Software Testing / 代码检测
- **技术介绍**：随着 LLM 代码生成的快速增长，区分人类编写和 AI 生成代码对学术诚信、招聘评估和软件安全越来越关键。提出 SemEval-2026 Task 13 多语言机器生成代码检测系统，参与子任务 A（二元检测）和子任务 B（跨 10 个 LLM 家族的多类归因）。子任务 A 使用多视图训练框架微调 UniXcoder-base——结合领域特定结构前缀、去词法化与对称 KL 一致性损失、token dropout 和混合内容增强。系统在验证集上达到 0.993 宏 F1，测试集上 0.845 宏 F1。子任务 B 显示严重类别不平衡（88.4% 人类代码，221:1 多数对少数比）在标准微调下导致灾难性少数类失败——宏 F1 崩溃至 0.086。类权重扩展训练 3 个 epoch 将宏 F1 恢复至 0.345（+301% 相对）。
- **应用场景**：AI 生成代码检测、学术诚信保障
- **为什么值得关注**：SemEval-2026 任务，揭示多类归因需要不平衡感知训练策略

#### 13. AdaEdit: Structure-Aware and Adaptive Output Formats for LLM Code Editing
- **地址**：https://arxiv.org/abs/2604.27296

#### 1. CI-Repair-Bench: Repository-Aware CI 修复基准

**地址**：https://arxiv.org/abs/2604.27148

**项目介绍（300 字）**：
CI-Repair-Bench 是首个针对 CI 工作流验证的仓库级程序修复基准。传统程序修复基准主要面向测试、将修复限制在源代码、假设固定执行环境——这些假设在 CI 场景下完全不成立。CI 失败经常涉及非代码工件（配置文件、环境变量、依赖版本）、嘈杂的执行日志和复杂的工作流约束。CI-Repair-Bench 从真实 GitHub Actions 执行构建，包含 103 个仓库的 567 个 CI 失败实例，仅通过原始工作流下的完整 CI 重新执行来评估修复正确性。失败分为 12 种 CI 错误类型（格式化、linting、编译、测试、环境、依赖、配置等）， enabling fine-grained, error-type-aware evaluation。参考 CI 修复工作流分析 CI 日志以定位故障并生成候选补丁。实证结果表明，自动修复对格式化和 linting 等局部工具强制执行失败最有效（成功率 >50%），而环境、依赖和配置相关失败仍然极具挑战性——最佳 LLM 仅实现 18.9% 的整体修复成功率。

**技术架构**：
- 数据采集层：从 GitHub Actions API 抓取真实 CI 失败日志
- 失败分类层：12 种 CI 错误类型的自动分类
- 修复生成层：基于 LLM 的 CI 日志分析和补丁生成
- 验证层：完整 CI 工作流重新执行验证

**核心机制**：
- 仓库级修复：修复不限于源代码，包括 CI 配置文件、环境设置等
- 错误类型感知评估：按 12 种错误类型分别评估修复成功率
- 真实工作流验证：修复必须通过原始 CI 工作流的完整执行

**与现有技术对比**：
| 维度 | 传统程序修复基准 | CI-Repair-Bench |
|------|-----------------|-----------------|
| 修复范围 | 仅源代码 | 源代码 + CI 配置 + 环境 |
| 验证方式 | 单元测试 | 完整 CI 工作流 |
| 失败类型 | 测试失败 | 12 种 CI 错误类型 |
| 执行环境 | 固定 | 原始仓库环境 |

**优势与局限**：
- 优势：真实 CI 失败数据、仓库级验证、细粒度错误分类
- 局限：仅覆盖 GitHub Actions、567 个实例规模有限、环境类失败修复率低

**未来发展潜力**：
随着 CI/CD 管道的复杂化，CI 原生修复将成为刚需。CI-Repair-Bench 为这一方向奠定了基础，未来可扩展到 GitLab CI、Jenkins 等其他 CI 系统。

#### 2. Sliceformer: 数据流感知程序切片

**地址**：https://arxiv.org/abs/2604.26961

**项目介绍（300 字）**：
Sliceformer 是 ACL 2026 录用的创新性研究，将静态程序切片重新定义为序列到序列任务。静态程序切片是隔离与特定变量相关代码的基础技术——广泛应用于调试、程序理解和代码重构。传统方法基于精确的数据流分析但需要完整的程序表示（CFG/DFG），而最近的 LM 方法虽然灵活但遭受不准确的依赖建模和无约束生成的困扰。Sliceformer 引入两个关键创新：首先，设计数据流感知预训练目标——利用数据流图（DFG）通过数据流保持语句排列和数据流感知跨度损坏来教授模型数据依赖；其次，开发约束解码机制强制执行词法和语法约束以消除幻觉。在 Java 和 Python 程序切片基准上评估，比 SOTA 基线最多提升 22% 的 ExactMatch。该方法使用小型语言模型（CodeT5+），在保持高精度的同时显著降低计算成本。

**技术架构**：
- 预训练层：DFG 驱动的数据流感知预训练目标
- 编码层：CodeT5+ 编码器捕获程序语义
- 解码层：约束解码确保词法和语法正确性
- 输出层：精确的程序切片序列

**核心算法**：
- 数据流保持语句排列：在预训练时随机排列语句但保持数据流关系
- 数据流感知跨度损坏：基于数据流依赖选择损坏跨度
- 约束解码：同时强制执行词法（token 合法性）和语法（结构合法性）约束

**与现有技术对比**：
| 方法 | ExactMatch (Java) | ExactMatch (Python) |
|------|-------------------|---------------------|
| 传统数据流分析 | ~65% | ~60% |
| LM 基线 | ~55% | ~50% |
| Sliceformer | ~77% (+22%) | ~72% (+22%) |

**优势与局限**：
- 优势：ACL 2026 录用、DFG 预训练创新、约束解码消除幻觉、小型模型高效
- 局限：仅评估 Java 和 Python、未扩展到更复杂的数据流场景

**未来发展潜力**：
程序切片是程序分析的基础技术，Sliceformer 的方法可推广到程序合并、缺陷定位、代码克隆检测等任务。

**个人技术理解**：
Sliceformer 的核心洞察是：LM 在程序切片上的失败不是因为表达能力不足，而是因为预训练目标与数据流分析不匹配。通过 DFG 感知的预训练，LM 可以学习到传统数据流分析的核心能力——同时保持灵活性和泛化能力。

#### 3. AdaEdit: 结构感知自适应代码编辑

**地址**：https://arxiv.org/abs/2604.27296

**项目介绍（250 字）**：
AdaEdit 是 ACL 2026 Findings 录用的代码编辑格式创新。当前 LLM 代码编辑普遍采用全代码生成范式——效率瓶颈严重，对交互式编码助手构成挑战。本文系统研究传统 diff 格式，揭示脆弱的偏移和碎片化的 hunks 使生成对 LLM 高度不自然。引入 BlockDiff 和 FuncDiff——两种结构感知 diff 格式，将变更表示为语法连贯单元（控制结构、函数）的块级重写。进一步提出 AdaEdit——通用自适应编辑策略，训练 LLM 动态选择在给定 diff 格式和全代码之间最 token 高效的格式。广泛实验表明，AdaEdit 配合结构感知 diff 格式始终匹配全代码生成的准确性，同时在长代码编辑任务上将延迟和成本降低超过 30%。

**技术架构**：
- 格式分析层：系统评估传统 diff 格式对 LLM 的友好度
- 结构设计层：BlockDiff（块级重写）和 FuncDiff（函数级重写）
- 自适应层：AdaEdit 动态格式选择策略
- 训练层：联合训练格式选择和内容生成

**核心机制**：
- 结构感知 diff：基于语法树的结构单元而非行级 diff
- 自适应选择：训练 LLM 根据代码长度和变更类型选择最优格式
- 联合优化：格式选择和内容生成联合训练

**优势与局限**：
- 优势：30%+ 延迟和成本降低、准确性不损失、通用性强
- 局限：需要额外的训练数据、对极短代码编辑收益有限

**个人技术理解**：
AdaEdit 揭示了一个被忽视的事实：编辑格式本身是 LLM 代码编辑效率的关键因素。结构感知的 diff 格式比传统的行级 diff 更适合 LLM 的生成模式——因为 LLM 更擅长生成语义连贯的单元而非精确的行偏移。

#### 1. Test Before You Deploy: LLM 供应链治理

**地址**：https://arxiv.org/abs/2604.27789

**论文介绍（300 字）**：
本文首次将 LLM 更新管理框架化为软件供应链治理问题。LLM 作为软件系统的核心依赖，其提供商侧的静默更新会导致行为漂移——功能回归、格式变化、安全约束失效等。现有方法聚焦回归测试或版本控制，但缺乏部署侧的兼容性治理机制。提出基于三个组件的部署侧治理框架：(1) 生产契约——明确定义模型行为规则；(2) 风险类别测试套件——按部署风险类别组织的聚焦测试；(3) 兼容性门控——除非满足定义的安全和性能标准否则阻止更新的发布检查点。通过多 LLM 版本的探索性验证，证明在特定风险区域的靶向测试可以发现整体指标遗漏的性能回归。识别若干开放研究挑战：如何系统构建有效测试套件、如何在非确定性系统中设置可靠性能阈值、如何在提供商提供有限透明度时检测和解释模型漂移。

**技术架构**：
- 生产契约层：定义模型可接受的行为边界
- 风险分类层：按部署风险将测试分为不同类别
- 测试执行层：针对每个风险类别执行聚焦测试
- 门控决策层：基于测试结果决定是否允许更新

**核心算法**：
- 风险类别识别：根据部署场景识别关键风险维度
- 靶向测试生成：针对每个风险类别生成高区分度测试
- 性能阈值设定：基于历史数据设定非确定性系统的性能边界
- 漂移检测：比较新旧版本在风险类别测试上的表现差异

**与现有技术对比**：
| 维度 | 传统回归测试 | LLM 供应链治理 |
|------|-------------|---------------|
| 关注点 | 功能正确性 | 行为一致性 |
| 版本控制 | 显式版本号 | 静默更新检测 |
| 测试策略 | 全面回归 | 风险靶向 |
| 决策机制 | 通过/失败 | 概率门控 |

**优势与局限**：
- 优势：首次提出 deployer-side 治理框架、风险靶向测试高效
- 局限：非确定性系统的性能阈值设定困难、提供商透明度有限

**未来发展潜力**：
随着 LLM 越来越多地作为核心依赖集成到软件系统中，供应链治理将成为刚需。本文提出的框架为这一方向奠定了基础。

**个人技术理解**：
LLM 供应链治理是一个被严重低估的问题。传统软件供应链有明确的版本号和变更日志，但 LLM 的静默更新使得这一问题变得复杂。风险靶向测试是一个聪明的思路——不需要全面回归，而是聚焦于部署场景中最关键的风险维度。

#### 2. The Grand Software Supply Chain of AI Systems

**地址**：https://arxiv.org/abs/2604.27781

**论文介绍（250 字）**：
本文将 AI 供应链作为一等分析对象，分解为四个架构层：数据获取、模型训练、模型推理和交叉基底。识别出传统供应链机制无法解决的四个结构性差距：可验证性、版本控制、可观测性和可追溯性。AI 系统在所有四个方面都存在不足：携带未声明的行为耦合、无法回退到已知工作组件、静默降级而非暴露破坏性变化、谱系几乎无法近似。通过测量 48 个生产级开源项目的参考栈进行说明——声明 4,664 个直接依赖，解析为 11,508 个传递包，总计约 3.92 亿行代码。这一规模远超传统软件项目的供应链复杂度。

**核心发现**：
- 48 个项目 → 4,664 直接依赖 → 11,508 传递包 → 3.92 亿行代码
- 四个结构性差距：可验证性、版本控制、可观测性、可追溯性
- AI 供应链的复杂度远超传统软件

**优势与局限**：
- 优势：首次量化 AI 供应链规模、识别结构性差距
- 局限：仅分析开源项目、未覆盖商业 AI 系统

**个人技术理解**：
3.92 亿行代码的传递依赖——这个数字令人震惊。AI 系统的供应链复杂度远超我们的直觉。传统软件供应链工具（如 SBOM）无法直接应用于 AI 系统，因为 AI 系统有独特的结构性差距。

#### 3. emote: EvoSuite 模块化测试增强

**地址**：https://arxiv.org/abs/2604.27112

**论文介绍（200 字）**：
本文评估和改进 EvoSuite 测试生成器的模块化测试模式。发现由于禁止调用非目标设置方法的严格限制，EvoSuite 的模块化测试效率低下。提出 emote 增强——放宽此限制，允许在测试前缀中包含非目标方法。修改 EvoSuite 的适应度函数以将分支覆盖率贡献集中在源自目标方法的调用链上。在 SF100 基准的子集上评估，显示目标方法覆盖率提高 15.15%。

**核心机制**：
- 放宽非目标方法调用限制
- 修改适应度函数聚焦目标调用链
- 从开发者模糊驱动中获得灵感

**个人技术理解**：
15.15% 的覆盖率提升在工业级基准上显著。这个改进的核心洞察是：模块化测试不应该完全隔离——适当的设置调用是必要的。

#### 4. LLM EOL 迁移框架

**地址**：https://arxiv.org/abs/2604.27082

**论文介绍（200 字）**：
提出当底层 LLM 达到 EOL 或需要替换时迁移生产 LLM 系统的框架。关键贡献是贝叶斯统计方法，将自动评估指标针对人类判断进行校准，即使在有限的人工评估数据下也能实现自信的模型比较。在服务于 530 万月交互的商业问答系统上演示，跨六个全球区域评估正确性、拒绝行为和风格遵守。

**核心机制**：
- 贝叶斯校准：自动指标 → 人类判断
- 有限人工评估下的自信比较
- 多区域、多维度评估

**个人技术理解**：
LLM EOL 迁移是一个即将成为现实的问题。随着模型快速迭代，生产系统的模型替换需求将越来越频繁。贝叶斯校准方法为这一挑战提供了 principled 的解决方案。

#### 5. PhantomRun: 嵌入式 CI 构建复现

**地址**：https://arxiv.org/abs/2604.27075

**论文介绍（200 字）**：
PhantomRun 提供统一的抽象层和公开可重用的数据集，标准化 CI 构建日志和元数据的检索、存储和复现。在 4,628 个失败的 CI 运行中重建了 91.8% 的构建并在 98% 的评估案例中保留了执行结果。提供两个核心能力：检索任何提交的构建日志和在受控环境中忠实重新执行相应的构建。EASE 2026 录用。

**核心机制**：
- 统一抽象层：标准化 CI 日志格式
- 构建复现：91.8% 重建率
- 公开数据集：4,628 个失败 CI 运行

**个人技术理解**：
91.8% 的构建重建率在嵌入式 CI 场景下是一个重大成就。嵌入式系统的 CI 失败诊断一直是一个痛点——PhantomRun 提供了可复现的解决方案。

## 技术趋势

### 趋势 1：LLM 供应链治理成为新刚需
从 "Test Before You Deploy" 和 "Grand Software Supply Chain" 两篇论文可以看出，LLM 作为软件核心依赖的供应链治理正在成为独立研究方向。传统软件供应链工具（SBOM、版本控制）无法直接应用于 AI 系统——AI 系统有独特的结构性差距（可验证性、版本控制、可观测性、可追溯性）。3.92 亿行代码的传递依赖揭示了 AI 供应链的惊人复杂度。

### 趋势 2：CI 原生修复从边缘走向主流
CI-Repair-Bench 和 PhantomRun 两篇论文共同指向一个方向：CI 工作流级别的自动化修复正在从边缘研究走向主流。传统程序修复基准假设固定执行环境和源代码级修复——这些假设在 CI 场景下完全不成立。CI 修复的 18.9% 成功率表明这一方向仍有巨大挑战，但也意味着巨大机会。

### 趋势 3：代码编辑格式优化被长期忽视
AdaEdit 揭示了一个被忽视的事实：编辑格式本身是 LLM 代码编辑效率的关键因素。结构感知的 diff 格式（BlockDiff/FuncDiff）比传统行级 diff 更适合 LLM 的生成模式。30%+ 的延迟和成本降低对交互式编码助手意义重大——这个方向值得持续跟踪。
