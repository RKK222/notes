#### 1. To Diff or Not to Diff: BlockDiff/FuncDiff/AdaEdit — 结构感知代码编辑格式
- **地址**：https://arxiv.org/abs/2604.27296
- **来源**：arXiv cs.SE, ACL 2026 Findings 录用
- **领域**：Software Testing / 代码编辑
- **技术介绍**：LLM 代码编辑目前主要采用全代码生成范式，存在严重的效率瓶颈——延迟高、成本高，难以满足交互式编码助手的需求。本文系统研究了传统 diff 格式的脆弱性：脆弱的偏移量和碎片化的 hunk 使得 LLM 生成极不自然。提出 BlockDiff 和 FuncDiff 两种结构感知 diff 格式，将变更表示为语法上连贯的单元（如控制结构和函数）的块级重写。进一步提出 AdaEdit——一种通用自适应编辑策略，训练 LLM 动态选择给定 diff 格式与全代码之间最 token 高效的格式。实验表明，AdaEdit 配合结构感知 diff 格式在准确率上匹配全代码生成，同时将长代码编辑任务的延迟和成本降低超过 30%。
- **应用场景**：AI 编码助手、代码编辑 LLM、交互式编程工具
- **为什么值得关注**：首次将 diff 格式本身作为优化目标而非模型能力，ACL 2026 录用，30%+ 成本降低在工业场景意义重大

#### 2. An Empirical Security Evaluation of LLM-Generated Cryptographic Rust Code
- **地址**：https://arxiv.org/abs/2604.27001
- **来源**：arXiv cs.CR/cs.SE, EASE 2026 工作坊
- **领域**：AI Testing / 代码安全
- **技术介绍**：开发者和组织越来越多地使用 LLM 生成安全关键代码，包括密码学解决方案。本研究对 240 个 Rust 代码样本（AES-256-GCM 和 ChaCha20-Poly1305 两种算法，由 Gemini 2.5 Pro、GPT-4o、DeepSeek Coder 三种 LLM 使用四种提示策略生成）进行了密码学安全的实证评估。结果显示仅 23.3% 的代码样本成功编译。在编译成功的代码中，CodeQL 仅产生 2 个误报，而专用的密码学分析器发现 57% 的编译样本存在漏洞且零误报。两种算法的编译成功率差异显著（AES-256-GCM 34.2% vs ChaCha20-Poly1305 12.5%）。提示策略显著影响结果（P=0.002），chain-of-thought 提示比 zero-shot 差 5 倍。所有三种模型都表现出系统性失败，包括 nonce 重用和 API 幻觉。
- **应用场景**：LLM 生成密码学代码的安全验证
- **为什么值得关注**：首次系统评估 LLM 生成密码学代码的安全性，57% 漏洞率揭示严重安全问题，chain-of-thought 反而更差的结果反直觉

#### 3. How Code Representation Shapes False-Positive Dynamics in Cross-Language LLM Vulnerability Detection
- **地址**：https://arxiv.org/abs/2604.27714
- **来源**：arXiv cs.CR/cs.SE
- **领域**：AI Testing / 漏洞检测
- **技术介绍**：代码表示格式如何影响跨语言 LLM 漏洞检测的误报行为仍不清楚。本文系统改变训练强度和代码表示格式，比较原始源代码与修剪的抽象语法树（AST）在训练和推理时的效果，使用两个 8B 参数 LLM（Qwen3-8B 和 Llama 3.1-8B-Instruct）在 NIST Juliet 测试套件（C/C++）上微调，并在 Java（OWASP Benchmark）和 Python（BenchmarkPython）上评估。发现跨语言 FPR 反映了训练时和推理时表示的联合效应——不是单独任何一个。文本微调单调推高 FPR（Qwen3-8B: 0.763→0.866→1.000），而 F1 保持稳定（0.637-0.688），掩盖了崩溃。表面线索记忆是主要机制。跨表示探测（将文本训练的权重应用于 AST 编码输入而不重新训练）隔离了这一效应：FPR 从 0.866 降至 0.583，37.2% 的误报在 AST 输入下恢复为真阴性。
- **应用场景**：跨语言 LLM 漏洞检测、代码安全分析
- **为什么值得关注**：揭示文本微调导致表面线索记忆的机制，提出无需重新训练的预部署一致性门控

#### 4. ClassEval-Pro: A Cross-Domain Benchmark for Class-Level Code Generation
- **地址**：https://arxiv.org/abs/2604.26923
- **来源**：arXiv cs.SE, AIWare 2026 录用
- **领域**：AI Testing / 代码生成基准
- **技术介绍**：LLM 在函数级代码合成和仓库级代码修改方面都取得了强劲成果，但介于两者之间的一种能力——组合式代码创建（即从规范构建完整、内部结构化的类）——仍未得到充分服务。当前评估要么局限于孤立函数，要么依赖手动策划的类级任务，这些任务扩展成本高且越来越容易受到数据污染。本文引入 ClassEval-Pro，一个包含 300 个类级任务的基准，涵盖 11 个领域，通过自动化三阶段管道构建，结合复杂性增强、跨领域类组合和集成 2025 年 1 月之后贡献的真实 GitHub 代码。每个任务都通过 LLM Judge Ensemble 验证，必须通过超过 90% 行覆盖率的测试套件。评估五个前沿 LLM 在五种生成策略下，最佳模型仅达到 45.6% 的类级 Pass@1，最强和最弱模型之间存在 17.7 个百分点的差距，确认了基准的区分力。策略选择强烈影响模型能力：结构化方法（如自底向上）可将较弱模型提升高达 9.4 个百分点，而组合式生成可能低至 1.3%。对 500 个手动注释失败的错误分析显示，逻辑错误（56.2%）和依赖错误（38.0%）占主导地位，跨方法协调是核心瓶颈。
- **应用场景**：类级代码生成评估、LLM 组合能力测试
- **为什么值得关注**：填补函数级和仓库级之间的评估空白，揭示跨方法协调是核心瓶颈

#### 5. CI-Repair-Bench: A Repository-Aware Benchmark for Automated Patch Validation via CI Workflows
- **地址**：https://arxiv.org/abs/2604.27148
- **来源**：arXiv cs.SE
- **领域**：AI Testing / CI 修复
- **技术介绍**：持续集成（CI）通过多阶段工作流强制执行仓库级正确性，是现代软件开发的核心，但诊断和修复 CI 失败仍然具有挑战性。与传统的程序修复不同，CI 失败通常涉及非代码工件、环境和依赖问题、嘈杂的执行日志以及工作流级约束。现有的程序修复基准在这种设置下存在不足：它们主要是测试中心的，将修复限制在源代码中，假设固定的执行环境，并在不反映真实仓库级验证的简化 CI 工作流下进行评估。本文引入 CI-Repair-Bench，一个用于 CI 验证的仓库级程序修复基准，从真实的 GitHub Actions 执行中构建。它包含来自 103 个仓库的 567 个 CI 失败实例，仅通过原始工作流下的完整 CI 重新执行来评估修复正确性。失败分为 12 种 CI 错误类型，支持细粒度的错误类型感知评估。实证结果表明，自动化修复对于本地化的、工具执行的失败（如格式化和 linting）最有效，而环境、依赖和配置相关的失败仍然具有挑战性；最佳 LLM 达到 18.9% 的修复成功率。
- **应用场景**：CI 工作流自动化修复、仓库级程序修复
- **为什么值得关注**：首个专注于 CI 工作流级别修复的基准，揭示环境/依赖问题的修复难度

#### 6. On the Effectiveness of Modular Testing with EvoSuite (emote)
- **地址**：https://arxiv.org/abs/2604.27112
- **来源**：arXiv cs.SE
- **领域**：Software Testing / 模块化测试
- **技术介绍**：本文探讨了面向对象 Java 程序中模块化随机测试的有效性。模块化测试涉及隔离测试程序的单个组件。通常，为了有效的测试生成，必须包含一系列非目标设置调用以获得对目标组件的高覆盖率。本文评估并改进了 EvoSuite 测试生成器的模块化测试。发现由于严格限制禁止调用非目标设置方法，EvoSuite 的模块化测试模式效率低下，通常导致低分支覆盖率。提出 emote（Effective Modular Testing with EvoSuite）：对 EvoSuite 的增强，放宽此限制，允许在测试前缀中包含非目标方法。此修改从开发者编写的模糊驱动中汲取灵感，这些驱动通常在测试目标方法之前调用设置方法来正确初始化状态。为确保有意义的测试生成，修改了 EvoSuite 的适应度函数，将分支覆盖率贡献集中在源自目标方法的调用链上。在 SF100 基准的子集上评估 emote，显示目标方法的覆盖率提高了 15.15%。
- **应用场景**：Java 模块化测试、EvoSuite 增强
- **为什么值得关注**：解决模块化测试中设置方法调用的核心问题，15%+ 覆盖率提升

#### 7. PhantomRun: Reproducing Build Failures in Embedded Open Source Software
- **地址**：https://arxiv.org/abs/2604.27075
- **来源**：arXiv cs.SE, EASE 2026
- **领域**：Software Testing / CI 构建复现
- **技术介绍**：由于嵌入式系统中的硬件 - 软件协同开发，持续集成（CI）构建经常因复杂的交叉编译、板配置和工具链约束而失败。尽管 CI 构建日志包含有价值的诊断信息，但由于异构运行器、工具链和日志格式，它们寿命短且难以重用。为解决这些挑战，本文提出 PhantomRun，一个统一的抽象层和公开可重用的数据集，标准化 CI 构建日志和元数据的检索、存储和复现。在 4628 个失败的 CI 运行中，重建了 91.8% 的构建，并在 98% 的评估案例中保留了执行结果。PhantomRun 提供两个核心能力：检索任何提交的构建日志和在受控环境中忠实重新执行相应的构建。通过以统一的、机器可读的格式暴露所有构建工件和元数据，PhantomRun 实现了 CI 失败的可重现和纵向研究。实证评估表明，重建的构建与原始构建紧密匹配，通常仅在时间戳或轻微的非确定性重排序上有所不同，证明了大规模历史 CI 重建的可行性。
- **应用场景**：嵌入式 CI 构建复现、历史 CI 失败分析
- **为什么值得关注**：解决嵌入式系统 CI 失败复现的核心难题，91.8% 重建率

#### 8. LLM Variability in Evidence Screening for Software Engineering SLRs
- **地址**：https://arxiv.org/abs/2604.27006
- **来源**：arXiv cs.SE/cs.AI
- **领域**：AI Testing / LLM 变异性
- **技术介绍**：系统文献综述中的研究筛选成本高、易出错且风险不对称，因为假阴性可能损害有效性。尽管 LLM 快速采用，但关于这些模型在研究筛选阶段的行为证据有限，特别是关于特定 LLM 的选择及其与经典模型的比较。本文评估了来自 4 个提供商（OpenAI、Google Gemini、Anthropic、Llama）的 12 个 LLM 和 4 个经典模型（逻辑回归、支持向量分类、随机森林和朴素贝叶斯）在 2 个真实系统文献综述（SLR）上的表现，总计 518 篇论文。实验设计调查了三个关键维度：（i）LLM 性能变异性，（ii）输入特征组成（摘要、标题和关键词）对 LLM 性能的影响，以及（iii）使用 LLM 而不是更传统分类模型的实际收益。结果表明，LLM 表现出实质性的异质性和即使在温度为零时的残余非确定性。摘要可用性是决定性的：移除它一致地降低了性能，而添加标题和/或关键词到摘要没有产生稳健的收益。与经典模型相比，性能差异不够一致以支持可泛化的 LLM 优越性。
- **应用场景**：SLR 研究筛选、LLM 变异性评估
- **为什么值得关注**：揭示 LLM 在学术筛选中的变异性问题，温度为零仍存在非确定性

#### 9. Adaptive and AI-Augmented Security Testing: A Systematic Survey
- **地址**：https://arxiv.org/abs/2604.27000
- **来源**：arXiv cs.SE/cs.CR/cs.PL
- **领域**：AI Testing / 安全测试综述
- **技术介绍**：现代软件系统在快速持续集成和部署（CI/CD）管道中开发，在发布前确保安全提出了重大的技术和组织挑战。传统的静态和动态分析工具提供了有价值的结构和行为洞察，但它们通常在非自适应工作流中运行，并产生大量需要手动分类的警告。反馈驱动的模糊测试和基于搜索的测试方法展示了由执行信号引导的迭代输入精炼的力量，而 LLM 在自动化测试生成方面显示出潜力，但通常缺乏对程序结构的语义基础。本文对五个领域的自适应和 AI 增强安全测试研究进行了系统综述：（1）用于漏洞检测的结构程序分析，（2）DevSecOps 和持续安全测试，（3）反馈驱动的模糊测试和基于搜索的测试，（4）基于 LLM 的自动化测试生成，以及（5）将程序分析与自适应学习集成的新兴混合系统。分析了来自四个主要数据库的系统搜索产生的 22,088 条原始记录中的 55 项同行评审研究。分析揭示了结构程序表示（AST、CFG 和 CPG）与自适应测试机制之间的持续脱节。将其特征化为结构 - 自适应碎片化：一个单独的范式都无法解决的系统性分离。没有现有系统将人类分类信号作为反馈用于精炼结构模型。最后确定了五个开放研究挑战，并概述了语义基础、反馈驱动、多语言安全测试框架的统一议程。
- **应用场景**：安全测试综述、AI 增强测试
- **为什么值得关注**：首个系统综述 AI 增强安全测试，识别结构 - 自适应碎片化核心问题

#### 10. UCSC-NLP at SemEval-2026 Task 13: Multi-View Generalization and Diagnostic Analysis of Machine-Generated Code Detection
- **地址**：https://arxiv.org/abs/2604.26990
- **来源**：arXiv cs.SE, SemEval-2026
- **领域**：AI Testing / 代码检测
- **技术介绍**：随着 LLM 在代码生成方面的快速增长，区分人类编写的和 AI 生成的代码对于学术诚信、招聘评估和软件安全变得越来越关键。本文介绍了 SemEval-2026 Task 13：多语言机器生成代码检测的系统，参与子任务 A（二元检测）和子任务 B（跨 10 个 LLM 家族的多类归因）。对于子任务 A，使用多视图训练框架微调 UniXcoder-base，该框架促进生成器不变表示。该框架结合领域特定结构前缀、带对称 KL 一致性损失的词法去除、token 丢弃和混合内容增强。系统在验证集上达到 0.993 宏 F1，在测试集上达到 0.845 宏 F1，涵盖未见语言和领域。对于子任务 B，表明严重类别不平衡（88.4% 人类代码，221:1 多数对少数比率）在标准微调下导致灾难性少数类失败，宏 F1 崩溃至 0.086，尽管准确率为 88.4%。训练 3 个 epoch 的类权重扩展将宏 F1 恢复至 0.345（+301% 相对），确认多类归因需要不平衡感知训练策略。
- **应用场景**：AI 生成代码检测、学术诚信
- **为什么值得关注**：SemEval 竞赛方案，揭示多类归因中的类别不平衡问题

#### 11. Static Program Slicing Using Language Models With Dataflow-Aware Pretraining and Constrained Decoding (Sliceformer)
- **地址**：https://arxiv.org/abs/2604.26961

#### 3. What Makes a Good Terminal-Agent Benchmark Task: A Guideline for Adversarial, Difficult, and Legible Evaluation Design
- **地址**：https://arxiv.org/abs/2604.28093
- **论文介绍**：终端代理基准已成为测量 LLM 编码和系统管理能力的主要信号。随着评估环境市场的增长，快速交付任务的压力往往没有经过对验证逻辑的彻底对抗性审查。本文是从一年多的 Terminal Bench 贡献和审查经验中得出的编写良好基准任务的指南。大多数人以编写提示的方式编写基准任务——他们不应该。提示旨在帮助代理成功；基准旨在发现它是否能。本文认为好的任务是对抗性的、困难的和可读的，并指出一大类常见失败模式——AI 生成的指令、过度规范的规格、文书难度、假设隐藏知识的 oracle 解决方案、验证错误事物的测试、可奖励黑客的环境——是将任务编写视为提示编写的可预测后果。本文编目了这些失败模式，认为真正的困难是概念性的而非环境性的，并讨论了最近的实证证据，表明流行终端代理基准中超过 15% 的任务是可奖励黑客的。
- **技术架构**：
  - 对抗性设计：防止 AI 生成指令绕过
  - 困难性：概念性困难而非环境复杂性
