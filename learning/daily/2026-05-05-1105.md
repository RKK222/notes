#### 1. AI-Generated Smells: Code and Architecture in LLM and Agent-Driven Development
- **地址**：https://arxiv.org/abs/2605.02741
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 代码质量
- **技术介绍**：LLM 生成代码的功能正确性被广泛关注，但长期可维护性被忽视。本文对 AI 生成软件中的技术债务进行系统性审计，发现 AI 并未消除缺陷，而是引入了独特的"机器签名"缺陷模式。多尺度分析揭示了一个根本性的 Reasoning-Complexity Trade-off：模型越强大，生成的代码越臃肿和耦合。这种架构退化如此显著，以至于建立了 Volume-Quality Inverse Law——代码量几乎是结构退化的完美预测因子。关键发现：功能正确性和详细提示都无法缓解这种退化。
- **应用场景**：AI 生成代码质量评估、技术债务管理
- **为什么值得关注**：挑战了"AI 生成代码只要功能正确就好"的假设，架构复杂度管理才是核心问题

#### 2. SSDE: Structured Spec-Driven Engineering for Repository-Level Code Generation
- **地址**：https://arxiv.org/abs/2605.02455
- **来源**：arXiv (cs.SE) / FSE Companion '26
- **领域**：AI Testing / 仓库级代码生成
- **技术介绍**：LLM 在函数级代码生成上表现优秀，但扩展到仓库级系统时质量显著下降。SSDE 提出利用结构化制品引导 LLM 生成，认为结构化规格作为 LLM 输入使高质量仓库级代码生成成为可行目标，同时提供更好的可验证性。通过生成 MVC 业务逻辑的试点研究，使用 5 个 LLM 在 3 个软件系统上验证可行性，并勾勒出 SSDE 的潜力、挑战和未来路线图。
- **应用场景**：仓库级代码生成、结构化软件工程
- **为什么值得关注**：FSE Companion '26 接受，从自然语言提示转向结构化规格是仓库级生成的关键方向

#### 3. Causal Software Engineering: A Vision and Roadmap
- **地址**：https://arxiv.org/abs/2605.02454
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 因果推理
- **技术介绍**：软件工程越来越多地在不确定性下做出高风险决策。AI 驱动的工具（异常检测、预测分析、AIOps、LLM Agent）增强了工程师检测模式和合成内容的能力，但许多关键问题是干预性或反事实的："改变负载均衡策略的预期影响是什么？""不同的发布计划能否避免故障？"相关模型回答"什么倾向于同时发生"，但难以回答"如果我们采取行动会怎样"。CSE 提出因果模型和因果推理系统性地告知软件生命周期各阶段的活动。
- **应用场景**：软件决策因果分析、反事实诊断
- **为什么值得关注**：从相关性到因果性的范式转变，对 AI 辅助软件决策有深远影响

#### 4. AOCI: Symbolic-Semantic Indexing for Repository-Scale Code Understanding
- **地址**：https://arxiv.org/abs/2605.02421
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 代码理解
- **技术介绍**：LLM 在理解超出一定规模的代码库时遇到困难——数十万行代码的仓库。现有方法（检索、摘要、Agent 探索）在查询时构建不同视图，运行之间变化，持久化通常是临时的。AOCI 提出符号-语义仓库表示——一种结构化蓝图，LLM 可以在单次传递中读取以获得系统架构、依赖关系和关键设计决策的完整仓库级图景。每个条目配对符号标签和语义内容。索引维护是增量的：代码变更时仅重新生成受影响的条目。
- **应用场景**：大仓库代码理解、LLM 上下文工程
- **为什么值得关注**：解决了 LLM 理解大仓库的根本瓶颈，增量维护使其可规模化

#### 5. AIDev: How Humans Review AI-Generated Pull Requests
- **地址**：https://arxiv.org/abs/2605.02273
- **来源**：arXiv (cs.SE) / EASE 2026
- **领域**：AI Testing / 代码审查
- **技术介绍**：使用 AIDev 数据集分析 GitHub 上 AI 生成 PR 的代码审查交互，并与同一仓库中人类编写的 PR 进行比较。发现大多数 AI 生成的 PR 没有收到审查，当被审查时，主要由 AI Agent 而非人类主导。人类编写的 PR 更可能收到纯人工审查并吸引直接人类反馈。相比之下，AI 生成 PR 的审查更多采取自动化中介交互形式，人类参与通常通过 Agent 引导而非独立评估表达。
- **应用场景**：AI 辅助开发流程、代码审查质量
- **为什么值得关注**：EASE 2026 接受，揭示了 Agentic 工作流中审查活动的系统性差异

#### 6. CommitSuite: Benchmark for Commit Classification and Message Generation
- **地址**：https://arxiv.org/abs/2605.02256
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 提交消息
- **技术介绍**：CommitSuite 基准包含来自 7 种编程语言 243 个开源仓库的 63,533 个符合 Conventional Commits 规范的提交。每个提交标注了 CCS 类型，并用 AST 级代码变更和 LLM 辅助语义注释（捕获变更的"什么"和"为什么"）进行丰富。提出无参考评估框架，基于五个二元指标：合理性、全面性、非冗余性、真实性和逻辑性。实验表明 LLM 可有效支持生成和评估，评估达到 0.849 Cohen's Kappa 一致性。
- **应用场景**：提交消息质量评估、变更日志自动化
- **为什么值得关注**：最大规模 CCS 基准，语义级评估框架无需人工参考

#### 7. DocSync: Agentic Documentation Maintenance via Critic-Guided Reflexion
- **地址**：https://arxiv.org/abs/2605.02163
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 文档维护
- **技术介绍**：软件文档经常随代码库演变而偏离可执行逻辑，造成降低可维护性的技术债务。DocSync 将文档维护构建为结构化的迭代生成任务。通过融合 AST 表示和 RAG 提供依赖感知上下文，弥合语法变更和自然语言描述之间的差距。引入基于 Reflexion 范式的 critic-guided 精炼循环，使模型能够针对源代码自我纠正候选更新。使用 LoRA 适配小语言模型在代理代码到文本维护任务上评估。
- **应用场景**：自动化文档维护、代码-文档一致性
- **为什么值得关注**：将 AST+RAG+Reflexion 结合解决文档漂移问题

#### 8. Foundation Models as Oracles for Refactoring Correctness Detection
- **地址**：https://arxiv.org/abs/2605.02096
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 重构正确性
- **技术介绍**：主流 IDE 中的重构工具可能引入意外行为变更或编译错误。传统检测方法依赖手工前置条件和静态/动态分析，适应性有限。本研究检验 Foundation Models 作为 Java 程序重构 Bug 检测预言机的潜力。零样本提示覆盖来自 IntelliJ-IDEA、Eclipse 和 NetBeans 的 226 个真实重构 Bug，跨越 47 种重构类型。GPT-5.4 达到 93.8% 准确率，Gemini-3.1-Pro-Preview 在所有模型中表现最佳。变形测试用于进一步验证。
- **应用场景**：IDE 重构质量保证、自动化重构验证
- **为什么值得关注**：Foundation Models 在重构正确性检测上超越传统方法，GPT-5.4 达 93.8%

#### 9. GHA Compliance: Checklist-Based Study with LLM-Assisted Auditing
- **地址**：https://arxiv.org/abs/2605.02091
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / CI/CD 合规
- **技术介绍**：GitHub Actions CI 工作流是关键基础设施，但当前工具仅提供语法或启发式检查，不强制执行安全、可维护性或性能的最佳实践。提出文档驱动的 GHA 合规检查清单，涵盖 4 个工作流部分和 8 个主题的 30 项标准。在 95 个真实 Java 工作流（2,850 项评估）上使用 4 个开源 LLM 评估，发现整体合规率仅 28%，权限控制降至 4%。引入多层裁决框架，GPT-5 解决模型冲突，减少 81% 验证工作量同时保持 87% 专家一致性。
- **应用场景**：CI/CD 安全合规、自动化审计
- **为什么值得关注**：揭示 CI 工作流安全合规的巨大缺口，权限控制仅 4%

#### 10. HEJ-Robust: Robustness Benchmark for LLM-Based Automated Program Repair
- **地址**：https://arxiv.org/abs/2605.02215
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 程序修复鲁棒性
- **技术介绍**：现有 APR 基准在单一规范形式的 Bug 代码上评估模型，不反映真实软件中常见的语法变化。HEJ-Robust 基于 HumanEval-Java-Bug 使用 8 种语义保持代码变换构建，产生 1,450 个变换实例。评估 5 个微调 LLM，发现在多种变换下模型性能下降超过 50%，表明当前 LLM 修复模型对微小语法变化缺乏鲁棒性。
- **应用场景**：APR 鲁棒性评估、代码修复泛化能力
- **为什么值得关注**：揭示 APR 模型在真实场景中的脆弱性，50%+ 性能下降

#### 11. Addressing False Failures in LLM-Based Code Translation
- **地址**：https://arxiv.org/abs/2605.02195
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 代码翻译评估
- **技术介绍**：LLM 代码翻译中大量报告的失败并非由于逻辑错误，而是评估引起的错误——不当的编译标志、缺失的库链接和未配置的运行时环境。在 5 种编程语言（C、C++、Java、Python、Go）和 3 个基准（Avatar、CodeNet、EvalPlus）上进行大规模实证研究，覆盖 GPT-4o、DeepSeek-Coder 和 Magicoder 生成的 6,164 个翻译。识别并分类常见假阴性，区分影响任何模型的管道失败和因 LLM 而异的模型依赖行为。
- **应用场景**：代码翻译评估标准化、评估框架质量
- **为什么值得关注**：揭示代码翻译评估中的系统性假阴性问题

#### 12. FeedbackLLM: Multi-Agentic Language Agnostic Test Case Generator
- **地址**：https://arxiv.org/abs/2605.01264
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 测试用例生成
- **技术介绍**：FeedbackLLM 提出两阶段语言无关测试用例生成框架。第一阶段从源代码提取输入约束并生成测试用例。第二阶段由两个专用 LLM 反馈 Agent 评估质量：(i) Line Feedback Agent 提取未执行行的元数据；(ii) Branch Feedback Agent 提取未执行分支条件的元数据。两个 Agent 在 tandem 中操作，过程重复 k 步。引入进化提示机制持续改进提示质量。
- **应用场景**：自动化测试生成、覆盖率优化
- **为什么值得关注**：多 Agent 反馈循环 + 进化提示实现持续覆盖率提升

#### 13. ClarifySTL: Interactive LLM Agent Framework for STL Transformation
- **地址**：https://arxiv.org/abs/2605.01209
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 形式化规范
- **技术介绍**：STL 是描述 CPS 实时行为的正式语言。ClarifySTL 提出交互式 LLM Agent 框架，通过需求澄清增强 STL 转换。首先检测需求中的模糊表达式，如果检测到模糊性，生成定向澄清查询引导用户补充需求。然后检测歧义，形成聚焦的歧义澄清查询，基于用户反馈更新需求直到所有歧义解决。最后使用 LLM 将澄清后的需求转换为 STL 规范。
- **应用场景**：CPS 形式化规范生成、需求工程
- **为什么值得关注**：通过交互式澄清解决模糊需求到形式化规范的转换

#### 14. LiveFMBench: Contamination-Aware Formal Specification Generation
- **地址**：https://arxiv.org/abs/2605.01394
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 形式化规范
- **技术介绍**：首个系统且污染感知的 LLM/Agent 形式化规范生成研究。引入 LiveFMBench，持续演进的 630 个 ACSL 标注 C 程序基准，包括 360 个新收集案例以减轻数据泄漏。评估直接提示（不同采样大小）、推理启用推理、Agent 管道，并进行细粒度失败分析。发现朴素评估大幅高估性能，因为直接提示下的模型可能表现不忠实行为，如欺骗自动证明器或忽略代码上下文。
- **应用场景**：形式化规范生成评估、污染检测
- **为什么值得关注**：首次污染感知评估，揭示形式化规范生成的真实能力上限

#### 16. ACDL: Agentic Context Description Language
- **地址**：https://arxiv.org/abs/2605.01920
- **来源**：arXiv (cs.SE)
- **领域**：AI / Agent 上下文工程
- **技术介绍**：LLM 越来越多地在更大系统（"LLM Agent"）中使用，这些系统按顺序进行 LLM 调用，每次调用提供指令、观察和交互历史的组合。上下文设计在系统质量中起核心作用，但没有标准来传达 LLM 上下文的组成及其随时间的演变。ACDL 提供指定上下文方面的构造：角色消息序列、动态内容、时间索引引用、条件或迭代结构。
- **应用场景**：Agent 上下文标准化、上下文工程文档
- **为什么值得关注**：首个 Agent 上下文描述语言，填补标准化空白

#### 17. X-CCD: Stabilized Knowledge Distillation for Cross-Language Code Clone Detection
- **地址**：https://arxiv.org/abs/2605.02860
- **来源**：arXiv (cs.SE)
- **领域**：AI / 代码克隆检测
- **技术介绍**：跨语言代码克隆检测（X-CCD）具有挑战性，因为不同语言编写的语义等价程序通常几乎没有表面相似性。提出知识蒸馏框架，将 DeepSeek-R1 的推理能力转移到紧凑开源学生模型用于 X-CCD。使用 Project CodeNet 的跨语言代码对构建推理导向合成训练数据，用 LoRA 适配器微调 Phi3 和 Qwen-Coder。引入响应稳定化方法：强制结论提示、二元分类头和对比分类头。
- **应用场景**：跨语言代码克隆检测、知识蒸馏
- **为什么值得关注**：将推理模型能力蒸馏到紧凑模型，解决 X-CCD 的规模化问题

#### 18. QSAF: Quantum Software Architecture Framework
- **地址**：https://arxiv.org/abs/2605.01800

#### 1. AI-Generated Smells: LLM 生成代码的"机器签名"

**地址**：https://arxiv.org/abs/2605.02741

**项目介绍**：本文对 AI 生成软件中的技术债务进行系统性审计，揭示了一个令人不安的发现：AI 并未消除缺陷，而是引入了独特的"机器签名"缺陷模式。多尺度分析从单文件算法任务到复杂的 Agent 生成系统，识别出一个根本性的 Reasoning-Complexity Trade-off——模型越强大，生成的代码越臃肿和耦合。作者建立了 Volume-Quality Inverse Law：代码量几乎是结构退化的完美预测因子。最关键的是，功能正确性和详细提示都无法缓解这种退化。

**技术架构**：分析跨越多个尺度——单文件算法任务、多文件 Agent 生成系统。使用静态分析工具检测代码异味（过长方法、God Class、Feature Envy 等），结合复杂度指标（Cyclomatic、Cognitive、Maintainability Index）量化架构退化。

**核心算法**：Volume-Quality Inverse Law 的统计验证——代码量与结构退化指标之间的相关性分析。Reasoning-Complexity Trade-off 的实证证明——更强模型生成更复杂代码的因果关系。

**与现有技术对比**：现有研究关注 AI 生成代码的功能正确性（如 SWE-bench、HumanEval），本文首次系统关注长期可维护性和架构质量。传统代码异味检测工具不区分人类和 AI 生成的异味模式。

**优势与局限**：优势是首次系统性揭示 AI 生成代码的架构退化问题；局限是分析主要基于静态指标，未涵盖运行时行为。

**未来发展**：需要开发专门针对 AI 生成代码的架构质量评估工具，以及缓解 Reasoning-Complexity Trade-off 的策略（如结构化提示、架构约束）。

**个人理解**：这篇论文代表了一个重要转向——从"AI 生成代码能否通过测试"到"AI 生成代码能否长期维护"。Volume-Quality Inverse Law 如果得到更多验证，将迫使整个行业重新思考 AI 辅助开发的架构治理策略。

#### 2. AOCI: 符号-语义代码索引

**地址**：https://arxiv.org/abs/2605.02421

**项目介绍**：AOCI（AI-Oriented Code Indexing）提出了一种全新的仓库级代码理解范式。现有方法（检索、摘要、Agent 探索）在查询时构建不同视图，运行之间变化，持久化通常是临时的。AOCI 提出符号-语义仓库表示——一种结构化蓝图，LLM 可以在单次传递中读取以获得系统架构、依赖关系和关键设计决策的完整仓库级图景。每个条目配对符号标签（架构坐标）和语义内容（功能、依赖、约束）。索引维护是增量的：代码变更时仅重新生成受影响的条目。

**技术架构**：AOCI 索引由编码规则和条目组成，每个代码单元（文件或数据库表）一个条目。符号组件提供架构坐标（模块层次、依赖关系、接口定义），语义组件携带功能描述、依赖关系和约束。AOCI Platform 自动化增量维护过程。

**核心创新**：首次提出"一次索引，多次使用"的仓库级代码理解模式，增量维护使其可规模化到大型仓库。

**与现有技术对比**：相比 CodeGraph、RepoCoder 等基于检索的方法，AOCI 提供一致的、可预测的仓库级视图。相比 Agent 探索（如 SWE-agent），AOCI 避免了重复探索的 token 消耗。

**优势与局限**：优势是增量维护和单次传递读取；局限是索引构建和维护的初始成本。

**个人理解**：AOCI 代表了代码理解从"查询时构建"到"预构建+增量更新"的范式转变。这对于 LLM 处理大型仓库至关重要——当前方法在数十万行代码的仓库上要么 token 爆炸，要么信息丢失。

#### 3. MCP Workflow Engine: 分离智能与执行

**地址**：https://arxiv.org/abs/2605.00827

**项目介绍**：MCP Workflow Engine 提出新型 MCP 原生编排层，将智能（决定做什么）与执行（执行）解耦。在现有架构中，Agent 必须在每个会话中推理每个工具调用，消耗与操作数成比例的 token——即使任务已被解决过。MCP Workflow Engine 允许 Agent 推理一次生成声明性工作流蓝图（JSON 文档），指定 MCP 工具调用的有向序列、参数化模板、循环、并行分支和数据管道。后续执行通过单个 run_workflow 工具调用触发，无论蓝图内部复杂度如何，只消耗一次调用的 token。

**技术架构**：MCP Mediator 模式——MCP Server 同时作为下游 MCP Server 的 Client。工作流蓝图是 JSON 文档，包含工具调用序列、参数模板、循环结构、并行分支和数据管道。执行引擎解析蓝图并按序执行，无需 LLM 参与。

**核心创新**：首次将"一次推理，多次执行"模式引入 MCP 生态，解决重复任务的 token 浪费问题。

**与现有技术对比**：相比每次调用都经过 LLM 推理的传统 MCP 使用方式，MCP Workflow Engine 在重复任务上可节省 90%+ token。相比硬编码脚本，工作流蓝图保持声明性和可组合性。

**优势与局限**：优势是大幅降低重复任务的 token 消耗；局限是工作流蓝图需要 Agent 预先推理，不适合完全未知的任务。

**个人理解**：MCP Workflow Engine 是 MCP 生态的重要基础设施补充。当前 MCP Agent 的 token 消耗是实用化的主要瓶颈之一，"一次推理多次执行"模式可显著改善。

#### 4. Causal Software Engineering: 因果推理范式

**地址**：https://arxiv.org/abs/2605.02454

**论文介绍**：软件工程越来越多地在不确定性下做出高风险决策。AI 驱动的工具（异常检测、预测分析、AIOps、LLM Agent）增强了工程师检测模式和合成内容的能力，但许多关键问题是干预性或反事实的："改变负载均衡策略的预期影响是什么？""不同的发布计划能否避免故障？"相关模型回答"什么倾向于同时发生"，但难以回答"如果我们采取行动会怎样"。CSE 提出因果模型和因果推理系统性地告知软件生命周期各阶段的活动，用显式假设、不确定性感知效应估计和反事实诊断增强现有实践。

**技术架构**：(1) 因果优先工作流视图——覆盖开发和运营；(2) 分阶段工具和组织采用路线图；(3) 评估和基准议程。

**核心算法**：因果推理三层次——关联（看到什么）→ 干预（如果做会怎样）→ 反事实（如果过去不同会怎样）。CSE 将这三个层次引入软件工程决策。

**与现有技术对比**：现有 AI 辅助 SE 工具主要基于相关性（预测分析、异常检测），CSE 引入因果性（干预效果估计、反事实诊断）。

**优势与局限**：优势是提供了更可靠的决策基础；局限是因果推断需要更强的假设和更多数据。

**个人理解**：CSE 代表了 AI 辅助软件工程从"预测"到"因果"的范式转变。这对于自动化决策系统（如自动扩展、自动修复）尤为重要——系统需要知道"如果我这样做会怎样"而非"什么倾向于同时发生"。

#### 5. DeFi HKG: 分层知识图谱驱动的漏洞利用合成

**地址**：https://arxiv.org/abs/2605.02868

**论文介绍**：DeFi 智能合约漏洞每年造成数十亿美元损失。识别漏洞 ≠ 证明可利用——手动 PoC 构造劳动密集，大多数已披露漏洞未经验证。HKG 是知识驱动的 Agent 系统，用于端到端合约漏洞检测和利用合成。核心洞察：利用合成不是代码生成任务而是结构化推理问题，需要协议语义、故障根因和利用原语的基础知识。HKG 组织为分层知识图谱作为 LLM 引导多跳推理的结构化记忆。两阶段验证框架：SMT 求解检查利用路径可达性，资产级状态模拟检查利润可实现性。

**技术架构**：(1) 分层知识图谱（协议语义 → 故障根因 → 利用原语）；(2) LLM 引导多跳推理；(3) SMT 求解验证路径可达性；(4) 资产级状态模拟验证利润可实现性。

**核心创新**：将利用合成从"代码生成"重新定义为"结构化推理"，知识图谱提供结构化记忆，双重验证确保逻辑和经济可行性。

**与现有技术对比**：现有方法（如 SmartCheck、Mythril）基于模式匹配或符号执行，HKG 使用知识图谱引导的多跳推理，更接近人类安全研究者的思维方式。

**优势与局限**：优势是双重验证确保 PoC 的实际可利用性；局限是知识图谱构建需要领域专家参与。

**个人理解**：HKG 代表了智能合约安全审计从"模式匹配"到"知识驱动推理"的转变。分层知识图谱作为结构化记忆是关键创新，使 LLM 能够进行多跳推理而非简单的模式匹配。

#### 6. Refactoring Oracle: Foundation Models 作为重构正确性预言机

**地址**：https://arxiv.org/abs/2605.02096

**论文介绍**：主流 IDE 中的重构工具可能引入意外行为变更或编译错误，这是长期存在的挑战。传统检测方法依赖手工前置条件和静态/动态分析，适应性有限。本研究检验 Foundation Models 作为 Java 程序重构 Bug 检测预言机的潜力。零样本提示覆盖来自 IntelliJ-IDEA、Eclipse 和 NetBeans 的 226 个真实重构 Bug，跨越 47 种重构类型。GPT-OSS-20B 首次运行达到 80.5% 准确率，GPT-5.4 达到 93.8%。Gemma-4-31B 在开源模型中表现最强，Gemini-3.1-Pro-Preview 在所有模型中表现最佳。变形测试用于进一步验证。

**技术架构**：零样本提示框架——无需任务特定训练。输入：重构前后的代码对。输出：是否存在行为变更。变形测试用于增强验证可靠性。

**核心算法**：零样本提示 + 变形测试组合。变形测试通过代码变换生成变体，验证重构在变体上是否保持一致行为。

**与现有技术对比**：传统方法（如 Refactoring Miner）基于模式匹配和静态分析，Foundation Models 在准确率上显著超越（93.8% vs ~70%）。

**优势与局限**：优势是零样本、无需训练、覆盖 47 种重构类型；局限是 API 调用成本和开源模型准确率较低。

**个人理解**：Foundation Models 在重构正确性检测上达到 93.8% 准确率，这意味着未来的 IDE 可以集成 LLM 作为重构预言机，在用户确认重构前自动检测潜在的行为变更。

## 技术趋势

1. **AI 生成代码质量从功能正确性转向架构可维护性**：AI-Generated Smells 揭示 LLM 生成代码的"机器签名"缺陷模式和 Volume-Quality Inverse Law，SSDE 提出结构化规格驱动工程。行业焦点正在从"AI 能否生成正确代码"转向"AI 生成的代码能否长期维护"。

2. **因果推理引入软件工程**：Causal Software Engineering 提出从相关性到因果性的范式转变。现有 AI 辅助 SE 工具主要基于预测分析，CSE 引入干预效果估计和反事实诊断，这对自动化决策系统至关重要。

3. **Agent 基础设施从"每次推理"到"一次推理多次执行"**：MCP Workflow Engine 提出声明性工作流蓝图，解决重复任务的 token 浪费。AOCI 提出预构建+增量更新的代码索引模式。Agent 基础设施正在从"交互式推理"向"预计算+增量更新"转变。

4. **代码审查的自动化中介趋势**：AIDev 研究发现 AI 生成 PR 的审查主要由 AI Agent 主导而非人类，人类参与通过 Agent 引导而非独立评估。这引发了关于 AI 辅助开发中人类监督有效性的新问题。

5. **测试评估从单一基准到鲁棒性和污染感知**：HEJ-Robust 揭示 APR 模型对语法变化的脆弱性，LiveFMBench 揭示形式化规范生成的污染问题，Addressing False Failures 揭示代码翻译评估中的假阴性。评估质量本身成为研究焦点。
