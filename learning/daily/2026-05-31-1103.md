### AI Testing / Software Testing（11 项）

#### 1. RADAR: Automating Low-Risk Code Review at Meta
- **地址**：https://arxiv.org/abs/2605.30208
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 代码审查自动化
- **技术介绍**：Meta 工业级代码审查自动化系统 RADAR（Risk Aware Diff Auto Review）。在多阶段漏斗中：按作者身份和来源分类 diff → 资格门控 → 静态启发式 → ML Diff Risk Score → LLM 自动代码审查 → 确定性验证 → 自动落地。覆盖 535K+ RADAR 审查的 diff，着陆 331K+。将 Diff Risk Score 阈值从 25th 百分位放宽到 50th，批准率升至 60.31%。RADAR 审查 diff 的回滚率是非 RADAR diff 的 1/3，生产事故率是 1/50。中位关闭时间降低 330%+，审查墙时间降低 35%。AI 生成代码占 Meta 新增代码的 80%+，审查带宽成为瓶颈。
- **应用场景**：企业级代码审查自动化、AI 生成代码质量控制
- **为什么值得关注**：Meta 工业级实证，首次系统展示风险感知分层自动化在代码审查中的规模化效果

#### 2. EvoRepair: Experience-Based Self-Evolving AVR Agent
- **地址**：https://arxiv.org/abs/2605.30105
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 漏洞修复
- **技术介绍**：首个基于经验自进化的自动漏洞修复（AVR）Agent 框架。EvoRepair 采用循环的 learn-and-repair 流程：检索相关历史经验指导修复 → 从修复轨迹提取新经验 → 使用质量感知评分更新经验库。在 PATCHEVAL 和 SEC-bench 上使用 GPT-5-mini 评估，总体达到 90.46%。超越最新 LLM 基线 LoopRepair 39.56%（PATCHEVAL）和 33.50%（SEC-bench），超越 IntentFix 70.86% 和 50.50%。在 VUL4J 上的迁移实验证明跨模型、语言、数据集的鲁棒性。
- **应用场景**：自动化漏洞修复、安全代码生成
- **为什么值得关注**：首次将自进化机制引入 AVR，经验积累和跨漏洞知识复用显著提升修复率

#### 3. Projectional Decoding: Semantic-Aware LLM Generation
- **地址**：https://arxiv.org/abs/2605.30054 (FSE 2026 IVR)
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 语义约束生成
- **技术介绍**：将领域语义直接集成到 LLM 生成过程中的概念框架。维护一个部分图模型作为主要工件表示，实现增量语义验证——显式捕获不确定性并原生支持错误检测。为 LLM 生成的软件工件提供可证明的语义有效性保证。在程序生成任务上的初步结果展示了该方法改善语义有效性的潜力。
- **应用场景**：代码生成语义验证、可验证自动化
- **为什么值得关注**：桥接 LLM 生成文本与 SE 语义验证推理，为可验证代码生成提供新范式

#### 4. RePoT: Recoverable Program-of-Thought via Checkpoint Repair
- **地址**：https://arxiv.org/abs/2605.30052
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 推理恢复
- **技术介绍**：一次性 PoT 发出一个 Python 程序打印原始行动计划；单个无效操作会静默使整个轨迹失效。RePoT 提供确定性验证重放：将计划遍历到第一个无效转换，然后一次 LLM 调用从验证前缀恢复。在 ~14% PoT 失败的问题上最多增加一次额外 LLM 调用。在 PuzzleZoo-775 上超越 PoT +3 到 +11pp，gpt-5.4-mini-medium 上达到 96.9% vs 86.3%。在 PlanBench Blocksworld 和四个开放权重模型上复制验证。Derail-550 对照实验显示 checkpoint 信息是关键的恢复信号。
- **应用场景**：AI 编码代理推理恢复、复杂任务规划
- **为什么值得关注**：低成本高效恢复机制，checkpoint 信息比错误反馈本身更有价值

#### 5. Agora: Autonomous Bug Detection in Consensus Protocols

#### 7. TRAILS: Inferring Code Correctness from Specification

#### 12. Compositional Incoherence in Multi-Component LLM Agents
- **地址**：https://arxiv.org/abs/2605.30335 (ICML 2026 Workshops)
- **来源**：arXiv (cs.AI)
- **领域**：AI / 多 Agent 一致性
- **技术介绍**：多组件 LLM Agent 从各只看到部分联合问题的组件组装概率声明；即使每个组件局部一致，组合也可能违反基本概率公理。形式化"局部一致、全局不一致"失败，通过组合残差 eps*（组合报价到联合一致多面体的 L2 距离）度量。分层 Boyle-Dykstra 投影确定性修复组合。1876 个集成 clique 上 eps* > 0 出现在 33-94% 的 clique 上。三种直觉 LLM 侧缓解（检索、分区感知提示、聚合 LLM）均失败或退化。
- **应用场景**：多 Agent 系统可靠性、概率推理一致性
- **为什么值得关注**：揭示多组件 Agent 系统性违反概率公理，对 Agent 集成架构提出根本挑战

#### 13. MIRA: Mid-training Rubric Anchoring for Source-Aware Data Selection
- **地址**：https://arxiv.org/abs/2605.30288
- **来源**：arXiv (cs.AI)
- **领域**：AI / 训练数据选择
- **技术介绍**：基于自锚定 rubric 发现的源感知过滤框架。MIRA 首先为每个源组发现应评估什么，然后将这些判断蒸馏为可扩展的学生评分器用于全语料库过滤。在 21 源 5 源组的代码导向 mid-training 上，MIRA 在九个代码基准上超越选择基线，仅使用一半 token 即匹配全语料库运行。
- **应用场景**：LLM 训练数据选择、代码模型优化
- **为什么值得关注**：将 rubric 构建作为数据选择的一部分，源自适应语义标准

#### 14. MMPO: Meta-Cognitive Memory Policy Optimization
- **地址**：https://arxiv.org/abs/2605.30159
- **来源**：arXiv (cs.AI)
- **领域**：AI / Agent 记忆优化

#### 1. Microsoft Agent Governance Toolkit — Agent 治理从 Prompt 到确定性代码

**地址**：https://github.com/microsoft/agent-governance-toolkit

**项目介绍（300 字）**：
Microsoft 出品的 AI Agent 治理工具包，覆盖 OWASP Agentic Top 10 全部 10 项风险。核心哲学是"Prompt-level safety 是对随机系统的礼貌请求，不是控制面"。AGT 在每个工具调用、消息发送和委托到达网络之前，在确定性应用代码中拦截。被 AGT kernel 拒绝的 Actions 不是"不太可能"发生，而是"结构上不可能"。

架构分为三层：
1. **Policy Enforcement**：YAML 策略定义，每工具调用检查、日志记录、执行
2. **Zero-Trust Identity**：多 Agent 系统中每个 Agent 的独立身份追踪
3. **Execution Sandboxing + SRE**：执行隔离和可靠性工程

两行代码即可治理任何工具函数：`safe_tool = govern(my_tool, policy="policy.yaml")`。支持 Claude Code 作为 marketplace 插件安装。

**技术架构**：
- Python 核心库 + TypeScript SDK + .NET NuGet 包
- 策略引擎：YAML 声明式策略 → 确定性评估 → GovernanceDenied 异常
- 身份层：每个 Agent 独立身份，解决"五个 Agent 共享一个 API key"的归因问题
- 审计层：防篡改记录——哪些策略生效、Agent 请求了什么、为什么允许或拒绝

**核心算法/系统设计**：
- 策略评估在工具调用前执行，不是事后审计
- 分层治理：工具级 → 消息级 → 委托级
- 与任何框架兼容：不绑定特定 Agent 框架

**与现有技术对比**：
| 维度 | Prompt 级安全 | AGT |
|------|-------------|-----|
| 保证强度 | 概率性 | 结构性不可能 |
| 可审计性 | 低 | 防篡改日志 |
| 身份归因 | 共享 API key | 每 Agent 独立身份 |
| OWASP 覆盖 | 部分 | 10/10 |

**优势与局限**：
- 优势：Microsoft 背书、OWASP 全覆盖、两行代码集成、跨框架
- 局限：Public Preview 阶段、可能有 breaking changes、策略设计需要安全专业知识

**未来发展潜力**：
Agent 治理将成为企业级 Agent 部署的标准基础设施。AGT 的"结构性不可能"哲学可能成为 Agent 安全的基准范式。

**个人技术理解**：
AGT 代表了 Agent 安全从"软约束"到"硬约束"的范式转变。当 Agent 开始自主决策时，prompt 级别的"请遵守规则"本质上是对随机系统的礼貌请求——这在安全关键场景中是不可接受的。AGT 将治理下沉到确定性应用代码层，使违规操作在结构上不可能发生，这是工程化的安全思维。

### 学术论文

#### 3. RADAR: Automating Low-Risk Code Review at Meta — 风险感知分层自动化

**地址**：https://arxiv.org/abs/2605.30208

**论文介绍（350 字）**：
Meta 工业级代码审查自动化的系统性实证研究。AI 辅助编码工具已改变软件生产——Meta 每人落地 diff 的代码行数同比增长 105.9%，每开发者 diff 量增长 51%，其中 80%+ 的增长由 agentic AI 驱动。但收到及时审查的 diff 比例在下降，暴露了代码供应和审查带宽之间不断扩大的差距。

RADAR（Risk Aware Diff Auto Review）是多阶段漏斗：
1. 按作者身份和来源类型分类每个 diff
2. 应用资格门控
3. 静态启发式
4. ML Diff Risk Score
5. LLM 自动代码审查
6. 确定性验证
7. 自动落地合格变更

**技术架构**：
- 535K+ RADAR 审查 diff 的遥测数据
- 观测前后比较 + 差分分析效率结果
- 风险阈值可调（25th → 50th 百分位）

**核心发现**：
- 批准率 60.31%（50th 百分位阈值）
- 回滚率是非 RADAR diff 的 1/3
- 生产事故率是非 RADAR diff 的 1/50
- 中位关闭时间降低 330%+
- 审查墙时间降低 35%

**与现有技术对比**：
传统代码审查完全依赖人工，RADAR 的风险感知分层自动化将低风险 diff 自动审查落地，人工审查聚焦高风险变更。

**优势与局限**：
- 优势：Meta 工业级规模验证、风险分层设计、可观测性强
- 局限：需要大量历史数据训练 Risk Score、对新型代码模式可能不够敏感

**未来发展潜力**：
RADAR 代表了代码审查从"全人工"到"风险感知自动化"的范式转变。随着 AI 生成代码占比持续上升，此类系统将成为开发基础设施的标准组件。

**个人技术理解**：
RADAR 的核心洞察是"不是所有 diff 都需要同等审查"。通过风险分层，将有限的人工审查带宽聚焦在真正需要人类判断的高风险变更上。1/50 的生产事故率证明风险感知分层自动化的有效性。这对 AI 编码代理的质量控制有直接启示——不是所有 AI 生成代码都需要人工审查，但需要可靠的风险评估。

#### 4. Agora: Autonomous Bug Detection in Consensus Protocols — 领域感知多 Agent 协议验证
