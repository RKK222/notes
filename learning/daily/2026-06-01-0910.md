## 技术扫描（25 项新增）

### AI Testing / Software Testing（9 项）

#### 1. RedundancyBench: Detecting Redundant Steps in Agent Trajectories
- **地址**：https://arxiv.org/abs/2605.29893
- **来源**：arXiv (cs.AI)
- **领域**：AI Testing / Agent 轨迹效率
- **技术介绍**：首次提出 Agent 轨迹冗余步骤检测这一新研究方向。现有评估协议主要关注任务成功，忽略了执行效率这一关键方面。RedundancyBench 包含多样化任务和精心标注的轨迹，每个步骤根据对任务完成的贡献进行标注。开发了 3 种代表性方法评估步骤是否冗余，结果最佳方法仅 24.88% 得分，某些方法比随机猜测还差。揭示了任务的复杂性和进一步研究的必要性。
- **应用场景**：Agent 执行效率评估、资源优化
- **为什么值得关注**：首次系统关注 Agent 执行效率而非仅任务成功，对降低 Agent 运行成本有直接价值

#### 2. Moment-KV: Momentum-Based KV Cache Compression
- **地址**：https://arxiv.org/abs/2605.29873
- **来源**：arXiv (cs.AI)
- **领域**：AI Testing / 推理优化
- **技术介绍**：针对长生成任务中 KV 缓存瓶颈的解码时压缩方法。分析注意力动态发现强时间模式：关键 token 在长范围内持续获得注意力，而局部推理涉及短促爆发。提出基于动量的时间注意力聚合，将 token 重要性建模为连续演化状态，注意力通过衰减聚合，同时捕获长期影响和近期相关性。在长生成任务中显著提升生成保真度 2.3-3.2%，同时保持解码延迟。
- **应用场景**：长文本生成、Agent 长上下文推理
- **为什么值得关注**：首次利用注意力时间动态模式进行 KV 缓存压缩，比静态启发式方法更智能

#### 3. CFGzip: Accelerating Constrained Decoding with Token Space Compression
- **地址**：https://arxiv.org/abs/2605.29986 (EMNLP 2026 under review)
- **来源**：arXiv (cs.AI)
- **领域**：AI Testing / 约束解码优化
- **技术介绍**：针对 CFG 约束解码引擎的离线 token 搜索空间压缩技术。当前 CFG 约束解码引擎高度优化，但每步搜索空间（整个 token 词汇表）的固有成本导致复杂 CFG 下开销难以承受。CFGzip 通过压缩 token 搜索空间大幅减少 CFG 引擎开销，实验报告延迟降低高达两个数量级，与 SoTA 语法引擎结合产生高达 7.5x 的总约束生成时间加速。
- **应用场景**：代码生成、结构化输出、形式化验证
- **为什么值得关注**：使复杂 CFG 的约束解码在规模化场景中可行，对代码生成质量保障有直接价值

#### 4. OmniMatBench: Human-Calibrated Multimodal Reasoning Benchmark
- **地址**：https://arxiv.org/abs/2605.29833
- **来源**：arXiv (cs.AI)
- **领域**：AI Testing / 多模态推理基准
- **技术介绍**：材料科学领域人类校准多模态推理基准，包含 3,171 个专家策划的 QA 和计算问题，覆盖 19 个材料科学子领域。评估 13 个开源和闭源 MLLM，最佳模型总体得分仅 0.372，揭示当前材料科学推理的巨大差距。分析显示子领域间强差异、固定推理启发式、不均匀材料知识和有限的高级知识应用。
- **应用场景**：多模态科学推理评估、材料 AI
- **为什么值得关注**：首个覆盖 19 个子领域的材料科学多模态基准，揭示 MLLM 科学推理能力的系统性不足

#### 5. VirtualME: Personalized Code Intelligence from IDE Behaviors
- **地址**：https://arxiv.org/abs/2605.29372 (FSE 2026)
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 个性化代码智能
- **技术介绍**：IDE 嵌入式数据基础设施，通过持续捕获和解释开发者动态编程行为来建模开发者。包含三个组件：(1) 日志级行为提取：从 IDE 捕获开发者日志级行为；(2) 任务级行为识别：通过多 Agent 管道聚合为任务级行为；(3) 开发者个性测量：构建规则引擎提炼四维开发者画像（技术栈、能力、行为习惯、学习风格）。在仓库级知识 Q&A 中集成开发者画像，比通用基线平均提升 33.80%。
- **应用场景**：个性化代码助手、开发者体验优化
- **为什么值得关注**：首次将开发者行为数据系统性地融入代码智能，从"一刀切"到"因人而异"

#### 6. Code-QA-Bench: Separating Code Reasoning from Documentation Memorization
- **地址**：https://arxiv.org/abs/2605.29277
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 代码理解评估
- **技术介绍**：全自动框架，合成仓库级代码理解基准，分离真正的代码理解与文档回忆和预训练记忆化。两个方法学贡献：(1) 答案先生成管道，工具辅助 Agent 探索源代码产生验证的黄金答案再推导问题；(2) 三条件实验设计：闭卷（无仓库）、仅代码（移除文档）、有文档（完整仓库），delta 直接量化文档效用和记忆化。在 10 个 SWE-Bench Python 仓库上生成 528 个代码推导和 100 个文档依赖任务。
- **应用场景**：代码理解基准、Agent 代码推理评估
- **为什么值得关注**：首次系统分离代码理解和文档记忆化，对评估 Agent 真实代码理解能力至关重要

#### 7. Configurator UI Usability Analysis with MLLM
- **地址**：https://arxiv.org/abs/2605.29456
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / UI 可用性评估
- **技术介绍**：使用多模态大语言模型对配置器 UI 进行可扩展半自动化可用性分析。从文献中综合 18 个配置器特定可用性标准，应用于 16 个真实世界配置器的 MLLM 分析。每个标准单独评估生成可用性问题的严重性评级和可操作的改进建议。结果确认 MLLM 可以可靠识别配置器特定可用性问题并提供领域感知建议。
- **应用场景**：配置器 UI 测试、自动化可用性评估
- **为什么值得关注**：将 MLLM 应用于特定领域 UI 可用性分析，可扩展且成本效益高

#### 8. Generalized Software Product Line Extraction
- **地址**：https://arxiv.org/abs/2605.28989
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / 软件产品线
- **技术介绍**：通用、工作bench 无关的协议，从现有软件工件提取功能模型并配置派生软件产品。自底向上方法基于称为"原子"的轻量级依赖单元，围绕 SPL 服务器（工作bench 独立）和 SPL 客户端（工作bench 特定后端 + 通用前端）组织提取和配置过程。对底层软件工件假设很少，适用于各种 SPL。在语言产品线 (LPL) 上下文中演示适用性。
- **应用场景**：软件产品线工程、变体管理
- **为什么值得关注**：解耦 SPL 工程与特定技术栈，促进异构开发环境中的复用和演进

#### 9. LogDx-CI: Benchmarking Log Reduction for LLM Diagnosis
- **地址**：https://arxiv.org/abs/2605.28876
- **来源**：arXiv (cs.SE)
- **领域**：AI Testing / CI 日志诊断
- **技术介绍**：比较 11 种上下文缩减工具在 35 个真实 GitHub Actions 失败案例上的表现。发现：混合 grep+tail 路由器主导成本-质量 Pareto 前沿；在 agent-loop 机制中，缩减工具质量范围坍缩 7 倍，代理通过后续工具调用恢复弱上下文；跨家族 LLM 摘要-调试器配对优于同家族配对 +0.071。
- **应用场景**：CI/CD 故障诊断、日志分析自动化
- **为什么值得关注**：首次实证比较日志缩减工具对下游 LLM 诊断的影响

#### 3. On the Geometry of Games and their Solvers
- **地址**：https://arxiv.org/abs/2605.29919
- **来源**：arXiv (cs.AI)
- **领域**：AI / 博弈论
- **技术介绍**：通过求解器-博弈映射研究均衡计算，将博弈连接到有效求解器动态。形式化结构感知求解器合成：学习的结构识别器将每个博弈映射到低维求解器对齐表示，策略将此表示映射到有效原始机制。有界残差作为局部校正器和不完全求解器基或表示的诊断信号。固定原始机制表现出系统性机制不匹配，而学习表示将博弈空间组织为与求解器行为对齐的结构化制图。
- **应用场景**：GAN 训练、多 Agent 系统、均衡计算
- **为什么值得关注**：将均衡计算从离散分类转向连续几何视角，自适应求解器合成新范式

#### 4. HTP: Flexible Semantic Trajectory Generation with LLMs
- **地址**：https://arxiv.org/abs/2605.30014 (KDD 2026)
- **来源**：arXiv (cs.AI)
- **领域**：AI / 轨迹生成

#### 5. Compass: Expert-Guided LLM Agent for Marine Data Integration

## 深度研究

### GitHub 项目

#### 1. Streambed: Stream Postgres to Iceberg on S3
- **地址**：https://github.com/viggy28/streambed
- **项目介绍**：Streambed 是一个极简架构的数据流式传输工具，将 Postgres WAL 更改直接流式传输到 S3 上的 Apache Iceberg，支持通过嵌入式 DuckDB 从 psql 查询。作者曾是 Cloudflare Postgres 团队技术负责人，该项目源于解决实际挑战：BI 和仪表板团队需要运行长时间分析查询，传统方案是旋转额外的定制读副本或将 ETL 转储到分析数据库。Streambed 采用最少组件方法——作为逻辑复制订阅者连接 Postgres，将 WAL 更改直接流式传输到 Iceberg，支持 Postgres Wire 协议。
- **技术架构**：Postgres 逻辑复制 → WAL 流式传输 → Apache Iceberg on S3 → 嵌入式 DuckDB 查询
- **核心设计**：最简组件架构，避免传统 ETL 管道的复杂性
- **关键机制**：逻辑复制订阅、WAL 解析、Iceberg 写入、DuckDB 嵌入式查询
- **与现有技术对比**：比 Debezium + 独立 ETL 更简单，比 pg_lake 有更多 pushdown 能力
- **优势与局限**：优势是架构极简、组件少、维护成本低；局限是早期阶段、边缘情况处理待完善
- **未来发展潜力**：如果成熟，可能成为 Postgres 用户实时数据分析的标准基础设施
- **个人技术理解**：代表了"少即是多"的架构哲学，在数据工程领域越来越受欢迎

#### 2. Code-QA-Bench 框架（开源）
- **地址**：https://arxiv.org/abs/2605.29277
- **项目介绍**：全自动仓库级代码理解基准合成框架，分离真正的代码理解与文档回忆和预训练记忆化。答案先生成管道确保每个任务扎根于真实代码结构，三条件实验设计（闭卷/仅代码/有文档）直接量化文档效用和记忆化效应。开源适用于任何文档完善的 Python 仓库。
- **技术架构**：工具辅助 Agent 探索 → 验证黄金答案 → 问题生成 → 三条件评估
- **核心算法**：答案先生成管道、类别分区测试输入生成、LLM 评分聚合
- **关键实现机制**：SWE-Bench 仓库集成、LLM-as-judge 评分、delta 分析
- **与现有技术对比**：比传统代码理解基准更严格，首次分离代码理解和文档记忆
- **优势与局限**：优势是方法学严谨、开源可用；局限是当前仅支持 Python 仓库
- **未来发展潜力**：可扩展到其他语言，成为代码理解评估的标准基准
- **个人技术理解**：对 Agent 代码理解能力的评估至关重要，避免"假理解"

### 学术论文

#### 1. Teaching Values to Machines: Simulating Human-Like Behavior in LLMs
- **地址**：https://arxiv.org/abs/2605.30036
- **论文介绍**：使用已建立的心理价值观理论（Schwartz 价值观circumplex）在 LLM 中诱导人类-like 价值观，并通过超过 500 万问题的实验评估其与人类价值观结构的一致性。使用验证的心理问卷测量价值观结构和价值观-行为关系，与人类研究数据进行系统比较。发现价值观提示的 LLM 与人类在两个维度上都表现出强烈一致性，结合人类价值观分布可增强人口级模拟的保真度。
- **技术架构**：心理价值观理论 → LLM 价值观提示 → 大规模问卷实验 → 人类数据对比
- **核心算法**：Schwartz 价值观框架映射、价值观-行为关系建模、人口分布对齐
- **关键实现机制**：500 万+ 问题实验设计、多维度一致性评估、人口级模拟增强
- **与现有技术对比**：比之前的对齐研究更系统，首次在价值观结构层面进行大规模验证
- **优势与局限**：优势是规模巨大（500 万+ 问题）、理论扎实；局限是仅测试价值观提示，未涉及价值观学习
- **未来发展潜力**：为人口模拟、社会科学研究、价值观对齐提供新工具
- **个人技术理解**：这项研究将 AI 对齐从"安全"扩展到"价值观"，为 AI 社会集成提供理论基础

#### 2. Multi-Phase Inference Framework for World-Model Alignment
- **地址**：https://arxiv.org/abs/2605.29930
- **论文介绍**：提出多阶段推理机制 (MIM)，形式化异构世界模型如何通过阶段形成空间、前景场、主体特定轮廓状态和状态表示之间的对齐图产生。将世界模型对齐重新框架为使异构表示可相互处理，而非强迫一致或收敛到单一价值系统。连接到哲学分歧、认知类型学、社会碎片化和 AI 对齐，为 AI 系统提供建设性词汇帮助人类理解自我和他人。
- **技术架构**：阶段形成空间 → 前景场 → 主体轮廓状态 → 对齐图
- **核心算法**：多阶段推理、预测误差更新、世界模型对齐映射
- **关键实现机制**：异构表示可处理性、意义/价值/预测误差的可见化/可比化/可转换化
- **与现有技术对比**：比传统对齐研究更关注"差异"而非"一致"，提供新的理论框架
- **优势与局限**：优势是理论创新、跨学科整合；局限是偏理论，实证验证有限
- **未来发展潜力**：为 AI 辅助跨文化理解、社会计算、认知科学提供新视角
- **个人技术理解**：这项研究将 AI 对齐从技术问题解决转向社会认知问题，代表对齐研究的新方向

1. **Agent 执行效率评估兴起**：RedundancyBench 首次提出 Agent 轨迹冗余检测，代表 Agent 评估从"能否完成任务"向"完成任务的效率如何"转变。随着 Agent 部署成本成为实际关切，效率评估将成为与准确性同等重要的维度。
