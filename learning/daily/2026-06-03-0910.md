#### 1. headroom — LLM 工具输出压缩代理
- **地址**：https://github.com/chopratejas/headroom
- **来源**：GitHub Trending #1（6,445 stars，1,265 stars/日）
- **领域**：AI Testing / Agent 效率
- **技术介绍**：headroom 是一个 LLM 前置压缩工具，在工具输出、日志、文件和 RAG chunks 到达 LLM 之前进行压缩，实现 60-95% 的 token 减少，同时保持相同的答案质量。提供 Library、Proxy 和 MCP Server 三种部署模式。在 AI 编码工具按使用量计费（GitHub Copilot 新计费模式）的背景下，token 效率成为核心竞争力。headroom 代表了一种"在代理管道中插入压缩层"的新架构思路，与之前探索的 8v 统一二进制（-66% token）和 DeepClaude（17x 更便宜）形成互补的效率优化路径。
- **应用场景**：AI 编码代理 token 优化、RAG 系统成本降低、MCP 工具调用压缩
- **为什么值得关注**：日增 1,265 stars 反映社区对 token 效率的迫切需求，代理效率优化从"模型选择"扩展到"管道优化"

#### 2. ECC — Agent Harness 性能优化系统
- **地址**：https://github.com/affaan-m/ECC
- **来源**：GitHub Trending #3
- **领域**：AI Testing / Agent Harness
- **技术介绍**：ECC（Efficient Coding Companion）是一个面向 Claude Code、Codex、Opencode、Cursor 等编码代理的 harness 性能优化系统。提供 Skills、Instincts、Memory、Security 和 Research-First Development 五大模块。与之前探索的 jcode（Rust harness）、8v（统一二进制）、Dirac（AST 驱动）形成代理 harness 生态的多样化竞争格局。ECC 的独特之处在于"Instincts"概念 — 将编码模式编码为代理的本能反应，减少推理步骤。
- **应用场景**：AI 编码代理性能调优、多代理平台统一优化
- **为什么值得关注**：Agent harness 从"能用"到"高效"的进化，Instincts 概念代表编码模式知识化的新方向

#### 3. Monitoring Agentic Systems Before They're Reliable — 不可靠代理系统监控方法论
- **地址**：https://arxiv.org/abs/2606.02494
- **来源**：arXiv cs.SE（AgenticSE @ CAIS 2026 Workshop）
- **领域**：AI Testing / Agent 可靠性
- **技术介绍**：提出一个三维度（质量、适用性、效率）× 三监控范围（运行内、跨运行、结构）的代理系统监控方法论。关键发现：结构缺陷（而非任务级错误）主导早期失败模式；注入的任务级错误在结构缺陷存在时无法被检测到（CV=0 的结构监控完美识别集成缺口）；97% 的发现可自动化追踪，仅 2% 需人工调查。提出基于成熟度分级的监控演进模型：从结构特征化→错误检测→可靠性追踪。
- **应用场景**：多阶段代理工作流监控、受监管行业代理部署
- **为什么值得关注**：首个针对"不可靠阶段"代理系统的系统化监控框架，CV 基范围特征化方法可操作

#### 4. TEL-DRIFT — 深度研究代理轨迹 Span 级错误定位
- **地址**：https://arxiv.org/abs/2606.02060
- **来源**：arXiv cs.AI
- **领域**：AI Testing / Agent 评估
- **技术介绍**：收集 2,790 条真实深度研究代理轨迹（2 框架 × 3 模型 × 3 基准），构建 TELBench（1,000 实例）用于在正常探索、失败搜索、临时假设和无害噪声中识别错误 span。提出 DRIFT 框架：基于声明的审计方法，追踪代理声明、检查轨迹证据支持、标记影响答案路径的不支持/冲突声明 span。实验显示 DRIFT 将 span 级错误定位和首次错误准确率提升高达 30 个百分点。
- **应用场景**：深度研究代理过程级诊断、代理可靠性审计
- **为什么值得关注**：从"最终答案评估"升级到"轨迹 span 级诊断"，代表 Agent 评估从结果导向到过程导向的范式转变

#### 5. POIROT — 多代理系统自检故障检测协议
- **地址**：https://arxiv.org/abs/2606.02282
- **来源**：arXiv cs.AI（44 页，6 图）
- **领域**：AI Testing / 多代理安全
- **技术介绍**：POIROT 协议将多代理系统自身的代理重新利用为诊断层，利用架构中已有的认知多样性进行故障检测。实验显示 POIROT 在复杂问题上超越单 LLM 评估基线（OR=1.60, p=0.008），增益随问题复杂度、代理数量和故障维度扩展。在复合故障条件下仍保持有效。核心洞察：安全监督无需外部化 — 执行角色的代理携带足够的集体智能来审计自身。
- **应用场景**：安全关键领域多代理系统自检、AI 法规合规审计
- **为什么值得关注**："让代理审计代理"的自监督安全范式，与 EU AI Act 合规需求高度契合

#### 6. SBOM 组件包含机制分析 — 供应链安全基础漏洞
- **地址**：https://arxiv.org/abs/2606.02442
- **来源**：arXiv cs.SE/cs.CR
- **领域**：Software Testing / 供应链安全
- **技术介绍**：系统分析 cdxgen、syft、trivy、ORT、Microsoft sbom-tool 五种 SBOM 生成工具，覆盖 Python/Java/Go/PHP/Rust/C 六种语言。发现：没有工具覆盖所有组件包含机制（CIM）；工具间存在共同盲区；在当前模糊定义和工具下，SBOM 存在歧义和盲点，无法实现安全级 SBOM。结论：需要回到起点澄清"什么是组件"，否则供应链安全努力将失败。

#### 7. LLM Go 代码审查改进 — Issue-List + 上下文增强
- **地址**：https://arxiv.org/abs/2606.01859
- **来源**：arXiv cs.SE
- **领域**：Software Testing / 代码审查
- **技术介绍**：提出 Issue-List 审查范式（枚举所有潜在问题而非仅报告最重要问题），系统比较三种上下文增强（邻居、LSP 语义、IR 相似协同变更），集成无上下文和上下文增强生成结果，引入精炼引导剪枝。在 1,438 个 Go 审查实例上，最佳配置达到 28.00% 精炼精确匹配（+10.85pp vs 无上下文单问题），显著超越 CodeReviewer（15.02%），接近人类上限（36.09%）。剪枝将候选从 7.2 降至 3.1 同时保留几乎全部收益。
- **应用场景**：LLM 辅助代码审查、Go 语言项目
- **为什么值得关注**：Issue-List 范式 + 上下文增强的组合策略，为 LLM 代码审查提供可操作的改进路径

#### 8. AI 辅助需求工程工业实证 — 工具集成是瓶颈
- **地址**：https://arxiv.org/abs/2606.01772
- **来源**：arXiv cs.SE（19 页）
- **领域**：Software Testing / 需求工程
- **技术介绍**：在 XITASO（中型高科技软件工程企业）进行工业实证，结合 2024 全公司用例调查 + 两轮半结构化访谈（8 个产品负责人）。发现 15 种用例（产品待办管理、招标管理、需求理解、文档创建），三个关键发现：(1) AI 对 PO-开发者交互影响混合，单用户交互模式可能替代协作对话；(2) 工具集成（而非工具能力）是绑定约束；(3) AI 比组织系统进步更快，收益归于个人 PO 而团队流程和客户准备度仍是限制因素。

- **为什么值得关注**：首个 AI 辅助 RE 工业实证，揭示"工具集成 > 工具能力"和"AI > 组织"的结构性矛盾

#### 9. ML 代码异味实证 — 领域特异性质量策略
- **地址**：https://arxiv.org/abs/2606.01882
- **来源**：arXiv cs.SE
- **领域**：Software Testing / 代码质量
- **技术介绍**：在 279 个开源 ML 项目上，使用 CodeSmile（ML 代码异味）和 Pylint（通用 Python 异味）分析六种项目特征（规模、年龄、贡献者、提交频率、CI/CD 采用、领域）与代码质量的关系。发现：(1) ML 代码异味比通用 Python 异味少 41-94 倍；(2) 提交频率和领域与 ML 特异性质量显著相关，而项目规模/团队规模/年龄/CI/CD 不相关；(3) 通用 Python 异味与任何项目特征无关（系统性问题）；(4) MLOps 多配置问题、RL 多张量操作问题、CV 多 GPU 工作流问题。
- **应用场景**：ML 项目代码质量评估、领域特异性 CI/CD 质量门
- **为什么值得关注**：挑战"一刀切"代码质量策略，证明 ML 项目需要领域特异性质量保障

#### 10. 代理复现包质量评估 — 380 需求→51 标准→31 自动化
- **地址**：https://arxiv.org/abs/2606.02006
- **来源**：arXiv cs.SE（Emerging Results）
- **领域**：Software Testing / 研究可复现性
- **技术介绍**：将 34 个来源的 380 个开放科学需求整合为 51 个可复现性标准，其中 31 个可操作化为自动化评估。实现多代理原型自动检查复现包并生成证据驱动的改进报告。在 5 个复现包上达到 91.4% 运行间一致性和 75.4% 正确率（与人工基线微平均一致）。代理在结构标准（代码、环境、工件可用性）上表现最佳，但在定性/混合方法研究上挣扎。
- **应用场景**：SE 研究可复现性自动评估、期刊/会议工件审查
- **为什么值得关注**：将开放科学指南机器可验证化，代理辅助研究质量保障新方向

#### 11. LLM 生成多文件变更审查工作流 — 信任校准问题
- **地址**：https://arxiv.org/abs/2606.01969
- **来源**：arXiv cs.SE/cs.HCI（JetBrains 合作，ESEM SEIP 2026）
- **领域**：Software Testing / 代码审查
- **技术介绍**：与 JetBrains 合作的参与式设计研究（N=17 发现阶段 + N=7 开发阶段 + N=43 验证调查）。核心发现：审查 LLM 生成多文件变更是"信任校准问题"而非"diff 问题"。提出三级审查工作流（概览→文件分析→代码片段审查）和七个设计构件（chunk、risk-per-line、risk-per-file、judge、walk-through、zooming in/out、security cage）。63% 受访者预期减少总体审查工作量，52% 预期减少信任评估工作量。

- **为什么值得关注**：首个针对 LLM 生成多文件变更审查的实证研究，从"信任校准"角度重新定义问题

### AI / LLM 技术（13 项）

#### 12. SISA — SSM-Informed Softmax Attention（第三设计轴）
- **地址**：https://arxiv.org/abs/2606.02332
- **来源**：arXiv cs.AI/cs.CL/cs.LG（20 页，6 图，25 表）
- **领域**：AI / LLM 架构
- **技术介绍**：SISA（SSM-Informed Softmax Attention）将 SSM 导出的重要性项直接添加到注意力分数中，作为单个 SDPA 调用在增强的 query/key 向量上实现 — 无需循环状态、无需自定义 kernel。在 152M/5B token 上，SISA 达到 LAMBADA-greedy 17.3%（vs Transformer 13.9%、Mamba-3 15.5%），NIAH 从 1K 步达到 100%（比 Transformer 检索收敛快 7x）。定义了 SSM-Attention 混合的第三设计轴 — 分数级融合，超越 Jamba（块级）和 Hymba（头级）范式。
- **应用场景**：高效 LLM 架构设计、长上下文检索加速
- **为什么值得关注**：第三设计轴定义，单一 SDPA 调用实现 SSM-Transformer 融合，工程友好

#### 13. COMAP — 世界模型与代理策略协同进化
- **地址**：https://arxiv.org/abs/2606.02372
- **来源**：arXiv cs.AI/cs.CL
- **领域**：AI / 世界模型
- **技术介绍**：COMAP 框架通过闭环交互协同进化文本世界模型和代理策略。每步决策中，世界模型为候选动作预测未来状态反馈，代理进行未来感知反思（估计反馈可靠性并精炼动作）。结果 on-policy 轨迹通过自蒸馏更新世界模型，使其更好地匹配代理不断演化的交互分布。在具身任务规划、Web 导航和工具使用基准上持续超越基线（Qwen3-4B 上 +16.75% 相对提升）。协同进化循环随时间提高世界模型预测精度。
- **应用场景**：长程决策代理、Web 导航代理、具身 AI
- **为什么值得关注**：解决"固定世界模型无法适应演化代理"的核心问题，闭环协同进化新范式

#### 14. SIRI — 自内部化 RL 内在技能训练
- **地址**：https://arxiv.org/abs/2606.02355
- **来源**：arXiv cs.AI
- **领域**：AI / 强化学习
- **技术介绍**：SIRI 三阶段框架使代理发现、验证和内部化技能，无需外部技能生成器或推理时技能库。(1) 用 GiGPO 预热策略获取基本交互能力；(2) 自技能挖掘 — 当前策略从自身成功轨迹中总结紧凑技能并通过配对验证；(3) 仅将有益技能引导动作 token 蒸馏到纯策略中。推理时代理仅用原始 prompt 运行。在 ALFWorld 上从 0.908 提升到 0.930，WebShop 上从 0.728 提升到 0.813（Qwen2.5-7B）。
- **应用场景**：长程 LLM 代理技能训练、无外部依赖技能库
- **为什么值得关注**："技能内部化"而非"技能检索"，推理时零额外开销

#### 15. EAPO — 代理 RL 中抑制工具滥用
- **地址**：https://arxiv.org/abs/2606.02132
- **来源**：arXiv cs.AI
- **领域**：AI / 强化学习
- **技术介绍**：EAPO（Efficient Agentic Policy Optimization）解决代理 RL 中的工具滥用问题（模型对内部可解问题也过度使用外部工具）。引入：(1) 每 rollout 组包含无工具轨迹；(2) 难度感知奖励塑形 — 主要在简单查询上惩罚冗余工具调用；(3) 置信度感知 token 重加权。在 9 个数学和知识密集型推理基准上，相比 GRPO 平均性能提升 7-10%，同时工具调用减少 18-25%。证明代理可以学习"何时不用工具"。
- **应用场景**：代理推理效率优化、工具调用成本控制
- **为什么值得关注**：从"强制限制工具使用"到"学习何时不用"，代理效率优化新方向

#### 16. SafeSteer — 局部 On-Policy 安全对齐蒸馏
- **地址**：https://arxiv.org/abs/2606.02530
- **来源**：arXiv cs.AI
- **领域**：AI / 安全对齐
- **技术介绍**：SafeSteer 将 on-policy 蒸馏限制在安全 token 上。首先通过激活引导构建安全教师模型，然后开发安全 token 选择算法，训练时将反向 KL 惩罚限制在这些 token 上以保留通用能力。仅需 100 个有害样本（不使用任何通用数据，不到之前基线的 1%），在 7 个安全基准上达到强安全性能，5 个通用能力基准上仅有微小退化。核心论点：安全特征在输出分布中本质稀疏，对齐需要局部修改而非全局权衡。
- **应用场景**：LLM安全对齐、对齐税最小化
- **为什么值得关注**：100 样本对齐 vs 之前需要海量数据，对齐成本降低 100x+

#### 1. headroom — LLM 工具输出压缩代理

**标题**：headroom — Compress tool outputs, logs, files, and RAG chunks before they reach the LLM

**地址**：https://github.com/chopratejas/headroom

**项目介绍（300 字）**
headroom 是 2026 年 6 月 GitHub 上增长最快的项目之一（6,445 stars，日增 1,265 stars），代表 AI 编码工具链效率优化的新方向。在 GitHub Copilot 转向按使用量计费的背景下，token 效率从"nice-to-have"变成"must-have"。headroom 在工具输出、日志、文件和 RAG chunks 到达 LLM 之前进行智能压缩，实现 60-95% 的 token 减少，同时保持相同的答案质量。

headroom 提供三种部署模式：Library（直接集成到代码中）、Proxy（作为中间件拦截和压缩）、MCP Server（与 MCP 协议集成）。这种多模式设计使其可以灵活适配不同的代理架构。

与之前探索的效率优化工具有显著差异：8v 通过统一二进制减少代理原生工具调用（-66% token），DeepClaude 通过后端替换降低成本（17x），而 headroom 通过内容压缩减少输入 token。三者互补而非竞争，代表代理效率优化的三个正交维度。

**技术架构**
- 三层压缩管道：预处理 → 智能压缩 → 后处理
- Library 模式：直接 API 集成，最小延迟开销
- Proxy 模式：HTTP 中间件，透明拦截
- MCP Server 模式：与 Model Context Protocol 深度集成

**核心算法或系统设计**
- 上下文感知压缩：识别 LLM 可忽略的冗余信息（日志时间戳、重复结构、调试信息）

#### 4. TEL-DRIFT — 深度研究代理轨迹 Span 级错误定位

**标题**：Where Do Deep-Research Agents Go Wrong? Span-Level Error Localization in Agent Trajectories

**地址**：https://arxiv.org/abs/2606.02060

**论文介绍（300 字）**
TEL-DRIFT 是深度研究代理评估从"结果导向"到"过程导向"的标志性工作。当前深度研究代理评估仅看最终答案是否正确，但无法诊断"哪一步错了"。本文收集 2,790 条真实深度研究代理轨迹（2 框架 × 3 模型 × 3 基准），构建 TELBench（1,000 实例）用于在正常探索、失败搜索、临时假设和无害噪声中识别错误 span。

提出 DRIFT 框架：基于声明的审计方法，追踪代理声明、检查轨迹证据支持、标记影响答案路径的不支持/冲突声明 span。实验显示 DRIFT 将 span 级错误定位和首次错误准确率提升高达 30 个百分点。

**技术架构**
- 轨迹语义化：原始日志 → 语义 span
- 声明追踪：从轨迹中提取代理声明
- 证据检查：验证声明与轨迹证据的一致性
- 错误标记：标记影响答案路径的错误 span

**核心算法**
- Claim-centric auditing：以声明为中心的审计框架
- Span-level error localization：从轨迹级到 span 级的细粒度错误定位
- 证据支持检查：声明与轨迹证据的匹配算法

**与现有技术对比**
| 维度 | TEL-DRIFT | 传统评估 | TraceGraph |
|------|-----------|----------|------------|
| 粒度 | Span 级 | 轨迹级/答案级 | 决策图谱 |
| 诊断能力 | 首次错误定位 | 无 | 过程诊断 |
| 提升 | +30pp | - | +3.1% SWE-bench |

**优势与局限**
- 优势：细粒度错误定位、可操作诊断、30pp 显著提升
- 局限：需要人工标注训练数据、对复杂推理链的 span 边界定义模糊

**未来发展潜力**
- 可能成为深度研究代理的标准诊断工具
- 与代理训练循环集成，实现"诊断→修复→验证"闭环

**个人技术理解**
TEL-DRIFT 代表 Agent 评估从"通过率"到"过程诊断"的范式转变。与 TraceGraph（共享决策图谱）形成互补：TraceGraph 回答"哪些决策路径相似"，TEL-DRIFT 回答"哪一步错了"。两者结合可实现完整的代理过程级诊断基础设施。

#### 5. POIROT — 多代理系统自检故障检测

**标题**：Interrogating Agents for Failure Detection in Multi-Agent Systems

**地址**：https://arxiv.org/abs/2606.02282

**论文介绍（250 字）**
POIROT 协议将多代理系统自身的代理重新利用为诊断层，利用架构中已有的认知多样性进行故障检测。核心洞察：安全监督无需外部化 — 执行角色的代理携带足够的集体智能来审计自身。

实验显示 POIROT 在复杂问题上超越单 LLM 评估基线（OR=1.60, p=0.008），增益随问题复杂度、代理数量和故障维度扩展。在复合故障条件下仍保持有效。发布开源库和 BLAME 基准（多代理系统故障归因基准）。

**技术架构**
- 代理即诊断器：复用执行代理进行自检
- 认知多样性利用：不同代理的 epistemic diversity
- 复合故障检测：多故障条件下的鲁棒性

**核心算法**
- 代理间交叉验证：代理互相审计
- 认知多样性度量：量化代理间的知识差异
- 复合故障分解：多故障条件下的归因

**优势与局限**
- 优势：无需外部监督、利用现有认知多样性、开源
- 局限：需要代理间通信开销、对高度同质化代理效果有限

**未来发展潜力**
- 可能成为多代理系统安全审计的标准协议
- 与 EU AI Act 合规需求结合，成为法规合规基础设施

**个人技术理解**
POIROT 的"代理审计代理"范式与之前探索的"LLM-as-a-Judge"形成有趣对比：后者用外部 LLM 评估代理，前者用代理自身审计。在 EU AI Act 合规需求下，自审计可能比外部审计更具可操作性。

#### 6. SISA — SSM-Informed Softmax Attention（第三设计轴）

**标题**：Importance-Aware Attention Is All You Need

**地址**：https://arxiv.org/abs/2606.02332

**论文介绍（250 字）**
SISA（SSM-Informed Softmax Attention）将 SSM 导出的重要性项直接添加到注意力分数中，作为单个 SDPA 调用在增强的 query/key 向量上实现 — 无需循环状态、无需自定义 kernel。定义了 SSM-Attention 混合的第三设计轴 — 分数级融合，超越 Jamba（块级）和 Hymba（头级）范式。

在 152M/5B token 上，SISA 达到 LAMBADA-greedy 17.3%（vs Transformer 13.9%、Mamba-3 15.5%），NIAH 从 1K 步达到 100%（比 Transformer 检索收敛快 7x）。

**技术架构**
- 分数级融合：SSM 重要性直接注入注意力分数
- 单一 SDPA 调用：无需自定义 kernel
- 增强 query/key：SSM 信息编码到向量中

**核心创新**
- 第三设计轴：超越块级/头级，定义分数级融合
- 工程友好：标准 SDPA 调用，无需自定义实现
- NIAH 7x 加速：检索收敛速度显著提升

**优势与局限**
- 优势：工程友好、性能提升、标准 API 兼容
- 局限：在更大模型上（369M）LAMBADA 略逊 Mamba-3

**未来发展潜力**
- 可能成为 SSM-Transformer 混合的标准范式
- 与长上下文 LLM 结合，解决检索效率问题

**个人技术理解**
SISA 的"第三设计轴"定义值得重视 — 它将 SSM 的"知道什么重要"能力直接注入 Transformer 的"看到一切"能力，在分数层面实现融合而非在块或头层面。单一 SDPA 调用的工程实现使其易于集成到现有框架中。
