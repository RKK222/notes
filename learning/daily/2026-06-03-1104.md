### AI Testing / Software Testing（9 项）

#### 1. Flare — LLM 代码细粒度诊断反馈
- **地址**：https://arxiv.org/abs/2606.03852
- **来源**：arXiv cs.SE/cs.AI
- **领域**：AI Testing / 代码调试
- **技术介绍**：Flare 通过轻量级诊断模型预测行级可疑度信号用于 Bug 定位和代码精炼。搜索 top-k 可疑区域并根据执行结果选择最佳候选。在 LiveCodeBench 和 BigCodeBench 上用 5 个基础 LLM 评估，k=1 时超越最强基线 1.72%-7.42%，搜索 10 个候选时平均提升 8.50%。
- **应用场景**：LLM 生成代码自动调试、代码精炼迭代优化
- **为什么值得关注**：从粗粒度测试失败反馈到行级精确诊断的范式转变

#### 2. DDOR — Delta Debugging 过度拒绝测试与修复
- **地址**：https://arxiv.org/abs/2606.03601
- **来源**：arXiv cs.SE/cs.AI
- **领域**：AI Testing / LLM安全测试
- **技术介绍**：全自动可解释框架，应用 delta debugging 定位最小拒绝触发片段（mRTFs），提供短语级可解释证据。每模型生成约 1K 案例测试套件。利用 mRTFs 执行定向提示修复，在保持安全性的同时大幅减少过度拒绝。
- **应用场景**：LLM安全护栏评估、过度拒绝问题诊断与修复
- **为什么值得关注**：将经典 delta debugging 引入 LLM安全测试

#### 3. Human-AI Collaboration — SE 工作转型框架

### GitHub 项目

#### 1. MAI-Code-1-Flash — Microsoft 编码模型

**地址**：https://microsoft.ai/news/introducingmai-code-1-flash/

**项目介绍**：Microsoft 发布的 MAI-Code-1-Flash 编码模型代表了 AI 编码工具竞争的新方向。与以往优化基准分数的策略不同，MAI-Code-1-Flash 直接在 GitHub Copilot 生产 harness 中训练，使模型学习如何与周围工具和系统交互。核心创新是自适应解决方案长度控制，使模型能根据任务复杂度调整响应深度。在 SWE-Bench Pro 上 51.2% vs Claude Haiku 4.5 的 35.2%（+16 点），同时解决更难问题时使用少 60% token。

**技术架构**：
- 训练环境：GitHub Copilot 生产 harness
- 评估维度：SWE-Bench Verified / Pro / Multilingual + Terminal Bench 2
- 核心机制：自适应解决方案长度控制
- 效率优化：价值/Token 最大化

**核心创新**：
1. **生产对齐训练**：直接在 Copilot harness 中训练，而非独立训练后适配
2. **自适应长度控制**：简单任务简洁响应，复杂任务深度推理
3. **效率优先**：更高准确率 + 更少 token，而非单纯追求准确率

**与现有技术对比**：
- 相比 Claude Haiku 4.5：SWE-Bench Pro +16 点，token 效率 -60%
- 相比传统编码模型：从"基准优化"转向"生产工作流优化"

**优势与局限**：
- 优势：生产环境直接可用、token 效率极高、与 Copilot 生态深度集成
- 局限：仅通过 Copilot 访问、基准覆盖范围有限（4 个基准）

**未来发展潜力**：如果 MAI-Code-1-Flash 的成功模式被验证，将推动编码模型从"基准竞赛"转向"价值/Token 竞赛"，这对按 token 计费的商业模式有深远影响。

**个人技术理解**：MAI-Code-1-Flash 的发布标志着 AI 编码工具进入"效率竞争"阶段。当准确率差距缩小时，token 效率成为差异化关键。这与我们之前观察到的 headroom（token 压缩）、8v（统一二进制）等工具形成互补的效率优化生态。

#### 2. Flare — 细粒度代码诊断反馈

**地址**：https://arxiv.org/abs/2606.03852

**项目介绍**：Flare 是一个迭代式 LLM 代码精炼框架，通过轻量级诊断模型预测行级可疑度信号。给定诊断预测的不确定性，Flare 搜索 top-k 可疑区域并根据执行结果选择最佳候选。在 LiveCodeBench 和 BigCodeBench 上，k=1 时超越最强基线 1.72%-7.42%，搜索 10 个候选时平均提升 8.50%。

**技术架构**：
- 诊断模型：轻量级，预测行级可疑度
- 搜索策略：top-k 可疑区域搜索
- 选择机制：基于执行结果选择最佳候选
- 迭代精炼：诊断→搜索→验证→修复循环

**核心创新**：
1. **行级可疑度**：超越粗粒度测试失败反馈
2. **不确定性感知搜索**：诊断预测不确定性驱动搜索策略
3. **执行引导选择**：用实际执行结果而非启发式选择修复位置

**优势与局限**：
- 优势：精确诊断、无需候选搜索即有效、轻量级诊断模型
- 局限：依赖可执行测试环境、对非确定性 Bug 效果有限

**个人技术理解**：Flare 代表 AI 代码调试从"知道有错"到"知道哪里错"的关键一步。行级诊断 + 不确定性搜索的组合策略为 LLM 代码精炼提供了可扩展路径。

### 学术论文

#### 3. VEPO — 视觉锚定 Token 选择 RL

**地址**：https://arxiv.org/abs/2606.03937

**论文介绍**：VEPO（Vision-Entropy token-selection for Policy Optimization）揭示了 token 级熵在文本 RL 中有效但在视觉推理中崩溃的根本原因：遗漏了视觉敏感但天然低熵的 token。通过乘法耦合将视觉敏感性与 token 熵整合，将梯度信用重定向到同时视觉接地和高信息的 token。7B 尺度超越熵基线 2.28 点，3B 尺度超越 3.15 点。

**技术架构**：
- 核心机制：视觉敏感性 × token 熵 乘法耦合
- 梯度路由：重定向到视觉接地且高信息 token
- RL 框架：基于策略优化，无需额外架构修改

**核心算法**：
1. 视觉敏感性度量：识别对视觉输入敏感的 token
2. 熵值计算：标准 token 级熵
3. 乘法耦合：VEPO = 视觉敏感性 × 熵
4. 梯度重定向：基于 VEPO 分数分配信用

**与现有技术对比**：
- 熵基线 RL：在视觉推理中崩溃（遗漏低熵视觉 token）
- 现有多模态 RL：缺乏系统视觉测量或忽略熵驱动语义探索
- VEPO：principled 乘法耦合，同时满足视觉接地和信息性

**优势与局限**：
- 优势： principled 理论基础、无需架构修改、显著性能提升
- 局限：仅适用于视觉推理任务、视觉敏感性度量依赖特定模型

**未来发展潜力**：VEPO 揭示的"视觉敏感 token 在 RL 中的关键作用"可能扩展到更多多模态任务。乘法耦合的思想也可能启发其他模态的 RL 优化。

**个人技术理解**：VEPO 的核心洞察是"熵不是万能的信用分配指标"。在视觉推理中，最重要的 token 往往是那些对视觉输入敏感但本身熵值低的 token。这一发现挑战了当前 RLVR 的默认假设。

#### 4. Hedge-Bench — 对冲基金代理推理基准

**地址**：https://arxiv.org/abs/2606.03918

**论文介绍**：Hedge-Bench 1.0 是 102 个真实在职任务基准，基于专业对冲基金分析师的显式推理轨迹。使用确定性分级（而非模型评判）验证专家步骤。前沿模型和代理得分低于 16%。

**技术架构**：
- 数据来源：真实对冲基金分析师工作流
- 评估方式：确定性分级（验证专家步骤）
- 任务类型：开放域金融推理

**核心创新**：
1. **真实工作流**：基于实际分析师任务，非合成
2. **确定性分级**：避免模型评判的噪声和循环性
3. **专家轨迹**：显式推理步骤可验证

**优势与局限**：
- 优势：真实场景、确定性评估、专家验证
- 局限：规模有限（102 项）、仅覆盖对冲基金场景

**未来发展潜力**：如果 Hedge-Bench 的成功模式被验证，可能推动更多"真实工作流基准"的出现，替代当前主流的合成基准。

**个人技术理解**：Hedge-Bench 揭示了一个重要趋势：金融代理评估正从"子技能测试"转向"完整工作流评估"。前沿模型得分低于 16% 表明开放域金融推理仍是巨大挑战。

#### 5. CER Framework — AI 损失重建与保险

**地址**：https://arxiv.org/abs/2606.03777

**论文介绍**：CER 框架处理 AI 系统损失重建问题。C（控制边界）系统是否有可执行的运行包络？E（证据重建）系统状态和因果链能否从保留工件重建？R（保险响应）重建的损失是否可投保？处理 prompt injection、RAG 中毒、恶意工具输出、凭证滥用、数据中毒等外部触发失败。

**技术架构**：
- C 维度：控制边界验证（可执行运行包络）
- E 维度：证据重建（系统状态 + 因果链）
- R 维度：保险响应（市场覆盖 + 索赔证明）

**核心创新**：
1. **AI 特定重建**：区分事件重建和状态重建
2. **三维度框架**：技术安全 → 工程证据 → 金融保险
3. **实用导向**：引用真实案例（PocketOS/Replit 数据库删除）

**优势与局限**：
- 优势：连接技术安全和金融保险、实用案例驱动
- 局限：概念框架为主、缺乏量化验证

**未来发展潜力**：随着 AI 系统部署规模扩大，CER 框架可能成为 AI 风险管理的标准框架。特别是当 AI 保险市场成熟时。

**个人技术理解**：CER 框架代表 AI 安全从"技术防御"到"全链路风险管理"的范式转变。当 AI 系统造成真实经济损失时，仅靠技术防护不够，需要完整的重建和保险机制。
