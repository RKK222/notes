### AI Testing / Software Testing（10 项）

#### 1. NovelAPIBench — LLM 工具使用知识缺口诊断
- **地址**：https://arxiv.org/abs/2606.03657
- **来源**：arXiv cs.AI
- **领域**：AI Testing / LLM 工具使用
- **技术介绍**：NovelAPIBench 是一个全自动动态基准，针对任何基础模型和目标库，自动发现新 API、提取分解知识包、生成可执行编码任务，并将失败样本分配到六个诊断类别。在约 1.9K 任务、四个基础模型和五个领域上，比较通过检索注入的知识与通过参数适应内化的知识。发现知识组件不可互换：使用示例是最强的独立信号，最佳双组件配置因领域而异。添加更多上下文（尤其是源代码）可能通过增加导入路径错误而降低性能。参数适应也不替代检索 — 微调主要教会模型如何使用提供的知识包，且这种能力可迁移到未见过的库。
- **应用场景**：LLM 代码生成工具使用诊断、API 知识注入策略优化
- **为什么值得关注**：从"能否调用 API"到"为什么失败"的诊断范式转变，揭示检索与微调的互补关系

#### 2. StepFinder — 多代理系统故障归因
- **地址**：https://arxiv.org/abs/2606.03467
- **来源**：arXiv cs.AI（KDD 2026）
- **领域**：AI Testing / 多代理可靠性
- **技术介绍**：LLM 多代理系统对单步执行错误高度敏感，错误通过代理交互传播导致级联失败。StepFinder 是一个轻量级故障归因框架，仅在特征构建阶段使用 LLM 将执行日志编码为时间语义序列，然后用参数高效的时序建模和注意力模块捕获轨迹的序列演化和跨步依赖，最后通过多尺度差异和位置偏差精炼步级错误分数。在 Who&When 基准上，StepFinder 在步级故障归因上超越 LLM 方法，推理时间减少 79%，无文本生成开销。
- **应用场景**：多代理工作流调试、级联故障根因分析
- **为什么值得关注**：用非生成式方法替代 LLM 推理归因，79% 加速且精度更高

#### 3. MedCUA-Bench — 临床计算机使用代理基准
- **地址**：https://arxiv.org/abs/2606.03203
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 医疗代理
- **技术介绍**：计算机使用代理可自动化重复性屏幕临床工作，但其在医疗 GUI 上的可靠性未经验证。MedCUA-Bench 覆盖 10 个医疗领域的 18 个临床场景，从真实产品手册和开源医疗系统重建，捕获真实临床界面。每个任务配有意图和步骤级目标，将临床推理与 UI 执行解耦，由确定性检查器在任务完成和五个临床安全维度上评估。23 个代理中，最佳闭源模型达到 54.2% 严格成功率，在真实 OpenEMR 上所有模型低于 9%，开源代理平均仅 2.5%。
- **应用场景**：医疗软件自动化评估、临床代理可靠性验证
- **为什么值得关注**：揭示当前代理与可靠临床软件使用之间的巨大鸿沟

#### 4. ClinicalMC — 多轮临床决策基准
- **地址**：https://arxiv.org/abs/2606.03157
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 医疗 LLM
- **技术介绍**：现有基准主要评估 LLM 在单次就诊场景中的表现，缺乏多轮场景评估。ClinicalMC 包含 1,275 中文和 5,804 英文样本，覆盖从入院到出院的四个阶段。英语数据集中患者平均经历 5.11 个临床轮次，中文数据集 3.42 轮次。构建多代理评估框架（患者、检查者、医生代理），设计单次静态和多轮动态两种实验设置，评估闭源 LLM、开源 LLM 和医疗 LLM 三大类。
- **应用场景**：医疗 LLM 多轮决策能力评估
- **为什么值得关注**：首个多轮临床决策基准，更贴近真实医疗场景

#### 5. GTBench — 图论数学研究助理基准
- **地址**：https://arxiv.org/abs/2606.03144
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 数学推理
- **技术介绍**：GTBench 是课程驱动的图论研究助理基准，包含 63 个问题，分为三个难度递增组：本科定义和基本性质、算法追踪和结构推理、研究生级证明构建。问题来自 Diestel 图论等验证学术材料。评估五个前沿模型，揭示明显性能层次：GPT-5 在 Group 1 接近天花板（95.8%），在研究生证明上保持 82% 精度，而 Llama 在 Group 3 零样本下达到 0%。失败模式分析显示正确算法错误执行主导 Group 1-2，Group 3 额外暴露不完整推理失败。
- **应用场景**：数学研究 AI 工具治理、教育 AI 可靠性评估
- **为什么值得关注**：首个课程驱动的图论推理评估框架

#### 6. Terminal-Lego — 交互轨迹教学有效性
- **地址**：https://arxiv.org/abs/2606.03461
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 代理训练
- **技术介绍**：更强代码代理通常被认为是更好的后训练教师，但这与任务难度、harness 设计和学生能力的关系未解耦。Terminal-Lego 将多域真实问题转化为环境验证的代理任务。发现"教学悖论"：Claude Opus 4.6 在 Terminal-Bench 2.0 上得分更高，但 DeepSeek-V3.2 轨迹微调的学生泛化更强。归因于"环境接地监督"（EGS）：通过 harness 可见交互显式暴露检查-行动-验证行为的轨迹，让学生内化鲁棒的问题解决例程而非脆弱动作序列。仅 15.3K 轨迹使 Qwen3-32B 达到 24.3%。
- **应用场景**：代理后训练数据选择、教学代理质量评估
- **为什么值得关注**：揭示"更强代理 ≠ 更好教师"的教学悖论

#### 7. TSQAgent — 时间序列数据质量评估
- **地址**：https://arxiv.org/abs/2606.03629
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 数据质量
- **技术介绍**：TSQBench 评估 LLM 在两个渐进能力上：(i) 理解并识别相关质量维度，(ii) 在特定维度下执行质量比较。发现当前 LLM 在维度识别和证据接地质量比较上持续挣扎。TSQAgent 由三个协作角色组成：Perceiver（聚焦维度选择）、Inspector（维度级定量分析）、Adjudicator（聚合精炼最终判断）。引入代理推理策略使模型能识别和优先排序最相关质量维度，配合外部分析工具实现精确定量比较。
- **应用场景**：时间序列数据质量评估、数据选择优化
- **为什么值得关注**：从手动预定义质量维度到代理自主识别的范式转变

#### 8. 基准污染检测可靠性缺口
- **地址**：https://arxiv.org/abs/2606.03305
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 基准审计
- **技术介绍**：基准污染威胁 LLM 评估有效性。识别两个研究不足的失效模式：分布偏移（当可疑集和验证集违反 IID 假设时）和规模约束（基准比预训练语料库小几个数量级）。在 27 个模型（Pythia、OLMo 2、专业文化和医疗 LLM，最高 27B）和多个前沿行业模型上系统评估三种领先范式（LLM Dataset Inference、Post-Hoc Dataset Inference、CoDeC）。335 次评估中仅 199 次产生正确结果。LLM Dataset Inference 在分布偏移下产生假阳性，Post-Hoc Dataset Inference 在基准规模下功效不足，CoDeC 仅提供粗糙来源信号。
- **应用场景**：基准污染检测、模型数据溯源审计
- **为什么值得关注**：揭示统计检测无法替代透明数据溯源的系统性可靠性缺口

#### 9. Proof-Refactor — 形式化证明重构
- **地址**：https://arxiv.org/abs/2606.03743
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 形式化验证
- **技术介绍**：LLM 生成的形式化证明在可读性、模块化、可维护性和可重用性上不如成熟形式数学库中的证明。Proof-Refactor 将证明重构分解为四个阶段：提取候选证明片段、设计辅助声明、形式化证明提取和设计组件、使用验证组件修复原始证明。在 PutnamBench 和 Putnam2025 的生成 Lean 证明上，Proof-Refactor 在签名质量和人类可读性上超越 Claude Code 基线。
- **应用场景**：形式化证明质量改进、数学库维护
- **为什么值得关注**：过程引导式重构不以证明长度为代理目标

#### 10. ChemCoTBench-V2 — 化学推理过程级评估
- **地址**：https://arxiv.org/abs/2606.03660
- **来源**：arXiv cs.AI
- **领域**：AI Testing / 化学推理
- **技术介绍**：大多数化学基准仍仅评分最终答案，掩盖了"模型输出正确分子但推理违反化学逻辑"的关键失败模式。ChemCoTBench-V2 是规则可验证的诊断基准，涵盖分子理解、分子编辑、分子优化和反应预测，5,620 个评估样本、18 个报告任务。模型必须在专家设计的模板中暴露关键中间步骤，这些步骤用确定性化学规则检查。报告三个独立信号：最终答案正确性、模板遵守性和逐步验证器正确性。
- **应用场景**：化学 LLM 过程级诊断、化学推理质量评估
- **为什么值得关注**：从最终答案评分到过程级可验证评估的范式转变

### AI / LLM 技术（12 项）

#### 11. ThoughtFold — 推理链折叠
- **地址**：https://arxiv.org/abs/2606.03503
- **来源**：arXiv cs.AI
- **领域**：AI / 推理效率
- **技术介绍**：长 CoT 自然包含试错，主流 RLVR 方法选择结果正确 CoT 轨迹进行记忆，不可避免地强化了长 CoT 中的冗余探索，导致 LRM 过度思考问题。ThoughtFold 利用细粒度偏好学习缓解冗余探索。采用内省策略识别每条正确轨迹内的冗余，产生候选子轨迹谱。引入掩码偏好优化目标，显式惩罚冗余探索并鼓励模型直接桥接基本推理段，有效折叠推理链为更简洁路径。DeepSeek-R1-Distill-Qwen-7B token 使用减少约 56%，同时保持 SOTA 精度。
- **应用场景**：推理模型效率优化、过度思考缓解
- **为什么值得关注**：从"鼓励短轨迹"到"识别冗余探索"的范式升级

#### 12. LEAP — 形式化数学代理框架
- **地址**：https://arxiv.org/abs/2606.03303
- **来源**：arXiv cs.AI
- **领域**：AI / 形式化证明
- **技术介绍**：LLM 展示强大的非正式数学推理能力，但在生成 Lean 等正式语言中机械可验证证明方面挣扎。LEAP 利用基础模型能力（非正式推理、指令遵循、迭代自我精炼），将复杂问题分解为更小单元，通过持续与 Lean 编译器交互桥接正式证明构建与非正式蓝图。引入 Lean-IMO-Bench（IMO 风格问题正式化为 Lean）。在 Putnam 2025 竞赛上，LEAP 解决全部 12 题，匹配 recent fr 的突破。
- **应用场景**：自动定理证明、形式化数学研究
- **为什么值得关注**：通用基础模型在形式化定理证明上达到 SOTA

#### 13. CRGC — 约束关系图补全
- **地址**：https://arxiv.org/abs/2606.03624
- **来源**：arXiv cs.AI
- **领域**：AI / 指令遵循
- **技术介绍**：LRM 在可靠遵循多条指令方面挣扎，要么无法单独满足各约束，要么难以平衡竞争约束。形式化为约束遵循问题（CAP）。CRGC 将指令表示为结构化约束知识图谱，显式建模约束间关系、识别遵循挑战、发现"桥接约束"帮助模型更好聚焦和调和需求。桥接约束作为辅助指令使主要约束更显著和兼容。在三个流行指令遵循数据集上，约束违规减少 39%。
- **应用场景**：多约束指令遵循、复杂任务分解
- **为什么值得关注**：利用模型自身知识创建更好生成路径

#### 14. 跨语言 Token 套利
- **地址**：https://arxiv.org/abs/2606.03618
- **来源**：arXiv cs.AI（EMNLP 2026）
- **领域**：AI / 编码代理效率
- **技术介绍**：AI 编码代理受输入 token 成本瓶颈。两个原始人类输入病理驱动大部分开销：非英语文本的 tokenization 低效和对话提示的结构熵。引入飞行前边缘侧提示重写中间件，在开发者和云代理之间运行。本地 Llama 3.2（3B）执行跨语言翻译为英语、结构重写为紧凑任务导向格式、regex 验证重写 + 回退保障确保优化提示永不大于原始提示。在 OMH-Polyglot 多语言编码基准上，prompt token 减少 34-47%，总 token 减少最多 18.8%。
- **应用场景**：多语言编码代理成本优化、提示工程
- **为什么值得关注**：主动提示优化可大幅降低推理成本

#### 15. CP-Agent — 细胞形态分析多模态代理
- **地址**：https://arxiv.org/abs/2606.03435

#### 1. StepFinder — 多代理系统故障归因（KDD 2026）

**地址**：https://arxiv.org/abs/2606.03467 | https://github.com/taiyu-zhu/StepFinder

**项目介绍**（200-400 字）：
StepFinder 是 KDD 2026 接收的轻量级多代理故障归因框架。LLM 多代理系统在执行复杂多步任务时，单步错误会通过代理交互传播导致级联失败。现有方法依赖 LLM 对原始执行轨迹进行推理归因，推理成本高、延迟大，且受冗余和噪声执行日志干扰，难以准确识别真正根因步骤。StepFinder 的关键洞察是：故障归因不需要生成式推理 — 只需从执行日志中提取结构化特征并用高效模型分类。StepFinder 仅在特征构建阶段使用 LLM 将执行日志编码为时间语义序列，然后用参数高效的时序建模和注意力模块捕获轨迹的序列演化和跨步依赖，最后通过多尺度差异和位置偏差精炼步级错误分数。在 Who&When 基准上，StepFinder 在步级故障归因上超越 LLM 方法，推理时间减少 79%，无文本生成开销。

**技术架构**：
- 特征提取层：LLM 将原始执行日志编码为时间语义序列
- 时序建模层：参数高效时序模型捕获序列演化
- 注意力层：跨步依赖建模
- 精炼层：多尺度差异 + 位置偏差

**核心算法**：
- 时间语义序列编码：将异构执行日志统一为语义向量
- 多尺度差异：在不同时间粒度上计算错误分数
- 位置偏差校正：消除序列位置对错误评分的干扰

**与现有技术对比**：
| 维度 | LLM 推理归因 | StepFinder |
|------|-------------|------------|
| 推理方式 | 生成式 | 分类式 |
| 推理时间 | 高 | -79% |
| 文本生成 | 需要 | 无 |
| 噪声鲁棒性 | 低 | 高 |

**优势与局限**：
- 优势：79% 推理加速、无文本生成开销、噪声鲁棒
- 局限：依赖 LLM 进行特征构建（虽然只在一次阶段使用）

**未来发展潜力**：
可扩展到更多代理框架（LangChain、AutoGen 等），结合因果推断提供更细粒度归因。

**个人技术理解**：
StepFinder 代表了一个重要趋势：用非生成式方法替代 LLM 推理密集型任务。故障归因本质上是分类问题，不需要 LLM 的生成能力。这种"LLM 仅用于特征工程"的范式在多个场景中都有效。

#### 2. ThoughtFold — 推理链折叠

**地址**：https://arxiv.org/abs/2606.03503

**项目介绍**（200-400 字）：
ThoughtFold 解决 LRM 过度思考问题。长 CoT 自然包含试错，主流 RLVR 方法选择结果正确的 CoT 轨迹进行记忆，不可避免地强化了长 CoT 中的冗余探索。ThoughtFold 采用内省策略识别每条正确轨迹内的冗余，产生候选子轨迹谱。引入掩码偏好优化目标，显式惩罚冗余探索并鼓励模型直接桥接基本推理段。DeepSeek-R1-Distill-Qwen-7B token 使用减少约 56%，同时保持 SOTA 精度。

**技术架构**：
- 内省分析器：识别正确轨迹中的冗余段
- 子轨迹生成器：产生候选子轨迹谱
- 掩码偏好优化器：显式惩罚冗余探索

**核心算法**：
- 内省冗余检测：分析轨迹中重复/无效的推理步骤
- 掩码偏好优化：对冗余探索施加惩罚梯度

**与现有技术对比**：
| 维度 | 传统 RLVR | ThoughtFold |
|------|----------|-------------|
| 优化目标 | 结果正确 | 结果正确 + 无冗余 |
| 轨迹长度 | 长 | -56% |
| 冗余探索 | 强化 | 惩罚 |

**优势与局限**：
- 优势：56% token 减少、保持精度、方法通用
- 局限：需要足够多的正确轨迹进行内省分析

**未来发展潜力**：
可推广到其他推理模型（Gemini、Claude 等），结合结构化推理进一步提升效率。

**个人技术理解**：
ThoughtFold 的关键洞察是"短轨迹 ≠ 高效轨迹"。传统方法简单地偏好短轨迹，但可能丢弃有用的探索步骤。ThoughtFold 通过内省识别真正的冗余，保留必要的探索，实现更智能的折叠。

#### 3. NovelAPIBench — LLM 工具使用诊断

**地址**：https://arxiv.org/abs/2606.03657

**项目介绍**（200-400 字）：
NovelAPIBench 是全自动动态基准，针对任何基础模型和目标库，自动发现新 API、提取分解知识包、生成可执行编码任务，并将失败样本分配到六个诊断类别。在约 1.9K 任务、四个基础模型和五个领域上，比较通过检索注入的知识与通过参数适应内化的知识。关键发现：知识组件不可互换 — 使用示例是最强的独立信号，最佳双组件配置因领域而异。添加更多上下文（尤其是源代码）可能通过增加导入路径错误而降低性能。参数适应也不替代检索 — 微调主要教会模型如何使用提供的知识包，且这种能力可迁移到未见过的库。

**技术架构**：
- API 发现器：自动识别目标库中的新 API
- 知识包提取器：分解为签名、机制、示例等组件
- 任务生成器：生成可执行编码任务
- 诊断分类器：六类失败原因分类

**核心算法**：
- 知识组件解耦分析：独立评估各组件的贡献
- 检索 vs 微调对比实验

**与现有技术对比**：
| 维度 | 静态基准 | NovelAPIBench |
|------|---------|---------------|
| API 覆盖 | 固定 | 动态发现 |
| 评估粒度 | pass/fail | 六类诊断 |
| 知识来源 | 单一 | 检索 + 微调对比 |

**优势与局限**：
- 优势：全自动、诊断细粒度、揭示检索-微调互补性
- 局限：依赖目标库的可访问性

**未来发展潜力**：
可扩展到更多编程领域（数据科学、Web 开发等），结合实时 API 变化追踪。

**个人技术理解**：
NovelAPIBench 的最大贡献是揭示了"检索和微调互补"而非"替代"的关系。这为 LLM 工具使用优化提供了明确方向：检索提供挥发性 API 内容，微调提高程序性集成能力。

#### 1. LEAP — 形式化数学代理框架

**地址**：https://arxiv.org/abs/2606.03303

**论文介绍**（200-400 字）：
LEAP 是 Google DeepMind 等机构提出的形式化数学代理框架，使通用基础模型在自动形式化定理证明上达到 SOTA。LLM 展示强大的非正式数学推理能力，但在生成 Lean 等正式语言中机械可验证证明方面挣扎。LEAP 利用基础模型能力（非正式推理、指令遵循、迭代自我精炼），将复杂问题分解为更小单元，通过持续与 Lean 编译器交互桥接正式证明构建与非正式蓝图。引入 Lean-IMO-Bench（IMO 风格问题正式化为 Lean，短陈述但高度非例行和多步证明）。在 Putnam 2025 竞赛上，LEAP 解决全部 12 题，匹配 recent fr 的突破。

**技术架构**：
- 非正式推理器：生成非正式证明蓝图
- 问题分解器：将复杂问题分解为更小单元
- Lean 编译器交互器：持续验证和迭代
- 自我精炼器：迭代改进证明

**核心算法**：
- 蓝图引导证明构建：非正式推理指导正式证明
- 编译器反馈循环：持续验证和修正

**与现有技术对比**：
| 维度 | 专用证明器 | LEAP |
|------|----------|------|
| 基础模型 | 专用 | 通用 |
| 推理方式 | 搜索 | 蓝图引导 |
| Putnam 2025 | 部分解决 | 全部 12 题 |

**优势与局限**：
- 优势：通用基础模型、SOTA 性能、可扩展
- 局限：依赖 Lean 编译器交互，计算成本高

**未来发展潜力**：
可扩展到其他形式化语言（Coq、Isabelle），结合 RLVR 进一步提升性能。

**个人技术理解**：
LEAP 的关键洞察是"非正式推理 + 正式验证"的协作范式。通用 LLM 擅长非正式推理，形式化验证器擅长机械检查，两者协作超越专用证明器。

#### 2. MedCUA-Bench — 临床计算机使用代理基准

**地址**：https://arxiv.org/abs/2606.03203

**论文介绍**（200-400 字）：
MedCUA-Bench 是首个临床计算机使用代理交互式基准。计算机使用代理可自动化重复性屏幕临床工作，但其在医疗 GUI 上的可靠性未经验证。现有基准聚焦通用 Web 或桌面任务，代表性不足医疗软件（需要领域知识、UI 设计与主流应用显著不同、缺乏公共测试环境、需要超出任务完成的安全验证）。MedCUA-Bench 覆盖 10 个医疗领域的 18 个临床场景，从真实产品手册和开源医疗系统重建。每个任务配有意图和步骤级目标，将临床推理与 UI 执行解耦。23 个代理中，最佳闭源模型达到 54.2% 严格成功率，在真实 OpenEMR 上所有模型低于 9%，开源代理平均仅 2.5%。

**技术架构**：
- 场景重建器：从产品手册重建临床界面
- 意图-步骤解耦评估器
- 确定性安全检查器
- 多代理评测框架

**核心算法**：
- 意图 vs 步骤级评估分离
- 五维度临床安全检查

**与现有技术对比**：
| 维度 | 通用 CUA 基准 | MedCUA-Bench |
|------|-------------|-------------|
| 领域覆盖 | 通用 Web | 10 医疗领域 |
| 评估粒度 | 任务完成 | 意图 + 步骤 + 安全 |
| 最佳成功率 | 80%+ | 54.2% |

**优势与局限**：
- 优势：真实临床场景、多维度评估、可复现
- 局限：场景来自开源系统，可能不完全反映商业系统

**未来发展潜力**：
可扩展到更多医疗系统（Epic、Cerner 等），结合多模态代理进一步提升性能。

**个人技术理解**：
MedCUA-Bench 揭示了通用 Web 代理与医疗专用代理之间的巨大鸿沟。医疗 GUI 的领域知识需求和安全性要求远超通用 Web 任务，需要专门的代理设计和评估框架。

#### 3. 基准污染检测可靠性缺口

**地址**：https://arxiv.org/abs/2606.03305

**论文介绍**（200-400 字）：
基准污染威胁 LLM 评估有效性。统计工具检测训练数据成员资格存在，但几乎仅在受控学术环境中验证：大型同质预训练语料库和透明单阶段训练管道。本文识别两个研究不足的失效模式：分布偏移（当可疑集和验证集违反 IID 假设时）和规模约束（基准比预训练语料库小几个数量级）。在 27 个模型（Pythia、OLMo 2、专业文化和医疗 LLM，最高 27B）和多个前沿行业模型上系统评估三种领先范式。335 次评估中仅 199 次产生正确结果。LLM Dataset Inference 在分布偏移下产生假阳性，Post-Hoc Dataset Inference 在基准规模下功效不足，CoDeC 仅提供粗糙来源信号。

**技术架构**：
- 分布偏移测试：违反 IID 假设的场景
- 规模约束测试：基准 vs 语料库规模差异
- 三范式对比评估

**核心算法**：
- LLM Dataset Inference
- Post-Hoc Dataset Inference
- CoDeC

**与现有技术对比**：
| 维度 | 受控验证 | 实际审计 |
|------|---------|---------|
| 正确率 | 高 | 59% (199/335) |
| 分布假设 | IID | 非 IID |
| 规模 | 大语料 | 小基准 |

**优势与局限**：
- 优势：系统性评估、揭示实际审计中的可靠性缺口
- 局限：未提出新的检测方法

**未来发展潜力**：
需要开发更鲁棒的污染检测方法，结合数据溯源和透明度要求。

**个人技术理解**：
这篇论文揭示了一个关键问题：基准污染检测的学术验证与实际审计之间存在系统性差距。统计检测无法替代透明数据溯源，这对 AI 评估的可信度有深远影响。

## 技术趋势

1. **推理效率从"短轨迹"到"无冗余"**：ThoughtFold 揭示传统 RLVR 简单偏好短轨迹的局限，内省式冗余识别代表推理效率优化的新方向。结合 headroom（11:04 覆盖）的 token 压缩和 Cross-Lingual Token Arbitrage 的 prompt 优化，推理效率优化正从单一维度扩展到多正交维度。

2. **代理评估从"结果"到"过程"**：NovelAPIBench（六类诊断）、StepFinder（步级归因）、ChemCoTBench-V2（过程级验证）、Terminal-Lego（教学有效性）共同代表代理评估从"pass/fail"到"为什么失败"的范式转变。过程级诊断比聚合分数更有指导价值。

3. **形式化证明从"专用"到"通用+协作"**：LEAP 用通用基础模型 + 非正式推理蓝图 + 正式验证器协作，解决 Putnam 全部 12 题。Proof-Refactor 进一步改进生成证明的可读性和模块化。形式化证明正从专用证明器走向通用 LLM 协作范式。

4. **医疗 AI 从"问答"到"多轮决策+GUI 操作"**：ClinicalMC（多轮临床决策）、MedCUA-Bench（临床 GUI 操作）、CP-Agent（细胞形态分析）共同代表医疗 AI 从单轮问答向多轮决策和 GUI 操作的演进。医疗场景的复杂性和安全性要求远超通用 AI 任务。

5. **基准审计可靠性缺口**：基准污染检测在受控验证与实际审计之间存在系统性差距（59% 正确率）。统计检测无法替代透明数据溯源，这对 AI 评估的可信度有深远影响。

## 新领域收获

1. **单细胞多组学翻译基准（scTranslation）**：AI 在生物信息学中的应用从基因组扩展到多组学整合。scTranslation 是首个系统基准，揭示了特征选择、特征质量和少样本设置对模型性能的关键影响。

2. **合规违规知识图谱模式（VSP）**：将合规违规作为持久图对象而非瞬态查询结果，审计历史即图遍历。这种模式将合规检测从"一次性检查"升级为"持续审计"。

3. **机器心智理论形式定义**：首次提出机器心智理论的严格形式定义，基于认知心理学、神经科学和 AI 证据。这为 AI 认知能力评估提供了理论基础。
