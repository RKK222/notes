## 深度研究

### GitHub 项目

#### 1. headroom: LLM Token Compression Proxy
- **地址**: https://github.com/chopratejas/headroom
- **Stars**: 9,772 (日增 3,530)
- **介绍**: headroom 是一个 LLM 工具输出压缩代理，通过 Library/Proxy/MCP 三种模式实现 60-95% 的 token 压缩，同时保持答案质量不变。它与 8v/DeepClaude 形成互补，专注于工具输出端的压缩而非编码代理桥接。
- **技术架构**: 三层架构——Library 模式直接集成到代码中、Proxy 模式作为中间件拦截、MCP 模式通过 MCP 协议集成。核心是一个轻量级压缩模型，学习识别和压缩冗余工具输出。
- **核心创新**: 将压缩从"事后处理"变为"预过滤"，在工具输出到达 LLM 之前就消除冗余。与 8v（统一二进制替代原生工具）和 DeepClaude（代理桥接）形成完整的 token 效率优化栈。
- **应用场景**: AI 编码代理、RAG 系统、多工具调用场景
- **为什么值得关注**: 日增 3530 stars，反映社区对 token 效率的强烈需求。与 8v/DeepClaude互补，可能成为 Agent 效率优化的标准组件。

#### 2. ECC: Agent Harness Performance Optimization
- **地址**: https://github.com/affaan-m/ECC
- **Stars**: GitHub Trending
- **介绍**: ECC 是一个 Agent harness 性能优化系统，引入 Skills+Instincts+Memory+Security+Research-First 五层架构，支持 Claude Code、Codex、Opencode、Cursor 等多平台。
- **技术架构**: 五层架构——Skills（编码技能）、Instincts（将编码模式编码为本能）、Memory（上下文管理）、Security（安全边界）、Research-First（研究优先开发流程）。
- **核心创新**: "Instincts" 概念将编码模式编码为代理的"本能"，减少重复推理。多平台兼容性设计使其成为跨工具的统一优化层。
- **应用场景**: 多平台 AI 编码代理性能优化
- **为什么值得关注**: 代表 Agent harness 从单一工具优化向系统化性能工程演进。

### 学术论文

#### 1. Reasoning Structure of LLMs (ICML 2026)
- **地址**: https://arxiv.org/abs/2606.03883
- **介绍**: 本文提出将 LLM 推理转化为可测量的图结构，通过逻辑谜题基准和推理图管道，将非结构化推理轨迹转换为可验证的推理图（claims + dependencies）。定义了推理效率指标，量化模型逻辑流的集中程度。
- **技术架构**: 逻辑谜题基准→推理轨迹捕获→推理图构建→拓扑分析。将推理从"黑盒"变为"可测量对象"。
- **核心算法**: 推理效率指标（Reasoning Efficiency Metric），衡量逻辑流的集中程度，超越 token 计数和准确率。
- **关键实现**: 自动将非结构化推理轨迹转换为有向图，节点为声明，边为依赖关系。
- **与现有技术对比**: 传统评估只看最终答案准确率或 token 消耗，本文揭示相同分数可能隐藏完全不同的推理结构。
- **优势与局限**: 优势是首次提供推理过程的结构性度量；局限是仅针对逻辑谜题，泛化到其他任务需要进一步验证。
- **未来发展潜力**: 推理图拓扑分析可能成为 Agent 评估的新标准维度，与 TraceGraph 互补。
- **个人技术理解**: 推理效率是 Agent 评估从"结果导向"向"过程导向"演进的关键一步。将推理图结构作为评估维度，有助于诊断代理的推理质量而非仅看最终结果。

#### 2. VEPO: Vision-Anchored RL for Visual Reasoning
- **地址**: https://arxiv.org/abs/2606.03937
- **介绍**: VEPO 揭示 token 级熵在视觉推理 RL 中崩溃的问题——视觉敏感 token 天然低熵，被传统熵基线忽略。通过乘法耦合视觉敏感性和熵，将梯度信用重定向到同时具有视觉接地和高信息量的 token。
- **技术架构**: 视觉敏感性测量 + token 熵计算 → 乘法耦合 → 策略优化。
- **核心算法**: VEPO 目标函数，将视觉敏感性权重与 token 熵相乘，确保视觉关键 token 获得足够的梯度信用。
- **关键实现机制**: 在 7B 规模上超越熵基线 2.28 点，3B 规模上超越 3.15 点。
- **与现有技术对比**: 现有多模态 RL 方法要么缺乏系统视觉测量，要么忽略熵主要驱动语义探索的事实。
- **优势与局限**: 优势是 principled 的乘法耦合设计；局限是仅针对视觉推理，扩展到多模态需要进一步研究。
- **未来发展潜力**: 视觉锚定 RL 可能成为多模态 Agent 训练的标准方法。
- **个人技术理解**: 这是一个重要的理论发现——熵基线在视觉推理中失效。乘法耦合视觉敏感性和熵的设计简洁而有效，为多模态 RL 提供了新方向。

#### 3. Hedge-Bench: Financial Reasoning Benchmark
- **地址**: https://arxiv.org/abs/2606.03918
- **介绍**: 102 个真实对冲基金分析师任务的基准，基于专家推理轨迹，确定性分级。前沿模型得分 <16%，揭示开放域金融推理的巨大挑战。
- **技术架构**: 真实任务采集→专家推理轨迹→确定性分级规则→自动化评估。
- **核心创新**: 从"模型评判"转向"确定性分级"，消除评估噪声和循环性。
- **关键实现**: 基于真实对冲基金工作流构建任务，每个任务有验证的专家步骤。
- **与现有技术对比**: 现有金融基准评估孤立子技能或最终答案，Hedge-Bench 评估开放域推理能力。
- **优势与局限**: 优势是真实场景、确定性评估；局限是 102 个任务规模有限。
- **未来发展潜力**: 可能成为金融代理评估的事实标准。
- **个人技术理解**: 前沿模型 <16% 得分说明开放域推理仍是巨大挑战。确定性分级方法值得其他领域借鉴。

#### 4. DDOR: Delta Debugging for Overrefusal Testing
- **地址**: https://arxiv.org/abs/2606.03601
- **介绍**: 将经典 delta debugging 引入 LLM 过度拒绝测试，自动定位最小拒绝触发片段（mRTF），提供短语级可解释证据。每模型生成约 1K 测试用例，并支持定向 prompt 修复。
- **技术架构**: Delta Debugging→mRTF 定位→多样化 prompt 生成→多 Oracle 验证→定向修复。
- **核心算法**: 最小拒绝触发片段（mRTF）定位算法，逐步缩小触发拒绝的最小文本片段。
- **关键实现**: 黑盒设置（仅输入输出可访问），无需访问内部安全机制。
- **与现有技术对比**: 现有方法依赖 LLM 评判或人工标注，DDOR 是全自动、可解释的。
- **优势与局限**: 优势是全自动+可解释+支持修复；局限是仅针对过度拒绝，不覆盖其他安全问题。
- **未来发展潜力**: Delta debugging 方法可能扩展到其他 LLM安全测试场景。
- **个人技术理解**: 经典软件工程方法（delta debugging）在 LLM安全测试中的成功应用，体现了跨领域方法迁移的价值。

#### 5. SPOQ: Specialist Orchestrated Queuing
- **地址**: https://arxiv.org/abs/2606.03115
- **介绍**: 多代理软件工程方法论，结合波次拓扑调度、双验证门、Human-as-an-Agent。三阶代理层级（Opus 工人/Sonnet 审查员/Haiku 调查员）。17 仓库 8589 commits 纵向研究，99.87% 测试通过率。
- **技术架构**: 波次调度→规划验证→执行→代码验证→人类审查。
- **核心算法**: 波次拓扑调度，从任务依赖图计算并行执行波次，接近关键路径下限（比率 1.03-1.11）。
- **关键实现**: 三阶代理层级优化成本-质量权衡，双验证门减少返工。
- **与现有技术对比**: 现有方法有协调开销、质量控制缺口、人类监督有限。
- **优势与局限**: 优势是大规模实证验证（17 仓库）；局限是复杂度高，需要精心调优。
- **未来发展潜力**: 可能成为多代理软件工程的默认架构。
- **个人技术理解**: 17 仓库 8589 commits 的纵向研究提供了罕见的工业级实证数据。波次调度+双验证门的组合设计体现了工程化思维。
