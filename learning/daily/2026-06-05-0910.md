## 深度研究

### GitHub 项目

#### 1. DCIChecker: MCP 描述-代码不一致检测

**地址**: https://arxiv.org/abs/2606.04769
**来源**: arXiv (cs.CR)
**领域**: AI Testing / Agent Security

**技术介绍**:
MCP（Model Context Protocol）已成为 LLM 使用外部工具的关键标准。LLM 依赖 MCP 服务器提供的自然语言描述来选择和执行函数，但这一交互隐含假设工具描述忠实反映底层实现，实践中却未被强制验证。DCIChecker 提出了一个全面的研究框架：首先形式化定义了描述-代码不一致（DCI）问题，提出了包含功能不一致和未声明副作用的分类学，然后开发了自动化检测框架，结合结构感知静态分析与 Direct-Reverse-Arbitration 提示方法，交叉验证工具描述与实际代码实现。在 2,214 个真实 MCP 服务器的 19,200 描述-代码对上，发现 9.93% 存在不一致。

**应用场景**: MCP 安全审计、Agent 工具链验证、供应链安全

**为什么值得关注**: MCP 生态快速增长（2,214 服务器仅采样），9.93% 不一致率意味着每 10 个工具就有 1 个描述与实际不符。这创造了从操作失败到隐蔽恶意行为的安全盲区。DCIChecker 的静态分析 + 提示交叉验证方法为 MCP 安全提供了首个系统性检测工具。

#### 2. Ekka: LLM 推理静默错误自动诊断

**地址**: https://arxiv.org/abs/2606.04594
**来源**: arXiv (cs.DC) — ICML 2026
**领域**: AI Testing / LLM Infrastructure

**技术介绍**:
LLM 推理框架快速发展，复杂软件栈和大量优化引入静默错误——输出质量静默退化而无任何显式错误信号。Ekka 将静默错误诊断形式化为差分调试问题：利用语义正确的参考实现，系统性地对齐和比较目标框架与参考框架的中间执行状态来定位根因。构建了来自流行推理框架的真实静默错误基准，Ekka 达到 80% pass@1 / 88% pass@5 诊断准确率，超越 SOTA 系统，并发现 4 个新静默错误全部被开发者确认。

**应用场景**: LLM 推理框架质量保障、CI/CD 静默错误检测、推理基础设施回归测试

**为什么值得关注**: 静默错误是 LLM 推理基础设施中最危险的缺陷类型——没有错误信号，只能通过输出质量下降间接发现。Ekka 的差分调试方法为推理框架质量保障提供了系统化方法，ICML 2026 录用确认了其学术价值。

### 学术论文

#### 3. Self-Reflective APIs: Structure Beats Verbosity for AI Agent Recovery

**地址**: https://arxiv.org/abs/2606.05037
**来源**: arXiv (cs.SE)
**领域**: AI Testing / API Design

**技术介绍**:
当 AI 代理调用 API 遇到验证错误时，它需要的不只是"哪里错了"，而是"接下来做什么"。自反思 API 在验证失败时返回机器可读的恢复建议负载，使代理能够修复请求并重试而无需外部推理。在泄漏审计的试点实验中（30 样本/单元，3 个 LLM，10 个对抗任务），结构化建议将任务完成率提升 +36.7~40.0pp（Anthropic 模型），per-success token 效率 1.8~2.2x。在 gpt-4o-mini 上效果不显著（p=0.435），计费 API 上的二次域复制确认了该模式。研究还审计发现了 LLM 基准中两类未文档化的答案泄漏。

**应用场景**: Agent-API 交互协议设计、代理错误恢复、API 可观测性

**为什么值得关注**: 这是首个系统验证"结构化恢复建议优于自然语言诊断"的研究，对 Agent-API 交互设计有直接指导意义。泄漏审计工具 audit_leakage.py 作为可复用 CI 基础设施开源，具有实用价值。

#### 4. The Invisible Lottery: How Subtle Cues Steer Algorithm Choice in LLM Code Generation

**地址**: https://arxiv.org/abs/2606.04057
**来源**: arXiv (cs.SE)
**领域**: Software Testing / LLM Code Generation

**技术介绍**:
46,535 个受控实验，11 个任务、19 种提示类型（18 通道 + 记忆化语义/表面抽象保持含义改变排版标点）、15 个模型配置。发现 incidental prompt cues（任务规范之外的上下文词或元数据）会导致算法族分布发生高达 100pp 的系统性偏移，与提示语义一致，包括限速等实际应用任务。直接算法命名是最可靠的缓解方法。意外上下文创造了性能、安全性和可维护性的"隐形彩票"。

**应用场景**: LLM 代码生成质量保障、提示工程最佳实践、代码生成评估协议

**为什么值得关注**: 首次系统量化了提示中意外上下文对算法选择的巨大影响（100pp 偏移），这对 LLM 代码生成的可靠性有深远影响。所有输出通过相同测试但算法选择不同，意味着功能正确性不足以评估代码质量。

#### 5. PCAA: Proof-Carrying Agent Actions — Model-Agnostic Runtime Governance

**地址**: https://arxiv.org/abs/2606.04104
**来源**: arXiv (cs.SE)
**领域**: Agent Security / Runtime Governance

**技术介绍**:
代理系统通过差异极大的运行时执行：本地编码工具、框架 SDK、托管代理平台、API 网关、仅观察集成。高风险行动（如外部发布数据）在不同运行时表现为 shell 命令、工具调用、托管会话转换，使治理问题难以一致回答。PCAA 提出运行时中立治理模型，以行动证书而非供应商原生会话记录为核心，组织 5 个检查点：行动前可接受性、行动开放、假设捕获、审批、结果关闭。绑定可移植行动信封、运行时/审批收据、可重放证明。证书是外部性感知的（携带目的地可见性和账户来源等边界事实），审批由显式可执行类描述而非单一审阅/未审阅位。在扩展基准（24 可执行种子→96 轨迹，4 运行时家族）上验证。

**应用场景**: 企业代理治理、合规审计、多运行时代理安全

**为什么值得关注**: 代理运行时碎片化是治理的核心挑战。PCAA 首次提出不依赖特定供应商的运行时中立治理模型，以行动证书统一跨运行时治理，对多运行时代理生态的安全合规有重要意义。

#### 6. An Empirical Catalog of 63 LLM-Agent Budget-Overrun Incidents

**地址**: https://arxiv.org/abs/2606.04056
**来源**: arXiv (cs.SE)
**领域**: AI Testing / Agent Reliability

**技术介绍**:
LLM 代理预算超支是已记录的生产失败类：单次重试循环可在操作员注意前花费数千美元。研究提供了 63 起已确认生产事故目录（21 编排框架，2023-2026），每个都有引用的 GitHub issue 和美元损失，组织为 8 类失败分类学（评分者间 Cohen's kappa = 0.837），加上 47 个补充结构条目。作为针对此分类学的缓解评估，构建了 token-budgets Rust crate（1,180 行，无 unsafe），将仿射所有权操作化，使克隆/双重花费/委托后使用预算成为编译错误而非运行时危险。单代理工作负载上 4 行 Python 计数器匹配 crate 的 0/30 超调，区别在于多代理委托下的不可绕过性：11 起事故中记录的委托扇出竞赛在编译时被 borrow checker 拒绝。

**应用场景**: 代理预算控制、多代理委托安全、生产部署可靠性

**为什么值得关注**: 首个系统化的代理预算超支实证研究，63 起真实事故提供了前所未有的失败模式全景。Rust affine 类型缓解方案将运行时风险转化为编译时错误，为多代理系统的预算安全提供了可验证的方法。
