#### 1. Nous Hermes Agent — 自改进 AI 代理的完整学习闭环

**项目地址**：https://github.com/NousResearch/hermes-agent

**项目介绍**：
Nous Research 推出的自改进 AI 代理，是目前唯一内置学习循环的开源代理系统。与传统代理"每次从零开始"不同，Hermes Agent 构建了完整的经验积累机制：从复杂任务中自动创建技能、在使用过程中自我改进、通过周期性推送保持知识持久化、搜索历史对话实现跨会话回忆、并构建深度用户模型。它运行在 $5 VPS 到 GPU 集群的任何基础设施上，支持 Telegram/Discord/Slack/WhatsApp/Signal/CLI 六平台，兼容 200+ 模型。

**技术架构**：
- **学习循环**：Agent-curated memory + periodic nudges → 自主技能创建 → 使用中自我改进
- **记忆系统**：FTS5 会话搜索 + LLM 摘要实现跨会话回忆
- **用户模型**：Honcho 辩证用户建模，跨会话深化理解
- **终端后端**：local/Docker/SSH/Singularity/Modal/Daytona 六种
- **技能标准**：兼容 agentskills.io 开放标准

**核心创新**：
- 首次将"学习循环"作为代理的一等公民，而非事后附加
- 服务器端持久化：Daytona/Modal 提供 serverless 持久化，空闲时休眠
- 研究就绪：批量轨迹生成 + 轨迹压缩用于训练下一代工具调用模型

**与现有技术对比**：
- vs ECC：ECC 侧重 harness 性能优化，Hermes 侧重代理自我进化
- vs CodeGraph：CodeGraph 提供代码知识图谱，Hermes 提供跨领域学习记忆
- vs oh-my-pi：oh-my-pi 侧重编辑质量，Hermes 侧重长期学习能力

**优势与局限**：
- 优势：学习闭环完整、多平台部署、模型无关、研究就绪

**未来发展潜力**：

#### 2. Tangram — 非均匀 KV 缓存的系统级解决方案

**项目地址**：https://github.com/aiha-lab/TANGRAM

**项目介绍**：
Tangram 是多轮 LLM 服务中非均匀 KV 缓存的开创性系统。传统 KV 缓存压缩要么均匀压缩所有 token（信息损失），要么非均匀压缩但引入内存碎片和调度复杂性。Tangram 通过三个核心技术解决了非均匀压缩的系统级挑战：确定性预算分配消除动态调度开销、头分组页表最大化物理内存回收、AOT 负载均衡确保均匀 GPU 利用。实验显示吞吐 +2.6x，精度完全保留。

**技术架构**：
- **确定性预算分配**：基于注意力头的内在模式分配静态内存足迹
- **头分组页表**：相似保留需求的头聚类，独立向量化页表管理
- **AOT 负载均衡**：静态预算配置文件确保无运行时开销的均匀利用

**核心算法**：
非均匀 KV 缓存的异构性导致内存碎片、调度复杂、内核利用率下降。Tangram 将异构性"静态化"：在 prefill 阶段就确定每个头的内存需求，避免运行时动态分配。

**与现有技术对比**：
- vs headroom：headroom 压缩输入 token，Tangram 优化 KV 缓存存储
- vs 均匀 KV 压缩：Tangram 保留更多信息，吞吐 +2.6x
- vs vLLM PagedAttention：Tangram 在 PagedAttention 之上增加非均匀管理

**优势与局限**：
- 优势：吞吐提升显著、精度无损、系统级完整方案
- 局限：需要 prefill 阶段额外分析、对动态注意力模式适应性待验证

**未来发展潜力**：
随着多轮对话成为 LLM 服务的主流场景，KV 缓存效率是核心瓶颈。Tangram 的非均匀压缩方案可能成为 vLLM/SGLang 等 serving 框架的内置选项。

#### 3. CodeGraph — 预索引代码知识图谱

**项目地址**：https://github.com/colbymchenry/codegraph

**项目介绍**：
CodeGraph 为 AI 编码代理提供预索引的代码知识图谱。当 Claude Code 等代理探索代码库时，传统方式通过 grep/glob/Read 逐文件扫描，消耗大量 token。CodeGraph 提供符号关系、调用图、代码结构的预索引知识图谱，代理直接查询图谱而非扫描文件。在 7 个真实开源代码库 × 7 种语言的测试中，平均节省 token 16%，工具调用显著减少。支持 Claude Code/Cursor/Codex/opencode/Hermes/Gemini 等主流代理自动配置。

**技术架构**：
- **预索引**：一次性构建代码知识图谱（符号关系 + 调用图 + 结构）
- **MCP 集成**：自动配置各代理的 MCP server
- **100% 本地**：无云端依赖，数据完全本地

**核心创新**：
- 将"文件扫描"范式转变为"图谱查询"范式
- 零配置安装：codegraph install 自动检测并配置所有代理

**优势与局限**：
- 优势：token 节省显著、多代理兼容、本地部署
- 局限：索引构建需要时间、大型代码库索引体积可能较大

**未来发展潜力**：
随着代理编码成为主流，代码知识图谱可能成为每个项目的标配基础设施，类似今天的 node_modules 或 .git 目录。

### 学术论文

#### 1. The End of Software Engineering: How AI Agents Are Fundamentally Restructuring the Software Paradigm

**论文地址**：https://arxiv.org/abs/2606.05608

**论文介绍**：
这篇论文提出了一个激进但论证充分的观点：AI 代理的出现不是软件工程的增量改进，而是根本性的范式重构。传统软件中"代码是决策逻辑的载体"，而代理系统中"代码是 LLM 推理循环的临时工具"。论文从复杂度缩放的第一性原理出发，形式化区分了传统软件与代理系统的本质差异，追溯了从许可软件→SaaS→Agent-as-a-Service (AaaS) 的历史演进，提出"Agent Engineering"作为新兴学科。通过 SWE-bench Verified、EvoClaw、LangChain 多代理协调研究等证据，论文展示了代理范式的变革潜力和当前局限，并提出了四阶段自进化代理生态系统路线图。

**技术架构**：
- **范式区分**：代码作为决策载体 vs 代码作为推理工具
- **复杂度分析**：从第一性原理分析复杂度缩放差异
- **历史演进**：许可软件→SaaS→AaaS 的复杂度转移轨迹
- **四阶段路线图**：自进化代理生态系统的演进路径

**核心论点**：
1. 代理系统不是"更好的软件工程"，而是"不同的工程学科"
2. 控制模型从静态代码流变为动态推理循环
3. 人类角色从"编写代码"变为"定义目标 + 审查代理输出"

**与现有技术对比**：
- vs MicroSkill：MicroSkill 关注技能封装，本文关注范式转移
- vs 传统 SE 文献：首次系统论证"代理工程"作为独立学科

**优势与局限**：
- 优势：理论框架完整、历史视角清晰、路线图具体
- 局限：部分论点仍为概念性，需要更多实证支撑

**未来发展潜力**：
这篇论文可能成为"代理工程"领域的奠基性文献，类似于 1968 年 NATO 软件工程会议的历史地位。

#### 2. Willing but Unable: Separating Refusal from Capability in Code LLMs via Abliteration

**论文地址**：https://arxiv.org/abs/2606.05396

**论文介绍**：
这篇论文研究了代码 LLM 中安全对齐与生成能力的可分离性。当要求 LLM 从安全代码注入指定 CWE 漏洞时，安全对齐的代码 LLM 系统性地拒绝此类提示。论文使用"abliteration"——一种低秩权重编辑技术，正交投影移除残差流中的拒绝方向——作为案例研究（Python + CWE-89 SQL 注入，Qwen2.5-Coder 3B/7B/14B）。关键发现：(1) 拒绝强烈依赖模型大小和提示上下文（14B 100% 拒绝，7B 73%/5%，3B 几乎从不拒绝）；(2) abliteration 将拒绝降至零或接近零，同时保持语法有效性 >93%；(3) 注入率仍受容量限制（14B 88-97%，7B 89-90%，3B 25-48%），表明"意愿"（abliteration 解锁）与"能力"（随参数缩放）是可分离的。

**技术架构**：
- **Abliteration**：低秩权重编辑，正交投影移除拒绝方向
- **评估框架**：三工具检测器集成（CodeQL + Semgrep + Bandit）+ 人工裁决

**核心发现**：
- 安全拒绝与代码生成能力在低秩空间中可正交分离
- 小模型（3B）本质上是"willing but unable"——不拒绝但能力不足
- 大模型（14B）拒绝是安全训练的副作用，而非能力不足

**优势与局限**：
- 优势：首次实证分离安全与能力、方法论严谨
- 局限：仅研究单一 CWE 类型、未评估 abliteration 对其他安全属性的影响

**未来发展潜力**：
这项技术可能成为代码 LLM 安全评估的标准工具，帮助区分"模型太安全"和"模型不够强"两种情况。

#### 3. DeployBench: Benchmarking LLM Agents for Research Artifact Deployment

**论文地址**：https://arxiv.org/abs/2606.05238

**论文介绍**：
DeployBench 是首个面向研究工件部署的多领域基准。现有环境设置基准不覆盖研究工件部署的完整范围：多语言工具链、系统级依赖（GPU/CUDA/内核配置）、遗留工件兼容性。DeployBench 包含 51 个部署任务，覆盖 AI/ML、计算机系统、科学计算三大领域。每个任务由隐藏管道验证，执行论文指定实验并检查结果。使用 OpenHands + 4 个 SOTA LLM 评估，pass rate 7.8%-51.0%。关键发现：154 个失败中 97 个是"完成判断"问题——代理的预完成检查验证了比论文特定任务要求更弱的目标。

**技术架构**：
- **多领域覆盖**：AI/ML + 系统 + 科学计算
- **隐藏验证管道**：执行论文实验 + 检查结果
- **OpenHands 集成**：统一代理框架评估

**核心发现**：
- 代理的"完成判断"能力是部署失败的主因（63%）
- 当前代理在自主部署方面与人类专家差距巨大
- 需要更精细的任务完成标准验证

**优势与局限**：
- 优势：真实场景覆盖、隐藏验证、多领域
- 局限：51 个任务规模有限、OpenHands 单框架

**未来发展潜力**：
作为科研自动化基础设施，DeployBench 可能推动"自主科研代理"领域的发展，类似于 SWE-bench 对编码代理的推动作用。

## 技术趋势

1. **Agent 学习闭环成为基础设施**：Hermes Agent 的自学习循环、MemOp 的无标注记忆优化、MicroSkill 的自主技能提取，三者共同指向"代理自我进化"成为标配能力。学习循环不再是研究概念，而是工程基础设施。
