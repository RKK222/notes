## 技术扫描（35 项）

### AI Testing / Software Testing（10 项）

1. **Code2LoRA: Hypernetwork-Generated Code Model Adapters** (arXiv 2606.06492, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06492
   - 领域：AI Testing / 代码模型适配
   - 技术介绍：利用超网络（Hypernetwork）为代码生成模型动态生成 LoRA 适配器，而非手动微调每个任务。超网络从任务描述中预测适配器权重，实现零样本任务适配。在 HumanEval、MBPP 等基准上验证，接近全量微调性能但推理成本大幅降低。
   - 应用场景：多任务代码生成、快速任务适配
   - 为什么值得关注：将 Hypernetwork 思想引入代码模型适配，为"一个基础模型服务多种代码任务"提供新方案。

2. **HarnessFix: Diagnosing and Repairing Agent Harness Failures from Trajectories** (arXiv 2606.06324, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06324
   - 领域：Agent Harness 故障诊断
   - 技术介绍：首次系统性地从失败轨迹中诊断 Agent Harness 故障。提出 HTIR（Harness Trajectory Intermediate Representation）中间表示和 ETCLOVG 七层分类法，覆盖从环境配置到执行逻辑的故障类型。实验显示最佳修复 LLM 仅 18.9% 成功率，揭示 Harness 修复的困难程度。
   - 应用场景：Agent 调试基础设施、Harness 质量保障
   - 为什么值得关注：Agent Harness 故障诊断从行为级到架构级，为 Agent 调试提供系统性方法论。

3. **Goedel-Architect: Blueprint Generation and Refinement via Formal Theorem Proving** (arXiv 2606.06468, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06468
   - 领域：形式化验证 / 架构蓝图生成
   - 技术介绍：将形式化定理证明引入软件架构蓝图生成。通过 Lean4 证明架构约束的正确性，确保生成的蓝图在结构上无矛盾。结合 LLM 生成能力和形式化验证的严格性，实现可验证的架构设计。
   - 应用场景：安全关键系统架构设计、形式化架构验证
   - 为什么值得关注：形式化验证从代码级扩展到架构级，为 AI 生成架构提供数学保证。

4. **TLA-Prover: Verifiable TLA+ Specification Synthesis via Preference-Optimized LoRA** (arXiv 2606.06133, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06133
   - 领域：形式化验证 / TLA+ 规格合成
   - 技术介绍：TLA+ 是分布式系统和安全关键协议的形式规格语言。25 个 LLM 中最佳公开基线仅 26.6% 语法解析和 8.6% 语义模型检查。TLA-Prover 是 20B 参数模型，结合 SFT 和基于修复的 GRPO，TLC 直接提供奖励信号。达到 30% pass@1 Gold/Diamond，约为未调优基线 8.6% 的 3.5 倍。
   - 应用场景：分布式系统形式化验证、安全关键协议规格生成
   - 为什么值得关注：TLA+ 在 Amazon/Microsoft 等工业界广泛使用，TLA-Prover 大幅降低形式化验证门槛。

5. **Sem: New Primitive for Code Understanding** (https://ataraxy-labs.github.io/sem/, HN 47 points)
   - 来源：https://ataraxy-labs.github.io/sem/
   - 领域：代码理解 / 语义抽象
   - 技术介绍：Sem 不是 LSP（Language Server Protocol），而是在 Git 之上的实体抽象层。它提供代码的语义级理解，超越文本和 AST 层面，直接操作代码实体（函数、类、变量）及其关系。支持跨文件、跨仓库的语义查询和推理。
   - 应用场景：代码智能、Agent 代码理解、语义搜索
   - 为什么值得关注：代表代码理解从"文本处理"向"语义理解"的范式转变，对 Agent 代码操作有深远影响。

6. **Metamorphic Testing with the Rashomon Set: Explanation Faithfulness in ML** (arXiv 2606.06056, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06056
   - 领域：ML 测试 / 解释可信度
   - 技术介绍：多个 ML 模型可在同一任务上达到接近等效的预测性能，但提供不同的基于特征的解释（Rashomon 效应）。提出基于 metamorphic testing 的框架，在不要求 ground-truth 标签的情况下评估解释的忠实度。五个 metamorphic 关系形式化了模型行为与特征归因之间的预期一致性属性。
   - 应用场景：ML 解释可信度评估、XAI 质量保障
   - 为什么值得关注：Rashomon 效应揭示了"性能相同的模型可能给出完全不同的解释"这一根本问题。

7. **Asuka-Bench: Benchmarking Code Agents on Underspecified User Intent and Multi-Round Refinement** (arXiv 2606.05920, cs.SE)
   - 来源：https://arxiv.org/abs/2606.05920
   - 领域：AI Testing / 多轮代码生成
   - 技术介绍：Asuka-Bench 配对 underspecified 用户意图与多轮精炼，基于浏览器渲染行为。Code Agent 生成 Web 项目，UI Agent 在部署站点上执行测试用例，User LLM 将评估结果转化为自然语言反馈用于下一轮。50 个 Web 任务、784 个评估标准。即使最强模型三轮后也只完成 52% 的项目。
   - 应用场景：Web 开发多轮 Agent 评估、用户意图渐进式理解
   - 为什么值得关注：真实 Web 开发中用户很少一开始就写出完整规范，Asuka-Bench 模拟了"看到中间结果后才明确需求"的真实场景。

8. **Microskill Architecture: A Modular Skill-Driven Framework for AI-Native Code Generation** (arXiv 2606.05720, cs.SE)
   - 来源：https://arxiv.org/abs/2606.05720
   - 领域：Agent 架构 / 知识封装
   - 技术介绍：将微服务设计理念应用于知识封装而非服务分解。将知识分区为原子、范围精确的技能胶囊，动态路由器仅选择语义相关的胶囊用于任务。企业内容管理系统 15 个复杂功能，token 消耗减少 90% 以上，首次编译成功率几乎翻倍，完全消除架构违规。
   - 应用场景：AI 原生代码生成系统、大规模项目知识管理
   - 为什么值得关注：与 COLLEAGUE.SKILL 形成互补——MicroSkill 关注"如何高效路由知识"，COLLEAGUE.SKILL 关注"如何蒸馏专家知识"。

9. **DragOn: A Benchmark and Dataset for Drag-Based GUI Interactions** (arXiv 2606.06322, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06322
   - 领域：GUI 代理测试 / 拖拽交互
   - 技术介绍：GUI agents 在 click-grounding 上已有百万级数据集，但 drag grounding（拖拽、滑动、高亮）数据小一个数量级。DragOn 覆盖四个域：文本高亮、单元格选择、元素调整、滑块操作。286K 训练截图、3.5M 训练任务、2000 例评估集。评估 GPT/Claude/Qwen/Kimi/Holo 等模型。
   - 应用场景：GUI 代理拖拽操作评估、Computer-Use 测试
   - 为什么值得关注：填补 GUI 代理拖拽交互评估空白，对 Computer-Use Agent 的实用化有直接推动作用。

10. **Human Oversight and Overload: Two Hidden Burdens of AI-Assisted SE** (arXiv 2606.05770, cs.SE)
    - 来源：https://arxiv.org/abs/2606.05770
    - 领域：SE 研究 / AI 辅助工程人类负担
    - 技术介绍：刻画两个常被忽视的负担：(1) 对 AI 生成工件持续的人类监督和检查需求；(2) 软件工程师从 AI 工具接收大量建议而产生的认知过载。人类监督不是可选的——工程师必须审查、验证、有时重新处理 AI 产生的内容。
    - 应用场景：AI 辅助软件工程的人因工程研究
    - 为什么值得关注：大多数研究关注 AI 能做什么，这篇论文关注 AI 给人类带来的隐性成本。

### AI / LLM 技术（12 项）

11. **Vortex: Efficient and Programmable Sparse Attention Serving for AI Agents** (arXiv 2606.06453, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06453
    - 领域：LLM 推理优化 / 稀疏注意力
    - 技术介绍：稀疏注意力对长上下文 LLM 服务越来越重要，但部署和评估新算法工程成本极高。Vortex 结合 Python 嵌入前端语言和页级张量抽象，与主流 LLM 服务栈紧密集成。AI 代理使用 Vortex 自动生成和精炼算法，最佳达到 3.46x 吞吐提升。在 GLM-4.7-Flash 上 4.7x，MiniMax-M2.7 (229B) 上 1.37x。
    - 应用场景：长上下文 LLM 推理加速、稀疏注意力算法探索
    - 为什么值得关注：将稀疏注意力从理论研究推向工程实践，AI 代理可直接参与算法设计。

12. **RedKnot: Head-Aware KV Reuse and SegPagedAttention for Long-Context LLM Serving** (arXiv 2606.06256, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06256
    - 领域：LLM 推理优化 / KV 缓存管理
    - 技术介绍：现有服务系统依赖单体 KV 缓存抽象，将 KV 缓存视为同质 token 级内存块。RedKnot 打破这一抽象，沿 KV head 分解 KV 缓存——不同 head 具有不同功能角色、注意力距离和运行时重要性。支持位置无关 KV 复用、前缀 KV 压缩、冷热 KV 分离和分布式 KV 放置，无需模型重训。
    - 应用场景：长上下文 LLM 推理、KV 缓存优化
    - 为什么值得关注：将 KV 缓存从被动运行时工件转变为动态模型感知运行时基质，是 AI 基础设施的新基础。

13. **CMTF: Causal Minimal Tool Filtering for Reliable LLM Agents** (arXiv 2606.06284, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06284
    - 领域：Agent 工具选择 / 因果推理
    - 技术介绍：现有工具选择方法优化语义相关性，但相关性不足——工具可能与任务相关但在当前步骤不必要或过早。CMTF 使用轻量级前置条件-效果契约，仅暴露从当前状态推进到用户目标所需的最小下一步工具前沿。102 任务、100 工具、4 个 LLM 后端实验：将可见工具从 100 减少到每步 1 个，token 减少约 90%。
    - 应用场景：Agent 工具选择优化、多步工具使用
    - 为什么值得关注：从语义相关性到因果充分性的范式转变，是 Agent 工具治理的关键突破。

14. **Benchmark Agent: Benchmark Everything Everywhere All at Once** (arXiv 2606.06462, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06462
    - 领域：基准构建 / 评估自动化
    - 技术介绍：基准构建劳动密集且难以复用。Benchmark Agent 是完全自主的基准构建系统，编排从用户查询分析到子任务设计、数据标注和质量控制的完整流水线。实现 15 个代表性基准，覆盖文本理解、多模态理解和领域特定推理。人类评估、LLM-as-judge 和一致性检查验证高质量。
    - 应用场景：自动基准构建、模型评估基础设施
    - 为什么值得关注：基准自动化是"AI 评估 AI"闭环的关键一环，快速演进的基准可对抗模型饱和。

15. **MLEvolve: A Self-Evolving Framework for Automated ML Algorithm Discovery** (arXiv 2606.06473, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06473
    - 领域：AutoML / 自进化 Agent
    - 技术介绍：现有 MLE Agent 存在分支间信息隔离、无记忆搜索和缺乏层次控制的问题。MLEvolve 通过 Progressive MCGS 树搜索实现跨分支信息流，通过图基参考边支持经验检索和复用。引入 Retrospective Memory 结合冷启动领域知识库和动态全局记忆。将战略规划与代码生成解耦以支持长期迭代。
    - 应用场景：自动化机器学习算法发现、ML 工程 Agent
    - 为什么值得关注：将自进化能力引入 ML 算法发现，是 AutoML 从静态搜索到动态进化的关键一步。

16. **DataCOPE: Unsupervised Skill Discovery for Agentic Data Analysis** (arXiv 2606.06416, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06416
    - 领域：Agent 技能发现 / 数据分析
    - 技术介绍：推理时技能增强为数据分析师 Agent 提供轻量级改进方式，但发现有效技能仍然困难。DataCOPE 从探索轨迹中推导验证器信号，使用无监督验证器进行对比技能蒸馏。报告式分析使用自适应检查点验证器，推理式分析使用答案一致性验证器。平均提高报告式任务 9.71%、推理式任务 32.30%。
    - 应用场景：数据分析师 Agent 技能自动发现、无监督技能学习
    - 为什么值得关注：无需标注数据即可发现可复用技能，是 Agent 自主进化的重要方向。

17. **LLM Self-Recognition: Steering and Retrieving Activation Signatures** (arXiv 2606.06315, cs.AI) — ICML 2026
    - 来源：https://arxiv.org/abs/2606.06315
    - 领域：LLM 可解释性 / 内容溯源
    - 技术介绍：LLM 隐式编码了使其能识别自身输出的信号。通过在生成过程中用随机稀疏向量引导内部残差流，创建可检测指纹，实现特定 LLM 的文本归因。信号可从检测器 LLM 的激活中恢复，在多种检测设置下实现 >98% 准确率，同时保持生成文本质量。
    - 应用场景：AI 生成内容溯源、模型归因
    - 为什么值得关注：利用模型自然表示结构进行归因而非外部嵌入信号，为 AI 内容治理提供新方案。

18. **Where Should Knowledge Enter? A Layered Framework for Knowledge Infusion in Multimodal Generative Models** (arXiv 2606.06356, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06356
    - 领域：知识注入 / 多模态生成
    - 技术介绍：提出知识注入是干预层问题——生成过程作为内部状态轨迹展开，知识可作用于四个结构不同的组件：输入/输出边界、转换函数、中间状态和模型参数。映射为表面、轨迹、潜层和参数四个干预层。安全对齐实验中，三层累积将知识违规输出减少 70.97%。
    - 应用场景：多模态生成知识注入、安全对齐
    - 为什么值得关注：首次将知识注入统一为干预层框架，为多模态安全对齐提供系统性方法。

19. **Characterization and System Implications of Stateful Long-Horizon Workloads** (arXiv 2606.06448, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06448
    - 领域：Agent 记忆系统 / 系统特征分析
    - 技术介绍：首个对 Agent 记忆的系统级特征分析。引入面向系统的分类法，沿四个轴分类 Agent 记忆系统。构建阶段感知分析工具归因成本到构建、检索和生成。表征 10 个代表性系统，推导 10 条系统建议，覆盖构建调度、能力底线、查询量摊销、新鲜度-延迟权衡和集群管理。
    - 应用场景：Agent 记忆系统设计、长程工作负载优化
    - 为什么值得关注：首次系统级分析 Agent 记忆，为记忆系统设计提供实证指导。

20. **TokenMizer: Graph-Structured Session Memory for Long-Horizon LLM Context Management** (arXiv 2606.06337, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06337
    - 领域：Agent 记忆 / 上下文管理
    - 技术介绍：将 LLM 会话历史建模为类型化知识图谱，14 种节点类型和 7 种边类型。混合提取管道增量填充图，三级检查点系统序列化为紧凑恢复块。8 层压缩管道减少上下文开销。21 个会话基准：恢复块平均 78 token（基线 159-170），决策回忆率 +9-17 个百分点。保留"为什么"而不仅是"是什么"。
    - 应用场景：长程 LLM 会话管理、Agent 上下文压缩
    - 为什么值得关注：知识图谱替代文本保留，在 token 成本减半的同时保留决策理由。

21. **ALMANAC: Action-Level Mental Model Annotations for Agent Collaboration** (arXiv 2606.06388, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06388
    - 领域：人机协作 / 心理模型
    - 技术介绍：有效协作需要协作者持续维护和调整对自己推理、伙伴意图和共享目标的心理模型。ALMANAC 数据集来自 Map Task 经典二元路由任务，2987 个协作动作，每个配对理论指导的心理模型注释（自我推理、感知伙伴意图、感知团队目标）。基准测试 6 个 LLM 预测人类下一步行为和心理模型。
    - 应用场景：人机协作 Agent、心理模型对齐
    - 为什么值得关注：首个动作级心理模型注释数据集，为 Agent 协作能力提供过程级训练数据。

22. **TRACE: Temporal Conditional Estimation for Multimodal Time Series Foundation Models** (arXiv 2606.06285, cs.AI)

#### 1. Vortex: Efficient and Programmable Sparse Attention Serving for AI Agents

**地址**: https://arxiv.org/abs/2606.06453

**项目介绍（300 字）**
Vortex 是一个面向 AI Agent 的高效可编程稀疏注意力服务系统。随着 LLM 生成长度持续增长，稀疏注意力对推理效率越来越关键，但部署和评估新稀疏注意力算法的工程成本极高，阻碍了人类研究者和 AI Agent 对稀疏注意力设计的探索。Vortex 结合 Python 嵌入前端语言（基于页级张量抽象）和高效后端（紧密集成到现代 LLM 服务栈），使稀疏注意力算法能够快速原型化、部署和评估。AI Agent 使用 Vortex 自动生成和精炼多样化算法，最佳达到 3.46x 吞吐提升同时保持精度。扩展到 GLM-4.7-Flash（MLA 架构）达到 4.7x，MiniMax-M2.7（229B 参数）在 B200 GPU 上达到 1.37x。

**技术架构**
- 前端：Python 嵌入 DSL，页级张量抽象表达稀疏注意力模式
- 后端：与 vLLM/TGI 等服务栈集成，编译优化到 GPU kernel
- 中间层：自动调度、内存管理、kernel 融合

**核心算法/系统设计**
- 页级张量抽象：将注意力计算分解为页粒度，支持细粒度稀疏模式
- AI Agent 驱动算法搜索：Agent 自动生成候选算法，Vortex 编译部署并评估
- 渐进式优化：从全注意力开始，逐步引入稀疏模式并验证精度

**与现有技术对比**
- vs FlashAttention：FlashAttention 是固定稀疏模式，Vortex 支持可编程任意稀疏模式
- vs vLLM PagedAttention：PagedAttention 解决内存管理，Vortex 解决注意力计算效率
- vs 手工优化 kernel：Vortex 通过 Agent 自动搜索，无需手动调参

**优势与局限**
- 优势：Agent 可自动发现新算法、支持多种架构、无需模型重训
- 局限：需要服务栈集成、稀疏模式搜索空间大、精度-效率权衡需人工验证

**未来发展潜力**
Vortex 将稀疏注意力从"手工优化"推向"Agent 自动发现"，可能成为 LLM 推理优化的标准基础设施。随着 Agent 能力增强，自动发现的算法可能超越手工设计。

**个人技术理解**
Vortex 代表"AI 优化 AI 基础设施"的闭环——Agent 不仅使用推理服务，还参与优化推理服务本身。这种自优化循环可能是 AI 基础设施进化的关键模式。

#### 2. RedKnot: Head-Aware KV Cache Management

**地址**: https://arxiv.org/abs/2606.06256

**项目介绍（300 字）**
RedKnot 是一个头部感知的 KV 缓存管理系统，打破传统单体 KV 缓存抽象。现有服务系统将 KV 缓存视为同质 token 级内存块，用相似策略管理所有注意力头。RedKnot 观察到 KV 缓存效用沿 KV head 高度结构化——不同 head 具有不同功能角色、注意力距离和运行时重要性。因此完整 KV 缓存并非对每个 head、token 范围或服务场景都必要。RedKnot 沿 KV head 分解 KV 缓存，将其从单体张量抽象转变为结构化内存对象，统一支持位置无关 KV 复用、前缀 KV 压缩、冷热 KV 分离和分布式 KV 放置，保持输出保真度并提升资源效率，无需模型重训或微调。

**技术架构**
- Head 级分解：将 KV 缓存按注意力头分解为独立管理单元
- 功能角色分类：根据 head 的注意力模式分类（局部/全局/跨层等）
- 动态策略选择：根据服务场景选择最优缓存策略组合

**核心算法/系统设计**
- Head 重要性估计：运行时动态评估每个 head 的效用
- 分层缓存策略：热 head 保留完整 KV，冷 head 压缩或丢弃
- 分布式放置：根据 head 特性选择本地/远程存储

**与现有技术对比**
- vs vLLM PagedAttention：PagedAttention 是 token 级管理，RedKnot 是 head 级管理
- vs Stream of Thought：SoT 是 token 级淘汰，RedKnot 是 head 级差异化策略
- vs 传统 KV 缓存：RedKnot 提供 head 级差异化策略，传统方法是 uniform 的

**优势与局限**
- 优势：head 级差异化策略可显著提升缓存效率、支持分布式放置、无需模型修改
- 局限：head 特性分析开销、分布式放置的网络延迟、head 间依赖处理

**未来发展潜力**
RedKnot 代表 KV 缓存管理从"均匀处理"到"差异化处理"的范式转变。随着 LLM 规模持续增长，head 级管理可能成为标配。

**个人技术理解**
RedKnot 的核心洞察是"不是所有 attention head 都平等"——这与 MoE 中"不是所有 expert 都平等"的洞察类似。KV 缓存管理的 head 级差异化可能是长上下文推理的下一个关键优化方向。

### 学术论文

#### 1. ToolChoiceConfusion: Causal Minimal Tool Filtering for Reliable LLM Agents

**地址**: https://arxiv.org/abs/2606.06284

**论文介绍（300 字）**
本文揭示了 LLM 代理工具选择中的核心问题：工具菜单越大，代理可靠性越低。现有工具选择方法优化语义相关性，但相关性不足——工具可能与任务相关但在当前步骤不必要或过早。作者提出 Causal Minimal Tool Filtering (CMTF)，一种无训练的因果最小工具过滤方法。CMTF 使用轻量级前置条件-效果契约，仅暴露从当前状态推进到用户目标所需的最小下一步工具前沿。在 102 个任务、100 个工具、4 个 LLM 后端的基准测试中，CMTF 将可见工具从 100 减少到每步 1 个，token 使用减少约 90%，同时保持与最强因果基线相当的成功率。

**技术架构**
- 前置条件-效果契约：为每个工具定义轻量级前置条件和效果
- 因果图构建：从契约构建工具间因果依赖图
- 最小前沿计算：从当前状态计算可达的最小工具集

**核心算法或系统设计**
- 因果充分性判定：工具集 C 对状态 S 是因果充分的，当且仅当存在从 S 到目标的路径仅使用 C 中的工具
- 最小前沿算法：贪心算法计算最小因果充分工具集
- 契约自动推断：从工具描述和示例自动推断前置条件-效果契约

**关键实现机制**
- 轻量级契约表示：每个工具仅 3-5 个前置条件和效果
- 增量因果图更新：随状态变化增量更新因果图
- 零训练设计：无需训练数据，从工具描述直接推断

**与现有技术对比**
- vs 语义检索工具选择：CMTF 基于因果而非语义，显著减少误选
- vs 排名工具选择：CMTF 提供因果保证，排名只提供概率排序
- vs 所有工具暴露：CMTF 减少 99% 工具暴露，大幅降低 token 成本

**优势与局限**
- 优势：零训练、因果保证、token 成本极低、成功率不降
- 局限：契约质量依赖工具描述、复杂工具契约可能不准确

**未来发展潜力**
CMTF 代表工具选择从"相关性驱动"到"因果性驱动"的范式转变。随着工具生态增长，因果最小工具过滤可能成为标配。

**个人技术理解**
CMTF 的核心洞察是"工具选择不是检索问题，而是规划问题"——选择工具不是找最相关的，而是找完成当前步骤所需的最小集合。这与"最小特权原则"在安全领域的理念异曲同工。
