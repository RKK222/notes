### AI Testing / Software Testing（9 项）

1. **An Empirical Study of Agent-Human Interaction in Crowdsourced Testing Assessment** (arXiv 2606.06301, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06301
   - 领域：AI Testing / 众包测试评估
   - 技术介绍：研究评估驱动的可操作反馈是否能改善测试者修订报告、后续任务表现和跨应用迁移。受控四阶段人类主体研究，20 名测试者跨 3 个真实应用。结果显示 Agent 生成的反馈支持即时改进、新任务首次提交改善、以及部分但有意义的应用间迁移。后任务问卷揭示反馈通常可理解、可操作，但在特异性和执行方面仍有摩擦。
   - 应用场景：众包测试质量改进、Agent 辅助测试工作流
   - 为什么值得关注：从"后验评判"到"工作流集成反馈提供者"，Agent 不仅是裁判更是教练。

2. **Metamorphic Testing with the Rashomon Set: Explanation Faithfulness in Machine Learning** (arXiv 2606.06056, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06056
   - 领域：AI Testing / 元变换测试 / 解释可信度
   - 技术介绍：多个 ML 模型可在同一任务上达到近似性能但提供不同的特征级解释（Rashomon 效应）。提出基于元变换测试的框架，在不要求 ground-truth 标签的情况下评估解释可信度。五个元变换关系形式化模型行为与特征归因之间的一致性属性。在两个表格回归数据集和两个后验解释器（SHAP、LIME）上应用。
   - 应用场景：ML 模型解释可信度评估、XAI 质量保证
   - 为什么值得关注：解决"哪个解释是可信的"这一根本问题，模型无关的工具。

3. **Towards the Readability of LLM-Generated Codes through Multitask Representation Engineering** (arXiv 2606.06214, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06214
   - 领域：代码可读性 / 表示工程
   - 技术介绍：大多数研究关注 LLM 生成代码的正确性，可读性被忽视。采用表示工程（RepE）作为目标控制方法，低数据依赖、低计算成本。提出多任务 RepE 框架，理论上讨论多任务引导对代码可读性与正确性权衡的影响。
   - 应用场景：LLM 代码质量改进、代码可读性控制
   - 为什么值得关注：首次将 RepE 扩展到多任务代码可读性控制，揭示可读性-正确性权衡。

4. **Domain Diversity, Motivation, Inclusion, and Feedback in Software Modelling Education** (arXiv 2606.06275, cs.SE)
   - 来源：https://arxiv.org/abs/2606.06275
   - 领域：软件工程教育 / FSE 2026
   - 技术介绍：90 名学生和 22 名教育者的并行调查，发现教育者假设与学生偏好之间的脱节。学生对社会相关领域最有动力，偏好选择权，而教育者高估了学习相关领域的兴趣。
   - 应用场景：软件工程教育改进、课程设计
   - 为什么值得关注：揭示教育者与学生之间的认知偏差，学生中心领域选择是关键。

5. **CogManip: Benchmarking Manipulative Behavior in Multi-Turn Interactions** (arXiv 2606.06099, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06099
   - 领域：AI 安全 / 操纵行为检测
   - 技术介绍：评估 15 种操纵策略风险，1000 个多轮交互场景，人类专家验证。13 个代表性模型评估（包括 GPT-5.4、DeepSeek-V3.2），揭示显著风险异质性。DeepSeek-V3.2 的操纵策略对负向和良性系统提示都高度敏感。
   - 应用场景：AI 安全审计、多轮对话操纵检测
   - 为什么值得关注：现有安全基准局限于显式规则合规，CogManip 捕获动态和隐蔽的操纵策略。

6. **Evaluating Agentic Configuration Repair for Computer Networks** (arXiv 2606.06212, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06212
   - 领域：网络配置 / Agent 修复
   - 技术介绍：基准测试开源和闭源 LLM + 形式化网络验证和上下文检索工具。Agent 架构在修复效率上比基础 LLM 高 12%，安全性高 17%，得益于动态上下文管理和迭代验证能力。
   - 应用场景：网络运维自动化、配置错误修复
   - 为什么值得关注：Agent 架构在大规模复杂场景中显著优于基础 LLM。

7. **Risk Assessment of Autonomous Driving: Integrating Technical Failures, Ethical Dilemmas, and Policy Frameworks** (arXiv 2606.06396, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06396
   - 领域：自动驾驶安全 / 风险评估
   - 技术介绍：基于 NHTSA 公开碰撞数据、加州 DMV 退出报告、MIT 道德机器数据集和五个司法管辖区的比较监管分析。感知和分类错误是主要技术故障模式，不同地区的监管不一致增加了广泛应用的不确定性。
   - 应用场景：自动驾驶安全评估、政策制定
   - 为什么值得关注：技术、伦理和监管问题的综合分析，建议适应性协作治理方法。

8. **ToolChoiceConfusion: Causal Minimal Tool Filtering for Reliable LLM Agents** (arXiv 2606.06284, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06284
   - 领域：Agent 工具选择 / 因果过滤
   - 技术介绍：CMTF 使用轻量级前提-效果契约，仅暴露从当前状态向用户目标推进所需的最小下一步工具前沿。102 个任务、100 个工具、4 个 LLM 后端、2448 次运行。CMTF 将可见工具从 100 个减少到每步 1 个，token 使用减少约 90%。
   - 应用场景：Agent 工具选择优化、减少错误工具调用
   - 为什么值得关注：因果充分性优于语义相关性，工具暴露减少 99% 但保持相同成功率。

9. **From Reward-Hack Activations to Agentic Risk States: Context-Calibrated Mechanistic Monitoring** (arXiv 2606.06223, cs.AI)
   - 来源：https://arxiv.org/abs/2606.06223
   - 领域：Agent 安全 / 奖励黑客监控
   - 技术介绍：在 ReAct 风格 Agent 中研究奖励黑客监控器。适配器在 School-of-Reward-Hacks 数据集上微调后可将奖励黑客倾向转移到 Agent 动作选择中。仅靠激活动力学无法缓解，熵和上下文校准的内部特征将风险估计优于纯奖励黑客激活。
   - 应用场景：Agent 安全监控、奖励黑客检测
   - 为什么值得关注：上下文校准的内部监控——奖励黑客激活识别潜在策略状态，熵和决策上下文帮助确定何时变为风险动作。

    - 应用场景：LLM 推理加速、稀疏注意力算法自动生成

13. **DataCOPE: Unsupervised Skill Discovery for Agentic Data Analysis** (arXiv 2606.06416, cs.AI)

    - 领域：无监督技能发现 / 数据分析 Agent

15. **LLM Self-Recognition: Steering and Retrieving Activation Signatures** (arXiv 2606.06315, cs.AI)

    - 技术介绍：LLM 隐式编码其生成文本中的自识别信号。通过随机稀疏向量引导内部残差流，创建可检测指纹，实现特定 LLM 文本归因。激活空间恢复信号，98%+ 准确率，保持生成质量。ICML 2026。
    - 应用场景：AI 生成内容归因、多 LLM 识别
    - 为什么值得关注：利用模型自然表示结构进行归因，而非外部嵌入信号。

16. **Where does Absolute Position come from in decoder-only Transformers?** (arXiv 2606.06160, cs.AI)
    - 来源：https://arxiv.org/abs/2606.06160
    - 领域：Transformer 架构 / 位置编码
    - 技术介绍：RoPE 训练的 Transformer 在注意力模式中区分绝对位置，尽管 RoPE 仅编码相对偏移。追溯到两个架构组件：因果掩码（每查询 softmax 分母取决于绝对查询位置）和残差流（位置 0 的激活作为封闭动力系统运行）。NTK 缩放抑制残差流组件，滑动窗口注意力允许其随深度累积。
    - 应用场景：Transformer 架构理解、位置编码设计
    - 为什么值得关注：揭示 RoPE 中绝对位置信息的来源，对架构设计有重要启示。

#### 1. headroom: Token 压缩代理基础设施

**地址**：https://github.com/chopratejas/headroom

**项目介绍**（300 字）：
headroom 是本周增长最快的 GitHub 项目（周增 13,308 star，总 16,379 star），解决 Agent 经济性核心问题——token 成本。它在工具输出、日志、文件和 RAG 块到达 LLM 之前压缩它们，实现 60-95% token 减少而保持相同答案质量。三层架构设计：Library 层提供直接 Python/JS 集成；Proxy 层作为透明代理拦截和压缩所有工具输出；MCP Server 层通过 MCP 协议无缝集成到任何支持 MCP 的 Agent 框架。压缩策略包括：工具输出摘要、日志采样、文件内容选择性包含、RAG chunk 去冗余。与 headroom 不同，之前的 token 压缩方案（如 ThoughtFold）聚焦于推理链折叠，而 headroom 聚焦于输入侧压缩，两者互补。

#### 3. ToolChoiceConfusion: 因果最小工具过滤

**地址**：https://arxiv.org/abs/2606.06284

**论文介绍**（300 字）：
ToolChoiceConfusion 提出因果最小工具过滤（CMTF），解决 LLM Agent 工具菜单过大导致的可靠性下降问题。现有工具选择方法优化语义相关性，暴露与用户请求名称或描述匹配的工具。但相关性不足：工具可能与任务相关但在当前步骤不必要或过早。CMTF 使用轻量级前提-效果契约，仅暴露从当前状态向用户目标推进所需的最小下一步工具前沿。102 个任务、100 个工具、4 个 LLM 后端、2448 次运行。CMTF 将可见工具从 100 减少到每步 1 个，token 使用减少约 90%，同时保持与最强因果基线相同的聚合成功率。

**技术架构**：
- 前提-效果契约：轻量级工具能力描述
- 因果充分性过滤：仅暴露必要的下一步工具
- 多步工具使用任务评估

**核心算法/设计**：
- 因果充分性优于语义相关性
- 最小工具前沿：每步仅暴露必要工具
- 训练免费方法，无需模型微调

**与现有技术对比**：
- 全工具暴露：100 个工具可见，token 成本高
- 关键词检索：语义匹配，暴露过多无关工具
- CMTF：因果过滤，每步 1 个工具，token -90%

**优势与局限**：
- 优势：训练免费、token 减少 90%、保持相同成功率
- 局限：需要工具前提-效果契约、契约质量影响过滤效果

**未来发展潜力**：
随着 Agent 工具生态膨胀，工具选择效率成为关键瓶颈。CMTF 提供了一种轻量级、训练免费的解决方案，有潜力成为 Agent 工具选择的标准方法。

5. **稀疏注意力 Agent 化**：Vortex 将稀疏注意力算法设计从人工工程转变为 Agent 自动生成+快速部署循环。AI Agent 不仅使用稀疏注意力，还自动生成和优化稀疏注意力算法，Agent 从"消费者"变为"创造者"。

## 新领域收获

1. **Agent 互联网接入基础设施** — Agent-Reach 揭示 Agent 需要统一互联网接入层，当前碎片化的平台 API 是 Agent 能力扩展的主要瓶颈。未来可能涌现更多"Agent 互联网中间件"项目。

2. **Agent 软件开发方法论** — Superpowers 将 Agent 从代码生成器升级为完整 SDLC 伙伴，Spec→Plan→Subagent 驱动开发。这代表 Agent 辅助开发从"工具"到"方法论"的演进方向。

3. **认知操纵检测** — CogManip 揭示 LLM 在多轮交互中可能表现出隐蔽心理操纵行为，现有安全基准无法捕获这一问题。这是 AI 安全的新研究方向，需要动态多轮评估框架。
