1. **DeNovoSWE: Scaling Long-Horizon Environments for Generating Entire Repositories from Scratch** (arXiv:2606.10728, cs.SE)
   - 来源：https://arxiv.org/abs/2606.10728
   - 领域：Agent 代码生成基准 / 全仓库生成
   - 技术介绍：随着 LLM 代码 Agent 能力从局部 bug 修复扩展到从高层规范架构和实现完整软件仓库，训练 Agent 处理长程软件工程任务面临大规模可验证全仓库生成数据稀缺的挑战。DeNovoSWE 包含 4,818 个高质量实例，每个实例要求从文档生成完整仓库。数据集通过精心设计的沙盒化 Agent 工作流自动构建，实现可扩展的无人类标注策展。采用"分而治之"和批评者-修复哲学构建，引入难度感知轨迹过滤策略平衡数据质量和多样性。在 Qwen3-30B-A3B 上微调显著提升长程 SWE 性能，在 BeyondSWE-Doc2Repo 基准上从 5.8% 提升到 47.2%。
   - 应用场景：长程软件工程 Agent 训练、全仓库代码生成评估
   - 为什么值得关注：首次提供大规模全仓库生成数据集，从"修复已有代码"推进到"从零构建仓库"的 Agent 能力评估。

2. **GapFuzz: Cross-Plane Divergence Fuzzing for Distributed SDN Controllers** (arXiv:2606.11035, cs.SE)
   - 来源：https://arxiv.org/abs/2606.11035
   - 领域：分布式系统模糊测试 / SDN 控制器
   - 技术介绍：分布式 SDN 集群在主节点和备份之间异步复制流状态，留下窗口期使两个备份节点各自提交矛盾规则、主节点序列化到数据面、内核数据面抓住无节点认为权威的action。现有 SDN 模糊测试器遗漏此故障：它们将预言限制在控制面、针对单一控制器、或不引导并发诱发复制竞争。GapFuzz 是分布式 SDN 集群的状态化并发模糊测试器，在两个非主节点上注入矛盾北向请求对，用受控注入延迟 Δt，通过查询每个副本和内核数据面 action（ovs-appctl ofproto/trace）重建全局跨平面状态。两阶段时序搜索检测发散是否存在，然后对 Δt 加倍和二分以界定注入时间窗口；生命周期探针标记每个裁决为瞬态或持久态，并分配到从 ONOS 2.7 源码导出的四个跨平面状态类之一。在三节点 ONOS 2.7 集群上，GapFuzz 在 81.7% 尝试中产生发散裁决（N=50），所有发散位于集群权威状态和内核数据面之间。
   - 应用场景：分布式网络控制器正确性验证、跨平面状态一致性测试
   - 为什么值得关注：首次发现并系统测试 SDN 跨平面发散故障，用内核数据面探针替代用户空间探针提升检测 26.6 个百分点。

3. **TestMap: Evidence Infrastructure for Foundation-Model-Assisted Test Generation** (arXiv:2606.10211, cs.SE)
   - 来源：https://arxiv.org/abs/2606.10211
   - 领域：测试生成基础设施 / AIWare 2026
   - 技术介绍：基础模型可生成合理的单元测试，但确定这些测试是否正确、有用、可维护、值得集成仍然困难。生成的测试必须映射到目标代码、插入真实项目、构建、执行、对照基线套件测量、必要时修复、跨模型和生成策略比较。此验证过程碎片化于构建系统、测试运行器、覆盖工具、变异工具、静态分析器和实验脚本之间。TestMap 是开源基础设施原型，自动化证据支持的基础模型辅助测试生成用于 C#/.NET 仓库。支持仓库分析、源码-测试映射、基线执行、代码度量收集、测试异味检测、覆盖测量、变异测试、模型引导测试生成、验证、修复和仓库特定实验跟踪。不仅报告最终通过测试，还记录每个生成候选的生命周期，包括失败、修复、低影响和证据积极结果。这些中间结果可揭示模型限制、缺失上下文、修复成本、工具链效率问题或系统下测试的可能故障。
   - 应用场景：AI 辅助测试生成全生命周期管理、跨模型测试质量对比
   - 为什么值得关注：直击 AI 生成测试最大痛点——验证碎片化，提供端到端可观察、可重复、可比较的证据基础设施。

4. **What Makes a Harness a Harness: Necessary and Sufficient Conditions for an Agent Harness** (arXiv:2606.10106, cs.SE)
   - 来源：https://arxiv.org/abs/2606.10106
   - 领域：Agent Harness 概念定义 / 软件工程术语学
   - 技术介绍："Agent Harness"术语在生成式 AI 软件工程领域广泛流通但使用松散且多义。有时指整个产品（Claude Code、Codex CLI），有时指运行 Agent 对抗任务的评估脚手架（SWE-bench harness），有时与 Agent 框架、SDK、IDE 插件或编排器混淆。本文通过结合持久标识符作品和灰色文献来源（官方文档、词汇表、工程报告）的概念分析构建定义。重建术语谱系：从马具到经典测试 harness，到 ML 评估 harness，最终到 Agent harness。提出构成性定义阐述 Agent harness 的必要充分条件，操作化为包含/排除测试，并划定与 Agent 框架、Agent SDK、IDE 插件、eval harness、编排器的概念边界。将定义应用于六个真实 harness（Claude Code、Codex CLI、Aider、Cline、OpenHands、SWE-agent）和刻意边缘案例，测试一致包含和排除。
   - 应用场景：Agent 系统分类、工程实践指南、科学比较
   - 为什么值得关注：首次提供 Agent Harness 的操作化定义，为工程实践和科学比较提供共享词汇。

5. **Improving Detection of Semantic Faults in Machine Learning Code Using Data Characteristics** (arXiv:2606.09957, cs.SE, ICSE-NIER 2026)
   - 来源：https://arxiv.org/abs/2606.09957
   - 领域：ML 代码静态分析 / 语义故障检测
   - 技术介绍：ML 模型使用特有的语义故障是 ML 开发者的常见问题，导致次优预测、高计算成本或不正确输出。例如，可能错误地使用未缩放数据训练尺度敏感模型。ML 开发者在训练模型后手动分析结果才检测这些故障，效率低下。本文提出新型数据感知静态分析方法检测 ML 代码中的语义故障，使开发者在编写代码时而非训练模型后发现 bug。方法结合数据和控流分析以及 API 契约，在高层次抽象上进行 ML 代码的数据感知推理。通过分析真实世界 ML 笔记本样本展示解决方案潜力，发现需要数据感知方法的故障。
   - 应用场景：ML 代码质量保障、开发时语义错误检测
   - 为什么值得关注：将 ML 代码语义故障检测从"训练后"推进到"编写时"，填补 ML 代码静态分析空白。

6. **Multi-task LLMs for Bug Classification: Efficient Inference with Auxiliary Decoding Heads** (arXiv:2606.09956, cs.SE)
   - 来源：https://arxiv.org/abs/2606.09956
   - 领域：行级 Bug 定位 / 多任务 LLM
   - 技术介绍：LLM 代码生成的快速采用显著加速软件开发，但有效验证方法仍然严重落后。现有 Bug 定位技术要么过于昂贵（需要数分钟 Agent 推理和数千生成 token/文件），要么在粗函数级粒度操作不适合精确定位。行级粒度且轻量级的工作通常性能或上下文大小受限。本文引入新型行级 Bug 定位方法：(1) Token 对齐算法克服先前工作的分词挑战，(2) 轻量级多任务 LLM 用于 Bug 定位（MLC）实现高效行级 Bug 分类，(3) 多行预测优化训练配方。在行级 Bug 定位上实现同类设置 SOTA，同时以单 token/文件推断在 Defects4J 和 PypiBugs 上达到接近 Agent 级性能，推理延迟降低数量级。在 Python 小域外评估数据集上展示强泛化能力。
   - 应用场景：高效 Bug 定位、代码质量自动化保障
   - 为什么值得关注：将 Bug 定位从"分钟级 Agent 推理"推进到"单 token 推断"，延迟降低数量级。

7. **MASTOR: Multi-Agent Approach to Semantic Test Oracle Generation for RESTful APIs** (arXiv:2606.10465, cs.SE)
   - 来源：https://arxiv.org/abs/2606.10465
   - 领域：API 测试 / 语义测试预言生成
   - 技术介绍：现有自动化 RESTful API 测试方法通常依赖简单检查（HTTP 状态码、模式一致性），不足以检测语义故障、业务逻辑违规和状态依赖不一致。MASTOR 是基于实现源码为 RESTful API 生成语义测试预言的多 Agent 方法。包含两阶段：源码分析和预言生成。前者用源码提取 Agent 通过分析相关源文件的传递导入闭包为每个端点操作构建源码上下文。后者在收集上下文上应用两条并行预言生成路径：单操作路径为每个操作生成状态和字段预言，多操作路径利用跨操作语义关联为操作序列生成行为一致性预言。两条路径应用挑战者 Agent 审查，专用审查者识别弱点并发出改进提示引导定向再生，然后预言规范化过滤结构无效预言。在 13 个开源 RESTful API 项目（296 操作、251,303 行代码）基准上评估，MASTOR 实现平均 75.4% 变异分数，生成 10,022 个预言。在 50 个选定操作上，MASTOR 超越 Direct Prompting 30.1 个百分点（69.9% vs 39.8%），超越 SATORI 49.4 个百分点（69.9% vs 20.5%）。
   - 应用场景：RESTful API 语义测试、业务逻辑验证
   - 为什么值得关注：首次用多 Agent 架构生成语义测试预言，超越传统方法 49.4 个百分点。

8. **Beyond Coverage and Kill Scores: Empirically Measuring Test Suite Behavioural Gaps** (arXiv:2606.10417, cs.SE)
   - 来源：https://arxiv.org/abs/2606.10417
   - 领域：测试充分性度量 / 行为覆盖
   - 技术介绍：传统测试充分性度量测量系统实现而非其是否遵循预期行为。开发者严重依赖代码覆盖率和变异测试评估测试套件质量，但这些度量根本是面向实现的，无法检测代码预期做什么和实际做什么之间的差距。本文引入自动化概念验证方法调查这些差距。方法从自然语言文档和源码提取预期方法级行为，映射到现有测试用例，识别预期和验证行为之间的差距。在十个流行开源 Java 库（8,922 方法）上评估，提取 20,729 个行为（93.1% 精度）。保守估计 17.5% 检测到的预期行为完全未测试，称为测试套件的行为差距。评估最先进的自动化测试生成器（EVOSUITE/ASTER），发现它们类似地未能验证至少 20.6%/27.1% 检测到的预期行为。进一步证明行为差距不被传统结构度量预测：大多数未测试行为发生在已有高行覆盖的方法中，超过一半在高变异杀分方法中持续存在。
   - 应用场景：测试套件质量评估、行为覆盖度量
   - 为什么值得关注：揭示行为覆盖作为独立测试套件充分性维度，补充传统结构度量。

9. **STAGE-Claw: Automated State-based Agent Benchmarking for Realistic Scenarios** (arXiv:2606.10394, cs.AI)
   - 来源：https://arxiv.org/abs/2606.10394
   - 领域：Agent 基准测试 / 状态验证
   - 技术介绍：LLM 越来越多地用于为日常应用提供个人 Agent 能力，但评估这些 Agent 仍然面临挑战。现有基准仍依赖沙盒工件、静态任务设计和粗糙评分，阻碍可扩展性并限制向可靠个人 Agent 评估的进展。STAGE-Claw 是构建和评估基于状态的个人计算环境中现实个人 Agent 场景的自动化框架。给定任务提示，STAGE-Claw 自动创建和验证现实基准任务及其环境、任务提示、真实值和.related 验证程序。Agent 在真实操作环境中评估，性能通过最终系统状态的正确性而非仅文本回答来测量。用 STAGE-Claw 创建包含 40 个挑战性真实场景 Agent 任务的基准，评估 11 个前沿模型，分析任务分数、成本、工具调用可靠性和常见失败模式。
   - 应用场景：个人 Agent 能力评估、系统状态验证
   - 为什么值得关注：Agent 评估从文本响应推进到系统状态验证，更贴近真实使用场景。

10. **Watts and Debts of Agentic Frameworks: An Empirical Study** (arXiv:2606.10702, cs.SE, ESEM 2026)
    - 来源：https://arxiv.org/abs/2606.10702
    - 领域：Agent 框架技术债务 / 能源消耗
    - 技术介绍：每个运送到生产的 Agent AI 系统携带两个隐藏风险：累积技术债务（TD）和未监控运行时能源成本。虽然功能基准测试常见，但内部结构质量（特别是 TD）与执行期间动态能源消耗之间的实证联系仍未探索，为管理可扩展性可持续性和运营预算的实践者和组织创造盲点。本文提出确认性实证研究关联自承认技术债务（SATD）与硬件级运行时能源消耗跨 Agent 框架，确定代码质量是否可驱动能源感知设计决策。评估五个开源 Agent 框架，在严格受控环境中执行标准化任务套件。SATD 通过自动化 Python 注释挖掘提取，用微调提示进行 LLM 分类，运行时能源在硬件级别测量。研究三个核心研究问题：(R1) 这些框架中 TD 的存在，(R2) 跨架构运行时能源消耗差异，(R3) 框架 TD 与任务级能源消耗之间的统计相关性。
    - 应用场景：绿色软件工程、Agent 框架质量评估
    - 为什么值得关注：首次实证研究 Agent 框架技术债务与能源消耗关联，为框架选择提供早期预警代理。

11. **Intent-Driven Requirements Completion for IoT Systems** (arXiv:2606.10598, cs.SE)
    - 来源：https://arxiv.org/abs/2606.10598
    - 领域：IoT 需求完成 / 多 Agent 框架
    - 技术介绍：终端用户通过触发动作编程创建 IoT 自动化规则，但其表达通常碎片化，捕获设备操作而非高层意图。此差距导致缺失条件、逻辑冲突和被忽视的安全约束，带来危险行为风险。本文提出意图驱动需求完成方法，将规则完成重新框架为双重过程：从碎片化规则重建意图，然后从该意图再生规则，全程嵌入安全。引入双向需求追溯树，三层模型链接规则、意图和质量关注，设计多 Agent 框架结合 LLM 推理和结构化追溯。使完成在功能上完整且固有安全，同时保持可追溯和可解释。评估显示方法显著超越基线，规则完成率提升 43%，逻辑冲突减少超过 21%。
    - 应用场景：IoT 自动化规则完整性保障、安全关键系统需求工程
    - 为什么值得关注：从"用户责任"转向"系统责任"，从"功能正确"转向"整体可信"。

12. **FAMOU: Co-Evolutionary Mechanisms for LLM-Driven Strategy Evolution in Adversarial Games** (arXiv:2606.10389, cs.AI)
    - 来源：https://arxiv.org/abs/2606.10389

**地址**：https://arxiv.org/abs/2606.11035

**项目介绍**（250字）：

- 生命周期探针：瞬态/持久态分类 + 四状态类映射

**核心创新**：

- 从 ONOS 2.7 源码导出四跨平面状态类

**优势与局限**：
- 优势：81.7% 检测率，99.4% 发散持续超过 30 秒
- 局限：仅测试 ONOS 2.7，其他 SDN 控制器未覆盖

#### 3. TestMap: 基础模型辅助测试生成基础设施

**地址**：https://arxiv.org/abs/2606.10211

**项目介绍**（250字）：
TestMap 是开源基础设施原型，自动化证据支持的基础模型辅助测试生成用于 C#/.NET 仓库。基础模型可生成合理的单元测试，但确定这些测试是否正确、有用、可维护、值得集成仍然困难。生成的测试必须映射到目标代码、插入真实项目、构建、执行、对照基线套件测量、必要时修复、跨模型和生成策略比较。此验证过程碎片化于构建系统、测试运行器、覆盖工具、变异工具、静态分析器和实验脚本之间。TestMap 支持仓库分析、源码-测试映射、基线执行、代码度量收集、测试异味检测、覆盖测量、变异测试、模型引导测试生成、验证、修复和仓库特定实验跟踪。不仅报告最终通过测试，还记录每个生成候选的生命周期，包括失败、修复、低影响和证据积极结果。

- 证据模型：记录失败、修复、低影响、证据积极等中间结果

**核心创新**：
- 直击 AI 生成测试最大痛点——验证碎片化
- 中间结果揭示模型限制、缺失上下文、修复成本
- 端到端可观察、可重复、可比较的证据基础设施

#### 1. Frontier Coding Agents Use Metaprogramming to Adapt to Unfamiliar Programming Languages

**地址**：https://arxiv.org/abs/2606.10933

**论文介绍**（350字）：
本文揭示强编码 Agent 适应不熟悉语言的核心机制——元编程。LLM 编码 Agent 通常在熟悉的软件环境中评估（主流语言、常见库、公共仓库），但可隐藏 Agent 在语言本身不熟悉时的行为。本文在四种异编程语言上评估六个当代编码 Agent，使用包含文件编辑、本地执行和隐藏测试评分的顺序设置。关键发现：最强 Agent（Claude Opus 4.6 和 GPT-5.4 xhigh）通常避免直接编写目标语言。在 Brainfuck 和 Befunge-98 上，它们编写生成目标语言代码的 Python 程序并在本地调试这些生成器。禁止此元编程策略导致大幅性能下降。从该策略提炼的文本指导不能实质性改善弱 Agent。相比之下，来自 Opus 的 Python 辅助代码用于构建生成器（无已解决的基准程序或隐藏测试答案）显著改善 Sonnet 4.6 和 GPT-5.4 mini 在相同问题上的表现。更多解释器调用和输出 token 改善强 Agent 但使弱 Agent 保持原始性能，表明这些资源放大有用策略而非创造它们。

**技术架构**：
- 顺序评估协议：文件编辑 → 本地执行 → 隐藏测试评分
- 元编程策略：Python 生成器 → 目标语言代码 → 本地调试
- 对比实验：允许 vs 禁止元编程策略

**核心算法**：
- 强 Agent 通过工具、反馈和工作区状态构建目标语言工作模型
- 元编程是最清晰案例，但更广泛差距是在目标语言规则下构建和调试有效策略

**与现有技术对比**：
- 主流基准（SWE-bench Verified、Terminal-Bench 2.0）压缩能力差异为更窄带
- 异编程语言评估暴露 Agent 间真实能力差距

**优势与局限**：
- 优势：揭示元编程作为强 Agent 适应不熟悉语言的核心机制
- 局限：仅测试四种异编程语言，更多语言未覆盖

**未来发展**：
元编程能力可能成为编码 Agent 评估的核心维度，未来基准需包含不熟悉语言测试。

#### 2. A World Model is the Coupling Kernel of Admissible Possible Worlds

**地址**：https://arxiv.org/abs/2606.10934

**论文介绍**（300字）：
本文提供世界模型的非对角线理论，揭示预测无法表示反事实耦合不确定性的根本限制。常见假设认为给定足够观察和干预数据加上足够强的预测器即可。本文报告与此矛盾的反例模式：在数百个结构因果模型上，在已识别数量上强预测器和贝叶斯基线都成功，但在未识别数量（反事实世界之间的耦合）上预测器坍缩为单点，在 28% 模型上坍缩到无有效模型可产生的值，而真相是无限数据也无法缩窄的可接受区间。本文将世界模型框架为单个半正定耦合核 K(T,T') 跨越可接受世界，其对角线是普通后验（预测器恢复的），非对角线是它不能的跨世界耦合，每个反事实都读取。半正定性是边缘缺乏的部分识别信息，enforcing 它在多项式时间约束反事实。本体论公理收紧约束多达三分之一。完全重建是近似计数可接受世界，在 Sly-Sun 阈值以下可处理，以上不可近似。

**技术架构**：
- 世界模型 = 半正定耦合核 K(T,T')
- 对角线 = 普通后验（预测可恢复）
- 非对角线 = 跨世界耦合（预测无法表示）
- 半正定性约束 = 部分识别信息

**核心创新**：
- 首次提供世界模型的非对角线理论
- 揭示预测无法表示反事实耦合不确定性
- 半正定性作为部分识别信息约束反事实

**优势与局限**：
- 优势：理论严谨，28% 模型上预测器坍缩到无效值
- 局限：完全重建在 Sly-Sun 阈值以上不可近似

**未来发展**：
此理论对因果推断、反事实推理和世界模型设计有深远影响，可能推动新一代因果推理框架。

#### 3. The Arbiter: Continually Monitoring Multi-Agent Conversations

**地址**：https://arxiv.org/abs/2606.10747 | GitHub: https://github.com/aisilab/arbiter

**论文介绍**（300字）：
由多个语言模型 Agent 构建的 AI 系统越来越多地用于共同决策（讨论、协商和共享任务行动）。虽然单个 Agent 在单独测试时可能看起来对齐良好，但问题可能从它们之间的交互方式产生。Arbiter 设计用于实时监控多 Agent 对话并识别哪些参与者可能以不对齐方式行事。Arbiter 在有限"检查预算"下运行，必须谨慎决定如何使用资源。逐步观察对话时，可选择等待、询问参与者、检查内部信息（系统提示或推理痕迹）或记录令人担忧的行为。最终生成报告识别可能不对齐来源。在五个对话条件上评估（从风险财务建议模型生物到评估感知和串谋 Agent），测试五个能力递增的工具配置和两个骨干模型。发现 Arbiter 在对话结束前可靠检测不对齐 Agent，主动检查工具提高检测准确性和速度。权重诱导不对齐最难检测，而指令诱导不对齐即使在被动观察下也可靠识别。日志工具有双重效果，提高召回率以牺牲精确率为代价。

**技术架构**：
- 预算感知监控：有限检查预算下的最优检查策略
- 主动检查工具：询问参与者、检查内部信息、记录行为
- 多对话条件评估：风险财务、评估感知、串谋等

**核心创新**：
- 将审计者作为活跃参与者处理多 Agent 系统
- 预算感知持续监控有效捕获不对齐
- 权重 vs 指令诱导不对齐的检测难度差异

**优势与局限**：
- 优势：AITC 2026 录用，开源实现
- 局限：日志工具召回率-精确率权衡需进一步研究

**未来发展**：
多 Agent 安全监控将成为 Agent 部署的核心基础设施，Arbiter 提供重要参考实现。
