### AI Testing / Software Testing（9 项）

1. **LLMKeyLens - LLM API 凭证泄露检测（iOS 应用）** (arXiv:2606.12212)
   - 来源：https://arxiv.org/abs/2606.12212
   - 领域：Software Testing / Security
   - 首次系统研究 iOS 生态中 LLM API key 泄露。构建 444 个 iOS 应用数据集，开发 LLMKeyLens 动态分析框架（流量拦截+提供商特定 key 提取+主动有效性确认），无需源码访问或二进制解密。282 个应用暴露可利用凭证，10+ 提供商。三种泄露模式：JWT token 泄露 (48%)、无认证后端代理访问 (33%)、明文 API key 传输 (19%)。负责任披露 3 个月后仅 28% 修复，72% 仍可被利用。
   - 应用场景：移动应用安全审计、LLM 集成安全指南
   - 为什么值得关注：揭示 LLM 集成安全系统性缺口，需要提供商和平台级执行

2. **GPU LLM 服务系统软件老化分析** (arXiv:2606.11916)
   - 来源：https://arxiv.org/abs/2606.11916
   - 领域：Software Testing / Reliability
   - 216 小时实验，6 个 co-located 部署，监控主机/设备/客户端指标，统计管道处理自相关和多重检验。所有部署存在统计显著的内存老化，泄漏率强烈依赖服务运行时和部署配置。提供可复现研究框架，开辟软件老化/ rejuvenation 与 LLM 服务交叉研究方向。
   - 应用场景：LLM 服务可靠性工程、GPU 集群运维
   - 为什么值得关注：首次将软件老化研究扩展到 GPU+CUDA 的 LLM 服务场景

3. **ScaleLens - 分布式系统可扩展性故障检测** (arXiv:2606.11815)
   - 来源：https://arxiv.org/abs/2606.11815
   - 领域：Software Testing / Distributed Systems
   - 444 个可扩展性问题报告分析，10 个大规模分布式系统。发现多数故障由维度代码片段和反模式协同导致。ScaleLens 结合动态和静态分析检测维度代码片段并匹配反模式。4.2x 多于基线。Cassandra/HDFS/Ignite 最新稳定版检测 334 个确认问题片段。
   - 应用场景：分布式系统质量保障、可扩展性测试
   - 为什么值得关注：首次全面研究可扩展性故障，提供实用检测工具

4. **Acoda - 对抗性代码混淆防御 LLM 分析** (arXiv:2606.11755)
   - 来源：https://arxiv.org/abs/2606.11755
   - 领域：Software Testing / Security
   - 遗传算法框架，利用 LLM 安全对齐和 token 处理机制设计 8 种语义保持混淆方法。迭代优化混淆策略生成对抗样本。基于 LLM 响应的定量评估框架（辅助 LLM + 4 评估指标）。7 个 SOTA LLM 上 ASR 达 70%，强跨模型迁移性，最小运行时开销。
   - 应用场景：代码知识产权保护、LLM 安全防御
   - 为什么值得关注：揭示 LLM 代码分析能力的安全边界，为代码保护提供新视角

5. **SentTrack - GitHub Issue 情感驱动瓶颈检测** (arXiv:2606.11476)
   - 来源：https://arxiv.org/abs/2606.11476
   - 领域：Software Testing / DevOps
   - 双管道框架：水平管道 LLM 摘要+UMAP/HDBSCAN 聚类 613 语义簇，垂直管道 ABCDE 协作框架分类。AvaloniaUI 9000 issue 线程 49% 停滞、13% 解决。加权评分引擎（负面+停滞+解决差距+线程长度）识别高摩擦讨论。
   - 应用场景：开源项目健康监控、开发者协作效率分析
   - 为什么值得关注：将社交信号引入仓库健康分析，早于传统标签方法发现风险

7. **Multisage - 代码翻译多语义增强** (arXiv:2606.11863)
   - 来源：https://arxiv.org/abs/2606.11863
   - 领域：Software Testing / Code Quality
   - 三模块框架：语义表示解析（数据流图+类型约束+API 信息）、多语义增强（代码摘要+函数级测试+API 描述）、语义一致性校准（语义保持变异+跨语义一致性验证）。HumanEval-X 提升 2.22x，小模型增益最大。持续优于 vanilla prompting、instruction-tuned LLM、CoT。
   - 应用场景：遗留系统迁移、多语言代码转换
   - 为什么值得关注：显式语义增强显著提升 LLM 代码翻译可靠性

8. **需求质量与追溯链接恢复实证** (arXiv:2606.11834)
   - 来源：https://arxiv.org/abs/2606.11834
   - 领域：Software Testing / Requirements Engineering
   - 189 用例描述 28 种质量缺陷标注，5 种 TLR 方法评估。某些缺陷损害 TLR 性能（如不以名词短语开头的句子），某些反而有益（如包含实现细节）。不同方法对缺陷响应不同。性能优化选择取决于数据集质量。
   - 应用场景：需求工程、追溯性管理
   - 为什么值得关注：首次量化需求质量缺陷对自动追溯的影响

9. **多文件变更探索结构研究** (arXiv:2606.11976)
   - 来源：https://arxiv.org/abs/2606.11976
   - 领域：Software Testing / Agent Evaluation
   - 线性顺序探索 vs 非线性领域范围并行探索。SWE-bench Pro + Ansible 评估。领域范围并行 Agent 在 Haiku 类模型中 micro F1 最高。文档演化是潜在依赖。文件系统访问可能因测试文件过度预测降低定位。Codex 5.5 High 在扩展基准上第二高。

   - 为什么值得关注：揭示 Agent 探索结构的结构性不匹配问题

    - 应用场景：数据库自然语言查询、企业数据分析
    - 为什么值得关注：无需模型参数更新即可大幅提升 Text-to-SQL

#### 2. APPO - 代理过程策略优化

**地址**：https://arxiv.org/abs/2606.12384

**项目介绍**：
APPO（Agentic Procedural Policy Optimization）将 Agentic RL 的分支和信用分配从粗粒度交互单元（工具调用边界或固定工作流）推进到序列中的细粒度决策点。作者通过 pilot 分析发现，有影响力的决策点广泛分布于生成序列中而非集中于工具调用处，且 token 熵本身不能可靠反映其对最终结果的影响。APPO 使用 Branching Score 选择分支位置（结合 token 不确定性和策略诱导的后续延续似然增益），并引入过程级优势缩放在分支 rollouts 间更好地分配信用。13 个基准实验显示 APPO 一致改善强 Agentic RL 基线近 4 点，同时保持高效工具调用和行为可解释性。

**技术架构**：
- 细粒度决策点检测器（Branching Score）
- 过程级优势计算器
- 与现有 Agentic RL 框架兼容的插件式架构

**核心算法或系统设计**：
- Branching Score = f(token uncertainty, policy-induced likelihood gain)
- 过滤虚假高熵位置，聚焦真正有影响力的决策点
- 过程级优势缩放替代粗粒度启发式信用分配

**关键实现机制**：
- 训练时动态决策点检测
- 分支 rollouts 的过程级优势计算
- 与 GRPO/PPO 等基线兼容

**与现有技术对比**：
- 传统方法：工具调用边界或固定工作流步作为信用分配单元
- APPO：序列中的细粒度决策点，更精确的信用归因
- 13 基准 +4 点改善显著

**优势与局限**：
- 优势：更精细的信用分配、保持工具调用效率、行为可解释性
- 局限：Branching Score 计算开销、超参数敏感性

**未来发展潜力**：
- 可能成为 Agentic RL 训练的标准组件
- 扩展到更多工具使用场景和多 Agent 协作

**个人技术理解**：
APPO 的核心洞察是 Agent 的"关键时刻"不在工具调用处而在序列中间。这与人类决策的认知科学研究一致——关键决策往往发生在信息整合的过程中而非动作执行的边界。将信用分配推进到细粒度决策点是 Agentic RL 从"能用"到"好用"的重要一步。

#### 1. Tahoe - Text-to-SQL 提示优化系统

**地址**：https://arxiv.org/abs/2606.12387

**论文介绍**：
Tahoe 将提示优化视为动态数据管理问题，而非静态工程任务。系统包含 Development 和 Deployment 两个阶段的错误驱动提示学习管道。编译器反馈蒸馏为可复用的语法提示（dialect-specific 规则），执行和用户反馈转换为语义提示（schema 和用户特定逻辑）。Strategy Layer 将冲突的用户意图建模为共享自然语言触发器下的竞争策略，带有近期信号和学习后归因统计（经验成功、危害、惰性、支持）。推理时检索相关提示引导 LLM 通过逻辑规划然后 SQL 合成。Spider 2.0-Snow GPT-5.5 61.95%→79.42% pass rate，pass-at-4 72.57%→87.61%，Snowflake 语法 100% 通过率，编译器反馈轮次 2.79→0.12。Hint Bank 迁移到弱模型 Doubao-2.0-lite 也有 19.7 个百分点增益。

**技术架构**：
- Hint Bank：结构化的提示知识库
- Strategy Layer：竞争策略管理
- 两阶段管线（Development + Deployment）
- 逻辑规划 → SQL 合成的推理流程

**核心算法或系统设计**：
- 错误驱动提示学习：从编译器/执行/用户反馈中提取结构化提示
- 语法提示蒸馏：编译器反馈 → dialect-specific 规则
- 语义提示转换：执行反馈 → schema/user-specific 逻辑
- 策略竞争模型：冲突意图作为竞争策略，归因统计指导选择

**关键实现机制**：
- Hint Bank 的增量更新和版本管理
- 策略归因统计（成功/危害/惰性/支持）
- 逻辑规划阶段的提示检索和组合

**与现有技术对比**：
- 传统方法：手动工程提示或 SFT 微调
- Tahoe：动态数据管理视角，Hint Bank 作为可演进的知识库
- Spider 2.0-Snow 61.95%→79.42% 提升显著

**优势与局限**：
- 优势：无需模型参数更新、Hint Bank 可迁移、策略归因提供透明度
- 局限：Deployment 阶段人类反馈更新留待未来工作

**未来发展潜力**：
- 可能成为 Text-to-SQL 生产部署的标准方法
- Hint Bank 概念可扩展到更多 NL-to-X 任务

**个人技术理解**：
Tahoe 的核心创新是将提示优化从"工程任务"转变为"数据管理问题"。这一视角转换使得提示可以像数据库一样被查询、更新和审计。Hint Bank 的归因统计机制尤其有价值——它让提示选择从经验法则变为数据驱动决策。

#### 2. LLMKeyLens - LLM API 凭证泄露检测

**地址**：https://arxiv.org/abs/2606.12212

**论文介绍**：
LLMKeyLens 是首个系统研究 iOS 生态中 LLM API key 泄露的工作。构建 444 个 iOS 应用数据集（从 1092 候选过滤），开发动态分析框架通过流量拦截、提供商特定 key 提取、主动有效性确认检测泄露，无需源码访问或二进制解密。282 个应用暴露可利用凭证，10+ 提供商。三种泄露模式：JWT token 泄露 (48%)、无认证后端代理访问 (33%)、明文 API key 传输 (19%)。负责任披露 3 个月后仅 28% 修复，72% 仍可被利用。持久性问题源于无认证后端和损坏的 JWT 实现。

**技术架构**：
- 流量拦截层：捕获 iOS 应用网络通信
- Key 提取层：提供商特定 API key 模式识别
- 有效性确认层：主动验证提取的 key 是否可用
- 无需源码或二进制访问

**核心算法或系统设计**：
- 提供商特定 key 格式解析
- 主动有效性验证（非被动模式匹配）
- 泄露模式分类（JWT/unauthenticated/plaintext）

**关键实现机制**：
- iOS 应用网络流量捕获
- 多提供商 key 格式支持
- 主动 API 调用验证

**与现有技术对比**：
- 传统方法：静态分析或 Android 专属
- LLMKeyLens：iOS 专属动态分析，主动验证
- 444 应用 282 暴露 = 63.5% 泄露率

**优势与局限**：
- 优势：无需源码、主动验证、覆盖 10+ 提供商
- 局限：iOS 生态特定、无法检测服务端泄露

**未来发展潜力**：
- 扩展到 Android 和其他移动平台
- 成为 LLM 集成安全审计的标准工具

**个人技术理解**：
63.5% 的泄露率和 72% 的未修复率揭示了 LLM 集成安全的系统性问题。这不仅是开发者意识问题，更是提供商和平台级安全指南缺失的结果。LLMKeyLens 的工作为移动端 LLM 安全审计提供了首个系统化工具。
