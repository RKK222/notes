### AI Testing / Software Testing

1. **Multi-LCB: 多语言代码生成基准**
   - 地址：https://arxiv.org/abs/2606.20517
   - 来源：arXiv (ICLR 2026)
   - 领域：Testing
   - 技术介绍：LiveCodeBench已成为LLM代码生成评估的事实标准，但仅限于Python。Multi-LCB将LCB任务转换为12种编程语言（包括Python），同时保持污染控制。评估24个LLM暴露了Python过拟合、语言特异性污染和多语言性能的显著差异。与原始LCB完全兼容，自动跟踪未来更新。
   - 应用场景：多语言代码生成评估、LLM编码能力基准测试
   - 为什么值得关注：直击LCB最大局限，暴露当前LLM在多语言编码中的系统性差距

2. **QMFOLBench: 可量化一阶逻辑推理基准**
   - 地址：https://arxiv.org/abs/2606.20227
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：提出自动化框架生成单体一阶逻辑推理任务，复杂度可控。使用合取/析取模式构建形式逻辑结构，精确控制推理深度、宽度、标签类型和干扰项。通过外部证明器进行往返验证确保逻辑一致性。构建2880个实例960种配置的基准。6个推理模型和2个LLM评估显示性能随逻辑复杂度退化。
   - 应用场景：LLM演绎推理能力评估、逻辑复杂度控制基准
   - 为什么值得关注：提供可控复杂度的推理基准，揭示模型对语义变化的敏感性

3. **BIM-Edit: IFC建筑信息模型编辑基准**
   - 地址：https://arxiv.org/abs/2606.20146
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：评估LLM对BIM/IFC格式的自然语言编辑能力。包含324个编辑任务，覆盖11个真实建筑模型和36个合成场景。三个指令类别：直接、空间、拓扑。三个评估维度：几何精度、语义有效性、拓扑一致性。最佳模型平均得分仅49.5%，没有模型完全解决超过3.4%的任务。
   - 应用场景：工程CAD自动化、BIM编辑能力评估
   - 为什么值得关注：揭示LLM在结构化工程设计工作流中的巨大能力差距

4. **LedgerAgent: 结构化状态管理工具调用Agent**
   - 地址：https://arxiv.org/abs/2606.20529
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：针对客服领域策略遵从型工具调用Agent，提出在推理时维护独立账本管理任务状态。解决两类失败模式：（1）基于过时/缺失/错误信息做决策；（2）语法正确但违反策略约束的工具调用。在4个客服领域和多种模型上，平均pass^k优于标准prompt方法。
   - 应用场景：客服Agent、策略驱动型工具调用、合规性Agent
   - 为什么值得关注：直击Agent状态管理隐式化根本缺陷，推理时无需训练

9. **UFP4: 统一4-bit训练配方**
   - 地址：https://arxiv.org/abs/2606.20381
   - 来源：arXiv
   - 领域：AI
   - 技术介绍：识别E2M1非均匀格式的Shrinkage Bias——由可表示箱的几何不对称性引起的系统负舍入误差。该偏差在层间累积并被RHT放大，统一解释了现有E2M1基FP4配方的训练不稳定。提出UFP4配方，对三个训练GEMM应用RHT，仅对dY限制随机舍入。在Dense 1.5B、MoE 7.9B和MoE 124B长期预训练中持续取得更低的BF16相对损失退化。
   - 应用场景：LLM低精度训练、量化训练配方、GPU加速
   - 为什么值得关注：挑战NVIDIA Blackwell/Rubin对E2M1的依赖，建议E1M2/INT4作为一等训练原语

13. **MACR: 知识冲突解决多Agent框架**
    - 地址：https://arxiv.org/abs/2606.20245
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：LLM的参数知识与外部上下文可能冲突，现有方法通常假设模型或上下文可靠，避免冲突而非主动解决。MACR提出显式冲突解决机制，基于多Agent推理。首先通过修改的语义熵度量量化LLM对自身答案的信心，自适应地外化内部知识或检索外部知识。然后引入三个专门Agent：归纳显式规则、分析潜在冲突、解决不一致。
    - 应用场景：RAG系统、知识密集型问答、多源信息融合
    - 为什么值得关注：超越"模型vs上下文"的二元选择，主动解决知识冲突

16. **FlowEdit: 流匹配TTS的终身发音适应**
    - 地址：https://arxiv.org/abs/2606.20518
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：流匹配TTS系统在部署后保持静态，对OOV专有名词的发音错误持续存在。FlowEdit在冻结的流匹配TTS上学习发音校正为潜条件编辑而非权重更新。将校正存储在Modern Hopfield Network中作为内容可寻址情景记忆。推理时通过软注意力和相似度门检索校正。在312个多语言专有名词上，目标词音素错误率降低92.7%，校正约15秒。

17. **Cross-Attention Attribution for Speech Diffusion**

#### 1. codebase-memory-mcp: 高性能代码知识图谱MCP服务器

**项目介绍**（300字）：
codebase-memory-mcp是一个革命性的代码智能MCP（Model Context Protocol）服务器，它将整个代码库索引为持久知识图谱，实现毫秒级查询。支持158种编程语言，查询延迟亚毫秒级，相比传统基于token的代码理解方式减少99%的token消耗。整个系统编译为单个静态二进制文件，零运行时依赖。

该项目的核心创新在于将代码理解从"将代码作为文本发送给LLM"的范式转变为"将代码索引为知识图谱后通过结构化查询获取信息"。传统方法需要将所有相关代码作为上下文发送给LLM，导致token爆炸；而codebase-memory-mcp通过知识图谱实现精确的代码关系查询，只返回相关信息。

- 单二进制分发：C语言实现，零依赖部署

- 增量索引：代码变更时只更新受影响的部分

### 学术论文

#### 1. UFP4: 统一4-bit训练配方

**论文**：https://arxiv.org/abs/2606.20381
**标题**：Rethinking Shrinkage Bias in LLM FP4 Pretraining: Geometric Origin, Systemic Impact, and UFP4 Recipe

**论文介绍**（350字）：
FP4训练承诺大幅降低LLM预训练的内存和计算成本，但当前的FP4硬件路径（包括NVIDIA Blackwell/Rubin和AMD MI350系列GPU）都以E2M1数据元素为中心。本文识别了E2M1的根本局限：非均匀格式如E2M1天生 suffers from Shrinkage Bias——由其可表示箱的几何不对称性引起的系统负舍入误差。

作者证明这种偏差在层间累积并被随机Hadamard变换（RHT）放大，为现有E2M1基FP4配方的训练不稳定提供了统一解释。相比之下，均匀网格（E1M2/INT4）绕过这一网格几何错误，更好地将RHT改进的桶利用率转化为更高的量化质量。

基于这一发现，作者提出UFP4——一种均匀4-bit训练配方，对三个训练GEMM应用RHT，仅对dY限制随机舍入。在Dense 1.5B、MoE 7.9B和MoE 124B长期预训练中，UFP4持续取得比强E2M1基线更低的BF16相对损失退化。

**技术架构**：
- Shrinkage Bias理论分析：几何不对称性→负舍入误差→层间累积→RHT放大
- 均匀网格（E1M2/INT4）vs 非均匀网格（E2M1）对比
- UFP4配方：RHT应用到三个GEMM + dY随机舍入

**核心算法**：
- 几何误差分析：证明E2M1的bin不对称性导致系统性偏差
- 统一RHT应用：三个GEMM都使用RHT而非仅部分
- 选择性随机舍入：仅对dY使用随机舍入

**与现有技术对比**：
- NVIDIA E2M1 FP4：训练不稳定，损失退化大
- UFP4 E1M2/INT4：训练稳定，损失退化小

**优势与局限**：
- 优势：理论解释清晰、实验验证充分（124B MoE）、scaling law支持
- 局限：需要硬件支持E1M2/INT4格式，当前GPU主要支持E2M1

**发展潜力**：
本文对FP4训练的理解具有深远影响。如果未来加速器支持E1M2/INT4作为一等训练原语，将显著降低LLM训练成本。建议NVIDIA/AMD重新考虑FP4格式设计。

#### 2. Multi-LCB: 多语言代码生成基准

**论文**：https://arxiv.org/abs/2606.20517
**标题**：Extending LiveCodeBench to Multiple Programming Languages

**论文介绍**（350字）：
LiveCodeBench（LCB）已成为LLM代码生成评估的事实标准，通过竞赛编程题目、持续新增题目和按发布日期过滤提供污染感知评估。然而，LCB仅限于Python，留下了LLM是否能泛化到现实软件工程所需的多种编程语言的问题。

本文引入Multi-LCB，一个跨12种编程语言（包括Python）评估LLM的基准。Multi-LCB将LCB数据集中的Python任务转换为其他语言中的等效任务，同时保持LCB的污染控制和评估协议。由于与原始LCB格式完全兼容，Multi-LCB将自动跟踪未来LCB更新。

评估24个LLM暴露了：（1）Python过拟合——模型在Python上表现显著优于其他语言；（2）语言特异性污染——某些语言的数据可能在训练中被污染；（3）多语言性能的显著差异——模型在不同语言间的表现差距巨大。

**技术架构**：
- 任务转换：Python→11种其他语言的自动化转换
- 污染控制：继承LCB的日期过滤和污染检测
- 自动跟踪：自动适配未来LCB更新

**核心发现**：
- Python过拟合是当前LLM的普遍问题
- 语言特异性污染影响评估公平性
- 多语言编码能力是LLM的真实瓶颈

**优势与局限**：
- 优势：与LCB完全兼容、自动跟踪更新、24模型全面评估
- 局限：任务转换可能引入语义偏差，某些语言特性无法完全映射

**发展潜力**：

#### 3. Thermodynamic Measure of Intelligence

**理论框架**：

**核心贡献**：
- 智能的通用可测量定义
- 从被动物质到LLM到人类的统一尺度

2. **多语言/多模态评估成为新焦点**：Multi-LCB暴露LLM在Python之外的编码能力差距；BIM-Edit揭示LLM在结构化工程设计中的巨大能力鸿沟。评估基准正从单一语言/模态向多语言/多模态扩展。

3. **形式化验证与RL融合**：Process-Verified RL via Lean将证明助手从评估验证器升级为训练过程级奖励oracle；Talos将Lean形式化验证扩展到Wasm模块。形式化方法正从"事后验证"走向"训练内嵌"。

4. **均匀量化挑战非均匀主导**：UFP4揭示E2M1非均匀格式的Shrinkage Bias，建议E1M2/INT4均匀网格作为一等训练原语。这可能影响NVIDIA/AMD下一代GPU的量化格式设计。

5. **Agent编排从学术走向企业规模**：Enterprise Multi-Agent Orchestration首次系统评估200 Agent规模下的性能退化，发现"规模而非复杂度"主导编排性能。企业级Agent系统进入工程化阶段。

## 新领域收获

1. **热力学智能理论**：将智能定义为"合法放大罕见但有效未来"，递归自我模拟是智能的必要条件。这一框架连接信息论、热力学和智能理论，可能成为AGI理论基础。

2. **超图语义通信**：HISR利用超图表示语义知识实体间的高阶多实体关系，在6G语义通信中提升36.6%精度。将AI图推理扩展到通信系统。

3. **流式生成+进化优化**：Residual-Space Evolutionary Optimization将进化算法引入流式生成模型，分离自花授粉（局部开发）和异花授粉（广泛探索），扩展到晶体数据等科学域。
