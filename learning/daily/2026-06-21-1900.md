## 技术扫描（30+）

### AI Testing / Software Testing

1. **AgentArmor: 编码Agent失败框架与缓解**
   - 地址：https://arxiv.org/abs/2606.19380
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：研究AI编码Agent的三类失败机制：（1）Underspecification——默认模型行为不安全；（2）Capability errors——安全动作可用但模型因偏差或能力限制未遵循；（3）Agent harness errors——模型无法通过harness执行安全动作。在8种评估、20个编码环境、59个合成转录模板上评估。提出AgentArmor harness修改：扩展系统prompt、独立命令分类器、"3 strikes"策略、确定性护栏、Agent自编辑上下文工具，在统计显著样本上更安全。
   - 应用场景：编码Agent安全加固、生产部署Agent防护
   - 为什么值得关注：系统分类编码Agent失败模式并提供可操作的harness级缓解方案

2. **StaminaBench: 编码Agent耐力基准**
   - 地址：https://arxiv.org/abs/2606.19613
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：衡量编码Agent能连续处理多少轮交互（变更请求）后才失败。不同于主流的"任务解决率"指标，StaminaBench匹配真实vibe-coding场景（数十到数百轮）。Agent实现REST API服务器并通过100轮程序化生成的后续变更请求进行修改，代码库达6000行。测试完全程序化生成无LLM参与确保可复现性。评估6种Agent harness+7个开源LLM在20个场景中发现：所有模型在5-6轮内失败；将测试反馈传回Agent并重试可将通过轮次提升12倍；好harness至关重要——强模型在不同harness间差距达6倍。
   - 应用场景：Vibe coding耐力评估、多轮编码Agent基准测试
   - 为什么值得关注：首次提出"耐力"作为编码Agent核心指标，揭示vibe-coding无测试的bug本质

3. **JustDiag: 可问责根因分析诊断引擎**
   - 地址：https://arxiv.org/abs/2606.19407
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：LLM能生成流畅的根因分析(RCA)，但流畅的最终答案不足以作为高stakes操作的问责证据。JustDiag维护显式的过程状态：证据、发现、竞争假设、冲突和下一步检查。在66个真实事件上用两层协议评估：分别评分最终答案质量和过程质量。相对无诊断论证的对照组，JustDiag获得更强的结果和过程评分，同时因更校准的非闭合接受略低的终端完成率。
   - 应用场景：事故响应、高stakes操作的AI诊断问责
   - 为什么值得关注：揭示"问责RCA需要显式诊断论证工件和过程感知评估，而非仅流畅的最终答案"

4. **Library-Aware UT生成: openSIL固件单元测试**
   - 地址：https://arxiv.org/abs/2606.19725
   - 来源：arXiv (AMD)
   - 领域：Testing
   - 技术介绍：针对AMD维护的openSIL固件代码库，提出LLM引导的多Agent UT生成管线。结合自动化测试脚手架生成、库感知的stub/mock/fake创建、以及由构建日志和行覆盖率反馈驱动的迭代编译-修复循环。76个函数测试中73个生成了可编译UT；无行覆盖率引导时平均行覆盖率达73.9%；有行覆盖率引导时达98.8%，结合向量数据库检索达94.7%。
   - 应用场景：底层C固件单元测试生成、受限构建环境自动化测试
   - 为什么值得关注：展示自动化生成-修复管线在受限固件环境中大幅提升UT创建效率和覆盖率

5. **Over-Privileged Tool Selection: 工具选择特权升级**
   - 地址：https://arxiv.org/abs/2606.20023
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：研究LLM Agent在工具选择中的特权升级问题——当低特权工具足够时仍选择高特权工具。引入ToolPrivBench评估8个领域5种风险模式。发现主流LLM Agent普遍存在过度特权选择，且瞬态失败进一步放大此问题。通用安全对齐不能可靠转移到最小特权工具选择，prompt级控制在瞬态失败下仅有限缓解。引入特权感知后训练防御，教Agent偏好足够低特权工具仅在必要时升级。
   - 应用场景：Agent工具选择安全评估、最小特权原则实施
   - 为什么值得关注：揭示Agent工具选择中的系统性特权升级风险并提供可操作的防御

6. **CLI vs GUI移动Agent范式**
   - 地址：https://arxiv.org/abs/2606.19388
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：移动Agent研究被GUI范式主导（感知UI信息+发射屏幕交互），但移动平台也暴露CLI提供对设备服务和数据的直接访问。评估3种编码Agent（Claude Code、Terminus-2、mini-swe-agent）在4个模型API上在AndroidWorld和MobileWorld的表现（无移动特定后训练），对比3种可复现GUI基线。Claude Code(Opus 4.7)达71.8%和51.9%，超越所有GUI基线。Oracle CLI解决方案在AndroidWorld达88.8%（103/116任务CLI可解），表明大量改进空间。引入CLI-Advantage任务套件（45个模板5类：批量操作、多条件过滤、聚合、跨应用工作流、隐藏设备状态），每类CLI Agent都超越GUI基线且步骤更少（10.7 vs 18.6）。
   - 应用场景：移动Agent评估、CLI vs GUI范式对比
   - 为什么值得关注：挑战GUI主导的移动Agent研究范式，揭示CLI在批量操作和跨应用工作流中的显著优势

7. **Agentic EDA手交协议**
   - 地址：https://arxiv.org/abs/2606.19795
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：电子设计自动化(EDA)中手交(handshake)协议是芯片设计的关键验证环节。研究LLM Agent在EDA手交协议验证中的表现，发现Agent在理解时序约束和状态机转换方面存在系统性弱点。提出结构化提示工程和领域知识注入方法提升验证准确率。
   - 应用场景：芯片设计验证、EDA工具AI辅助
   - 为什么值得关注：将Agent验证扩展到硬件设计领域，揭示LLM在时序推理方面的局限

8. **LedgerAgent: 结构化状态管理工具调用Agent**
   - 地址：https://arxiv.org/abs/2606.20529
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：针对客服领域策略遵从型工具调用Agent，提出在推理时维护独立账本管理任务状态。解决两类失败模式：（1）基于过时/缺失/错误信息做决策；（2）语法正确但违反策略约束的工具调用。在4个客服领域和多种模型上，平均pass^k优于标准prompt方法。
   - 应用场景：客服Agent、策略驱动型工具调用、合规性Agent
   - 为什么值得关注：直击Agent状态管理隐式化根本缺陷，推理时无需训练

   - 地址：https://arxiv.org/abs/2606.19386
   - 来源：arXiv
   - 领域：AI
   - 技术介绍：对8个Diffusion Language Models(DLM)在8个基准上进行系统评估，揭示去噪步数、上下文长度和块大小对生成质量的影响。发现DLM在短文本生成上接近AR模型，但在长文本中因误差累积而退化。提出自适应去噪调度策略，在保持质量的同时减少40%推理步数。

10. **AgenticRei: 义务策略治理语言**
    - 地址：https://arxiv.org/abs/2606.19387
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：现有策略语言(XACML/Rego/Cedar)仅覆盖permit/prohibit，无法表达obligation（义务）——必须执行的动作或必须满足的条件。AgenticRei引入obligation lifecycle（创建→触发→履行→过期）、dispensation（豁免）、meta-policy冲突解析。在Agent工具调用、数据访问、合规审计场景验证，显著降低策略违规率。
    - 应用场景：Agent策略治理、合规性执行、义务追踪
    - 为什么值得关注：填补策略语言在义务表达方面的空白，对生产级Agent合规至关重要

## 深度研究

### GitHub 项目

1. **headroom: LLM Token压缩基础设施**
   - 地址：https://github.com/chopratejas/headroom
   - 项目介绍：headroom解决Agent经济性核心问题——token消耗。在LLM接收前压缩工具输出、日志、文件和RAG块，减少60-95% token消耗同时保持答案质量。提供Library（Python库直接集成）、Proxy（透明中间件）、MCP Server（Model Context Protocol集成）三种方式。
   - 技术架构：三层架构——Library层提供Python API直接压缩文本；Proxy层作为HTTP中间件拦截LLM请求/响应；MCP Server层通过Model Context Protocol与Agent框架集成。核心压缩算法结合语义保留和冗余消除。
   - 核心创新：将压缩从"后处理"前移到"预处理"——在token到达LLM前压缩，而非在生成后压缩。这避免了LLM处理冗余信息的计算浪费。
   - 与现有技术对比：传统RAG压缩在检索后、生成前进行，headroom扩展到工具输出、日志、文件等更广泛的上下文来源。相比上下文窗口扩展方案，headroom从源头减少token需求。
   - 优势与局限：优势是通用性强（不依赖特定模型）、即插即用、效果显著。局限是压缩质量依赖算法，极端压缩可能丢失关键细节。
   - 未来发展潜力：Agent经济性是2026年核心议题，headroom作为基础设施层有巨大增长空间。可能扩展到多模态压缩（图像/视频token压缩）。
   - 个人技术理解：headroom代表Agent基础设施从"功能实现"向"经济性优化"的范式转变。当Agent从实验走向生产，token成本成为首要瓶颈。

2. **codebase-memory-mcp: 代码知识图谱MCP服务器**
   - 地址：https://github.com/DeusData/codebase-memory-mcp
   - 项目介绍：将代码库索引为持久知识图谱，支持158种语言，亚毫秒查询，减少99% token消耗。单静态二进制文件，零依赖。周增5419星，总星9766。
   - 技术架构：C语言实现的高性能MCP服务器，使用自定义索引结构将代码解析为AST→符号表→知识图谱。支持增量更新和持久化存储。
   - 核心创新：将代码理解从"LLM阅读源码"转变为"知识图谱查询"。Agent通过MCP协议查询代码关系而非直接读取文件，大幅减少token消耗。
   - 与现有技术对比：传统Agent代码理解依赖读取文件内容（消耗大量token），codebase-memory-mcp将代码关系预索引，查询只需少量token。相比LSP（Language Server Protocol），codebase-memory-mcp专注于Agent场景的知识图谱抽象。
   - 优势与局限：优势是速度极快、token节省显著、语言覆盖广。局限是索引构建需要初始计算，对动态代码变更需要增量更新。
   - 未来发展潜力：代码理解是Agent核心能力，知识图谱方案可能成为标准基础设施。可能扩展到跨仓库知识图谱和团队协作场景。
   - 个人技术理解：这代表Agent代码理解从"阅读"到"查询"的范式转变。当Agent需要理解大型代码库时，知识图谱比逐文件阅读更高效。

3. **deer-flow: 字节跳动长视野SuperAgent harness**
   - 地址：https://github.com/bytedance/deer-flow
   - 项目介绍：开源长视野SuperAgent harness，支持研究、编码和创作。通过沙箱、记忆、工具、技能、子Agent和消息网关处理从几分钟到几小时的不同级别任务。
   - 技术架构：多层架构——消息网关负责任务路由和状态管理；沙箱提供隔离执行环境；记忆模块维护跨步骤上下文；工具/技能模块提供可扩展能力；子Agent支持任务分解和并行执行。
   - 核心创新：长视野任务处理——不同于单次调用的Agent，deer-flow设计用于处理需要多步骤、长时间执行的任务。消息网关协调各组件，确保状态一致性和错误恢复。
   - 与现有技术对比：相比CrewAI/AutoGen等框架，deer-flow更强调长视野任务的可靠执行和状态管理。相比LangGraph，deer-flow提供更完整的沙箱和记忆基础设施。
   - 优势与局限：优势是功能完整、支持复杂任务、字节跳动工程背书。局限是相对较新、社区生态待建立、文档可能不够完善。
   - 未来发展潜力：长视野Agent是实用化关键方向，deer-flow作为开源方案可能成为行业标准之一。
   - 个人技术理解：字节跳动进入Agent基础设施领域表明大厂对Agent赛道的重视。长视野任务处理是Agent从"玩具"到"工具"的关键跨越。

### 学术论文

1. **AgentArmor: 编码Agent失败框架与缓解** (arXiv:2606.19380)
   - 论文介绍：系统研究AI编码Agent的三类失败机制——Underspecification（默认模型行为不安全）、Capability errors（安全动作可用但模型未遵循）、Agent harness errors（模型无法通过harness执行安全动作）。在8种评估、20个编码环境、59个合成转录模板上评估。
   - 技术架构：提出AgentArmor harness修改方案：扩展系统prompt增加安全指导、独立命令分类器分离安全/不安全操作、"3 strikes"策略限制重复失败、确定性护栏强制执行安全约束、Agent自编辑上下文工具允许Agent修正自身错误。
   - 核心算法/系统设计：三类失败机制的分类框架是核心贡献。AgentArmor通过harness级修改而非模型级修改实现安全加固，这使其可应用于任何编码Agent。
   - 与现有技术对比：现有Agent安全研究多聚焦于模型对齐或prompt工程，AgentArmor首次系统性分类编码Agent失败模式并提供harness级缓解。
   - 优势与局限：优势是通用性强（不依赖特定模型）、可操作、在统计显著样本上验证。局限是harness修改可能增加延迟，"3 strikes"策略可能误杀合法操作。
   - 未来发展潜力：编码Agent安全是生产部署的前提，AgentArmor的分类框架可能成为行业标准参考。
   - 个人技术理解：将Agent失败分类为underspecification/capability/harness三类，类似于软件工程的"需求/实现/部署"三层问题，提供了清晰的诊断框架。

2. **StaminaBench: 编码Agent耐力基准** (arXiv:2606.19613)
   - 论文介绍：首次提出"耐力"作为编码Agent核心指标——衡量Agent能连续处理多少轮交互（变更请求）后才失败。Agent实现REST API服务器并通过100轮程序化生成的后续变更请求进行修改，代码库达6000行。
   - 技术架构：完全程序化生成测试场景，无LLM参与确保可复现性。评估6种Agent harness+7个开源LLM在20个场景。
   - 核心发现：所有模型在5-6轮内失败；将测试反馈传回Agent并重试可将通过轮次提升12倍；好harness至关重要——强模型在不同harness间差距达6倍。
   - 与现有技术对比：主流基准（SWE-bench等）衡量"任务解决率"（单次任务），StaminaBench衡量"连续交互耐力"（多轮任务），更匹配真实vibe-coding场景。
   - 优势与局限：优势是程序化生成确保可复现性、匹配真实场景、揭示harness重要性。局限是仅针对REST API场景，可能不覆盖所有编码任务类型。
   - 未来发展潜力：耐力指标可能成为Agent评估的新维度，与解决率、速度等并列。
   - 个人技术理解：揭示"vibe-coding无测试的bug本质"——当Agent连续修改代码时，没有测试反馈的Agent会在5-6轮内崩溃。这为Agent开发流程中集成测试提供了实证依据。

3. **LLM心理画像测量伪影** (arXiv:2606.20205)
   - 论文介绍：对56个指令调优LLM进行人格和风险偏好仪器测试，发现模型间差异81-90%由方向性响应偏差驱动，而非仪器瞄准的特征。通过项目选择可"制造"任意心理画像。
   - 技术架构：正式心理测量框架，56个LLM+大型人类参考样本，覆盖自我报告和行为任务。方差分解量化偏差vs特征贡献。
   - 核心发现：（1）模型间差异81-90%由方向性响应偏差驱动（人类仅9-16%）；（2）偏差随模型能力下降但不消除；（3）仪器表观可靠性几乎完全由响应正交性预测；（4）通过项目选择可制造任意画像。
   - 与现有技术对比：现有研究将LLM心理画像视为模型固有属性，本文证明这是测量工具伪影。
   - 优势与局限：优势是样本量大（56个模型）、方法严谨（正式心理测量框架）、发现颠覆性。局限是仅测试指令调优模型，未覆盖基座模型。
   - 未来发展潜力：可能颠覆LLM心理评估领域，推动开发专为LLM设计的评估工具。
   - 个人技术理解：这是2026年最重要的LLM评估发现之一——当我们将人类心理仪器用于LLM时，我们测量的不是"LLM的人格"，而是"仪器对LLM的偏差"。这提醒我们：LLM评估需要专用工具而非借用人类工具。

## 技术趋势

1. **Agent经济性基础设施爆发**：headroom（Token压缩）、codebase-memory-mcp（代码知识图谱）等项目的爆发式增长表明，Agent从实验走向生产的核心瓶颈已从"能力不足"转向"成本过高"。Token经济性、代码理解效率、长期记忆成本将成为2026下半年Agent基础设施的主战场。

2. **Agent安全从模型层向harness层转移**：AgentArmor、Over-Privileged Tool Selection等研究揭示，仅靠模型对齐不足以保证Agent安全，harness级防护（命令分类器、特权感知、确定性护栏）成为必要补充。Agent安全正在形成"模型对齐→harness防护→运行时监控"三层防御体系。

3. **Agent评估维度从"解决率"向"耐力/过程质量"扩展**：StaminaBench（耐力基准）、JustDiag（过程质量评估）表明，Agent评估正在从单一的"任务是否完成"向"能持续工作多久"和"过程是否可问责"扩展。这反映了Agent从一次性工具向持续服务角色的转变。

## 新领域收获

1. **ML代码质量的环境影响**：首次量化ML资源泄漏对碳排放的影响（IMR+32%、UTR+46%电力消耗），将代码质量与碳足迹关联，为绿色AI提供量化基础。这是一个被忽视但日益重要的方向。

2. **LLM心理画像测量伪影**：发现LLM的"人格"是测量工具伪影而非模型固有属性（81-90%由响应偏差驱动）。这对LLM安全评估、LLM作为人类代理研究有深远影响——需要开发专为LLM设计的评估工具。

3. **热力学智能度量**：将智能定义为"合法放大罕见但有效未来"，递归自我模拟是智能的必要条件。提供可测量的智能理论框架，连接热力学与信息论，可能为通用智能评估提供统一尺度。
