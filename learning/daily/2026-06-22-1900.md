### AI Testing / Software Testing (10)

1. **Multi-LCB: 扩展LiveCodeBench到多编程语言**
   - 地址：https://arxiv.org/abs/2606.20517
   - 来源：arXiv (ICLR 2026)
   - 领域：Testing
   - 技术介绍：LiveCodeBench已成为评估LLM代码生成能力的广泛采用的基准，但仅限于Python。Multi-LCB将LCB任务转换为12种等价的编程语言任务，同时保持污染控制评估协议。评估24个LLM在指令和推理方面的多语言代码生成能力，发现Python过拟合、语言特定污染和多语言性能差异的证据。Multi-LCB与原始LCB格式完全兼容，将自动跟踪未来LCB更新。
   - 应用场景：多语言代码生成能力评估、LLM泛化能力测试
   - 为什么值得关注：直接解决LCB的主要局限性，暴露当前LLM能力的关键差距

2. **CWE-Trace: 系统软件漏洞检测微调LLM的局限性诊断**
   - 地址：https://arxiv.org/abs/2606.20502
   - 来源：arXiv (cs.CR/cs.SE)
   - 领域：Testing
   - 技术介绍：构建834个手动策划的Linux内核样本，覆盖74个CWE，强制执行严格的时间分割（pre-2025历史集/post-cutoff无泄漏集）。评估8个原始LLM和15个LoRA微调变体。发现：数据污染无可测量优势（84%的污染样本无可用记忆信号）；骨干方向性先验主导微调（DFI从-85.5到+94.8pp）；微调改变输出阈值但不改变决策策略——"校准而非理解"。最佳检测分数仅达52.1%（比随机高2.1pp），精确CWE排名Top-1准确率低于1.3%。
   - 应用场景：LLM漏洞检测能力评估、安全推理能力诊断
   - 为什么值得关注：揭示当前LLM缺乏可靠的系统软件安全推理能力，无论微调策略如何

3. **ToolPro: 工具程序作为灵活Agentic Web服务接口**
   - 地址：https://arxiv.org/abs/2606.19992
   - 来源：arXiv (ICML 2026)
   - 领域：Testing
   - 技术介绍：在Agentic Web时代，LLM Agent越来越多地调用Web服务作为工具，但大多数接口仍是静态端点，无法表达长程工作流（循环、条件、连接、重试）。ToolPro将Agent的工具意图表示为可执行工具程序，紧凑编码多步服务交互。结合约束引导的程序构造、效应感知重放（用于恰好一次状态修改调用）和配置驱动策略。在MCP风格服务上实例化，带WebAssembly沙箱。端到端延迟降低高达53.4%，客户端流量降低高达96.1%。
   - 应用场景：Agentic Web服务接口优化、MCP协议增强
   - 为什么值得关注：将静态端点升级为可执行程序，显著提升Agent调用Web服务的效率

4. **SolidityBench: 仓库级Solidity代码生成基准**
   - 地址：https://arxiv.org/abs/2606.19988
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：引入SolidityBench基准，包含5,470个仓库级Solidity智能合约配对自然语言描述。提出SolidityScore——一个Solidity感知语义指标，强调领域关键构造（安全修饰符、合约声明、Solidity特定关键字）。评估Qwen2.5-Coder、DeepSeek-Coder、CodeLlama等在零样本提示、思维链推理、上下文学习、RAG和监督微调下的表现。发现通用模型在仓库级Solidity生成中表现出系统性结构缺陷，SFT通过内部化Solidity特定约束实现最大改进。
   - 应用场景：智能合约代码生成评估、Web3安全
   - 为什么值得关注：首个仓库级Solidity代码生成基准，揭示SFT+高质量领域数据是最有效策略

5. **JamBench: 专业游戏引擎项目级代码框架基准**
   - 地址：https://arxiv.org/abs/2606.19830
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：基于Godot引擎的文本格式和无头执行模式，从240,000+仓库中蒸馏出8,133个验证项目。300个手动验证项目形成JamBench。定义主题生成和代码补全任务，通过编译通过率、结构完整性分数(SCS)和行为对齐分数(BAS)评估。9个前沿模型评估显示能力悬崖：小型项目运行时通过率80.4%，大型项目暴跌至5.7%。代码Agent改善编译率但在运行时行为质量上无增益——瓶颈在架构设计而非语法正确性。
   - 应用场景：游戏开发AI代码生成评估、项目级代码工程基准
   - 为什么值得关注：首个专业游戏引擎项目级基准，揭示架构设计是AI代码生成的真正瓶颈

6. **CoRaCommit: 带示例检索的VS Code提交消息生成扩展**

7. **Probe-and-Refine: 编码Agent仓库引导调优**
   - 地址：https://arxiv.org/abs/2606.20512
   - 来源：arXiv
   - 领域：Testing
   - 技术介绍：LLM编码Agent需要关于仓库的高层操作知识（哪些文件包含哪些子系统、如何运行测试套件、哪些工作流历史上导致错误修复）。引入probe-and-refine调优：使用合成bug修复探针迭代诊断和修补仓库引导文件，通过单次LLM调用，调优期间无Agent循环或工具使用。在SWE-bench Verified上，Qwen3.5-35B-A3B达33.0%平均解决率vs 28.3%静态知识库vs 25.5%无引导基线。改进来自覆盖率而非精度。
   - 应用场景：编码Agent仓库级引导优化、SWE-bench性能提升
   - 为什么值得关注：证明引导生成方式是关键变量，合成探针调优显著提升编码Agent性能

8. **Qiskit代码迁移: LLM+RAG自动化量子代码版本迁移**
   - 地址：https://arxiv.org/abs/2606.20173
   - 来源：arXiv

   - 应用场景：量子软件代码迁移、API过时缓解
   - 为什么值得关注：展示数据驱动方法在量子软件工程中的潜力，缓解API过时问题

9. **RL环境家族模型驱动开发: 混合遗传算法生成RL训练环境**
   - 地址：https://arxiv.org/abs/2606.20324
   - 来源：arXiv
   - 领域：Testing

   - 应用场景：RL训练环境自动化生成、课程学习

### AI / LLM 技术 (12)

11. **安全对齐LLM从混合合规演示中学到什么**
    - 地址：https://arxiv.org/abs/2606.20508
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：先前工作表明上下文演示可以jailbreak语言模型，但模型如何解释不同类型的合规演示仍不清楚。研究混合良性合规演示（非有害请求+有帮助响应）与有害合规演示（有害请求+有帮助响应），测试三种假设。发现良性和有害演示不可互换：良性演示可减少或增加有害合规取决于模型。偏好优化是防止良性演示增加有害合规的关键训练阶段。演示排序表现出强烈的近因偏差。模型在拒绝时与上下文学习的交互方式不同：有些采用演示格式即使拒绝，有些在拒绝时覆盖所有上下文信号。
    - 应用场景：LLM安全对齐、Jailbreak防御、上下文学习分析
    - 为什么值得关注：超越展示演示jailbreak有效，表征它是如何工作的

12. **企业级事件驱动多Agent编排**
    - 地址：https://arxiv.org/abs/2606.20058
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：企业AI旨在向持续事件监控、检测和跨专业Agent行动发展，但现有多Agent系统主要假设离散请求-响应工作流。评估DAG Plan and Execute和ReAct在208个生产派生企业场景上的表现，涵盖Persona(<10 Agent)、Department(20-80)和Enterprise(200)规模。引入Task Manager通过优先级推断、相关事件合并和抢占实现持续操作。发现规模而非任务复杂度主导编排性能：两种架构在小规模表现良好但在企业规模退化，Agent发现噪声成为主要瓶颈。Task Manager将高优先级队列延迟降低14-75%。
    - 应用场景：企业级多Agent系统、事件驱动编排
    - 为什么值得关注：首个企业规模多Agent编排实证研究，揭示Agent发现噪声是关键瓶颈

13. **UFP4: 重新思考LLM FP4预训练中的收缩偏差**
    - 地址：https://arxiv.org/abs/2606.20381
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：FP4训练承诺大幅降低LLM预训练的内存和计算成本，但当前FP4硬件路径（NVIDIA Blackwell/Rubin、AMD MI350）仍以E2M1数据元素为中心。识别E2M1的根本限制：非均匀格式固有遭受Shrinkage Bias——由可表示bin的几何不对称引起的系统负舍入误差。该偏差跨层累积并被随机Hadamard变换(RHT)放大。均匀网格(E1M2/INT4)绕过此网格几何误差。提出UFP4统一4位训练配方，将RHT应用于所有三个训练GEMM，同时将随机舍入限制在dY。在Dense 1.5B、MoE 7.9B和MoE 124B长程预训练上持续取得更低的BF16相对损失退化。
    - 应用场景：FP4训练、LLM量化、GPU加速
    - 为什么值得关注：揭示E2M1格式的几何根本限制，为未来加速器设计提供方向

14. **SoftSkill: 上下文适应的行为压缩**
    - 地址：https://arxiv.org/abs/2606.20333
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：Agent技能通常以自然语言Markdown文件部署，编码答案策略、证据使用习惯和任务程序。这些文件可读且可移植，但被间接消费：对每个任务实例，冻结语言模型必须将长文本工件转换为生成时行为。提出SoftSkill——冻结骨干方法，将自然语言技能初始化为紧凑连续上下文对象，通过可训练软delta精炼，同时基础模型保持冻结。在Qwen3.5-4B上，长度32的SoftSkill前缀比无技能提示在SearchQA上提升8.3分、LiveMath上提升42.1分、DocVQA上提升1.3分。相对于SkillOpt在SearchQA上提升5.2分、LiveMath上提升12.5分，同时将数百到数千Markdown技能token替换为几个虚拟token。
    - 应用场景：Agent技能压缩、上下文效率优化
    - 为什么值得关注：证明某些任务技能更好地被视为紧凑潜在控制而非额外Markdown

15. **AutoPass: 证据引导LLM Agent用于编译器性能调优**
    - 地址：https://arxiv.org/abs/2606.20373
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：LLM在代码编译任务中展现前景，但将其应用于运行时性能调优因复杂的微架构效应和嘈杂的运行时测量而困难。提出AutoPass——编译器性能调优的多Agent框架，使用编译器和运行时证据引导LLM生成的优化决策。不同于将编译器视为黑盒的先前自动调优方案，AutoPass向LLM打开编译器，使其能查询编译器内部优化状态并分析中间表示来编排编译器选项。搜索过程迭代精炼优化配置，使用测量的运行时反馈诊断回归并引导延迟改进编辑。在LLVM编译器上实现，x86-64上1.043x、ARM64上1.117x超越LLVM -O3。
    - 应用场景：编译器自动调优、LLM系统优化
    - 为什么值得关注：打开编译器内部而非黑盒调优，在真实硬件上超越专家调优启发式

16. **Multi-Agent审议中的隐藏锚点**
    - 地址：https://arxiv.org/abs/2606.19494
    - 来源：arXiv
    - 领域：AI
    - 技术介绍：多Agent LLM审议（Agent在多轮中交换和修订答案）越来越多地用于改进推理和准确性，但其工作原理很少被建模。将多Agent审议建模为封闭回路动力系统，每个Agent携带隐藏的内在信念（锚点），持续拉回其意见无论邻居如何。证明锚点可从审议本身恢复，并解释经典共识规则禁止的行为：Agent对正确答案的信心可攀升超过任何Agent的起点，逃逸初始信念形成的空间（凸包）。恢复的锚点也预测保留运行（泛化），提供模型是否真正受锚点驱动的简单测试。在三个开放权重模型家族中这是频谱而非全有或全无。
    - 应用场景：多Agent系统理论建模、审议动力学分析
    - 为什么值得关注：首个封闭回路动力学模型解释多Agent审议为何有效

17. **DeXposure-Claw: DeFi风险监督的Agentic系统**
    - 地址：https://arxiv.org/abs/2606.19501

   - 核心创新：将长程工作流（循环、条件、连接、重试）编码为紧凑可执行程序
   - 与现有技术对比：静态端点调用 → 可执行程序批量执行；逐步骤调用 → 端到端优化
   - 优势与局限：优势是延迟降低53.4%和流量降低96.1%；局限是需要服务方支持程序执行
   - 未来发展潜力：可能成为Agentic Web的新接口标准

### 学术论文

1. **安全对齐LLM从混合合规演示中学到什么**
   - 地址：https://arxiv.org/abs/2606.20508
   - 论文介绍：研究上下文演示如何影响安全对齐LLM的有害合规行为。混合良性合规演示（非有害请求+有帮助响应）与有害合规演示（有害请求+有帮助响应），测试三种假设。在四个模型上实验发现良性和有害演示不可互换，偏好优化是关键训练阶段，演示排序有强烈近因偏差。
   - 技术架构：演示混合实验设计 → 四模型对比 → 偏好优化阶段分析 → 排序效应测试
   - 核心算法：混合演示组合 + 多模型对比评估
   - 关键实现机制：通过控制演示内容和顺序，测量对有害合规的影响
   - 与现有技术对比：先前仅展示jailbreak有效 → 本文表征jailbreak如何工作
   - 优势与局限：优势是系统化的机制分析；局限是仅测试四个模型
   - 未来发展潜力：为jailbreak防御提供理论基础

2. **热力学智能度量**
   - 地址：https://arxiv.org/abs/2606.20231
   - 论文介绍：提出智能可定义为"罕见但有效未来的合法放大"。智能系统必须建模世界及其自身位置，导致递归自模拟。核心结果给出必要性陈述和条件近充分性陈述，连接递归自模拟架构到精确的热力学度量。
   - 技术架构：世界建模 → 递归自模拟 → 热力学度量 → 通用智能尺度
   - 核心算法：罕见有效提升度量 + 递归自模拟保真度
   - 关键实现机制：从热力学第一原理推导智能的数学定义
   - 与现有技术对比：传统基准测试 → 热力学第一原理定义
   - 优势与局限：优势是理论优雅和通用性；局限是实际测量可能复杂
   - 未来发展潜力：可能成为智能理论的基础框架

3. **Process-Verified RL用于定理证明(Lean)**
   - 地址：https://arxiv.org/abs/2606.20068
   - 论文介绍：证明Lean证明助手可作为符号过程预言机，在训练期间提供结果级和细粒度tactic级验证反馈。证明尝试被解析为tactic序列，Lean的elaboration标记局部健全步骤和最早失败步骤。将这些结构化奖励纳入GRPO风格RL目标。
   - 技术架构：证明解析 → Lean elaboration → tactic级奖励 → GRPO训练
   - 核心算法：第一错误传播 + 第一token信用方法
   - 关键实现机制：利用Lean的类型理论提供验证器基础的信用信号
   - 与现有技术对比：结果级RLVR → 过程级RLVR；二元信号 → 细粒度信号
   - 优势与局限：优势是 dense 验证信号；局限是依赖特定证明助手
   - 未来发展潜力：可扩展到其他证明助手，成为形式化推理RL的标准方法

5. **开源模型生态成熟**: Hacker News上"切换到开源模型几乎没有downside"(245 points)和Apertus主权AI(406 points)反映开源模型生态成熟度里程碑。趋势：frontier模型垄断被打破，开源+主权AI成为重要方向。
