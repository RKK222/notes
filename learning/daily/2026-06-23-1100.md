### AI Testing / Software Testing (9)

1. **Probe-and-Refine Tuning: 仓库引导文件迭代优化**
   - 地址：https://arxiv.org/abs/2606.20512
   - 来源：arXiv (cs.SE)
   - 领域：Testing
   - 技术介绍：提出probe-and-refine tuning方法，使用合成bug修复探针迭代诊断和修补仓库的AGENTS.md引导文件。通过单 shot LLM调用（无需agent循环或工具使用）识别引导文件中的不足并修复。在SWE-bench Verified上，Qwen3.5-35B-A3B在200步下达到33.0%平均解决率，对比静态知识基线28.3%和无引导基线25.5%。改进来自coverage提升（多14.5pp实例生成可评估补丁）而非precision提升（~59%保持稳定）。引导使agent能更有效利用更大步预算。
   - 应用场景：编码Agent仓库引导优化、SWE-bench性能提升
   - 为什么值得关注：揭示AGENTS.md质量是决定性变量，提供无需agent循环的轻量级优化方法

2. **ToolPrivBench: LLM Agent过度特权工具选择**
   - 地址：https://arxiv.org/abs/2606.20023
   - 来源：arXiv (cs.SE/cs.AI)
   - 领域：Testing
   - 技术介绍：研究LLM Agent在工具选择中的过度特权问题——当低特权工具足够时仍选择高特权工具。构建ToolPrivBench基准，覆盖8个领域5种风险模式，评估初始选择和瞬态工具失败后的升级行为。发现主流LLM Agent普遍存在过度特权选择，且瞬态失败会放大此行为。一般安全对齐不自然迁移到最小特权工具选择，提示级控制仅在瞬态失败下有限缓解。提出特权感知后训练防御，显著减少不必要的高特权工具使用同时保持通用能力。
   - 应用场景：Agent安全评估、最小特权原则验证
   - 为什么值得关注：揭示Agent工具选择安全新维度，安全对齐≠最小特权

3. **SolidityBench: 仓库级Solidity代码生成基准**
   - 地址：https://arxiv.org/abs/2606.19988
   - 来源：arXiv (cs.SE)
   - 领域：Testing
   - 技术介绍：构建5470个仓库级Solidity智能合约配对自然语言描述的基准，提出SolidityScore领域感知语义度量（强调安全修饰符、合约声明、Solidity特定关键字）。评估Qwen2.5-Coder、DeepSeek-Coder、CodeLlama等 across zero-shot、CoT、ICL、RAG、SFT。发现通用模型在仓库级Solidity生成中存在系统性结构缺陷。RAG在非参数方法中最佳，ICL超过2例因上下文饱和退化。SFT通过内化Solidity特定约束到模型参数实现最大改进。
   - 应用场景：智能合约代码生成评估、区块链开发安全
   - 为什么值得关注：高 stakes领域基准缺失，SFT+高质量领域数据是最有效策略

## 深度研究

### GitHub 项目

#### 1. Probe-and-Refine Tuning: 仓库引导文件迭代优化
**地址**: https://arxiv.org/abs/2606.20512

**项目介绍 (200-400字)**:
Probe-and-Refine Tuning是一项针对编码Agent仓库引导文件(AGENTS.md)的系统化优化方法。传统方法中，仓库引导文件要么手动编写（质量参差不齐），要么由LLM一次性生成（缺乏迭代改进）。该方法创造性地使用合成bug修复探针作为诊断工具——通过让Agent尝试修复已知bug来暴露引导文件中的不足，然后用单 shot LLM调用修补这些问题。整个过程无需Agent循环或工具使用，极其轻量。在SWE-bench Verified上，Qwen3.5-35B-A3B在200步下从28.3%提升到33.0%解决率。关键洞察是：改进来自coverage提升（更多实例生成可评估补丁）而非precision提升（每补丁精度稳定在~59%）。这意味着更好的引导帮助Agent找到正确的文件，而非改进代码变更质量。

**技术架构**:
- 探针生成层：从仓库历史自动生成合成bug修复任务
- 诊断层：运行Agent探针，分析失败模式
- 修补层：单 shot LLM调用修复引导文件不足
- 迭代循环：重复直到收敛

**核心算法或系统设计**:
探针-修补循环：生成合成bug → 运行Agent → 分析失败 → 修补引导 → 验证改进。关键创新是用bug修复作为引导质量的诊断信号，而非直接优化Agent性能。

**关键实现机制**:
- 合成bug从仓库真实issue自动提取
- 单 shot修补避免Agent循环开销
- 覆盖度监控作为收敛信号

**与现有技术对比**:
- vs 手动AGENTS.md：系统化、可量化改进
- vs LLM一次性生成：迭代改进而非一次性
- vs Agent-in-the-loop优化：无需工具使用，成本极低

**优势与局限**:
优势：轻量级、可量化、无需训练。局限：依赖仓库有足够issue历史、对全新仓库效果有限。

**未来发展潜力**:
可扩展到多仓库引导优化、跨语言引导迁移、引导文件版本控制。

**个人技术理解**:
这项工作的核心洞察是"coverage > precision"——更好的引导让Agent找到正确位置，而非写出更好代码。这与工程直觉一致：找对文件比写好代码更难。

#### 2. ToolPrivBench: LLM Agent过度特权工具选择
**地址**: https://arxiv.org/abs/2606.20023

**项目介绍**:
ToolPrivBench是首个系统化评估LLM Agent工具选择中过度特权问题的基准。传统工具选择研究关注安全性不可知的元数据偏好，而ToolPrivBench聚焦于当低特权工具足够时Agent仍选择高特权工具的行为。基准覆盖8个领域5种风险模式，测量初始选择和瞬态工具失败后的升级行为。发现主流LLM Agent普遍存在过度特权选择，且瞬态失败会放大此行为。关键发现：一般安全对齐不自然迁移到最小特权工具选择，提示级控制仅在瞬态失败下有限缓解。提出特权感知后训练防御。

**技术架构**:
- 工具特权分层定义层
- 选择行为测量层
- 瞬态失败注入层
- 防御策略评估层

**核心算法或系统设计**:
工具特权分级模型 + 瞬态失败注入 + 后训练特权感知微调。

**关键实现机制**:
- 8领域工具特权映射
- 5种风险模式分类
- 瞬态失败注入协议

**与现有技术对比**:
- vs 通用安全评估：聚焦工具选择特权维度
- vs 提示工程防御：后训练方法更有效

**优势与局限**:
优势：首次系统化评估、多领域覆盖。局限：依赖工具特权定义完整性。

**未来发展潜力**:
可扩展到运行时特权监控、动态特权调整、企业级Agent安全策略。

**个人技术理解**:
这项研究揭示了安全对齐的一个重要盲区——"最小特权原则"在Agent工具选择中被系统性忽视。这对生产部署至关重要。

#### 1. Calibrated MoE Under Distribution Shift (ICML 2026)
**地址**: https://arxiv.org/abs/2606.20544

**论文介绍**:
这篇ICML 2026论文研究MoE模型在分布偏移下的校准行为。校准对齐模型预测不确定性与经验结果频率，对理解信任报告概率至关重要。论文发现专家级校准足以确保硬路由模型在广泛分布偏移下的校准，但对软路由模型不足。提出对抗性重加权方法惩罚路由聚合的校准误差，在平均和困难子集上改善精度-校准权衡。

**技术架构**:
- 专家校准分析层
- 路由机制交互分析层
- 对抗性重加权层
- 校准-精度权衡优化层

**核心算法或系统设计**:
对抗性重加权：在分布偏移下惩罚路由聚合的校准误差，改善精度-校准权衡。

**关键实现机制**:
- 硬路由vs软路由校准差异分析
- 对抗性重加权损失函数
- 多任务多分布验证

**与现有技术对比**:
- vs 单层校准：专家级+聚合级双层校准
- vs 静态路由：动态校准感知路由

**优势与局限**:
优势：ICML 2026录用、理论+实验双重验证。局限：主要关注分类任务、未覆盖生成模型。

**未来发展潜力**:
可扩展到生成模型校准、多模态MoE校准、在线校准适应。

**个人技术理解**:
这篇论文的核心贡献是揭示了路由机制与校准的交互——硬路由天然受益于专家校准，软路由需要额外处理。这对MoE模型部署有直接指导意义。

#### 2. RAINbow: 具身对话导航数据增强
**地址**: https://arxiv.org/abs/2606.19948

**论文介绍**:
RAINbow针对具身对话导航DialNav训练数据稀缺(仅2K episodes)问题，提出自动生成分支构建238K episodes大规模数据集。将现有VLN数据集转换为多轮对话，创建成本高效高质量训练数据。引入双策略训练对齐导航训练与动态对话-导航循环，以及利用VLN知识的定位模型。Val Seen +89%、Val Unseen +100%成功率提升，建立新SOTA。

**技术架构**:
- VLN数据集转换层
- 多轮对话生成层
- 双策略训练层
- VLN知识定位层

**核心算法或系统设计**:
自动数据生成管线 + 双策略训练 + VLN知识迁移。

**关键实现机制**:
- VLN到对话格式自动转换
- 双策略训练对齐导航-对话循环
- VLN知识注入定位模型

**与现有技术对比**:
- vs 手工数据收集：119x数据量提升
- vs 单策略训练：双策略对齐导航-对话循环

**优势与局限**:
优势：大规模数据、新SOTA。局限：依赖现有VLN数据质量。

**未来发展潜力**:
可扩展到多模态对话导航、跨场景迁移、真实机器人部署。

**个人技术理解**:
从2K到238K episodes的数据量飞跃，展示了数据工程在具身AI中的关键作用。双策略训练对齐对话-导航循环的设计很巧妙。
