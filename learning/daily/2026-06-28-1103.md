## 技术扫描（30+）

### AI Testing / Software Testing（10个）

1. **How Much Static Structure Do Code Agents Need? A Study of Deterministic Anchoring** (arXiv:2606.26979, ISSTA 2026) - 研究轻量级静态分析如何为 LLM 代码 Agent 提供确定性锚点。发现静态结构不是让 Agent "更聪明"，而是让导航"更有纪律和可重现"。调用/继承拓扑改善函数级定位（+2.2pp Func@5），缩短轨迹（-1.6 交互轮次），将链接跟随率从 0.15-0.18 提升到 0.21-0.24，运行间方差减半。对 Agent 导航可测试性有直接价值。

2. **To Run or Not to Run: Analyzing the Cost-Effectiveness of Code Execution in LLM-Based Program Repair** (arXiv:2606.26978, ISSTA 2026) - 对 SWE-bench 7,745 条 Agent 轨迹和 3,000 次端到端修复实验的分析。发现代码执行平均每次任务 8.8 次测试运行，但执行限制对修复成功率影响很小（仅 1.25 个百分点差距且不显著）。当前 Agent 不加区分地应用执行，在几乎没有收益的实例上支付成本。建议将执行视为具有明确成本效益权衡的资源。

3. **A Deterministic Control Plane for LLM Coding Agents (Rel(AI)Build)** (arXiv:2606.26924) - 对 10,008 个 GitHub 仓库的调查发现 10.1% 的 Agent 配置路径是跨仓库的精确重复（75.5% 跨越组织边界）。配置极少修订（58% 单次提交），极少声明权限边界（<1% vs Actions 的 33%）。提出 Rel(AI)Build 框架：SHA-256 内容寻址、HMAC 锁文件、哈希链审计日志、分层权限、阶段状态机、提示漂移检测。Agent 配置治理必须确定性和工具无关。

4. **Spec-Anchored, Code-Coupled, Drift-Enforced Architecture for AI-Assisted Software Development** (arXiv:2606.27045) - 提出 Spec Growth Engine 框架解决 AI 编码 Agent 的两个结构性失败模式：上下文爆炸（Agent 必须一次推理整个仓库）和静默规范-代码漂移（代码演进但规范不变）。通过机器可读规范图、Spine 上下文组装器、垂直切片增长协议和漂移门实现。

5. **Trade-offs in LLM Quantization for Automated Program Repair** (arXiv:2606.27205, ICSME 2026) - 对 13 种量化配置（不同位宽、方法、目标组件）在 APR 任务上的实证评估。发现基础模型和量化模型可以提供不同的修复问题集合，重叠很少。量化减少 85% 内存占用但增加推理时间和能耗。48% 的配置被严格支配。对 AI 编码工具部署的成本效益分析有直接价值。

6. **Are LLMs Ready for Anti-Pattern Detection in Microservice Architectures?** (arXiv:2606.26927, ICSME 2026) - 评估 LLM 在微服务架构反模式检测中的能力。LLM 在局部、异构或语义丰富的证据上表现竞争性，但在需要显式结构或跨服务依赖证据的反模式上有明显局限。LLM 尚未替代传统分析器，但已是架构评估的有希望补充工具。

7. **Evaluation-Strategy Gap in Fault Diagnosis of Deep Learning Programs** (arXiv:2606.26492, ICSME 2026) - 发现 DL 程序故障诊断技术在程序内评估和跨程序评估之间存在 0.190 的平衡准确率差距。曲率特征对未见程序的不稳定性检测有用，而优化器和激活特征仅对训练期间见过的程序有帮助。对 AI 系统测试的泛化能力评估有重要启示。

8. **An Empirical Study of LLM-Generated Specifications for VeriFast** (arXiv:2606.26490) - 评估 LLM 生成分离逻辑验证器 VeriFast 规范的能力。8 种提示方法、10 个 LLM、3 种输入类型。LLM 保留源代码和规范的功能行为（均 >91%），但验证成功率仅 31.4%。94% 的错误来自 LLM 在分离逻辑领域特定知识上的失误。

9. **Eye Tracking Insights into Reviewing LLM-Labelled Code** (arXiv:2606.26505, ASE 2026) - 眼动追踪实验发现开发者在审查标注为 LLM 生成的代码时花费更多时间注视，但审查彻底性不变。标签本身影响注意力。从业者调整审查策略，使用特定标准（如逻辑正确性）或提示来引导审查。揭示审查者意图与实际行为之间的差距。

10. **Safety-Aware Mutation Testing for Autonomous Driving Systems** (arXiv:2606.26456) - 提出 SAMT 范式：从传统组件级突变测试转向在 ADS 模块间消息中注入时间有界故障。从 STPA 安全工程框架导出突变生成规则。将系统思维嵌入突变测试管道。

## 深度研究

### GitHub 项目

#### 1. Rel(AI)Build — LLM 编码 Agent 的确定性控制平面
- **地址**: https://arxiv.org/abs/2606.26924
- **类型**: 论文（45页，9张图，13张表）
- **领域**: AI Testing / Agent 安全治理
- **技术介绍**: Rel(AI)Build 提出将 LLM 编码 Agent 的配置层（规则文件、Agent 定义、IDE 特定 Markdown）视为受管理的供应链，而非未声明的共享组件。对 10,008 个 GitHub 仓库的调查显示，10.1% 的 Agent 配置路径是跨仓库的精确重复（75.5% 跨越组织边界），配置极少修订（58% 单次提交），极少声明权限边界（<1%）。Rel(AI)Build 提供：SHA-256 内容寻址、HMAC 锁文件、哈希链审计日志、分层权限和攻击派生阻止列表、阶段状态机（需求→文件→测试可追溯性）、单一规范到 7 个 IDE 目标的编译、提示漂移检测（Jaccard 相似度）。每个机制通过注入违规的符合性测试验证。
- **应用场景**: 企业级 AI 编码 Agent 部署、开源项目 Agent 协作、合规敏感的软件开发
- **为什么值得关注**: 首次系统性诊断 Agent 配置层的治理真空。Agent 配置正在以未声明依赖的方式在组织间传播，但缺乏版本控制、权限声明和审计能力。Rel(AI)Build 将软件供应链安全的成熟实践（内容寻址、锁文件、审计日志）适配到 Agent 配置层，为 Agent 安全治理提供了可操作的基础设施。

#### 2. ConcoLixir — LLM + Concolic 混合测试
- **地址**: https://arxiv.org/abs/2606.26545
- **类型**: 论文
- **领域**: AI Testing / 软件测试
- **技术介绍**: ConcoLixir 将 LLM 定位为"发现预言机"而非求解器替代品。在 Python concolic 测试中，LLM 在三个关键时刻介入：(1) 生成初始种子 (2) 在求解器失败后提出具体输入 (3) 在覆盖停滞时针对未覆盖代码。每个候选输入通过 concolic 执行验证，仅观察覆盖率和收集的路径约束指导后续探索。在合成、真实世界和库目标上，平均行覆盖率比基线提升 8.6-17.0 个百分点。 gains 在语义屏障和库边界最强。完整评估成本仅 $1.63 API 费用。
- **应用场景**: Python 程序测试、语义屏障突破、库边界测试
- **为什么值得关注**: 解决了 LLM 在软件测试中的角色定位问题——不是替代传统方法，而是作为互补的发现引擎。$1.63 的成本展示了 LLM 辅助测试的经济可行性。对 Agent 测试基础设施的成本效益分析有直接参考价值。

#### 3. CLIR — Cranelift 编译器差分测试框架
- **地址**: https://arxiv.org/abs/2606.26977
- **类型**: 论文
- **领域**: 软件测试 / 编译器测试
- **技术介绍**: CLIR 针对 Cranelift 编译器（WebAssembly + Rust 实验后端）的差分测试框架。三大创新：(1) 语法保持的分层生成策略保证 SSA 有效性 (2) 活性引导的指令精炼机制最大化计算密度 (3) 诊断引导的跨架构适配方案促进异构后端的根因分析。72 小时内发现 24 个 bug（21 个确认，9 个修复），比 cranelift-fuzzgen 多 8 倍、比 wasm-smith 多 24 倍、比 WASMaker 多 8 倍。
- **应用场景**: 编译器测试、WebAssembly 安全、Rust 工具链验证
- **为什么值得关注**: 编译器是 AI 基础设施的关键组件。CLIR 展示了结构化测试策略如何显著超越通用模糊测试。对 AI 生成的编译器代码的测试有直接参考价值。

### 学术论文

#### 1. Knowledge-Based Pull Requests (KPR) — Agent 协作新范式
- **地址**: https://arxiv.org/abs/2606.26721
- **类型**: 论文
- **领域**: AI / 软件工程 / Agent 协作
- **技术介绍**: KPR 重新定义外部协作的 PR 流程。传统 PR 将代码作为默认合并候选，KPR 将外部协作方的代码、测试和清理后的 Agent 交互轨迹视为"知识源"。Agent 将这些源提炼为人类确认的知识包（设计备忘录、风险清单、测试计划、实现简报），然后项目拥有的内部可信编码 Agent 在接收项目环境中重新生成候选代码。关键创新：分离"知识是否应进入项目"和"特定实现是否应合并"两个决策。通过 7 个合并的公开 PR 的受控模拟验证，在描述消融、diff 消融和合成中毒补丁条件下压力测试。
- **应用场景**: 开源项目 Agent 协作、企业间代码共享、供应商驱动开发
- **为什么值得关注**: 当 AI Agent 成为软件协作的主要参与者时，传统 PR 模型面临信任边界问题。KPR 提供了跨信任边界的 Agent 协作框架，将"知识传递"与"代码合并"解耦，为 AI 时代的软件协作基础设施提供了新范式。

#### 2. To Run or Not to Run — 代码执行成本效益分析
- **地址**: https://arxiv.org/abs/2606.26978
- **类型**: 论文（ISSTA 2026）
- **领域**: AI Testing / 软件测试
- **技术介绍**: 对 SWE-bench 7,745 条 Agent 轨迹和 3,000 次端到端修复实验的深入分析。发现：(1) 代码执行在所有 Agent 和模型中使用，平均每次任务 8.8 次测试运行，频率从 2 到 19 不等，晚期执行持续比早期执行有更高的成功率 (2) 执行限制对修复成功率影响很小：商业 Agent 上受限制与无限制的解决率差距仅 1.25 个百分点且不显著，但受限制节省大量 token 和时钟成本 (3) 执行收益集中而非均匀分布。当前 Agent 不加区分地应用执行，在几乎没有收益的实例上支付成本。
- **应用场景**: AI 编码 Agent 优化、测试成本控制、Agent 执行策略
- **为什么值得关注**: 直接挑战"执行越多越好"的默认假设。揭示当前 Agent 在代码执行上的资源浪费，为 Agent 测试基础设施的成本优化提供了实证依据。对 Agent 执行策略的智能调度有直接指导价值。

#### 3. How Much Static Structure Do Code Agents Need — 确定性锚定效应
- **地址**: https://arxiv.org/abs/2606.26979
- **类型**: 论文（ISSTA 2026）
- **领域**: AI Testing / 软件工程
- **技术介绍**: 研究轻量级静态分析（调用图、继承层次、配置依赖）如何为 LLM 代码 Agent 提供确定性锚点。发现"确定性锚定效应"：静态结构不是让 Agent "更聪明"，而是让导航"更有纪律和可重现"。三个关键观察：(1) 锚定有效：轻量级调用/继承拓扑改善函数级定位（+2.2pp Func@5），缩短轨迹（-1.6 交互轮次）(2) 锚定对规模敏感：最佳粒度和方向性取决于仓库特征，密集语义显示递减回报，集线器密集型项目从反向链接受益 (3) 锚定稳定化：标签将链接跟随率从 0.15-0.18 提升到 0.21-0.24，运行间方差减半，单运行可靠性提升（Pass@1 +3.4pp），代价是约 10% 更多输入 token。
- **应用场景**: Agent 导航优化、代码仓库理解、Agent 可测试性
- **为什么值得关注**: 揭示了 Agent 导航的核心问题不是"智能"而是"纪律"。为 Agent 代码理解提供了结构化的改进方向，对 Agent 测试基础设施的可预测性和可重现性有直接价值。

## 技术趋势
1. **Agent 配置治理成为新焦点**: Rel(AI)Build 揭示 Agent 配置层存在严重的治理真空——配置以未声明依赖的方式在组织间传播，但缺乏版本控制、权限声明和审计能力。这标志着 AI 编码 Agent 从"能力竞赛"进入"治理竞赛"阶段。
2. **LLM 在测试中的角色重新定位**: ConcoLixir 和"Run or Not to Run"两篇论文共同指向一个趋势：LLM 在软件测试中不应替代传统方法，而应作为互补的发现引擎和执行优化器。$1.63 的成本展示了 LLM 辅助测试的经济可行性。
3. **跨信任边界 Agent 协作**: KPR 工作流提出将"知识传递"与"代码合并"解耦，为 AI 时代的软件协作基础设施提供了新范式。当 Agent 成为主要协作参与者时，传统 PR 模型面临信任边界问题。

## 新领域收获
1. **Rel(AI)Build 确定性控制平面**: Agent 配置供应链治理。SHA-256 内容寻址 + HMAC 锁文件 + 哈希链审计日志，将软件供应链安全实践适配到 Agent 配置层。对 Agent 安全治理有根本影响。
2. **ConcoLixir LLM+Concolic 混合测试**: LLM 作为发现预言机（非求解器替代），在语义屏障和库边界处突破。$1.63 成本展示经济可行性。
3. **KPR 知识驱动 PR 协作**: 将外部协作从"代码合并"推进到"知识传递"，分离知识进入和项目内部再生成两个决策。
